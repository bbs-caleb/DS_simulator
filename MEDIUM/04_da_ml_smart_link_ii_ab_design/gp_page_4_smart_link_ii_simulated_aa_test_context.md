```python
def aa_test(

```

Функция должна:

3. сравнивать их функцией `t_test`;
4. отмечать случаи ложной статистической значимости;
5. возвращать долю таких случаев.

# 2. Что такое A/A-тест

В обычном A/B-тесте сравниваются разные варианты:

```text
A — текущая версия;
B — новая версия.
```

В A/A-тесте обе группы получают одинаковое воздействие:

```text
A₁ — текущая версия;
A₂ — та же самая текущая версия.
```

Или, как в этой симуляции, обе выборки создаются из одного распределения с одинаковыми параметрами.

То есть мы заранее знаем:

```text
реального эффекта нет.
```

Поэтому статистически значимый результат в отдельной симуляции является ошибкой.

---

# 3. Бизнесовый смысл A/A-теста

A/A-тест проверяет не продуктовую гипотезу, а саму экспериментальную систему.

Он помогает ответить:

- корректно ли делится трафик;
- правильно ли работает статистический тест;
- не теряются ли события одной группы;
- не задваиваются ли данные;
- правильно ли считаются метрики;
- соответствует ли частота ложных побед выбранному `alpha`;
- не появляется ли систематический сдвиг между одинаковыми группами;
- достаточно ли хорошо работают симуляции дизайна.

Для OrangeChicken это особенно важно.

Если A/A-тест постоянно показывает разницу между одинаковыми Smart-Link системами, возможны проблемы:

- некорректный сплит;
- разные источники трафика;
- неодинаковая атрибуция;
- задержки конверсий;
- потери кликов;
- задвоение выплат;
- неверная агрегация CPC;
- неподходящий статистический критерий.

---

# 4. Ошибка I рода

Нулевая гипотеза A/A-теста:

```text
H0: средние CPC двух групп равны.
```

Поскольку обе группы генерируются одинаково, H0 действительно верна.

Но статистический тест работает со случайными выборками.

Иногда случайный шум создает достаточно большую разницу средних, чтобы:

```text
p-value < alpha
```

Тогда тест отвергает истинную H0.

Это называется:

```text
ошибка I рода
```

или:

```text
false positive
```

---

# 5. Аналогия с пожарной сигнализацией

Представим сигнализацию.

```text
H0: пожара нет.
```

Если сигнализация сработала, хотя пожара нет, это ложная тревога.

В A/A-тесте:

```text
реального эффекта нет;
тест объявляет эффект;
получается ложная тревога.
```

---

# 6. Что означает alpha

Часто задают:

```python
alpha = 0.05
```

Это означает, что статистическая процедура допускает примерно 5% ложных отклонений H0 при многократном повторении корректного эксперимента.

Это не значит:

```text
каждый набор из 20 тестов обязательно содержит ровно одну ошибку.
```

На маленьком числе повторений результат случайный.

Например, среди 20 симуляций может быть:

- 0 ложных побед;
- 1 ложная побед;
- 2 ложные победы;
- иногда больше.

Но при 10 000 симуляциях доля должна стабилизироваться около 0.05.

---

# 7. Что делает симуляция

Для каждой итерации:

```text
1. Генерируется CPC_A.
2. Генерируется CPC_B.
3. Параметры A и B одинаковы.
4. Выполняется t-test.
5. Если p-value < alpha, записывается ошибка I рода.
```

После всех повторений:

```text
type I error rate =
число ложных значимых результатов / число симуляций
```

---

# 8. Пример

Допустим:

```text
n_simulations = 10
```

Результаты тестов:

```text
False
False
True
False
False
False
False
True
False
False
```

Две симуляции дали ложную значимость.

Тогда:

```text
type_1_errors_rate = 2 / 10 = 0.20
```

На десяти повторениях это очень шумная оценка.

При 10 000 симуляциях она будет намного стабильнее.

---

# 9. Почему используются одинаковые параметры

Обе группы получают:

```python
n_samples
conversion_rate
reward_avg
reward_std
```

с одинаковыми значениями.

Например:

```text
A:
CVR = 10%
reward mean = 10
reward std = 2

B:
CVR = 10%
reward mean = 10
reward std = 2
```

Следовательно:

```text
E[CPC_A] = E[CPC_B]
```

Между распределениями нет заложенного продуктового эффекта.

---

# 10. Почему выборки должны генерироваться отдельно

Нельзя сделать так:

```python
cpc_a = cpc_sample(...)
cpc_b = cpc_a
```

Тогда массивы будут идентичны элемент в элемент.

P-value почти всегда будет равен 1, и реальная частота ошибки I рода не будет оценена.

Правильно дважды вызвать генератор:

```python
cpc_a = cpc_sample(...)
cpc_b = cpc_sample(...)
```

Обе выборки:

- независимы;
- имеют одинаковое распределение;
- содержат разный случайный шум.

---

# 11. Массив `type_1_errors`

В шаблоне:

```python
type_1_errors = np.zeros(n_simulations)
```

Если:

```text
n_simulations = 5
```

создается:

```python
[0., 0., 0., 0., 0.]
```

После каждой симуляции записывается:

```text
1 — тест зафиксировал ложную разницу;
0 — тест не зафиксировал разницу.
```

Пример:

```python
[0., 0., 1., 0., 1.]
```

Среднее:

```text
(0 + 0 + 1 + 0 + 1) / 5 = 0.4
```

Среднее массива из нулей и единиц равно доле единиц.

---

# 12. Почему `bool` можно записать в числовой массив

В Python:

```text
False соответствует 0;
True соответствует 1.
```

Поэтому:

```python
type_1_errors[index] = is_significant
```

автоматически превращает:

```text
False → 0.0
True  → 1.0
```

---

# 13. Ожидаемый результат

При корректном A/A-тесте:

```text
type_1_errors_rate ≈ alpha
```

Проверочная симуляция дала:

```text
alpha = 0.01 → около 0.011
alpha = 0.05 → около 0.050
alpha = 0.10 → около 0.095
```

Значения не обязаны совпадать с alpha идеально.

Симуляция сама является случайной.

---

# 14. Почему результат колеблется около alpha

Каждая симуляция похожа на Bernoulli-испытание:

```text
ошибка есть     с вероятностью примерно alpha;
ошибки нет      с вероятностью примерно 1 - alpha.
```

Число ошибок среди `n_simulations` примерно подчиняется биномиальному распределению.

Стандартная ошибка оцененной доли:

```text
sqrt(alpha × (1 - alpha) / n_simulations)
```

Для:

```text
alpha = 0.05
n_simulations = 10 000
```

получаем примерно:

```text
sqrt(0.05 × 0.95 / 10 000)
≈ 0.00218
```

То есть обычны результаты около:

```text
0.05 ± несколько тысячных
```

---

# 15. Что будет при увеличении alpha

Если увеличить:

```text
alpha: 0.05 → 0.10
```

порог становится менее строгим.

Теперь значимыми считаются все результаты:

```text
p-value < 0.10
```

а не только:

```text
p-value < 0.05
```

Поэтому ложные победы происходят чаще.

---

# 16. Что будет при увеличении n_simulations

`n_simulations` не меняет истинную ошибку I рода статистического теста.

Оно меняет точность нашей оценки.

При:

```text
100 симуляций
```

оценка может сильно колебаться.

При:

```text
10 000 симуляций
```

она обычно ближе к alpha.

При:

```text
100 000 симуляций
```

она становится еще стабильнее, но вычисления занимают больше времени.

---

# 17. Что будет при изменении n_samples

В идеальном непрерывном тесте с корректными предпосылками ошибка I рода должна оставаться около alpha.

Но CPC имеет сложное распределение:

- много нулей;
- редкие конверсии;
- возможный тяжелый хвост;
- иногда почти нулевая дисперсия.

При очень маленьком `n_samples` и низком CVR:

- в выборке может не быть конверсий;
- t-test может получить нестабильную оценку;
- p-value может быть `nan`;
- фактическая ошибка I рода может отличаться от alpha.

При достаточно больших выборках поведение обычно улучшается.

---

# 18. Распределение p-value в корректном A/A-тесте

При истинной H0 и корректно калиброванном непрерывном статистическом тесте p-value должны быть приблизительно распределены:

```text
Uniform(0, 1)
```

То есть равномерно от 0 до 1.

Это означает:

- около 10% p-value меньше 0.10;
- около 5% меньше 0.05;
- около 1% меньше 0.01;
- примерно одинаковое число значений попадает в интервалы одинаковой ширины.

Например, интервалы:

```text
0.0–0.1
0.1–0.2
...
0.9–1.0
```

должны иметь примерно одинаковую высоту на гистограмме.

---

# 19. Почему равномерное распределение дает ошибку alpha

Если p-value равномерно распределен на `[0, 1]`, то длина интервала:

```text
[0, alpha)
```

равна `alpha`.

Следовательно:

```text
P(p-value < alpha) = alpha
```

Именно поэтому примерно 5% A/A-тестов значимы при `alpha=0.05`.

---

# 20. Как читать гистограмму p-value

## Корректная картина

Столбцы примерно одинаковой высоты.

Вывод:

```text
p-values близки к Uniform(0, 1).
```

## Избыток значений около нуля

Возможные причины:

- группы на самом деле различаются;
- сплит сломан;
- стандартные ошибки занижены;
- наблюдения зависимы;
- есть утечка;
- статистический тест некорректен.

## Избыток значений около единицы

Возможные причины:

- выборки искусственно связаны;
- одна выборка скопирована из другой;
- тест чрезмерно консервативен;
- данные дискретны или вырождены;
- есть слишком много одинаковых значений.

# 21. A/A на реальных данных

На production-данных A/A может выглядеть так:

```text
50% пользователей → старая версия A1;
50% пользователей → та же старая версия A2.
```

Проверяются:

- ratio групп;
- одинаковость метрик;
- логирование;
- атрибуция;
- повторные пользователи;
- задержанные конверсии;
- выбросы;
- качество статистического пайплайна.

Один A/A-тест не доказывает идеальность системы.

Для проверки калибровки часто нужны:

- множество симуляций;
- исторические данные;
- synthetic A/A;
- repeated random splits.

---

# 22. A/A в разных индустриях

## Финтех

Можно случайно разделить текущий кредитный интерфейс на две одинаковые версии и проверить:

- application rate;
- approval rate;
- disbursement;
- technical errors.

Если одинаковые версии систематически различаются, проблема может быть в assignment или данных.

## E-commerce

A/A проверяет:

- purchase conversion;
- revenue per user;
- checkout events;
- attribution заказов;
- возвраты.

## Marketplace

Нужно отдельно проверять две стороны:

- покупателей;
- продавцов.

Обычный user-level A/A может не заметить market-level interference.

## AI-продукты

A/A для одной версии модели помогает проверить:

- routing;
- логирование ответов;
- оценки качества;
- latency;
- стоимость токенов;
- повторные запросы одного пользователя.

## iGaming

Помимо revenue проверяются:

- responsible gaming;
- fraud;
- deposits;
- withdrawals;
- self-exclusion;
- регуляторные ограничения.

---

# 23. Главный продуктовый вывод

Если доля ошибок I рода заметно превышает alpha, нельзя сразу доверять будущему A/B-тесту.

Нужно искать причины:

1. ошибка статистического метода;
2. зависимость наблюдений;
3. некорректная единица анализа;
4. сломанный сплит;
5. разные eligibility-правила;
6. ошибки логирования;
7. many-to-many join;
8. peeking;
9. множественные проверки;
10. слишком маленькая выборка для выбранного теста.

# 24. Что реализовано в решении

В основном файле находятся три функции:

```python
cpc_sample(...)
t_test(...)
aa_test(...)
```

Первые две сохранены с предыдущего шага.

`aa_test`:

1. создает массив нулей;
2. запускает `n_simulations` повторений;
3. генерирует независимые A и A выборки;
4. выполняет t-test;
5. сохраняет `True/False`;
6. берет среднее;
7. возвращает обычный Python `float`.

Дополнительная визуализация вынесена в отдельный файл, чтобы не менять API основного решения.
