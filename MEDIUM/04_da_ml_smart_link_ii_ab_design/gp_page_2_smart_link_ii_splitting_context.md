# Smart-Link II: Splitting — бизнесовый и технический контекст

## 1. Что именно требуется сделать

На этой странице уже есть конкретное программное задание.

Нужно реализовать класс `Experiment`, который распределяет каждый входящий клик по экспериментальным группам.

Минимальный API:

```python
experiment = Experiment(experiment_id=1)
group_id, group_name = experiment.group(click_id=123)
```

Метод должен возвращать:

```python
(group_id, group_name)
```

Например:

```python
(0, "A")
```

или:

```python
(1, "B")
```

Задача выглядит небольшой, но в ней проверяется важная часть экспериментальной платформы: корректное назначение трафика в варианты.

---

# 2. Бизнесовый смысл сплитования

Допустим, Smart-Link получает клики пользователей и должен выбрать конечный оффер.

Мы хотим сравнить две системы:

- группа `A` — текущий алгоритм;
- группа `B` — новый алгоритм.

Для честного сравнения трафик должен быть разделен так, чтобы:

1. группы имели запланированные размеры;
2. назначение не зависело от результата пользователя;
3. один и тот же клик при повторной обработке попадал в ту же группу;
4. разные эксперименты по-разному перемешивали трафик;
5. распределение не шло блоками вида «сначала все A, потом все B».

Если сплитование работает неправильно, дальнейшая статистика теряет смысл.

Например, если первые 50 000 кликов направить в A, а следующие 50 000 в B, то группы могут различаться по времени:

- утренний и вечерний трафик;
- будние и выходные;
- разные рекламные кампании;
- разные страны;
- разные партнеры;
- разные лимиты офферов.

Тогда разница между A и B может быть вызвана не алгоритмом, а временем поступления кликов.

---

# 3. Что означает «случайное, но детерминированное» распределение

На первый взгляд здесь есть противоречие.

Случайное означает, что клики должны быть хорошо перемешаны.

Детерминированное означает, что один и тот же ввод всегда дает один и тот же результат.

Для A/B-платформы нужны оба свойства.

## 3.1. Почему нельзя использовать обычный `random.choice`

Например:

```python
import random

group = random.choice(("A", "B"))
```

При двух вызовах для одного `click_id` может получиться:

```text
первый вызов  → A
второй вызов  → B
```

Это плохо, потому что:

- результат невозможно воспроизвести;
- повторная обработка события может изменить группу;
- разные сервисы могут назначить разные варианты;
- анализ становится ненадежным.

## 3.2. Что используется вместо этого

Берется функция:

```text
group = f(experiment_id, click_id)
```

Для одинаковых аргументов она всегда возвращает одинаковую группу.

Но хеш-функция перемешивает последовательные идентификаторы так, что результат внешне похож на случайный.

---

# 4. Зачем нужна соль

Представим два эксперимента.

## Эксперимент 1

```text
experiment_id = 1
```

## Эксперимент 2

```text
experiment_id = 2
```

Если распределять клики только по `click_id`, то один и тот же клик будет попадать в одинаковую группу во всех экспериментах.

Например:

```text
click 100 → A в эксперименте 1
click 100 → A в эксперименте 2
click 100 → A в эксперименте 3
```

Это означает, что группы между экспериментами связаны.

Чтобы заново перемешивать трафик, к идентификатору добавляют соль.

В решении соль определяется через `experiment_id`:

```python
self.salt = sha256(
    f"experiment:{self.experiment_id}".encode("utf-8")
).hexdigest()
```

Для разных экспериментов получаются разные соли.

Затем хешируется уже сочетание:

```text
salt + click_id
```

Поэтому один клик может получить:

```text
эксперимент 1 → A
эксперимент 2 → B
эксперимент 3 → A
```

---

# 5. Почему соль должна быть детерминированной

Плохой вариант:

```python
self.salt = generate_random_string()
```

Если при каждом создании `Experiment(1)` соль генерируется заново, то один и тот же эксперимент начинает по-разному распределять клики.

Получится:

```text
Experiment(1), click 100 → A
Experiment(1), click 100 → B
```

Поэтому соль должна одновременно быть:

- уникальной для разных `experiment_id`;
- одинаковой для одного `experiment_id`;
- воспроизводимой после перезапуска программы.

Хеш от `experiment_id` удовлетворяет этим условиям.

---

# 6. Единица рандомизации в этой задаче

В тексте особо указано:

> Мы разделяем трафик, то есть клики, а не пользователей.

Поэтому методом принимается:

```python
click_id
```

Один пользователь с несколькими кликами теоретически может попасть в разные группы.

Например:

```text
user 10, click 1001 → A
user 10, click 1002 → B
```

Для учебной задачи это ожидаемое поведение.

В реальном продукте нужно заранее решить, допустимо ли такое смешение.

## Когда click-level splitting уместен

- клик является практически независимым событием;
- пользователи не имеют стабильного идентификатора;
- эффект краткосрочный;
- важна маршрутизация каждого запроса;
- пользователь не замечает смены варианта.

## Когда лучше user-level splitting

- один пользователь возвращается много раз;
- предыдущий опыт влияет на следующий;
- важна консистентность интерфейса;
- измеряется retention;
- treatment может обучать пользователя;
- повторные события сильно зависимы.

# 7. Требования задания и соответствие решения

## Требование 1. По умолчанию две группы

В конструкторе:

```python
groups=("A", "B")
```

Если веса не заданы, они становятся:

```python
[0.5, 0.5]
```

## Требование 2. Произвольные названия и число групп

Пример:

```python
Experiment(
    experiment_id=1,
    groups=("control", "model_1", "model_2"),
)
```

Без явных весов каждая группа получает:

```text
1 / 3
```

## Требование 3. Неравномерные веса

Пример:

```python
Experiment(
    experiment_id=1,
    groups=("A", "B", "C"),
    group_weights=[0.90, 0.05, 0.05],
)
```

Ожидаемое распределение:

```text
A ≈ 90%
B ≈ 5%
C ≈ 5%
```

## Требование 4. Веса неотрицательны

Запрещено:

```python
[0.8, 0.3, -0.1]
```

## Требование 5. Сумма весов равна 1

Запрещено:

```python
[0.7, 0.2]
```

Потому что сумма равна `0.9`.

## Требование 6. Число весов совпадает с числом групп

Запрещено:

```python
groups=("A", "B", "C")
group_weights=[0.5, 0.5]
```

## Требование 7. Детерминированность

Для одинаковых параметров:

```python
experiment.group(123)
```

всегда дает одинаковый ответ.

## Требование 8. Разные эксперименты перемешивают клики по-разному

В состав хешируемого ключа входит соль, зависящая от `experiment_id`.

# 8. Почему используется SHA-256

В Python есть встроенная функция:

```python
hash(...)
```

Но для этой задачи она менее удобна.

Python может использовать рандомизацию встроенного хеша между разными запусками процесса.

Это означает, что значение:

```python
hash("some text")
```

не обязательно будет одинаковым после перезапуска интерпретатора.

Для A/B-назначения нужна межпроцессная воспроизводимость.

Поэтому используется криптографическая хеш-функция из стандартной библиотеки:

```python
from hashlib import sha256
```

SHA-256:

- детерминирована;
- хорошо перемешивает входы;
- доступна без дополнительных библиотек;
- стабильно работает между запусками;
- дает большой диапазон значений.

Криптографическая стойкость здесь не является основной целью. Важны стабильность и равномерность.

---

# 9. Как хеш превращается в число от 0 до 1

Сначала строится ключ:

```python
click_key = f"{self.salt}:{click_id}".encode("utf-8")
```

Затем вычисляется бинарный хеш:

```python
digest = sha256(click_key).digest()
```

Берутся первые 8 байт:

```python
digest[:8]
```

Восемь байт содержат 64 бита.

Их можно представить целым числом от:

```text
0
```

до:

```text
2^64 - 1
```

Преобразование:

```python
hash_value = int.from_bytes(digest[:8], byteorder="big")
```

Нормализация:

```python
random_value = hash_value / 2**64
```

Получается число:

```text
0 <= random_value < 1
```

Например:

```text
0.124830...
0.762921...
0.503114...
```

Оно выглядит случайным, но полностью определяется парой:

```text
experiment_id + click_id
```

# 10. Как работают веса

Допустим:

```python
groups = ("A", "B", "C")
group_weights = [0.90, 0.05, 0.05]
```

Строятся накопленные границы:

```text
A: 0.00–0.90
B: 0.90–0.95
C: 0.95–1.00
```

В коде:

```text
[0.90, 0.95, 1.00]
```

Если `random_value` равен:

```text
0.40 → A
0.92 → B
0.98 → C
```

Именно поэтому доля диапазона каждой группы совпадает с ее весом.

---

# 11. Зачем используется `bisect_right`

После построения накопленных весов нужно найти первую подходящую границу.

```python
group_id = bisect_right(
    self._cumulative_weights,
    random_value,
)
```

Пример:

```text
границы = [0.90, 0.95, 1.00]
значение = 0.92
```

`0.92` находится после `0.90`, но до `0.95`.

Поэтому индекс равен:

```text
1
```

Группа с индексом 1 — `B`.

`bisect_right` также корректно пропускает группы с нулевым весом.

Например:

```python
group_weights=[0.0, 1.0]
```

Все клики попадут в группу с индексом `1`.

---

# 12. Почему нельзя требовать идеальные 50/50 на маленькой выборке

Случайное распределение не означает, что на первых десяти кликах обязательно будет:

```text
5 A и 5 B
```

Может получиться:

```text
6 A и 4 B
```

или:

```text
7 A и 3 B
```

Это нормальный случайный шум.

Чем больше кликов, тем ближе доли обычно становятся к заданным весам.

Для 100 000 кликов проверка решения дала примерно:

```text
A = 50.15%
B = 49.85%
```

Это очень близко к 50/50.

Для весов:

```text
0.90 / 0.05 / 0.05
```

на 200 000 кликов было получено примерно:

```text
A = 90.02%
B = 5.00%
C = 4.98%
```

# 13. Бизнес-примеры весов

## 13.1. Безопасный rollout

```text
A — старая версия: 95%
B — новая версия: 5%
```

Используется, если новая система рискованна.

## 13.2. Несколько treatment-вариантов

```text
A — контроль: 50%
B — модель 1: 25%
C — модель 2: 25%
```

## 13.3. Один большой контроль и два малых теста

```text
A — контроль: 80%
B — новая логика: 10%
C — новая логика + новый интерфейс: 10%
```

## 13.4. Canary

```text
A — production: 99%
B — canary: 1%
```

Canary не всегда является полноценным A/B-тестом. Его первая цель может быть технической: проверить ошибки, latency и стабильность.

# 14. Применение в разных индустриях

## Финтех

Сплит по клиенту:

- новый экран заявки;
- кредитный оффер;
- cashback;
- уведомление.

Нельзя бездумно сплитовать каждую заявку, если один клиент может многократно подавать заявки и менять варианты.

## E-commerce

Сплит по пользователю или сессии:

- ranking;
- рекомендации;
- checkout;
- промокоды.

## Marketplace

Сплит может проводиться по:

- покупателю;
- продавцу;
- географии;
- магазину;
- категории.

Нужно учитывать взаимодействие сторон.

## Ритейл

Эксперимент может делить магазины:

```text
магазины A → текущая выкладка
магазины B → новая выкладка
```

Тогда хешируется `store_id`, а не `click_id`.

## Dating

Лучше сохранять пользователя в одном варианте рекомендаций, иначе его опыт может постоянно меняться.

## AI-продукт

Обычно используется user-level assignment:

```text
A → текущая модель
B → новая модель
```

Это помогает сохранять консистентность и измерять долгосрочное использование.

## Реклама и Smart-Link

Click-level assignment может быть допустим, если каждый переход — отдельная возможность маршрутизации.

---

# 15. Что не делает это учебное решение

Решение сознательно минимальное.

Оно не реализует:

- хранение конфигурации экспериментов в базе;
- дату начала и окончания;
- eligibility;
- exposure logging;
- mutually exclusive layers;
- пересечение экспериментов;
- SRM-мониторинг;
- удаленное управление весами;
- rollout;
- rollback;
- cluster randomization;
- sticky assignment для пользователей.

Это нормально: задание требует только класса распределения кликов.

---

# 16. Итог

Реализованный алгоритм:

1. получает `experiment_id`;
2. детерминированно создает соль;
3. проверяет группы и веса;
4. строит накопленные интервалы;
5. соединяет соль и `click_id`;
6. вычисляет SHA-256;
7. превращает часть хеша в число `[0, 1)`;
8. определяет группу по интервалам;
9. возвращает индекс и название группы.

Такое решение:

- соответствует требуемому API;
- не использует сторонние библиотеки;
- детерминировано;
- поддерживает произвольное число групп;
- поддерживает неравномерные веса;
- хорошо перемешивает последовательные клики;
- дает разный сплит для разных экспериментов.
