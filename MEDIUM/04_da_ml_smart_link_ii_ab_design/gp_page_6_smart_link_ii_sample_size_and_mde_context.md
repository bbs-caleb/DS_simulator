# Smart-Link II: Sample size and MDE — бизнесовый и технический контекст

## 1. Что требуется реализовать

На этой странице нужно собрать в одном Python-файле функции предыдущих шагов:

```python
cpc_sample(...)
t_test(...)
aa_test(...)
ab_test(...)
```

и добавить две новые функции:

```python
select_sample_size(...)
select_mde(...)
```

`select_sample_size` получает сетку размеров групп. Для каждого размера функция оценивает ошибку I рода через A/A-симуляцию и ошибку II рода через A/B-симуляцию. Она возвращает первый размер, для которого одновременно выполнены условия:

```text
type_1_error <= alpha
type_2_error <= beta
```

`select_mde` получает фиксированный размер группы и сетку MDE. Для каждого MDE функция оценивает ошибку II рода и возвращает первый эффект, для которого:

```text
type_2_error <= beta
```

Это уже не просто статистическое упражнение. Это полноценный этап дизайна будущего эксперимента.

---

## 2. Бизнесовая проблема

До запуска реального A/B-теста OrangeChicken должна ответить на два вопроса.

### Вопрос 1. Сколько кликов требуется в каждой группе?

Если взять слишком мало наблюдений:

- полезный Smart-Link может не показать статистически значимый эффект;
- ошибка II рода будет высокой;
- команда может отказаться от действительно прибыльного алгоритма;
- эксперимент потратит время, но не даст надежного решения.

Если взять слишком много:

- тест будет длиться дольше;
- больше трафика останется на старой версии;
- полезное изменение позже выйдет на 100% пользователей;
- компания сможет провести меньше экспериментов за год;
- возрастут стоимость и сложность сопровождения.

### Вопрос 2. Какой минимальный эффект можно обнаружить при доступном трафике?

Иногда команда не может свободно выбрать размер выборки. Например, за две недели доступно только 20 000 eligible-кликов на группу. Тогда вопрос формулируется наоборот:

> Какой минимальный относительный рост CVR можно надежно обнаружить при `n_samples=20_000`?

Это и решает `select_mde`.

## 3. Sample size и MDE — две стороны одного дизайна

Если бизнес заранее определил минимально полезный эффект, подбирается размер выборки.

Пример:

```text
Рост CPC меньше 5% не окупает внедрение.
```

Тогда аналитик фиксирует:

```text
MDE = 0.05
```

и ищет минимальный `n_samples`, при котором мощность достаточна.

Если размер трафика фиксирован, подбирается MDE.

Пример:

```text
За допустимый срок можно набрать 10 000 кликов на группу.
```

Тогда аналитик ищет минимальный эффект, который такой дизайн способен обнаруживать.

---

## 4. Напоминание об ошибках

### Ошибка I рода

Реального эффекта нет, но статистический тест объявляет различие.

```text
False positive
```

Допустимый уровень задается параметром:

```text
alpha
```

Часто используется:

```text
alpha = 0.05
```

### Ошибка II рода

Реальный эффект есть, но статистический тест его не обнаруживает.

```text
False negative
```

Обозначается:

```text
beta
```

Часто используется:

```text
beta = 0.20
```

### Мощность

```text
Power = 1 - beta
```

Если `beta=0.20`, то мощность равна 80%.

Это означает: при эффекте, равном заданному MDE, примерно 80% повторений такого дизайна обнаружат различие.

---

## 5. Почему для размера выборки проверяются обе ошибки

Можно было бы проверять только ошибку II рода. Но это опасно.

Представим плохой статистический тест, который объявляет значимость почти всегда. Он будет редко пропускать эффект, то есть покажет высокую мощность. Но одновременно он будет давать огромное число ложных побед в A/A-тестах.

Поэтому хороший дизайн должен одновременно:

- контролировать ложноположительные решения;
- иметь достаточную вероятность обнаружения реального эффекта.

Условие:

```python
if type_1_error <= alpha and type_2_error <= beta:
```

проверяет обе стороны качества теста.

---

## 6. Почему `select_mde` проверяет только ошибку II рода

MDE не используется в A/A-симуляции. При фиксированных:

- размере выборки;
- метрике;
- статистическом тесте;
- уровне `alpha`;

изменение MDE влияет только на A/B-симуляцию.

Поэтому для каждого MDE пересчитывается только `type_2_error`.

Калибровку ошибки I рода нужно проверять отдельно для выбранного размера и статистического метода.

---

## 7. Почему сетку нужно задавать от меньшего к большему

Пример сетки размеров:

```python
n_samples_grid = [500, 1000, 2000, 5000, 10000]
```

Пример сетки MDE:

```python
mde_grid = [0.01, 0.02, 0.05, 0.10, 0.20]
```

Функции останавливаются на первом подходящем значении. Если сетка отсортирована, первый подходящий элемент является минимальным среди проверенных.

Внутри функции сетка специально не сортируется. Порядок является частью входного дизайна и влияет на порядок случайных вызовов.

## 8. Пример выбора sample size

Предположим, симуляции дали:

| Размер одной группы | Ошибка I рода | Ошибка II рода |
|---:|---:|---:|
| 500 | 0.048 | 0.70 |
| 1 000 | 0.052 | 0.46 |
| 2 000 | 0.047 | 0.19 |
| 5 000 | 0.050 | 0.05 |

Параметры дизайна:

```text
alpha = 0.05
beta = 0.20
```

Проверка:

- 500 не подходит: ошибка II рода слишком высокая;
- 1 000 не подходит: обе оценки выше ограничений;
- 2 000 подходит: `0.047 <= 0.05` и `0.19 <= 0.20`;
- 5 000 уже не нужно проверять.

Возвращается:

```python
(2000, 0.047, 0.19)
```

---

## 9. Пример выбора MDE

Пусть размер группы фиксирован. Симуляции дали:

| MDE | Ошибка II рода | Мощность |
|---:|---:|---:|
| 0.01 | 0.95 | 0.05 |
| 0.05 | 0.72 | 0.28 |
| 0.10 | 0.34 | 0.66 |
| 0.20 | 0.18 | 0.82 |
| 0.30 | 0.04 | 0.96 |

При `beta=0.20` первым подходящим значением будет:

```text
MDE = 0.20
```

Возвращается:

```python
(0.20, 0.18)
```

---

## 10. Почему используется симуляционный подход

Для простой бинарной метрики sample size можно вычислять аналитической формулой. Но CPC в этой задаче имеет сложную структуру:

- большая доля значений равна нулю;
- число конверсий случайно;
- reward имеет отдельный разброс;
- CVR и reward вместе определяют среднее и дисперсию CPC;
- при изменении CVR меняется не только среднее, но и форма распределения.

Симуляция позволяет:

1. воспроизвести процесс генерации данных;
2. применить тот же статистический тест, который будет использоваться в анализе;
3. напрямую оценить частоты ошибок;
4. не выводить вручную сложную формулу мощности.

## 11. Monte Carlo noise

Результаты симуляций сами случайны.

Даже если настоящий уровень ошибки I рода равен 5%, оценка по конечному числу симуляций может быть:

```text
0.044
0.051
0.058
```

То же относится к ошибке II рода.

Поэтому оценки могут быть не идеально монотонными. Например, для 2 000 наблюдений β иногда случайно получится чуть выше, чем для 1 500.

Чем больше `n_simulations`, тем стабильнее оценка, но тем дольше расчет.

В учебной задаче ограничения применяются буквально:

```python
type_1_error <= alpha
type_2_error <= beta
```

В промышленном дизайне можно дополнительно использовать доверительные интервалы для Monte Carlo-оценок и фиксированные seeds.

## 12. Почему нельзя использовать `np.random.default_rng`

В условии это отдельно запрещено.

Решение использует:

```python
np.random.binomial(...)
np.random.normal(...)
```

Это сохраняет совместимость с:

```python
np.random.seed(...)
```

и с ожидаемой последовательностью случайных чисел в автотестах.

Внутри функций seed не устанавливается. Управлять воспроизводимостью должен внешний код.

---

## 13. Зачем нужен `RuntimeError`

Если ни одно значение сетки не подходит, нельзя возвращать последнее значение как будто оно валидно.

Функция должна явно сообщить:

```text
сетка недостаточно широкая.
```

Для sample size нужно добавить более крупные размеры.

Для MDE нужно добавить более крупные эффекты или пересмотреть размер выборки.

---

## 14. Перевод sample size в длительность

Допустим, функция выбрала:

```text
5 000 кликов на группу.
```

При сплите 50/50 всего нужно:

```text
10 000 eligible-кликов.
```

Если приходит 2 000 eligible-кликов в день, механически требуется около пяти дней.

Но реальная длительность может быть больше из-за:

- необходимости покрыть полную неделю;
- сезонности по дням;
- задержанных конверсий;
- maturation window;
- ramp-up;
- фильтрации ботов;
- технических потерь.

Sample size и календарная длительность связаны, но не являются одним и тем же.

---

## 15. Как выбирать MDE бизнесово

MDE нельзя выбирать только по принципу «на это хватает трафика».

Нужно оценить:

```text
дополнительный CPC
× объем кликов
× ожидаемый срок жизни решения
- стоимость разработки
- стоимость инфраструктуры
- риски
- ухудшение guardrail-метрик.
```

Очень маленький эффект может быть статистически обнаружим, но экономически бессмыслен.

Наоборот, 1% на очень большом масштабе может приносить значительные деньги.

## 16. Применение в других индустриях

### Финтех

Подбирают размер для конверсии заявки, активации карты, выдачи кредита, repayment и fraud-метрик. Нельзя оптимизировать только выдачи без учета default и fraud.

### E-commerce

Используют conversion rate, revenue per visitor, gross profit per user и repeat purchase. Денежные метрики обычно шумнее бинарной конверсии.

### Marketplace

Нужно учитывать обе стороны рынка, общий inventory, seller exposure и interference. Простая независимая симуляция кликов может переоценить мощность.

### AI-продукты

Единицей может быть запрос, сессия, пользователь или организация. Если один пользователь делает много зависимых запросов, нельзя считать их полностью независимыми.

### Dating

MDE по meaningful conversations должен рассматриваться вместе с safety, reports, blocks и fairness of exposure.

### iGaming

Мощность не заменяет responsible-gaming ограничения, fraud-контроль и compliance.

## 17. Ограничения учебной модели

Симуляция предполагает:

- независимые клики;
- одинаковые размеры групп;
- фиксированный CVR;
- нормальный reward;
- отсутствие сезонности;
- отсутствие кластеризации;
- отсутствие interference;
- отсутствие peeking;
- корректный t-test.

В реальной системе могут потребоваться bootstrap, CUPED, кластерные стандартные ошибки, стратификация, sequential testing и моделирование delayed conversions.

## 18. Главный вывод

Если минимально полезный эффект известен:

```text
подбираем sample size.
```

Если доступный трафик фиксирован:

```text
подбираем обнаружимый MDE.
```

Хороший дизайн одновременно контролирует:

- ошибку I рода;
- ошибку II рода;
- мощность;
- длительность;
- стоимость;
- практическую ценность эффекта.
