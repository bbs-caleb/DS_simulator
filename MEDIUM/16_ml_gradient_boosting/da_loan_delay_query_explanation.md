# Объяснение запроса `query.sql` — выгрузка датасета для Gradient Boosting

## 1. Бизнесовый контекст

SigmaBank хочет прогнозировать **просрочку выплаты кредита** ещё до того, как она случится. Для этого нужен исторический датасет: по уже закрытым кредитам мы знаем, кто и на сколько дней просрочил платёж (или не просрочил вообще). На этом датасете позже будет обучаться градиентный бустинг.

Чтобы модель могла чему-то научиться, нам нужны:
- **признаки (features)** — характеристики клиента и кредита, которые были известны *на момент выдачи* кредита (возраст, доход, кредитный рейтинг и т.д.);
- **целевая переменная (target)** — `delay_days`, то есть то, что мы хотим предсказывать.

Сырые данные лежат в ClickHouse в таблице `default.loan_delay_days`. Наша задача — SQL-запросом достать из неё ровно те поля, которые нужны, и превратить две даты (`loan_deadline`, `loan_payed`) в понятные модели числа (срок кредита в днях и просрочку в днях).

## 2. Технический контекст

ClickHouse — колоночная СУБД. Для работы с датами в ней есть функция `dateDiff('unit', start, end)`, которая считает разницу между двумя датами в нужных единицах (у нас — дни). Важный нюанс: `dateDiff('day', a, b)` считает `b - a`, то есть **первый аргумент — это дата, из которой вычитают, второй — дата, до которой считают**. Порядок аргументов важен, перепутав их местами, получим отрицательные числа.

Также используется конструкция `CASE WHEN ... THEN ... ELSE ... END` — это аналог `if/else` в SQL, позволяющий вернуть разные значения в зависимости от условия.

## 3. Разбор запроса построчно

```sql
select 
        age,
        income,
        dependents,
        has_property,
        has_car,
        credit_score,
        job_tenure,
        has_education,
        loan_amount,
        dateDiff('day', loan_start, loan_deadline) as loan_period,
        case 
            when loan_payed <= loan_deadline then 0 
            else dateDiff('day', loan_deadline, loan_payed) 
        end as delay_days
from default.loan_delay_days
```

### Шаг 1 — простые поля

```sql
age, income, dependents, has_property, has_car,
credit_score, job_tenure, has_education, loan_amount
```

Эти 9 колонок просто перечисляются как есть — они уже хранятся в таблице в нужном виде и не требуют вычислений. Мы их берём один в один. Обратите внимание, что `id` в выгрузку **не** включён — по условию задачи он не нужен, а `loan_start`, `loan_deadline`, `loan_payed` не включены как отдельные колонки, потому что вместо них мы выводим *производные* от них величины (см. ниже).

### Шаг 2 — расчёт `loan_period`

```sql
dateDiff('day', loan_start, loan_deadline) as loan_period
```

Здесь мы считаем **плановый срок кредита в днях**: сколько дней должно было пройти от даты выдачи кредита (`loan_start`) до плановой даты полного погашения (`loan_deadline`).

Разбор по шагам:
1. `dateDiff` — функция ClickHouse для разницы дат.
2. `'day'` — единица измерения результата (дни, а не месяцы или часы).
3. `loan_start` — точка отсчёта (дата выдачи).
4. `loan_deadline` — конечная точка (плановая дата погашения).
5. Результат = `loan_deadline − loan_start`, в днях.
6. `as loan_period` — переименовываем результат вычисления в колонку `loan_period`, как требует условие задачи.

Пример: кредит выдан 1 января, дедлайн — 1 марта → `loan_period` = 59 (или 60, в зависимости от года).

### Шаг 3 — расчёт `delay_days`

```sql
case 
    when loan_payed <= loan_deadline then 0 
    else dateDiff('day', loan_deadline, loan_payed) 
end as delay_days
```

Это ключевая часть — целевая переменная. Логика следующая:

- **Если клиент заплатил вовремя или раньше срока** (`loan_payed <= loan_deadline`) — просрочки нет, значит `delay_days = 0`. Это прямо прописано в условии задачи ("если клиент оплатил кредит раньше дедлайна, просрочка = 0").
- **Иначе** (клиент заплатил позже дедлайна) — считаем, на сколько дней он опоздал: `dateDiff('day', loan_deadline, loan_payed)`, то есть `loan_payed − loan_deadline`.

Построчный разбор `CASE`:
1. `case` — начало условной конструкции.
2. `when loan_payed <= loan_deadline then 0` — первое условие: "если дата фактической оплаты меньше или равна плановой дате — верни 0".
3. `else dateDiff('day', loan_deadline, loan_payed)` — если условие выше не выполнилось (то есть клиент просрочил), вычисляем разницу в днях между дедлайном и фактической датой оплаты.
4. `end` — закрывает конструкцию `case`.
5. `as delay_days` — называем результат `delay_days`, как требует задача.

Важно, что здесь **аргументы `dateDiff` идут в обратном порядке** по сравнению с `loan_period`: сначала `loan_deadline`, потом `loan_payed` — потому что просрочка считается как "факт минус план", а не наоборот. Если перепутать порядок, получим отрицательные числа просрочки, что грейдер не пропустит.

### Шаг 4 — источник данных

```sql
from default.loan_delay_days
```

Просто указываем таблицу, из которой берём данные: `loan_delay_days` в базе (схеме) `default`.

## 4. ⚠️ Важное замечание — в текущем файле не хватает сортировки

В условии задачи явно сказано: **"Отсортировать по id"**. В вашем `query.sql` сейчас **нет `ORDER BY id`** — запрос заканчивается на `from default.loan_delay_days` без сортировки. Без ORDER BY ClickHouse не гарантирует порядок строк в результате (может отдавать их в любом, даже разном при повторных запусках, порядке), а грейдер, скорее всего, сравнивает построчно — значит без сортировки задача не даст 100 баллов.

Исправленная версия (добавлена только одна строка в конец):

```sql
select 
        age,
        income,
        dependents,
        has_property,
        has_car,
        credit_score,
        job_tenure,
        has_education,
        loan_amount,
        dateDiff('day', loan_start, loan_deadline) as loan_period,
        case 
            when loan_payed <= loan_deadline then 0 
            else dateDiff('day', loan_deadline, loan_payed) 
        end as delay_days
from default.loan_delay_days
order by id
```

Обратите внимание: `id` можно использовать в `ORDER BY`, даже не выводя его в `SELECT` — ClickHouse (как и большинство SQL-диалектов) это позволяет, потому что сортировка применяется к исходной таблице, а не к уже сформированному результату выборки.

## 5. Итог: что получится на выходе

| Колонка | Смысл |
|---|---|
| age, income, dependents, has_property, has_car, credit_score, job_tenure, has_education, loan_amount | признаки клиента и кредита "как есть" |
| loan_period | плановый срок кредита в днях (loan_deadline − loan_start) |
| delay_days | целевая переменная: 0, если оплачено вовремя/раньше; иначе фактическая просрочка в днях |

Отсортировано по `id` (после исправления) — датасет готов к следующему шагу: реализации градиентного бустинга над решающими деревьями.
