### 1.1. Сравниваем «ожидалось» и «получилось» посимвольно

Вот ключевой фрагмент из ошибки (я сократил тексты):

```
Ожидалось:  '"I just received my shipment... We love "slickers"... monitors."'
Получилось:  'I just received my shipment... We love "slickers"... monitors.'
```

Разница ровно одна: **в ожидаемом тексте есть кавычки в начале и в конце**, в нашем — нет.

Смотрим на все три текста из ошибки:

| Отзыв | Особенности содержимого | Ожидание грейдера |
|---|---|---|
| №1 «This was a really good idea...» | нет запятых, кавычек, переводов строк | без обрамляющих кавычек |
| №2 «I just received...» | есть запятые и кавычки (`"slickers"`) | **с** обрамляющими кавычками |
| №3 «When chatchis...» | есть переводы строк (абзацы) | **с** обрамляющими кавычками |

Закономерность очевидна: кавычки ожидаются ровно у тех отзывов, которые **в csv-файле были
взяты в кавычки**.

### 1.2. Почему так: два уровня кавычек в CSV

Здесь нужно понять, как устроен формат CSV изнутри.

Правило записи CSV (так называемый режим QUOTE_MINIMAL — «кавычки по минимуму»):

> Если значение поля содержит запятую, кавычку или перевод строки — всё поле оборачивается
> в двойные кавычки. А каждая кавычка **внутри** значения удваивается: `"` → `""`.

Пример. Отзыв в базе: `We love "slickers", great!`. В csv-файле он будет записан так:

```
"We love ""slickers"", great!"
```

Теперь два способа это прочитать:

**Способ А — стандартный парсер (`csv.reader`), как делали мы:**
снимает обрамляющие кавычки, превращает `""` обратно в `"`:

```
We love "slickers", great!        ← «чистое» значение
```

**Способ Б — сырое чтение (как читает эталон грейдера):**
склеивает многострочные записи, режет по запятым, схлопывает `""` → `"`, но **обрамляющие
кавычки оставляет как есть**:

```
"We love "slickers", great!"      ← значение с кавычками по краям
```

Мы читали способом А. Грейдер ожидает способ Б. Оба «работают», но грейдер сравнивает
результат **посимвольно** с эталоном — значит, наша задача не «прочитать правильно по
стандарту CSV», а **воспроизвести поведение эталона в точности**.

Это важный практический урок: *когда тебя проверяет автоматический грейдер, «правильно» —
это «так же, как эталон», а не «так, как по учебнику»*. То же самое в продакшене: если твой
компонент встраивается в существующий пайплайн, он обязан быть побайтово совместим с тем,
что ожидают соседние компоненты.

### 1.3. Косвенное подтверждение в токенах

В ошибке `test_batch_tokenized` видно то же самое на уровне чисел. Ожидаемые токены второго
отзыва начинаются так:

```
[101, 1000, 1045, 2074, 2363, ...]
```

`101` — это `[CLS]`, а `1000` — это индекс токена `"` (кавычки!) в словаре DistilBERT.
У нас же: `[101, 1045, 2074, ...]` — кавычки нет. То есть эталон **токенизирует кавычку как
часть текста**. Ещё одно доказательство, что кавычки должны оставаться в тексте отзыва.

## ЧАСТЬ 2. Новое решение: сырой построчный парсер

### 2.1. Идея

Отказываемся от `csv.reader` и читаем файл так, как это делает эталон:

1. Читаем файл **построчно** (физическими строками).
2. **Склеиваем** строки в логические записи: запись закончена, когда количество кавычек в ней
   чётное (об этом ниже).
3. Режем запись по запятым, но **только по первым четырём** — всё, что после четвёртой
   запятой, целиком является текстом отзыва (он последняя колонка!).
4. В тексте отзыва схлопываем `""` → `"`, а обрамляющие кавычки **не трогаем**.

### 2.2. Почему «чётное число кавычек = запись закончена»

Это красивый инвариант правильно записанного CSV. Посчитаем кавычки:

- Поле без кавычек → вносит 0 кавычек (чётно).
- Поле в кавычках → 2 обрамляющие + внутренние всегда парами (`""`) → чётно.

Значит, **любая завершённая запись содержит чётное число кавычек**.

А если перевод строки случился **внутри** закавыченного поля (многострочный отзыв)? Тогда на
момент конца физической строки мы прочитали открывающую кавычку (1 штука, нечёт) плюс,
возможно, сколько-то пар `""` (чётно) — итого **нечётное** число. Признак: запись не окончена,
надо приклеить следующую строку.

Пример с отзывом про chatchi:

```
Физическая строка 1: 3,...,negative,"When chatchis treat vanished, chatchi panicked.   ← 1 кавычка (нечёт) → клеим дальше
Физическая строка 2: (пустая)                                                          ← всё ещё 1 (нечёт) → клеим
Физическая строка 3: Fortunately, Amazon carries this candy."                          ← стало 2 (чёт) → запись готова!
```

### 2.3. Почему `split(",", 4)` безопасен

Метод `record.split(",", 4)` режет строку по запятым, но делает **максимум 4 разреза** —
получается ровно 5 кусков. Всё, что после четвёртой запятой (включая любые запятые внутри
отзыва!), попадает в пятый кусок целиком.

Это безопасно благодаря структуре нашей выгрузки с шага 2:

```
review_id , dt , rating , sentiment , review
   число    дата   число     слово     ← текст (может содержать что угодно)
```

Первые четыре колонки **гарантированно не содержат запятых**: число, дата в формате
`YYYY-MM-DD HH:MM:SS`, число, одно слово. Значит, первые четыре запятые в записи — это всегда
разделители колонок. (Вот и ещё один смысл подсказки №1 про порядок колонок: `review`
специально поставлен последним, чтобы его можно было отрезать таким способом.)

### 2.4. Почему `.replace('""', '"')`, но обрамляющие кавычки не трогаем

В файле внутренние кавычки задвоены (`""slickers""`), а эталон ожидает одинарные
(`"slickers"`) — значит, задвоение надо схлопнуть. А обрамляющие кавычки эталон ожидает на
месте — значит, их не снимаем. Ровно это и делает связка «сырой парсинг + `replace`».

## ЧАСТЬ 3. Итоговый код

```python
"""Batch data loader with tokenization for sentiment analysis."""

from dataclasses import dataclass
from itertools import islice
from typing import Generator, Iterator, List, TextIO, Tuple

from transformers import PreTrainedTokenizer

LABELS = {"negative": -1, "neutral": 0, "positive": 1}


def iter_records(file: TextIO) -> Iterator[str]:
    """Yield logical csv records, merging multiline quoted fields"""
    record = ""
    for line in file:
        record += line
        if record.count('"') % 2 == 0:  # record complete when quotes balanced
            yield record.rstrip("\n")
            record = ""


@dataclass
class DataLoader:
    """Load and tokenize reviews from a csv file batch by batch."""

    path: str
    tokenizer: PreTrainedTokenizer
    batch_size: int = 512
    max_length: int = 128

    def __iter__(self) -> Generator[List[List[int]], None, None]:
        """Iterate over batches"""
        for i in range(len(self)):
            yield self.batch_tokenized(i)

    def __len__(self) -> int:
        """Number of batches"""
        with open(self.path, encoding="utf-8") as file:
            next(file)  # skip header
            rows = sum(1 for _ in iter_records(file))
        return (rows + self.batch_size - 1) // self.batch_size  # ceil division

    def tokenize(self, batch: List[str]) -> List[List[int]]:
        """Tokenize list of texts"""
        return [
            self.tokenizer.encode(
                text,
                add_special_tokens=True,
                max_length=self.max_length,
                truncation=True,
            )
            for text in batch
        ]

    def batch_loaded(self, i: int) -> Tuple[List[str], List[int]]:
        """Return loaded i-th batch of data (text, label)"""
        start = i * self.batch_size
        with open(self.path, encoding="utf-8") as file:
            next(file)  # skip header
            records = islice(iter_records(file), start, start + self.batch_size)
            rows = [record.split(",", 4) for record in records]
        texts = [row[4].replace('""', '"') for row in rows]  # review
        labels = [LABELS[row[3]] for row in rows]  # sentiment
        return texts, labels

    def batch_tokenized(self, i: int) -> Tuple[List[List[int]], List[int]]:
        """Return tokenized i-th batch of data"""
        texts, labels = self.batch_loaded(i)
        tokens = self.tokenize(texts)
        return tokens, labels
```

## ЧАСТЬ 4. Построчный разбор того, что изменилось

### Функция `iter_records` (новая, на уровне модуля)

```python
def iter_records(file: TextIO) -> Iterator[str]:
    """Yield logical csv records, merging multiline quoted fields"""
    record = ""
    for line in file:
        record += line
        if record.count('"') % 2 == 0:  # record complete when quotes balanced
            yield record.rstrip("\n")
            record = ""
```

Пошагово:

- `record = ""` — буфер-накопитель для текущей записи. Изначально пуст.
- `for line in file:` — читаем файл **по одной физической строке**. Это ленивое чтение:
  Python не грузит весь файл в память, он отдаёт строки по мере надобности. Каждая `line`
  включает свой перевод строки `\n` на конце (Python его не отрезает).
- `record += line` — приклеиваем строку к буферу. Если отзыв многострочный, здесь склеиваются
  его куски, причём **внутренние переводы строк сохраняются** (они «сидят» внутри буфера).
- `if record.count('"') % 2 == 0:` — считаем кавычки в накопленном буфере. `%` — остаток от
  деления: `x % 2 == 0` означает «x чётное». Чётное число кавычек = все кавычки закрыты =
  запись целая (см. пункт 2.2).
- `yield record.rstrip("\n")` — отдаём готовую запись, отрезав **завершающий** перевод строки
  (тот, что закрывает последнюю физическую строку записи). `rstrip("\n")` срезает `\n` только
  с правого края. Внутренние переводы строк не затрагиваются: они стоят до закрывающей
  кавычки, а строка заканчивается кавычкой либо обычным символом.
- `record = ""` — очищаем буфер под следующую запись.

Почему функция на уровне модуля, а не метод класса? Ей не нужен `self` — она работает только
с переданным файлом. Выносить такие утилиты наружу — чистая практика, и API класса остаётся
ровно таким, каким его ожидает грейдер.

Почему это **генератор** (с `yield`)? Чтобы записи отдавались лениво, по одной, и `islice` мог
остановить чтение файла сразу после нужного батча — не дочитывая гигантский файл до конца.

### `__len__` — минимальное изменение

```python
with open(self.path, encoding="utf-8") as file:
    next(file)  # skip header
    rows = sum(1 for _ in iter_records(file))
```

Было: считали записи через `csv.reader`. Стало: считаем через наш `iter_records` — теперь
`__len__` и `batch_loaded` пользуются **одним и тем же определением записи**. Это принципиально:
если бы они считали записи по-разному, число батчей могло бы не сойтись с реальной нарезкой.

`next(file)` — забирает из файла первую строку (заголовок) и выбрасывает. Заголовок — одна
физическая строка без кавычек, так что читать его через `iter_records` необязательно.

### `batch_loaded` — сердце исправления

```python
start = i * self.batch_size
with open(self.path, encoding="utf-8") as file:
    next(file)  # skip header
    records = islice(iter_records(file), start, start + self.batch_size)
    rows = [record.split(",", 4) for record in records]
texts = [row[4].replace('""', '"') for row in rows]  # review
labels = [LABELS[row[3]] for row in rows]  # sentiment
```

- `islice(iter_records(file), start, start + self.batch_size)` — как и раньше, берём «срез»
  ленивого потока: пропускаем `start` записей, забираем `batch_size` штук. Файл читается ровно
  до конца нужного батча и ни строчкой дальше.
- `record.split(",", 4)` — режем запись максимум по четырём запятым: получаем 5 кусков,
  и пятый (`row[4]`) — это весь отзыв целиком, со всеми его внутренними запятыми
  (см. пункт 2.3).
- `row[4].replace('""', '"')` — схлопываем задвоенные кавычки внутри отзыва. Обрамляющие
  кавычки (если были) остаются — ровно как ожидает эталон.
- `LABELS[row[3]]` — сентимент по-прежнему в четвёртой колонке (индекс 3); он никогда не
  бывает в кавычках (это одно слово без спецсимволов), поэтому сырое разрезание для него
  безопасно.

### Что НЕ изменилось

`tokenize`, `batch_tokenized`, `__iter__` — не тронуты. `test_tokenize` уже давал 20/20,
а два других теста падали только из-за `batch_loaded` и починятся автоматически.

## Уроки этого фейла (самое ценное)

1. **Читай diff «Expected vs Got» посимвольно.** Вся разгадка была в паре кавычек по краям
   строки. Ошибки уровня «почти совпало» — всегда про формат, кодировку или экранирование.
2. **У CSV два уровня кавычек: кавычки-данные и кавычки-разметка.** Стандартный парсер снимает
   разметку; сырое чтение — нет. Нужно понимать, на каком уровне работает тот, с кем ты
   должен совпасть.
3. **«Правильно» для грейдера = «идентично эталону», а не «по стандарту».** Тесты сравнивают
   строки посимвольно. Твоя реализация должна воспроизводить поведение эталонной, даже если
   эталонная читает csv нестандартно.
4. **Инвариант чётности кавычек** — простой и надёжный способ склеивать многострочные
   csv-записи руками, когда стандартный парсер не подходит.
