# Page 3. Greedy Sampler: бизнесовый и технический контекст

## 1. Что строится на этой странице

На предыдущей странице Smart-Link выбирал один из допустимых офферов случайно. Это было полезно как первый технический baseline: сервис принимал запрос, разбирал список кандидатов и возвращал один `offer_id`.

Теперь сервис должен использовать обратную связь. Он будет:

1. запоминать, какой оффер показан для каждого `click_id`;
2. принимать результат клика через `/feedback/`;
3. накапливать статистику по офферам;
4. считать Conversion Rate и Revenue per Click;
5. первые 100 рекомендаций делать случайно;
6. начиная со 101-й рекомендации выбирать кандидата с максимальным RPC.

Такой выбор называется **greedy**, или жадным: алгоритм всегда использует вариант, который сейчас выглядит лучшим.

---

# 2. Бизнес-задача Smart-Link

У партнёрской сети есть поток пользовательских кликов и множество рекламных офферов. Для каждого клика кандидатная система уже отобрала только допустимые предложения по географии, устройству, источнику трафика, бюджету, юридическим ограничениям и другим правилам.

Из этого набора Smart-Link должен выбрать один оффер.

В этой задаче бизнес-цель формулируется так:

```text
выбирать оффер с максимальной средней выручкой на один клик
```

Каждый клик — ограниченный ресурс. Его можно отправить только на один оффер, поэтому системе важно понимать, сколько денег в среднем приносит один клик для каждого предложения.

---

# 3. Почему нельзя оптимизировать только конверсию

Представим два оффера.

## Оффер A

- 100 кликов;
- 20 конверсий;
- награда за каждую конверсию — 1 цент.

```text
CR = 20 / 100 = 0.20
Reward = 20 × 1 = 20
RPC = 20 / 100 = 0.20
```

## Оффер B

- 100 кликов;
- 5 конверсий;
- награда за каждую конверсию — 10 центов.

```text
CR = 5 / 100 = 0.05
Reward = 5 × 10 = 50
RPC = 50 / 100 = 0.50
```

Оффер A конвертируется лучше, но оффер B приносит больше денег на один клик. Если задача бизнеса — максимизировать выручку, выбирать нужно B.

Поэтому greedy-алгоритм сравнивает офферы по `RPC`, а не только по `CR`.

# 4. Основные сущности

## Click ID

`click_id` — уникальный идентификатор входящего клика.

Он нужен, чтобы позднее связать feedback с тем оффером, который был показан именно для этого клика.

## Offer ID

`offer_id` — идентификатор рекламного предложения.

## Reward

`reward` — деньги, полученные за клик. В условии награда передаётся в центах.

Если пользователь не совершил целевое действие:

```text
reward = 0
```

Если пользователь сконвертировался и платформа заработала 12.5 цента:

```text
reward = 12.5
```

## Conversion

В рамках задачи:

```text
is_conversion = reward > 0
```

Положительная награда означает конверсию, нулевая — отсутствие конверсии.

# 5. Почему feedback приходит отдельным запросом

Сначала пользователь кликает по Smart-Link. Сервис должен сразу выбрать оффер через `/sample/`.

Но целевое действие может произойти позже:

- регистрация — через несколько минут;
- покупка — через несколько часов;
- депозит — через день;
- подтверждённый заказ — после доставки;
- выплата — после антифрод-проверки.

Поэтому feedback невозможно вернуть в том же запросе. Он приходит отдельно через:

```text
PUT /feedback/
```

Причём события могут приходить вразнобой.

Например, sample-запросы пришли так:

```text
click 101
click 102
click 103
```

А feedback может прийти так:

```text
feedback для click 103
feedback для click 101
feedback для click 102
```

Нельзя хранить только «последний оффер». Нужно хранить точное соответствие:

```text
click_id -> offer_id
```

Именно для этого используется словарь `CLICK_TO_OFFER`.

# 6. Четыре глобальных словаря

Условие подсказывает использовать четыре словаря.

## 6.1. Количество рекомендаций по офферу

```python
OFFER_CLICKS = {}
```

Пример:

```python
{
    45: 120,
    67: 80,
}
```

Оффер 45 был рекомендован 120 раз, а оффер 67 — 80 раз.

## 6.2. Связь клика с оффером

```python
CLICK_TO_OFFER = {}
```

Пример:

```python
{
    1001: 45,
    1002: 67,
    1003: 45,
}
```

## 6.3. Суммарная награда

```python
OFFER_REWARDS = {}
```

Пример:

```python
{
    45: 84.0,
    67: 100.0,
}
```

## 6.4. Количество конверсий

```python
OFFER_CONVERSIONS = {}
```

Пример:

```python
{
    45: 12,
    67: 5,
}
```

# 7. Метрики `/stats/`

Endpoint:

```text
GET /offer_ids/{offer_id}/stats/
```

возвращает пять показателей.

## Clicks

Количество рекомендаций этого оффера.

## Conversions

Количество feedback с `reward > 0`.

## Reward

Сумма всех наград по офферу.

## Conversion Rate

```text
CR = conversions / clicks
```

Например:

```text
1 / 10 = 0.1
```

В API возвращается доля `0.1`, а не строка `"10%"`.

## Revenue per Click

```text
RPC = reward / clicks
```

Например:

```text
4.2 / 10 = 0.42
```

# 8. Что делать с оффером без статистики

Проверяющая система может запросить оффер, по которому ещё не было кликов.

Если обращаться к словарю напрямую:

```python
OFFER_CLICKS[999]
```

возникнет `KeyError`.

Поэтому решение использует:

```python
OFFER_CLICKS.get(999, 0)
```

Ожидаемый ответ для неизвестного оффера:

```json
{
  "offer_id": 999,
  "clicks": 0,
  "conversions": 0,
  "reward": 0.0,
  "cr": 0.0,
  "rpc": 0.0
}
```

Это одновременно защищает от деления на ноль.

---

# 9. Почему первые 100 кликов случайные

В начале RPC всех офферов равен нулю. Если сразу включить greedy, алгоритм будет постоянно выбирать первый оффер и никогда не соберёт данные по остальным.

Поэтому первые 100 рекомендаций используются как инициализация:

```text
exploration phase
```

На них оффер выбирается случайно.

Начиная со 101-го запроса включается:

```text
exploitation phase
```

Алгоритм начинает выбирать текущего лидера по RPC.

Точная проверка в коде:

```python
if total_clicks < 100:
```

До обработки первого запроса `total_clicks = 0`, а перед сотым запросом `total_clicks = 99`. Поэтому запросы с 1-го по 100-й включительно случайные. Перед 101-м запросом значение равно 100, и включается greedy.

---

# 10. Как работает greedy-выбор

После инициализации рассматриваются только кандидаты текущего запроса.

Например, глобально система знает офферы:

```text
1, 2, 3, 4, 5
```

Но на конкретный клик кандидатная модель передала:

```text
2,5
```

Greedy не имеет права выбрать 1, 3 или 4, даже если их RPC выше.

Код:

```python
offer_id = max(offers_ids, key=get_rpc)
```

Функция `max` вычисляет `get_rpc` для каждого переданного кандидата и возвращает ID с максимальным значением.

---

# 11. Почему при нулевой статистике возвращается первый кандидат

Пусть пришли:

```python
offers_ids = [45, 67]
```

И RPC обоих равен нулю.

Python-функция `max` при равных значениях ключа возвращает первый встретившийся элемент:

```python
max([45, 67], key=get_rpc)
```

вернёт `45`.

Это полностью соответствует условию.

Важно не сортировать кандидатов, потому что «первый» — первый в исходном запросе, а не минимальный ID.

# 12. API-контракт

## `/sample/`

Запрос:

```text
GET /sample/?click_id=1001&offer_ids=45,67
```

Ответ в random-фазе:

```json
{
  "click_id": 1001,
  "offer_id": 67,
  "sampler": "random"
}
```

Ответ в greedy-фазе:

```json
{
  "click_id": 1001,
  "offer_id": 45,
  "sampler": "greedy"
}
```

## `/feedback/`

Запрос:

```text
PUT /feedback/?click_id=1001&reward=12.5
```

Ответ:

```json
{
  "click_id": 1001,
  "offer_id": 45,
  "is_conversion": true,
  "reward": 12.5
}
```

Оффер не передаётся. Сервис находит его через `CLICK_TO_OFFER`.

## `/stats/`

Запрос:

```text
GET /offer_ids/45/stats/
```

Ответ:

```json
{
  "offer_id": 45,
  "clicks": 10,
  "conversions": 2,
  "reward": 20.0,
  "cr": 0.2,
  "rpc": 2.0
}
```

# 13. Почему глобальные словари допустимы здесь

Для учебной задачи это минимальное и понятное хранилище:

- все endpoints видят одно состояние;
- данные живут между запросами;
- после перезапуска статистика сбрасывается;
- решение не перегружено базой данных.

Но это не production-архитектура. При реальном запуске возникают проблемы:

- данные теряются при рестарте;
- разные worker-процессы имеют разное состояние;
- возможны race conditions;
- память растёт;
- нет транзакций и истории.

В production обычно применяют:

- Redis для online-state;
- Kafka для потока событий;
- PostgreSQL или ClickHouse для истории;
- отдельный сервис агрегации;
- дедупликацию callback;
- идемпотентность;
- мониторинг задержек feedback.

# 14. Ограничения greedy

После первых 100 кликов greedy почти перестаёт исследовать альтернативы.

Из-за этого возможны проблемы:

- ранний случайный шум выбирает неправильного победителя;
- новый оффер не получает трафик;
- изменение payout остаётся незамеченным;
- сезонность делает старую статистику неактуальной;
- долгий conversion lag занижает некоторые офферы;
- алгоритм может зафиксироваться на локально хорошем варианте.

Поэтому Greedy Sampler — промежуточный учебный шаг. Следующий естественный алгоритм — ε-greedy, который большую часть времени выбирает лидера, но иногда продолжает случайное исследование.

# 15. Бизнес-применение в разных индустриях

## Marketplace

Кандидаты:

- товары;
- продавцы;
- рекламные карточки;
- способы доставки.

Reward:

- комиссия;
- маржа;
- GMV;
- contribution margin.

## E-commerce

Кандидаты:

- баннеры;
- скидки;
- рекомендации;
- cross-sell предложения.

Оптимизировать только CR опасно: большая скидка может повысить конверсию, но снизить прибыль.

## Fintech

Кандидаты:

- кредитная карта;
- депозит;
- рассрочка;
- страховка;
- cashback-механика.

Reward лучше определять как risk-adjusted profit или ожидаемый NPV, а не простую выдачу кредита.

## Two-sided marketplace

Кандидаты:

- водитель;
- курьер;
- поставщик;
- производственный партнёр;
- исполнитель услуги.

Reward может учитывать маржу, скорость, качество и вероятность успешного выполнения.

## Dating

Кандидаты — анкеты и типы рекомендаций. Reward может быть взаимным match, содержательным диалогом или долгосрочным retention. Оптимизация только клика создаёт clickbait-рекомендации.

## iGaming и casino

Кандидаты — игры, бонусы и коммуникации. Нужны responsible gaming, возрастные ограничения, self-exclusion и compliance.

## Crypto

Кандидаты — образовательные карточки, торговые пары, способы пополнения или продукты earn. Нельзя оптимизировать только объём торговли без учёта риска и регулирования.

## Google и Meta

Аналогичная логика используется для выбора объявлений, креативов, рекомендаций и уведомлений. В production применяются намного более сложные модели, но идея оценки reward каждой альтернативы остаётся фундаментальной.

## Anthropic и AI-системы

Руками могут быть разные модели, промпты или инструменты агента. Reward может учитывать качество, стоимость и latency. Greedy-router выбирает конфигурацию с максимальной накопленной полезностью.

## SpaceX, BP и промышленность

Альтернативами могут быть настройки процесса, поставщики или режимы оборудования. В safety-critical системах exploration должен быть ограничен симуляциями, правилами безопасности и человеческим контролем.
