# Page 2. Random Sampler для Smart-Link

## 1. Что происходит на этой странице

На этой странице мы создаём **первую работающую модель выбора оффера** для Smart-Link.

Это ещё не ε-greedy, не полноценный многорукий бандит и не модель машинного обучения в привычном смысле. Здесь используется максимально простой baseline:

> для каждого входящего клика случайно выбрать один оффер из уже отфильтрованного набора кандидатов.

Такой простой механизм нужен не потому, что случайный выбор является лучшей бизнес-стратегией. Он нужен как первый технический этап, позволяющий проверить весь путь запроса:

1. backend получает пользовательский клик;
2. кандидатная система определяет доступные офферы;
3. backend вызывает ML-сервис;
4. ML-сервис получает список `offer_ids`;
5. сервис выбирает один идентификатор;
6. сервис возвращает ответ в формате JSON;
7. backend перенаправляет пользователя на выбранный оффер;
8. события клика и последующей конверсии можно сохранить для дальнейшего обучения.

То есть Random Sampler — это прежде всего **интеграционный baseline**.

---

# 2. Бизнес-контекст

## 2.1. Что является входящим объектом

В Smart-Link приходит пользовательский клик.

Пользователь мог перейти:

- по рекламному баннеру;
- по ссылке блогера;
- по ссылке из Telegram-канала;
- из мобильного приложения;
- из поисковой рекламы;
- с сайта арбитражника;
- из email-рассылки;
- из социальной сети.

Для этого пользователя кандидатная система уже отобрала офферы, которые технически можно показать.

Например, после фильтрации остались офферы:

```text
2,5,1
```

Это означает, что каждый из офферов `2`, `5` и `1` допустим для данного клика.

Наш сервис должен вернуть ровно один из них.

---

## 2.2. Зачем нужна кандидатная модель

Нельзя выбирать случайный оффер вообще из всех существующих офферов.

У каждого оффера могут быть ограничения:

- страна пользователя;
- язык;
- тип устройства;
- операционная система;
- источник трафика;
- возрастная категория;
- разрешённые рекламные каналы;
- дневной лимит;
- бюджет рекламодателя;
- доступность оффера;
- категория продукта;
- наличие товара;
- юридические ограничения.

Например, пользователь из Армении с Android-смартфоном не должен быть отправлен:

- на оффер, работающий только в Германии;
- на приложение только для iPhone;
- на оффер с исчерпанным бюджетом;
- на запрещённую для этого источника кампанию;
- на остановленный рекламодателем оффер.

Поэтому система логически разделена на две стадии.

### Стадия 1. Candidate generation

Формируется множество допустимых офферов.

### Стадия 2. Ranking или selection

Из допустимого множества выбирается один оффер.

На этой странице мы реализуем только вторую стадию.

---

## 2.3. Почему начинают со случайного выбора

Случайный выбор решает несколько важных задач.

### Проверка интеграции

Можно убедиться, что:

- FastAPI запускается;
- endpoint доступен;
- query-параметр правильно передаётся;
- строка преобразуется в список;
- ответ возвращается в ожидаемом формате;
- backend умеет прочитать `offer_id`.

### Сбор начальных данных

До запуска системы может не быть истории по новым офферам.

Случайное распределение трафика позволяет собрать первичные наблюдения:

- количество показов;
- количество кликов;
- количество конверсий;
- выручку;
- выплату;
- задержку до конверсии;
- статистику по странам и устройствам.

### Базовая точка сравнения

Позже ε-greedy или другой алгоритм необходимо сравнивать с простым baseline.

Можно задать вопрос:

> Насколько умная стратегия лучше равномерного случайного распределения?

Без baseline нельзя понять, действительно ли новая модель создаёт дополнительную ценность.

# 3. Почему исходный код неправильный

В заготовке используется строка:

```python
offer_id = sorted(offers_ids)[0]
```

Разберём её по частям.

```python
sorted(offers_ids)
```

Создаёт новый список, отсортированный по возрастанию.

Например:

```python
offers_ids = [2, 5, 1]
sorted(offers_ids)
```

Результат:

```python
[1, 2, 5]
```

После этого:

```python
[0]
```

берёт элемент с индексом `0`, то есть первый элемент списка.

Результат всегда будет:

```python
1
```

Следовательно, такой сервис не делает случайный выбор. Он всегда выбирает минимальный `offer_id`.

Это создаёт серьёзный систематический перекос:

- оффер с меньшим ID получает почти весь трафик;
- остальные офферы не получают данных;
- статистика становится непригодной для честного сравнения;
- рекламодатели получают неравномерное распределение;
- невозможно корректно перейти к многорукому бандиту.

---

# 4. Техническая постановка задачи

Нужно изменить только механизм выбора оффера.

Было:

```python
offer_id = sorted(offers_ids)[0]
```

Должно стать:

```python
offer_id = int(np.random.choice(offers_ids))
```

Функция `np.random.choice()` случайно выбирает один элемент из переданной последовательности.

Вход:

```python
[2, 5, 1]
```

Возможные результаты разных вызовов:

```text
2
1
5
2
5
...
```

При большом количестве вызовов каждый элемент будет выбираться примерно с одинаковой вероятностью:

```text
P(offer=1) ≈ 1/3
P(offer=2) ≈ 1/3
P(offer=5) ≈ 1/3
```

---

# 5. Почему используется `int(...)`

`np.random.choice()` может вернуть числовой объект NumPy, например `numpy.int64`.

FastAPI должен превратить ответ в JSON:

```json
{"offer_id": 2}
```

Стандартный JSON ожидает обычный Python-тип `int`.

Поэтому результат безопасно преобразуется:

```python
int(np.random.choice(offers_ids))
```

Внутри ответа будет обычное целое число Python.

---

# 6. Архитектура HTTP-запроса

Endpoint объявлен так:

```python
@app.get("/sample/")
```

Это означает, что сервис обрабатывает HTTP GET-запросы по пути:

```text
/sample/
```

Параметр функции:

```python
offer_ids: str
```

становится query-параметром.

Пример полного запроса:

```text
http://localhost:8000/sample/?offer_ids=2,5,1
```

FastAPI передаст функции строку:

```python
"2,5,1"
```

Далее выполняется:

```python
offer_ids.split(",")
```

Результат:

```python
["2", "5", "1"]
```

Затем list comprehension преобразует строки в целые числа:

```python
[int(offer) for offer in offer_ids.split(",")]
```

Результат:

```python
[2, 5, 1]
```

Из этого списка случайно выбирается один элемент.

---

# 7. Что возвращает сервис

Сервис создаёт словарь:

```python
response = {
    "offer_id": offer_id,
}
```

FastAPI автоматически преобразует Python-словарь в JSON.

Например:

```json
{
  "offer_id": 5
}
```

Ключ `offer_id` является контрактом между ML-сервисом и backend.

Backend не обязан знать, как именно был выбран оффер. Он должен получить идентификатор и продолжить обработку клика.

---

# 8. Что именно проверяет задание

## 10% — сервис не сломан

Ожидается, что:

- модуль импортируется;
- объект `app` существует;
- endpoint `/sample/` существует;
- функция принимает `offer_ids`;
- возвращается словарь;
- в словаре есть ключ `offer_id`;
- выбранный ID принадлежит исходному набору кандидатов.

## 90% — выбор действительно случайный

Тесты могут многократно вызвать функцию и проверить, что:

- результат не всегда одинаковый;
- выбираются разные элементы;
- нет постоянного выбора минимального элемента;
- офферы выбираются примерно равномерно;
- не возвращаются значения вне переданного списка.

Именно поэтому недостаточно, например, заменить первый элемент на последний. Требуется настоящий случайный выбор.

---

# 9. Почему это ещё не машинное обучение

Random Sampler:

- не обучается;
- не хранит статистику;
- не использует признаки пользователя;
- не оценивает конверсию;
- не знает стоимость action;
- не максимизирует прибыль;
- не адаптируется к изменениям.

Тем не менее он важен как baseline и источник начальной exploration-выборки.

Следующий шаг — использовать накопленные результаты и балансировать:

- exploration: исследовать разные офферы;
- exploitation: чаще выбирать уже успешные офферы.

Это и приводит нас к ε-greedy.

# 10. Бизнес-метрики, которые появятся позже

Случайный sampler можно оценивать не только по технической работоспособности.

## Conversion Rate

```text
CR = actions / clicks
```

## Revenue per Click

```text
RPC = revenue / clicks
```

## Earnings per Click

В affiliate-модели часто используется:

```text
EPC = payout / clicks
```

## Profit per Click

```text
profit_per_click =
    advertiser_revenue_per_click
    - publisher_payout_per_click
    - infrastructure_cost_per_click
```

## Fill Rate

Доля кликов, для которых удалось найти хотя бы один допустимый оффер.

## Latency

Время от запроса backend до ответа ML-сервиса.

Для online-рекомендации это критично: медленный выбор увеличивает время перенаправления пользователя и может снижать конверсию.

# 11. Применение аналогичной схемы в других индустриях

## Marketplace

Кандидатная модель отбирает доступные товары, а selection-модель выбирает товар для рекламного блока.

Random baseline случайно показывает один из доступных товаров.

## E-commerce

Из товаров:

- в наличии;
- доступных в регионе;
- подходящих под устройство;
- разрешённых для рекламной кампании

случайно выбирается один баннер.

## Fintech

Из доступных пользователю банковских продуктов выбирается:

- кредитная карта;
- рассрочка;
- депозит;
- cashback-предложение.

В реальном production случайный выбор должен учитывать compliance и eligibility.

## Two-sided marketplace

Для заказа выбирается один допустимый исполнитель:

- водитель;
- курьер;
- мастер;
- поставщик;
- производственный партнёр.

Candidate generation проверяет ограничения, selection выбирает кандидата.

## Dating

Из допустимых анкет случайно выбирается одна рекомендация. Позже выбор можно оптимизировать по вероятности взаимного match и качеству общения.

## iGaming и casino

Из разрешённых игр или бонусов выбирается один вариант. Здесь особенно важны responsible gaming, возрастные ограничения и локальное регулирование.

## Crypto

Из допустимых продуктов может выбираться:

- образовательная карточка;
- торговая пара;
- способ пополнения;
- staking-продукт.

Критично исключать недоступные и нерегуляторные предложения до этапа выбора.

## Big Tech

Та же двухступенчатая архитектура используется в:

- рекомендательных системах;
- рекламном ранжировании;
- поиске;
- feed ranking;
- выборе уведомлений;
- маршрутизации запросов между моделями;
- подборе инструментов AI-агента.

offer_id = int(np.random.choice(offers_ids))
