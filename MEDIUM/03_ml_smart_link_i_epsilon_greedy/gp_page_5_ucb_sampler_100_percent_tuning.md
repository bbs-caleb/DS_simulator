# 2. Почему предыдущий UCB недобрал 4.8 балла

Предыдущая формула:

```python
RPC + 0.5 * global_reward_scale * uncertainty
```

использовала один глобальный денежный масштаб для всех офферов.

Это создаёт две проблемы.

## Проблема 1. Коэффициент 0.5 слишком исследовательский

Алгоритм уже корректно находил сильные офферы, но продолжал достаточно часто возвращаться к менее выгодным кандидатам.

Результат 75.2 из 80 означает, что решение очень близко к порогу, поэтому полная смена алгоритма создаёт лишний риск.

Рациональный шаг — уменьшить bonus:

```text
0.50 → 0.30
```

## Проблема 2. Один bonus для разных payout

Представим два оффера.

```text
Оффер A: выплата за конверсию = 2
Оффер B: выплата за конверсию = 20
```

Неопределённость Conversion Rate имеет разную денежную цену.

Ошибка CR на 1 процентный пункт означает:

```text
для A: 0.01 × 2  = 0.02 RPC
для B: 0.01 × 20 = 0.20 RPC
```

Значит, confidence bonus должен масштабироваться payout конкретного оффера.

---

# 3. Новая формула

Теперь:

```text
UCB(offer)
=
RPC(offer)
+
0.30
× estimated_payout(offer)
× sqrt(log(total_clicks + 1) / offer_clicks)
```

Где:

```text
estimated_payout(offer)
=
total reward offer / conversions offer
```

Если у оффера пока не было конверсии, используется средняя выплата по всем уже наблюдаемым конверсиям.

---

# 4. Почему это должно уменьшить regret

## Для дешёвого слабого оффера

После первой конверсии сервис узнаёт низкий payout.

Его exploration bonus уменьшается.

Алгоритм перестаёт тратить на него лишние клики.

## Для дорогого малоизученного оффера

Если payout высокий, uncertainty в CR имеет большую денежную ценность.

Такой оффер получает достаточное дополнительное исследование.

## Для текущего лидера

Высокий RPC остаётся основной частью score.

После накопления данных confidence bonus уменьшается как:

```text
1 / sqrt(clicks)
```

и лидер получает всё большую часть трафика.

# 5. Почему не выбран полностью новый алгоритм

Thompson Sampling может быть очень эффективным и имеет теоретические гарантии логарифмического ожидаемого regret. citeturn839689search1turn839689academia25

Однако текущий UCB уже набрал 95.26%, а класс UCB является стандартным алгоритмом конечного времени для stochastic multi-armed bandit. citeturn839689search0

Поэтому безопаснее:

1. сохранить прошедший API;
2. сохранить прошедшую структуру UCB;
3. устранить конкретное избыточное exploration;
4. не добавлять случайность нового алгоритма.

# 7. Что проверено локально

Проверено:

- импорт файла;
- синтаксис;
- startup-reset;
- нулевая статистика;
- `clicks=1` после одного sample;
- feedback не увеличивает clicks второй раз;
- reward и conversions;
- CR и RPC;
- ответ sample ровно из двух полей;
- cold start всех новых кандидатов;
- выбор только из переданного candidate set;
- более высокий payout не применяется к чужому офферу;
- максимальный payout-aware UCB выбирается корректно.
