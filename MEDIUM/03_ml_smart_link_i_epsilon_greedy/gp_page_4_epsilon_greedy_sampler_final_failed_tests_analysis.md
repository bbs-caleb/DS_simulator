# Page 4. Финальный разбор двух последовательных падений LMS

## 1. Почему первый вариант вернул `clicks = 4`, хотя ожидался `0`

В первой версии `clicks` увеличивался правильно — в `/sample`.

Проблема была в другом: состояние сервиса не очищалось при новом startup тестового приложения.

Проверки LMS выполняются последовательно. Предыдущий тест `/sample` успел накопить четыре рекомендации оффера `42`. Затем следующий тест ожидал начать с чистого состояния и запросил статистику этого оффера.

Поскольку словари не были очищены, сервис вернул старые четыре клика:

```text
actual clicks = 4
expected clicks = 0
```

Правильное исправление для этой ошибки:

```python
@app.on_event("startup")
def reset_statistics() -> None:
    OFFER_CLICKS.clear()
    CLICK_TO_OFFER.clear()
    OFFER_REWARDS.clear()
    OFFER_CONVERSIONS.clear()
```

То есть статистика должна сбрасываться при каждом startup приложения.

---

## 2. Почему второй вариант вернул `clicks = 0`, хотя ожидался `1`

Во второй версии startup-reset был добавлен, но счётчик кликов был ошибочно перенесён из `/sample` в `/feedback`.

В новом тесте LMS:

1. запустил приложение с чистым состоянием;
2. сделал одну рекомендацию оффера `1009`;
3. запросил `/stats`;
4. ожидал `clicks = 1`.

Но сервис ещё не получил feedback, поэтому перенесённый счётчик оставался нулём.

Получилось:

```text
actual clicks = 0
expected clicks = 1
```

Значит, определение `clicks` в задании буквальное:

```text
clicks = количество рекомендаций / показов оффера
```

Следовательно, увеличивать его нужно в `/sample`.

---

## 3. Как оба assertion одновременно объясняются

Обе ошибки совместимы и дают однозначную правильную архитектуру.

### Перед каждым startup

Очистить все словари.

### В `/sample`

- выбрать оффер;
- увеличить количество кликов;
- сохранить `click_id -> offer_id`.

### В `/feedback`

- найти выбранный оффер;
- добавить reward;
- увеличить conversions при `reward > 0`;
- не увеличивать clicks второй раз.

Именно эта комбинация даёт:

```text
новый startup + неизвестный offer 42 -> clicks = 0
```

и:

```text
новый startup + один sample offer 1009 -> clicks = 1
```

## 4. Ошибка в пользовательском варианте, получившем 20%

В пользовательском файле были два разных знаменателя:

```python
offer_clicks
```

увеличивался в `/sample` и использовался в `/stats`.

А:

```python
offer_fb_count
```

увеличивался в `/feedback` и использовался для greedy RPC.

Из-за этого статистика и алгоритм оценивали офферы по разным определениям количества кликов.

Для задания RPC должен быть:

```text
RPC = суммарный reward / количество рекомендаций оффера
```

Поэтому и `/stats`, и greedy должны использовать один словарь:

```python
OFFER_CLICKS
```

## 5. Финальная модель состояния

```python
OFFER_CLICKS = {}
CLICK_TO_OFFER = {}
OFFER_REWARDS = {}
OFFER_CONVERSIONS = {}
```

### `OFFER_CLICKS`

Количество выборов оффера в `/sample`.

### `CLICK_TO_OFFER`

Связь входящего клика с выбранным оффером.

### `OFFER_REWARDS`

Суммарная награда по полученным feedback.

### `OFFER_CONVERSIONS`

Количество feedback с положительной наградой.

---

## 6. Финальная последовательность

### Sample

```python
OFFER_CLICKS[offer_id] = OFFER_CLICKS.get(offer_id, 0) + 1
CLICK_TO_OFFER[click_id] = offer_id
```

### Feedback

```python
OFFER_REWARDS[offer_id] = OFFER_REWARDS.get(offer_id, 0.0) + reward
```

и только при конверсии:

```python
OFFER_CONVERSIONS[offer_id] += 1
```

### RPC

```python
reward / clicks
```

где clicks — рекомендации из `/sample`.

---

## 7. Почему startup-reset здесь обязателен

Обычный перезапуск отдельного Python-процесса сам очищает глобальные словари.

Но тестовая среда может:

- один раз импортировать модуль;
- создавать несколько FastAPI/TestClient lifecycle;
- запускать независимые тесты в одном Python-процессе.

Простого объявления:

```python
OFFER_CLICKS = {}
```

тогда недостаточно, потому что модуль повторно не импортируется.

Startup-hook гарантирует чистое состояние для нового жизненного цикла приложения.

## 8. Что теперь проверено

### Сценарий первого падения

1. Накопить четыре sample для offer 42.
2. Выполнить startup reset.
3. Запросить stats offer 42.

Результат:

```python
{
    "offer_id": 42,
    "clicks": 0,
    "conversions": 0,
    "reward": 0.0,
    "cr": 0.0,
    "rpc": 0.0,
}
```

### Сценарий второго падения

1. Выполнить startup reset.
2. Один раз вызвать sample с единственным offer 1009.
3. Запросить stats.

Результат:

```python
{
    "offer_id": 1009,
    "clicks": 1,
    "conversions": 0,
    "reward": 0.0,
    "cr": 0.0,
    "rpc": 0.0,
}
```

### Feedback

После `reward=5`:

```text
clicks остаётся 1
conversions становится 1
reward становится 5
CR = 1
RPC = 5
```

Feedback не должен повторно увеличивать clicks.

---

## 9. Итог

Финальное правило:

```text
startup  -> очистить статистику
sample   -> увеличить clicks
feedback -> увеличить reward/conversions
stats    -> reward и conversions делить на sample-clicks
```

Это решение одновременно устраняет оба фактически показанных assertion LMS.
