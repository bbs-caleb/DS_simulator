# gp_page_5 — Residual Fairness: бизнесовый и технический контекст

## 1. Главная идея шага

На предыдущих шагах мы анализировали:

- величину ошибки;
- направление ошибки;
- нормальность остатков;
- смещение;
- постоянство дисперсии;
- диагностические графики.

Теперь появляется новый вопрос:

> Даже если средняя ошибка стала меньше, одинаково ли улучшение распределилось между объектами?

Это принципиально важный вопрос.

Модель может улучшить среднюю метрику двумя способами.

### Сценарий A — равномерное улучшение

Ошибки стали немного меньше почти у всех пользователей, товаров или магазинов.

Это обычно желательное изменение:

- большинство пользователей получают более качественный результат;
- нет сегмента, который заплатил за общее улучшение;
- поведение системы становится стабильнее;
- меньше риск негативного пользовательского опыта.

### Сценарий B — неравномерное улучшение

У небольшой части объектов ошибка резко уменьшилась, но у другой группы сильно выросла.

Средняя метрика может стать лучше, однако:

- один сегмент пользователей получает ухудшение;
- растёт число жалоб;
- ухудшается retention;
- может возникнуть дискриминация;
- появляются репутационные и регуляторные риски;
- бизнес-эффект средней метрики может быть перечёркнут другими KPI.

Именно эту неравномерность измеряет Residual Fairness.

---

# 2. Почему одной средней метрики недостаточно

Рассмотрим два набора абсолютных ошибок.

## Модель A

```text
[2, 2, 2, 2, 2]
```

Средняя ошибка:

```text
2
```

Ошибка распределена идеально равномерно.

## Модель B

```text
[0, 0, 0, 0, 10]
```

Средняя ошибка также:

```text
2
```

По средней метрике модели одинаковы.

Но пользовательский опыт принципиально разный:

- у модели A все получают небольшую ошибку;
- у модели B четыре объекта обслужены идеально, а один получил катастрофическую ошибку.

Именно поэтому средняя ошибка не описывает справедливость распределения.

# 3. Что означает fairness в этом уроке

Здесь fairness не означает классическую демографическую fairness вроде:

- равенства по полу;
- равенства по возрасту;
- равенства по национальности.

В этом шаге fairness определяется через концентрацию ошибок по объектам.

Вопрос звучит так:

> Не приходится ли большая часть общей ошибки на небольшое количество объектов?

Объектом может быть:

- пользователь;
- товар;
- магазин;
- кредитная заявка;
- рекламная кампания;
- водитель;
- ресторан;
- регион;
- устройство;
- промышленный агрегат.

---

# 4. Gini Index

Для измерения неравномерности используется индекс Джини.

Классически его применяют в экономике для анализа распределения доходов.

Интерпретация:

```text
Gini = 0
```

означает идеальное равенство.

```text
Gini близок к 1
```

означает сильную концентрацию.

В задаче определяется:

```text
fairness = 1 - Gini
```

Поэтому интерпретация меняется:

```text
fairness = 1
```

ошибки распределены максимально равномерно.

```text
fairness близка к 0
```

ошибки сильно сконцентрированы на малом числе объектов.

---

# 5. Почему нужно брать модуль псевдоостатков

Псевдоостатки LogLoss могут быть отрицательными:

```python
y_true * np.log(y_pred)
+ (1 - y_true) * np.log(1 - y_pred)
```

Логарифм вероятности от 0 до 1 является отрицательным.

Но индекс Джини должен измерять не направление ошибки, а её размер.

Поэтому сначала используется:

```python
np.abs(residuals)
```

Иначе положительные и отрицательные значения могли бы:

- компенсировать друг друга;
- дать некорректное среднее;
- разрушить интерпретацию через кривую Лоренца;
- привести к неправильному Gini.

# 6. Бизнес-применение

## 6.1 Retail и demand forecasting

Средняя WAPE может улучшиться, потому что модель стала лучше на массовых SKU, но при этом ухудшилась:

- на товарах первой необходимости;
- на высокомаржинальных товарах;
- на отдельных магазинах;
- на новых товарах;
- на промо;
- на товарах с высоким риском OOS.

Residual Fairness помогает проверить, не сконцентрировалась ли ошибка на небольшой группе SKU или магазинов.

Однако в реальном проекте одной объектной fairness недостаточно. Дополнительно нужно смотреть fairness по бизнес-сегментам:

- формат магазина;
- регион;
- категория;
- ABC/XYZ-класс;
- оборачиваемость;
- промо;
- OOS;
- срок годности.

---

## 6.2 Fintech

Модель кредитного риска может улучшить средний LogLoss, но стать хуже:

- для молодых клиентов;
- для самозанятых;
- для отдельных регионов;
- для клиентов без кредитной истории;
- для крупных кредитов.

Это создаёт:

- финансовый риск;
- регуляторный риск;
- риск дискриминации;
- репутационные потери.

Residual Fairness является первым сигналом концентрации ошибок, но затем обязательно нужен групповой fairness-анализ.

---

## 6.3 Marketplace

В ETA-модели средняя ошибка может уменьшиться, но ухудшиться:

- в удалённых районах;
- в часы пик;
- в плохую погоду;
- у новых ресторанов;
- у определённого типа курьеров.

Средний MAE не покажет, что один сегмент систематически страдает.

---

## 6.4 Рекомендательные системы и dating

Общий CTR или conversion rate может вырасти за счёт популярных пользователей и массовых интересов.

При этом система может хуже обслуживать:

- новых пользователей;
- пользователей с редкими предпочтениями;
- малые группы;
- создателей нового контента;
- пользователей из небольших городов.

В результате краткосрочно растёт метрика, но долгосрочно ухудшаются:

- diversity;
- retention;
- content supply;
- доверие пользователей.

---

## 6.5 Реклама

Модель может улучшить общий CTR, направляя почти весь трафик на несколько очень сильных сегментов.

Но это может:

- лишить другие сегменты достаточного exploration;
- ухудшить обучение модели;
- создать концентрацию рекламных показов;
- снизить долгосрочную ценность экосистемы.

---

## 6.6 iGaming и casino

Средняя модельная метрика может улучшаться, но ошибки могут концентрироваться на:

- пользователях с рискованным поведением;
- high-value игроках;
- новых пользователях;
- пользователях с признаками зависимости.

Здесь цена несправедливой ошибки может быть особенно высокой:

- финансовый вред;
- responsible gambling risk;
- лицензионный риск;
- регуляторные санкции.

---

## 6.7 Crypto и финансовые рынки

Средний прогноз может выглядеть хорошо в обычные периоды, но вся ошибка концентрироваться в редких рыночных шоках.

Это пример ситуации, где:

- средняя метрика выглядит приемлемо;
- fairness низкая;
- самые важные объекты получают самые крупные ошибки.

---

## 6.8 Industrial ML, энергетика и aerospace

Модель predictive maintenance может хорошо работать на большинстве оборудования, но систематически ошибаться на редком типе агрегатов.

Средняя метрика будет хорошей, но ошибка на редком оборудовании может привести к:

- аварии;
- простою;
- дорогостоящему ремонту;
- угрозе безопасности.

# 7. Что делает `best_prediction`

Функция выбирает не просто модель с минимальным LogLoss.

Модель должна выполнить два условия:

1. LogLoss меньше, чем у текущей лучшей допустимой модели.
2. Fairness не ниже разрешённого порога относительно baseline.

Порог:

```text
minimum_fairness =
baseline_fairness * (1 - fairness_drop)
```

При `fairness_drop = 0.05` разрешено ухудшение fairness максимум на 5%.

---

# 8. Почему baseline остаётся фиксированным

Baseline — это нулевой элемент:

```python
y_preds[0]
```

Именно относительно него задаётся допустимое падение fairness.

Если baseline fairness равна:

```text
0.80
```

а допустимое падение:

```text
5%
```

то минимально допустимая fairness:

```text
0.80 × 0.95 = 0.76
```

Этот порог остаётся фиксированным для всех кандидатов.

Нельзя каждый раз сравнивать новую модель с fairness предыдущего победителя, потому что условие задачи задано относительно baseline.

---

# 9. Почему нужно обновлять лучший LogLoss

Допустим:

```text
baseline loss = 0.50
model 1 loss = 0.45
model 2 loss = 0.47
```

Обе модели лучше baseline.

Но лучшая — модель 1, потому что:

```text
0.45 < 0.47
```

Если продолжать сравнивать все модели только с исходным `0.50`, функция может в конце выбрать модель 2 просто потому, что она встретилась позже.

Поэтому после выбора модели 1 нужно обновить:

```python
best_loss = current_loss
```

После этого модель 2 сравнивается уже с `0.45` и не заменяет настоящего победителя.

# 10. Как мыслить как продуктовый ML-специалист

Перед выкатыванием модели нужно ответить:

1. Улучшилась ли средняя метрика?
2. Для скольких объектов она улучшилась?
3. Где сконцентрировались ошибки?
4. Есть ли сегменты, которым стало хуже?
5. Насколько дорогие эти сегменты?
6. Допустимо ли падение fairness?
7. Не ухудшились ли guardrail-метрики?
8. Сохранится ли эффект после полного rollout?

Это и есть переход от «модель имеет лучший LogLoss» к полноценному продуктовому решению.
