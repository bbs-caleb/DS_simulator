# gp_page_7 — Adversarial Validation на остатках: полный контекст

## 1. Где мы находимся в Error Analysis

До этого мы прошли несколько уровней диагностики модели:

1. Научились считать остатки и псевдоостатки.
2. Проверили их верхнеуровневые свойства.
3. Построили диагностические графики.
4. Оценили равномерность ошибок через Residual Fairness.
5. Нашли лучшие и худшие отдельные объекты.
6. Научились анализировать заранее заданные corner cases.

Теперь возникает естественная проблема.

Допустим, у нас:

- сотни признаков;
- миллионы объектов;
- тысячи worst cases;
- сложные взаимодействия между признаками.

Человек уже не может надёжно просмотреть всё вручную.

Нам нужен способ автоматически найти свойства объектов, на которых основная модель ошибается сильнее всего.

Идея шага:

> Использовать одну ML-модель, чтобы исследовать ошибки другой ML-модели.

---

# 2. Классический Adversarial Validation

Изначально adversarial validation используют для сравнения двух выборок:

```text
train
test
```

Мы создаём искусственный таргет:

```text
объект из train → класс 0
объект из test  → класс 1
```

После этого обучаем классификатор отличать train от test.

## Если ROC-AUC около 0.5

Классификатор почти не умеет различать выборки.

Это означает, что по доступным признакам train и test похожи.

## Если ROC-AUC заметно выше 0.5

Выборки различимы.

Значит, есть признаки, распределения которых отличаются.

Feature importance вспомогательного классификатора показывает, какие признаки сильнее всего выдают различие.

---

# 3. Как перенести идею на анализ ошибок

В нашем случае сравниваются не train и test.

Мы создаём две группы внутри одной выборки:

```text
класс 0 — обычные, относительно хорошо предсказанные объекты
класс 1 — worst cases, объекты с крупнейшими ошибками
```

Затем обучаем вспомогательный классификатор:

```text
X_test → является ли объект worst case?
```

Если классификатор умеет предсказывать worst cases, значит ошибки не случайны.

Они зависят от признаков.

Это очень важный вывод:

> В данных существует область, где основная модель систематически работает хуже.

---

# 4. Что означает ROC-AUC вспомогательной модели

## ROC-AUC около 0.5

Worst cases почти неотличимы от остальных объектов.

Возможные причины:

- ошибки близки к случайным;
- в `X_test` нет признаков, объясняющих ошибку;
- нужные признаки существуют, но не были переданы во вспомогательную модель;
- ошибки связаны с шумом таргета;
- классификатор слишком слабый.

Это не означает автоматически, что улучшить модель невозможно.

Это означает только:

> Текущий вспомогательный классификатор не нашёл устойчивый разделяющий паттерн.

## ROC-AUC выше 0.6

Worst cases уже в некоторой степени различимы.

Это сигнал, что:

- ошибки зависят от признаков;
- у основной модели есть зона слабости;
- можно исследовать feature importance;
- есть направление для feature engineering;
- возможно, нужна сегментация или отдельная модель.

## ROC-AUC очень высокий

Например:

```text
0.9+
```

Это может означать сильный системный паттерн.

Но нужно быть осторожным.

Возможные объяснения:

- действительно существует чёткий проблемный сегмент;
- во вспомогательные признаки попала утечка;
- оценка сделана на той же выборке, где модель обучалась;
- классификатор переобучился;
- классы разделяются техническим артефактом данных.

По условию задания ROC-AUC считается на полной выборке. Это удобно для учебной диагностики, но в production даёт оптимистичную оценку.

# 5. Что делает параметр `quantile`

Параметр задаёт долю крупнейших ошибок, которым присваивается класс 1.

Пример:

```python
quantile = 0.12
```

означает:

```text
12% объектов с крупнейшими абсолютными ошибками → класс 1
остальные 88% → класс 0
```

Если объектов 1000:

```text
top_k = floor(1000 × 0.12) = 120
```

Worst cases — 120 объектов.

---

# 6. Почему используются абсолютные остатки

Остаток может быть:

```text
+50
```

или:

```text
-50
```

Первый означает недооценку, второй — переоценку.

Но сила ошибки одинакова:

```text
50
```

Поэтому для выделения worst cases используется:

```python
np.abs(residual)
```

Без модуля большой отрицательный остаток мог бы ошибочно считаться маленьким.

# 7. Параметр `func`

`func` играет ту же роль, что и на прошлом шаге.

Допустимые варианты:

```text
None
"residuals"
"squared_errors"
"logloss"
"ape"
"quantile_loss"
```

Если:

```python
func is None
```

используется:

```python
residuals.residuals
```

Если передано:

```python
func="logloss"
```

используется:

```python
residuals.logloss
```

Функция выбирается динамически через:

```python
getattr(residuals, function_name)
```

# 8. Что возвращает функция

Результат:

```python
{
    "ROC-AUC": roc_auc,
    "feature_importances": feature_importances,
}
```

## `ROC-AUC`

Насколько хорошо вспомогательная модель отличает worst cases.

## `feature_importances`

Какие признаки сильнее всего помогают отличать плохо прогнозируемые объекты.

Для деревьев используется:

```python
classifier.feature_importances_
```

Для линейных классификаторов:

```python
abs(classifier.coef_[0])
```

Модуль нужен, потому что нас интересует сила влияния, а не направление коэффициента.

# 9. Бизнес-применение

## Retail и demand forecasting

Вспомогательная модель может показать, что worst cases связаны с:

- промо;
- OOS;
- новыми товарами;
- резкими изменениями цены;
- определёнными форматами магазинов;
- короткой историей продаж.

Это даёт конкретный backlog:

- улучшить промо-календарь;
- создать cold-start признаки;
- очищать факт от OOS;
- учитывать изменение цены;
- разделить модели по сегментам.

## Fintech

Worst cases скоринга могут быть связаны с:

- короткой кредитной историей;
- нестабильным доходом;
- новыми регионами;
- крупными лимитами;
- нестандартной занятостью.

Такой анализ нужно дополнять fairness, регуляторными проверками и out-of-time validation.

## Marketplace и delivery

Для ETA ошибки могут зависеть от:

- дождя;
- часа пик;
- расстояния;
- района;
- возраста ресторана;
- загрузки;
- типа курьера.

## Recommendations и dating

Ошибки могут концентрироваться на:

- новых пользователях;
- редких интересах;
- пользователях с короткой историей;
- непопулярном контенте;
- небольших городах.

## iGaming и casino

Worst cases могут концентрироваться на:

- новых игроках;
- high-value пользователях;
- резких изменениях поведения;
- vulnerable users.

Здесь нужны responsible gambling guardrails и регуляторный контроль.

## Crypto

Основные ошибки могут быть связаны с:

- высокой волатильностью;
- низкой ликвидностью;
- regime change;
- новостными шоками;
- резкими изменениями объёма.

## Industrial ML и aerospace

Для predictive maintenance worst cases могут зависеть от:

- типа оборудования;
- возраста агрегата;
- температуры;
- нагрузки;
- версии датчика;
- пропусков телеметрии.

# 10. Граница применимости модели

Вспомогательную модель можно превратить в error detector:

```text
низкий риск ошибки → автоматическое решение
высокий риск ошибки → fallback / human review / другая модель
```

Это превращает Error Analysis в часть архитектуры ML-системы.

---

# 11. Главный вывод

Adversarial Validation на остатках отвечает на вопрос:

> Можно ли по признакам заранее предсказать, что основная модель ошибётся?

Если да:

- ошибки систематичны;
- существует проблемная область данных;
- feature importance даёт направление исследования;
- можно улучшать признаки, лосс, сегментацию или routing;
- можно построить отдельный error detector.
