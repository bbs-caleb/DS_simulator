## 1. Что пошло не так — честный разбор

Тестер дал 70.86%. Разберём каждую проблему — **это важные концептуальные ошибки**, а не опечатки, поэтому стоит понять их до конца.

### 1.1 `logloss` — я поставил лишний минус (2.8 / 20)

В задании чёрным по белому написано:

> "То сами `(y·log(p) + (1−y)·log(1−p))` представляют собой псевдоостатки"

Это значит: псевдоостаток = **ровно** `y*log(p) + (1-y)*log(1-p)`, **без внешнего минуса**. Минус в формуле LogLoss стоит **снаружи суммы**, когда мы усредняем эти термины в единую метрику:

```
logloss = -(1/n) * Σ (y·log(p) + (1−y)·log(1−p))
                    └──────────────┬──────────────┘
                          вот это — псевдоостаток
```

Я по привычке "красивого положительного лосса" добавил минус **внутрь** поэлементной функции — это и есть ошибка. Псевдоостаток **сам по себе отрицателен** (т.к. логарифм вероятности ≤ 0), и это нормально: чем ближе к 0 (то есть чем меньше по модулю), тем лучше предсказание; чем более отрицательный — тем хуже.

**Было (неверно):**
```python
return -(y_true * np.log(p) + (1 - y_true) * np.log(1 - p))
```

**Стало (верно):**
```python
return y_true * np.log(y_pred) + (1 - y_true) * np.log(1 - y_pred)
```

### 1.2 `logloss` — я скрывал граничные случаи вместо того, чтобы их ловить

Я использовал `np.clip(y_pred, eps, 1-eps)`, чтобы `p=0` или `p=1` не ломали `log()`. Это стандартная практика **внутри готовых библиотек** (sklearn это тоже делает), но в данном задании API **явно требует другого поведения**: если `y_pred` содержит значение `0` или `1` — функция должна **упасть с `ValueError`**, а не молча "подрихтовать" данные.

Почему это принципиально с точки зрения дизайна библиотеки Error Analysis: если модель выдала вероятность **ровно 0 или 1**, это, скорее всего, признак проблемы в самой модели (переобучение, баг в calibration) — и **скрывать** это через `clip` означает прятать баг вместо того, чтобы явно сообщить о нём аналитику. Библиотека анализа ошибок должна быть **строгой** — лучше упасть с понятной ошибкой, чем незаметно исказить данные.

```python
if np.any(y_pred <= 0) or np.any(y_pred >= 1):
    raise ValueError("y_pred must be strictly between 0 and 1")
```

### 1.3 `ape` — я взял модуль там, где нужен был знак (8 / 20)

В задании написано:

> MAPE term → `1 − y_pred/y_true`

Без `|...|` вокруг. Название функции `ape` (Absolute Percentage Error) — сбивает с толку, потому что по классическому определению **метрики** MAPE действительно берётся модуль. Но здесь мы считаем не саму метрику, а **псевдоостаток** — а псевдоостаток **обязан сохранять знак**, иначе (как и в случае с `residuals`) мы теряем информацию "модель занизила / завысила", которая критична для Error Analysis (вспомни план модуля: смещение/bias — это первое, что мы будем анализировать в следующем шаге, а посчитать смещение по данным без знака невозможно).

**Было (неверно):**
```python
return np.abs((y_true - y_pred) / y_true)
```

**Стало (верно):**
```python
return 1 - y_pred / y_true
```

### 1.4 `ape` — тоже не хватало валидации

MAPE в принципе определена только для неотрицательных величин (иначе "процент от чего" теряет физический смысл — например, для ETA или продаж отрицательных значений быть не может в принципе). Тест ожидает `ValueError`, если `y_true` или `y_pred` отрицательны:

```python
if np.any(y_true < 0) or np.any(y_pred < 0):
    raise ValueError("y_true and y_pred must be non-negative")
```

## 2. Итоговый разбор исправленного кода

### `logloss` — построчно

```python
def logloss(y_true: np.ndarray, y_pred: np.ndarray) -> np.ndarray:
    """LogLoss pseudo-residuals: per-object contribution to logloss."""
    y_true = np.asarray(y_true, dtype=float)
    y_pred = np.asarray(y_pred, dtype=float)
    if np.any(y_pred <= 0) or np.any(y_pred >= 1):
        raise ValueError("y_pred must be strictly between 0 and 1")
    return y_true * np.log(y_pred) + (1 - y_true) * np.log(1 - y_pred)
```

1. `np.asarray(y_true, dtype=float)` — на входе может прийти обычный Python-список (`[1, 0, 1]`), а не numpy-массив. `np.asarray` гарантированно превращает вход в numpy-массив нужного типа (`float`), чтобы дальше работали векторные операции. Если на входе уже numpy-массив — функция просто ничего не копирует лишний раз (в отличие от `np.array`, `asarray` не создаёт копию, если тип уже совпадает — это чуть эффективнее).
2. `np.any(y_pred <= 0)` — сравнение `y_pred <= 0` поэлементно даёт массив `True/False`, а `np.any(...)` возвращает `True`, если **хотя бы один** элемент — `True`. То есть "если хотя бы одно предсказание ≤ 0 — это невалидный вход".
3. `raise ValueError(...)` — останавливаем выполнение с понятным сообщением об ошибке, а не пытаемся "додумать" за пользователя API, что делать с некорректным входом.
4. Финальная строка — прямая реализация формулы `y·log(p) + (1−y)·log(1−p)` без внешнего минуса, поэлементно для всего массива сразу.

### `ape` — построчно

```python
def ape(y_true: np.ndarray, y_pred: np.ndarray) -> np.ndarray:
    """MAPE pseudo-residuals: signed relative error."""
    y_true = np.asarray(y_true, dtype=float)
    y_pred = np.asarray(y_pred, dtype=float)
    if np.any(y_true < 0) or np.any(y_pred < 0):
        raise ValueError("y_true and y_pred must be non-negative")
    return 1 - y_pred / y_true
```

Логика идентична `logloss`: привести типы → провалидировать вход → вернуть формулу как есть, без модуля. Знак результата теперь несёт смысл: `ape > 0` → модель **занизила** прогноз (`y_pred < y_true`), `ape < 0` → модель **завысила** прогноз.
