# gp_page_6 — Best/Worst-Case Analysis: бизнесовый и технический контекст

## 1. Где мы находимся в Error Analysis

До этого момента мы рассматривали ошибки модели сверху вниз:

1. Посчитали остатки и псевдоостатки.
2. Проверили распределение ошибок.
3. Проверили смещение и гомоскедастичность.
4. Построили диагностические графики.
5. Проверили равномерность распределения ошибок через Residual Fairness.

Теперь мы спускаемся ещё на один уровень ниже.

Нас уже интересует не только:

> Как выглядит распределение ошибок в целом?

Нас интересует:

> Какие конкретно объекты модель предсказывает лучше всего и хуже всего?

Это переход от агрегированной диагностики к объектному анализу.

---

# 2. Почему одна агрегированная метрика не объясняет проблему

Предположим, модель прогнозирования спроса имеет:

```text
WAPE = 20%
```

Это число не отвечает на вопросы:

- Какие товары создают основную часть ошибки?
- Ошибка связана с промо?
- Модель ломается на новых товарах?
- Проблема только в маленьких магазинах?
- Самые большие ошибки возникают в выходные?
- Модель систематически ошибается на товарах с низкими продажами?
- Ошибки связаны с OOS?
- Есть ли общие признаки у объектов, где модель работает идеально?

Чтобы ответить, нужно достать конкретные строки исходных данных.

Именно это делают функции:

```python
best_cases(...)
worst_cases(...)
```

# 3. Best Case Analysis

## 3.1 Что такое best cases

Best cases — это объекты с минимальной абсолютной ошибкой.

Например:

| Товар | Реальные продажи | Прогноз | Абсолютная ошибка |
|---|---:|---:|---:|
| Молоко A | 100 | 101 | 1 |
| Хлеб B | 80 | 79 | 1 |
| Сок C | 50 | 52 | 2 |

Такие объекты показывают:

- где модель уже работает хорошо;
- для каких данных её предположения выполняются;
- на каких сегментах модель можно безопаснее раскатывать;
- какие признаки хорошо описывают поведение таргета;
- где находится текущая зона применимости модели.

---

## 3.2 Зачем анализировать хорошие случаи

На первый взгляд кажется, что нужно смотреть только на ошибки.

Но best cases отвечают на важный инженерный вопрос:

> Почему здесь модель работает?

Возможные закономерности:

- товары имеют длинную и стабильную историю;
- отсутствуют промо;
- нет OOS;
- спрос имеет выраженную недельную сезонность;
- магазины имеют стабильный трафик;
- пользователи давно находятся в системе;
- объекты не относятся к cold start;
- признаки заполнены качественно.

Это помогает понять, какие условия делают задачу предсказуемой.

---

## 3.3 Progressive rollout

Допустим, новая Transformer-модель улучшила общую метрику, но её поведение сильно отличается от production-модели.

Вместо немедленного rollout на 100% трафика можно:

1. Найти сегмент, где новая модель даёт минимальные ошибки.
2. Проверить устойчивость эффекта.
3. Раскатить модель только на этот сегмент.
4. Постепенно расширять область применения.

Так best-case analysis становится инструментом управления риском rollout.

# 4. Worst Case Analysis

## 4.1 Что такое worst cases

Worst cases — объекты с максимальной абсолютной ошибкой.

Пример:

| Товар | Реальные продажи | Прогноз | Абсолютная ошибка |
|---|---:|---:|---:|
| Торт Promo | 400 | 100 | 300 |
| Новый SKU | 180 | 20 | 160 |
| Салат Vegan | 220 | 90 | 130 |

Если в топе повторяются общие паттерны, появляется гипотеза о причине ошибки.

---

## 4.2 Как worst cases превращаются в работу над системой

Допустим, в топе худших прогнозов находятся:

- промо-товары;
- веганские блюда;
- новые рестораны.

Это может привести к конкретному backlog:

### Проблема: промо

Действия:

- добавить промо-календарь;
- добавить глубину скидки;
- учитывать тип механики;
- добавить историю промо uplift;
- разделить regular и promo forecast.

### Проблема: тип кухни

Действия:

- добавить cuisine type;
- добавить dietary flags;
- учитывать локальные тренды;
- добавить embedding блюда.

### Проблема: cold start

Действия:

- использовать признаки категории;
- использовать похожие объекты;
- применять иерархические priors;
- строить отдельную cold-start стратегию.

Worst-case analysis превращает абстрактную фразу:

```text
модель ошибается
```

в конкретную:

```text
модель не умеет прогнозировать промо новых товаров
```

# 5. Corner Case Analysis

## 5.1 Отличие от worst-case analysis

Worst-case analysis задаёт вопрос:

> Какие признаки объединяют объекты с самыми большими ошибками?

Corner-case analysis задаёт обратный вопрос:

> Какие ошибки возникают у заранее выбранной группы объектов?

То есть направление анализа меняется.

---

## 5.2 Примеры corner cases

Можно заранее выбрать:

- новые товары;
- новые пользователи;
- маленькие магазины;
- определённый регион;
- промо;
- высокий чек;
- редкую категорию;
- пользователей из конкретного рекламного канала;
- ночные заказы;
- плохую погоду;
- товары около нулевого спроса.

После этого внутри сегмента ищутся лучшие или худшие случаи.

# 6. Роль параметра `mask`

Параметр `mask` — это булева маска pandas.

Пример:

```python
mask = X_test["is_promo"] == 1
```

Она выглядит примерно так:

```text
0     False
1      True
2     False
3      True
```

При передаче маски функция сначала оставляет только нужный сегмент:

```text
только промо-объекты
```

а уже затем выбирает top-k.

Это важно.

Нельзя сначала выбрать худшие случаи по всему датасету, а затем применить маску. В таком случае в выбранном сегменте может не оказаться нужного числа объектов.

Правильный порядок:

```text
1. Отфильтровать сегмент.
2. Посчитать остатки внутри сегмента.
3. Отсортировать.
4. Взять top-k.
```

# 7. Почему сортируем по модулю

Обычный остаток:

```text
residual = y_true - y_pred
```

может быть положительным или отрицательным.

Пример:

```text
-30 и +30
```

Оба значения означают одинаковую силу ошибки:

```text
30
```

Различается только направление:

- `+30` — недопрогноз;
- `-30` — перепрогноз.

Для best/worst selection нужен размер ошибки, поэтому используется:

```python
np.abs(residual)
```

В результате:

```text
|-30| = 30
|+30| = 30
```

# 8. Почему нужно переиспользовать модуль `residuals`

В первом задании уже были реализованы функции:

```python
residuals
squared_errors
logloss
ape
quantile_loss
```

Плохой вариант — копировать формулы заново.

Почему:

- появляется дублирование;
- исправление нужно делать в нескольких местах;
- реализации могут разойтись;
- растёт риск багов;
- нарушается принцип единственного источника истины.

Правильный подход:

```python
import residuals
```

и затем динамически получить функцию:

```python
getattr(residuals, function_name)
```

---

# 9. Что такое `getattr`

Пусть есть модуль:

```python
import residuals
```

И строка:

```python
function_name = "logloss"
```

Вызов:

```python
getattr(residuals, function_name)
```

эквивалентен:

```python
residuals.logloss
```

То есть строка превращается в ссылку на функцию.

После этого функцию можно вызвать:

```python
residual_function(y_test, y_pred)
```

Это позволяет не писать длинную цепочку:

```python
if func == "residuals":
    ...
elif func == "logloss":
    ...
elif func == "ape":
    ...
```

# 10. Применение в разных индустриях

## 10.1 Retail

Worst cases:

- товары с промо;
- fresh-категории;
- OOS;
- новые SKU;
- магазины после открытия;
- локальный ассортимент.

Best cases:

- стабильные товары;
- регулярный спрос;
- длинная история;
- отсутствие промо;
- высокая доступность.

---

## 10.2 Fintech

Worst cases скоринга:

- thin-file клиенты;
- новые регионы;
- нестандартный доход;
- крупные кредиты;
- нестабильная занятость.

Corner cases:

```python
mask = X_test["credit_history_months"] == 0
```

Анализируется ошибка только для клиентов без истории.

---

## 10.3 Marketplace

Worst cases ETA:

- дождь;
- час пик;
- удалённые районы;
- новые рестораны;
- batch delivery.

Corner case:

```python
mask = X_test["restaurant_age_days"] < 30
```

---

## 10.4 Recommendations и dating

Worst cases:

- новые пользователи;
- редкие интересы;
- малопопулярный контент;
- неполные профили;
- смена поведения.

Best cases помогают определить сегмент, где recommender уже хорошо персонализирован.

---

## 10.5 iGaming и casino

Нужно отдельно анализировать:

- high-value users;
- новых игроков;
- пользователей с рискованным поведением;
- резкие изменения активности.

Важно: лучшие бизнес-прогнозы не должны достигаться ценой вреда vulnerable-сегментам.

---

## 10.6 Crypto

Worst cases могут концентрироваться:

- в периоды высокой волатильности;
- на low-liquidity инструментах;
- во время market shocks;
- при regime change.

---

## 10.7 Industrial ML и aerospace

Worst cases predictive maintenance могут показывать:

- редкий тип оборудования;
- аномальные режимы температуры;
- недостаток истории;
- конкретную версию датчика;
- определённый режим нагрузки.

Здесь один worst case может быть важнее средней метрики по миллионам нормальных наблюдений.

# 11. Главное мышление специалиста

После получения top-k нельзя просто показать таблицу.

Нужно пройти цикл:

```text
Наблюдение
    ↓
Повторяющийся паттерн
    ↓
Гипотеза причины
    ↓
Проверка гипотезы
    ↓
Изменение данных/признаков/модели
    ↓
Повторный error analysis
```

Best/Worst-Case Analysis — не конечный отчёт, а генератор проверяемых гипотез.
