# gp_page_7 — Adversarial Validation на продуктовом и System Design интервью

## 1. Ответ за 30 секунд

> Я выделяю top-q объектов с максимальной абсолютной ошибкой, присваиваю им класс 1, остальным класс 0 и обучаю вспомогательный классификатор по исходным признакам. ROC-AUC показывает, насколько worst cases систематически отличаются от остальных объектов. Feature importance помогает найти признаки, связанные с областью слабости основной модели.

---

# 2. Как интерпретировать ROC-AUC

## Около 0.5

В доступных признаках нет явного разделяющего сигнала или вспомогательная модель его не нашла.

## 0.6–0.7

Есть заметный паттерн. Стоит исследовать важные признаки и сегменты.

## Очень высокий ROC-AUC

Worst cases сильно отличаются, но нужно проверить:

- переобучение;
- leakage;
- артефакты;
- качество валидации;
- стабильность во времени.

---

# 3. Почему это называется adversarial validation

В классическом варианте модель различает:

```text
train vs test
```

В анализе ошибок она различает:

```text
normal cases vs worst cases
```

В обоих случаях классификатор ищет признаки, по которым две выборки отличаются.

---

# 4. Почему нельзя просто смотреть feature importance основной модели

> Feature importance основной модели объясняет, какие признаки важны для предсказания таргета. Feature importance вспомогательной модели объясняет, какие признаки связаны с вероятностью ошибки основной модели. Это разные задачи.

Пример:

```text
price
```

может быть важен для прогноза спроса.

Но:

```text
days_since_launch
```

может быть главным признаком, объясняющим крупные ошибки.

---

# 5. Доказывает ли importance причину ошибки

Нет.

Он показывает предсказательную ассоциацию.

Признак может:

- быть proxy;
- коррелировать с настоящей причиной;
- отражать утечку;
- быть следствием другого процесса.

После adversarial validation нужна проверка гипотезы через:

- сегментный анализ;
- абляцию признаков;
- временную валидацию;
- эксперимент;
- дополнительный RCA.

---

# 6. Пример для demand forecasting

> Я беру top-10% SKU-store-day по абсолютной ошибке, обучаю классификатор отличать их от остальных и смотрю feature importance. Если важны promo, OOS и item age, я отдельно проверяю эти сегменты. Затем могу добавить промо-признаки, очистить target от OOS и создать cold-start модель.

---

# 7. Пример для fintech

> Я определяю worst cases по individual log-loss, строю вспомогательный классификатор и проверяю, связаны ли ошибки с thin-file клиентами, регионом или нестабильным доходом. Но дополнительно провожу fairness analysis и out-of-time validation, потому что высокий training ROC-AUC сам по себе недостаточен.

---

# 8. Пример для ETA

> Worst-case classifier может выявить дождь, час пик, расстояние и новый ресторан. Это показывает, где ETA-модель систематически недооценивает сложность заказа. Дальше я добавляю real-time нагрузку, погодные признаки или отдельный routing.

---

# 9. Как использовать результат в production

Adversarial classifier можно превратить в error-risk model.

```text
Основная модель даёт прогноз
           ↓
Error-risk model оценивает риск крупной ошибки
           ↓
Низкий риск → автоматическое решение
Высокий риск → fallback / human review / другой алгоритм
```

Примеры fallback:

- более консервативный прогноз;
- prediction interval;
- ручная проверка;
- отдельная модель сегмента;
- бизнес-эвристика;
- отказ от автоматического решения.

---

# 10. Production-архитектура

```text
Features
   ├── Main model → prediction
   └── Error model → probability of large error
                         ↓
                 Decision router
                 ├── normal path
                 └── fallback path
```

Нужно логировать:

- версию основной модели;
- версию error model;
- вероятность ошибки;
- фактическую ошибку;
- выбранный routing;
- бизнес-результат;
- segment;
- drift.

---

# 11. Почему training ROC-AUC недостаточно

В учебной задаче ROC-AUC считается на тех же данных, где классификатор обучен.

В production это может привести к завышенной оценке.

Правильнее использовать:

- cross-validation;
- out-of-fold predictions;
- temporal split;
- holdout;
- проверку stability;
- bootstrap confidence interval.

---

# 12. Leakage в анализе ошибок

Особенно опасно передать вспомогательной модели признаки, появившиеся после момента прогноза.

Примеры утечки:

- итоговый OOS, известный после периода;
- фактическое время доставки;
- статус дефолта после выдачи;
- будущая отмена;
- ручной override, сделанный после прогноза.

Error classifier покажет высокий ROC-AUC, но не сможет работать в момент реального решения.

---

# 13. Какие признаки можно использовать

## Режим диагностики

Можно использовать более широкий набор признаков, включая те, которые помогают RCA, но недоступны онлайн.

Цель:

```text
понять причину постфактум
```

## Режим production error detector

Можно использовать только признаки, доступные на момент основного прогноза.

Цель:

```text
предсказать риск ошибки заранее
```

Эти режимы нельзя смешивать.

---

# 14. Как выбирать quantile

Маленький quantile:

- концентрируется на экстремальных ошибках;
- создаёт сильный дисбаланс;
- может быть нестабилен.

Большой quantile:

- даёт больше объектов класса 1;
- включает умеренные ошибки;
- может размыть паттерн.

Выбор зависит от:

- бизнес-цены ошибки;
- объёма данных;
- стабильности хвоста;
- возможности обработки проблемных объектов;
- требуемого recall error detector.

---

# 15. Какие метрики нужны кроме ROC-AUC

Для production error detector полезны:

- PR-AUC;
- recall worst cases;
- precision;
- recall@capacity;
- lift in top decile;
- calibration;
- cost-based metric;
- coverage;
- доля ошибок, пойманных fallback-процессом.

ROC-AUC может выглядеть хорошо даже при редком положительном классе, поэтому одной метрики недостаточно.

---

# 16. Сильная итоговая формулировка

> Adversarial validation на остатках превращает неструктурированный список ошибок в задачу классификации. Оно показывает, являются ли крупные ошибки предсказуемыми по признакам, и помогает найти границу применимости основной модели. Но feature importance — это источник гипотез, а не причинное доказательство, и production-решение требует OOF-валидации, проверки leakage и бизнес-метрик.
