# gp_page_1 — ML: COCA-COLA

## Что это за страница

Это задача про постпроцессинг цен после ML-модели динамического ценообразования.

Модель может предсказать цены, которые математически выглядят нормально для модели, но нарушают бизнес-логику:

- большая упаковка не должна стоить дешевле маленькой;
- большая упаковка должна быть выгоднее в пересчёте на литр / грамм / штуку;
- товары одного объёма внутри горизонтальной линейки должны стоить одинаково.

## Главная идея решения

Есть два типа линеек.

### Вертикальная линейка

Один продукт в разных объёмах:

```text
0.5 л → 1.0 л → 2.0 л
```

Для соседних объёмов нужно:

```text
price_left < price_right
```

и:

```text
price_left / netto_left > price_right / netto_right
```

То есть большая упаковка стоит дороже, но дешевле за единицу объёма.

### Горизонтальная линейка

Товары одного объёма должны стоить одинаково:

```text
яблочный сок 250 мл = апельсиновый сок 250 мл = мультифрукт 250 мл
```

## Что минимизируется

Сначала минимизируется сумма абсолютных изменений:

```text
sum(abs(new_price - old_price))
```

Потом среди решений с одинаковой суммой минимизируется число изменённых цен:

```text
count(new_price != old_price)
```

## Важный API-контракт

Метод `transform` должен вести себя так, как в условии:

```python
df = self.fix_vlines(df)
df = self.fix_hlines(df)
return df
```

Это нельзя менять. Grader проверяет порядок вызовов.

## Главный профессиональный вывод

Production ML — это не только `model.predict`.

Модель предлагает цену, но финальная система должна применить бизнес-ограничения, проверить результат и только потом публиковать цену.
