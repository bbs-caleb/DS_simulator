# Страница 6. Masked Multi-Head Attention в GPT-2

## 1. Что реализуется

На предыдущей странице был обычный Self-Attention:

```text
Q, K, V
  ↓
QKᵀ / √dₖ
  ↓
Softmax
  ↓
взвешенная сумма V
```

GPT-2 использует более сложную версию:

```text
Masked Multi-Head Self-Attention
```

Нужно добавить две идеи:

1. **Causal mask** — каждый токен видит только себя и прошлые токены.
2. **Несколько голов** — embedding делится на независимые части, каждая из которых вычисляет собственное внимание.

Полный поток:

```text
x
 ↓
Linear(c_attn)
 ↓
Q, K, V
 ↓
разделение Q/K/V на n_head частей
 ↓
masked attention в каждой голове
 ↓
конкатенация голов
 ↓
Linear(c_proj)
 ↓
MHA output
```

---

# 2. Почему GPT нельзя смотреть в будущее

GPT обучается предсказывать следующий токен.

Для последовательности:

```text
The cake was sour
```

позиция `The` не должна видеть `cake`, `was`, `sour`.

Позиция `cake` может видеть:

```text
The cake
```

Позиция `was`:

```text
The cake was
```

Если будущие токены доступны, возникает утечка ответа. На обучении качество будет искусственно высоким, а при генерации модель не сможет повторить такой режим, потому что будущих токенов ещё нет.

Именно поэтому GPT относится к decoder-only архитектурам и использует causal attention.

---

# 3. Encoder Attention и Decoder Attention

## Encoder-like модель

BERT-подобная модель обычно использует двунаправленное внимание. Токен может смотреть и влево, и вправо.

Это удобно для:

- классификации;
- NER;
- извлечения информации;
- понимания уже готового текста;
- анализа тональности.

## Decoder-like модель

GPT генерирует слева направо. Токен может использовать:

- себя;
- все предыдущие позиции.

Будущие позиции запрещены.

---

# 4. Causal mask

Для четырёх токенов разрешённые связи выглядят так:

```text
[
    [1, 0, 0, 0],
    [1, 1, 0, 0],
    [1, 1, 1, 0],
    [1, 1, 1, 1],
]
```

NumPy создаёт нижнетреугольную матрицу:

```python
np.tri(n_seq)
```

Чтобы получить additive mask:

```python
causal_mask = (1 - np.tri(n_seq)) * -1e10
```

Результат:

```text
[
    [0, -1e10, -1e10, -1e10],
    [0,     0, -1e10, -1e10],
    [0,     0,     0, -1e10],
    [0,     0,     0,     0],
]
```

Разрешённые позиции добавляют 0.

Запрещённые позиции получают очень большое отрицательное число.

---

# 5. Почему маска добавляется до Softmax

Формула:

\[
Attention(Q,K,V,M)
=
softmax
\left(
\frac{QK^T}{\sqrt{d_k}} + M
\right)V
\]

При запрещённой позиции:

\[
e^{-10^{10}} \approx 0
\]

Следовательно, вероятность будущего токена практически равна нулю.

Правильно:

```python
softmax(scores + mask)
```

Неправильно:

```python
softmax(scores) + mask
```

Неправильно:

```python
softmax(scores) * mask
```

После Softmax распределение уже нормировано, поэтому причинное ограничение должно влиять именно на logits.

---

# 6. Masked Attention

```python
def attention(q, k, v, mask):
    scores = q @ k.T / np.sqrt(q.shape[-1])
    return softmax(scores + mask) @ v
```

Формы:

```text
q:    [n_q, d_k]
k:    [n_k, d_k]
v:    [n_k, d_v]
mask: [n_q, n_k]
```

После `q @ k.T`:

```text
[n_q, n_k]
```

После умножения весов на `v`:

```text
[n_q, d_v]
```

---

# 7. Causal Self-Attention

Сначала строится общая QKV-проекция:

```python
x = linear(x, **c_attn)
```

Форма:

```text
[n_seq, n_embd]
→
[n_seq, 3*n_embd]
```

Затем:

```python
q, k, v = np.split(x, 3, axis=-1)
```

Получаются три матрицы:

```text
q: [n_seq, n_embd]
k: [n_seq, n_embd]
v: [n_seq, n_embd]
```

Создаётся causal mask, выполняется attention и output projection:

```python
x = attention(q, k, v, causal_mask)
return linear(x, **c_proj)
```

---

# 8. Зачем несколько голов

Embedding GPT-2 содержит множество типов информации:

- грамматику;
- синтаксические связи;
- сущности;
- местоименные ссылки;
- эмоциональный оттенок;
- отрицания;
- числовые отношения;
- форматирование;
- дальние зависимости.

Одна Attention-матрица не обязана одинаково хорошо описывать все эти виды связей.

Multi-Head Attention позволяет нескольким подпространствам embedding независимо строить свои Attention-веса.

---

# 9. Размер одной головы

Пусть:

```text
n_embd = 768
n_head = 12
```

Тогда:

\[
d_{head} = 768 / 12 = 64
\]

Q, K и V формы:

```text
[n_seq, 768]
```

делятся на:

```text
12 × [n_seq, 64]
```

`n_embd` должен делиться на `n_head` без остатка.

---

# 10. Построение qkv_heads

После:

```python
qkv = np.split(x, 3, axis=-1)
```

структура:

```text
qkv = [Q, K, V]
```

Каждая матрица делится на головы:

```python
split_qkv = [
    np.split(part, n_head, axis=-1)
    for part in qkv
]
```

Получается:

```text
[
    [q₀, q₁, ..., qₕ₋₁],
    [k₀, k₁, ..., kₕ₋₁],
    [v₀, v₁, ..., vₕ₋₁],
]
```

Но Attention нужно запускать тройками одной головы:

```text
(q₀, k₀, v₀)
(q₁, k₁, v₁)
...
```

Поэтому:

```python
qkv_heads = list(zip(*split_qkv))
```

Результат имеет требуемый тип:

```text
List[tuple]
```

---

# 11. Attention в каждой голове

```python
out_heads = [
    attention(q, k, v, causal_mask)
    for q, k, v in qkv_heads
]
```

Каждая голова использует:

- свои Q;
- свои K;
- свои V;
- одну и ту же causal mask.

Маска одинакова, потому что разрешённость зависит от позиции токена, а не от номера головы.

---

# 12. Объединение голов

Каждый результат:

```text
[n_seq, d_head]
```

Головы соединяются по последней оси:

```python
x = np.hstack(out_heads)
```

Форма:

```text
[n_seq, n_head*d_head]
=
[n_seq, n_embd]
```

После этого применяется:

```python
linear(x, **c_proj)
```

`c_proj` смешивает признаки, полученные разными головами.

---

# 13. Почему не vstack

`np.vstack` добавляет строки:

```text
[n_head*n_seq, d_head]
```

Это разрушит структуру последовательности.

`np.hstack` сохраняет токены в строках и соединяет признаки голов по столбцам.

# 15. KV cache

При генерации прошлые токены не меняются.

На каждом Transformer-слое можно сохранить:

```text
K предыдущих токенов
V предыдущих токенов
```

Для нового токена вычислить:

```text
q_new
k_new
v_new
```

Новый Query сравнивается с накопленными Keys.

Values используются для взвешенной суммы.

Прошлые Queries хранить не требуется.

Это называется **KV cache**.

Плюс:

- ускоряет autoregressive inference.

Минус:

- занимает память;
- растёт с контекстом, batch size и числом слоёв.

---

# 16. Сложность

Для последовательности длины \(n\) Attention-матрица имеет:

```text
[n, n]
```

Память:

\[
O(n^2)
\]

Вычисления:

\[
O(n^2d)
\]

При удвоении длины контекста число элементов Attention-матрицы растёт примерно в четыре раза.

Современные оптимизации:

- FlashAttention;
- sliding-window attention;
- sparse attention;
- grouped-query attention;
- KV cache;
- memory-efficient kernels.

# 17. Критические ошибки

1. Добавить mask после Softmax.
2. Использовать 0/1 mask без большого отрицательного штрафа.
3. Создать верхнетреугольную разрешающую матрицу вместо запрещающей.
4. Делить Q/K/V по `axis=0`.
5. Смешать `q_i`, `k_j`, `v_k` разных голов.
6. Использовать `np.vstack`.
7. Суммировать головы вместо конкатенации.
8. Применять `c_proj` отдельно к каждой голове.
9. Делить scores на `sqrt(n_embd)`, а не на `sqrt(d_head)`.
10. Забыть output projection.

# 19. Ответ на интервью

## Что делает causal mask?

Causal mask запрещает токену на позиции `i` использовать Keys позиций `j > i`. Большое отрицательное значение прибавляется к logits до Softmax, поэтому вероятность будущих токенов становится практически нулевой.

## Что делает Multi-Head Attention?

Hidden dimension делится на несколько голов. Каждая голова независимо вычисляет Attention и может изучать свой тип связи. Результаты голов конкатенируются и смешиваются output projection.

## Что хранится в KV cache?

Keys и Values всех прошлых токенов для каждого слоя и каждой головы. Прошлые Queries для будущей генерации не нужны.

# 20. Финальная схема

```text
x: [n_seq, n_embd]
    ↓
Linear c_attn
    ↓
[n_seq, 3*n_embd]
    ↓
Q, K, V
    ↓
каждая матрица делится на n_head частей
    ↓
[(q₀,k₀,v₀), ..., (qₕ₋₁,kₕ₋₁,vₕ₋₁)]
    ↓
masked attention для каждой головы
    ↓
n_head × [n_seq, d_head]
    ↓
np.hstack
    ↓
[n_seq, n_embd]
    ↓
Linear c_proj
    ↓
[n_seq, n_embd]
```
