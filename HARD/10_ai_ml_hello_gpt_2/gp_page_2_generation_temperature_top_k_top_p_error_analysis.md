# Разбор ошибки после первой отправки

## Результат первой попытки

Первая версия получила **40 из 100 баллов**. Прошли импорт, сигнатура, форма результата, воспроизводимость, случайность и Pylint. Не прошли проверки `temperature`, `top_k` и `top_p`.

Это означает, что общий авторегрессивный цикл был правильным, но реализация фильтрации отличалась от конкретной логики, которую ожидают тесты курса.

## 1. Почему упал `test_top_p`

В первой версии было ограничение:

```python
if not 0 < top_p <= 1:
    raise ValueError(...)
```

Однако тесты передают `top_p=0`. В этой задаче ноль допустим и должен приводить к сохранению только самого вероятного токена.

Поэтому проверка, запрещающая `top_p=0`, удалена.

## 2. Почему не совпали частоты `temperature` и `top_k`

Первая версия сортировала вероятности так:

```python
np.argsort(-probabilities, kind="stable")
```

Математически это корректно. Но тесты курса ожидают конкретный порядок индексов:

```python
indices = np.argsort(probabilities)[-top_k:]
probabilities = probabilities[indices]
probabilities /= probabilities.sum()

order = np.argsort(probabilities)[::-1]
indices = indices[order]
probabilities = probabilities[order]
```

Разница проявляется при равных вероятностях. С фиксированным `random_state` порядок кандидатов влияет на то, как случайные числа сопоставляются token ID. Поэтому частоты отдельных токенов в тестах отличаются, даже если общий математический смысл похож.

## 3. Правильный порядок операций

```text
llm(input_ids)
    ↓
деление логитов на temperature
    ↓
Softmax
    ↓
top-k
    ↓
нормализация
    ↓
сортировка кандидатов по убыванию вероятности
    ↓
накопленная сумма
    ↓
первый порог, который достиг или превысил top-p
    ↓
нормализация
    ↓
np.random.choice
    ↓
добавление token ID в input_ids
```

## 4. Как теперь реализован `top_p`

```python
cumulative_probabilities = np.cumsum(probabilities)
threshold = np.min(
    cumulative_probabilities[
        cumulative_probabilities >= top_p
    ]
)
mask = cumulative_probabilities <= threshold
```

Пример:

```text
Вероятности:             [0.50, 0.30, 0.15, 0.05]
Накопленные вероятности: [0.50, 0.80, 0.95, 1.00]
top_p:                    0.75
```

Первое значение, достигшее порога, равно `0.80`. Поэтому остаются первые два токена.

При `top_p=0` первым подходящим порогом становится накопленная вероятность первого токена. Следовательно, остается только наиболее вероятный токен.

## 5. Как обрабатывается `top_k=0`

Используется:

```python
indices = np.argsort(probabilities)[-top_k:]
```

При `top_k=0` срез превращается в `[0:]`, то есть сохраняет весь словарь. В рамках ожидаемой реализации это означает отключение top-k фильтрации.

## 6. Что осталось правильным

Не изменились:

- повторный вызов `llm` на каждой итерации;

- добавление нового ID в `input_ids`;
- возврат строковых токенов;
- исходная сигнатура функции.
