# GP — страница 1: бизнесовый и технический контекст APPROXIMATE KNN

# 1. Какую реальную проблему мы решаем

Представим, что у продукта есть один пользовательский запрос и огромная коллекция объектов.

Объектами могут быть:

- товары;
- фильмы;
- песни;
- объявления;
- анкеты;
- вакансии;
- резюме;
- банковские транзакции;
- изображения;
- фрагменты документов;
- игровые сессии;
- технические инциденты;
- телеметрические последовательности.

Модель преобразует каждый объект в набор чисел — **вектор**, или **эмбеддинг**.

```text
товар A → [0.12, -0.44, 0.91, ...]
товар B → [0.10, -0.41, 0.88, ...]
товар C → [-0.82, 0.17, -0.10, ...]
```

Если embedding-модель обучена хорошо, похожие по смыслу объекты располагаются рядом в векторном пространстве.

Тогда поиск похожих объектов превращается в задачу:

> Найти K векторов, расположенных ближе всего к вектору запроса.

Это и есть задача ближайших соседей.

---

# 2. KNN здесь — не только классификатор

В базовом ML KNN часто объясняют как классификатор:

1. Берём новый объект.
2. Находим K ближайших обучающих объектов.
3. Смотрим их классы.
4. Выбираем класс голосованием.

В этой задаче используется более фундаментальный примитив:

```text
найти K ближайших векторов
```

После этого найденных соседей можно применять для:

- поиска;
- рекомендаций;
- matching;
- классификации;
- дедупликации;
- антифрода;
- поиска аномалий;
- Retrieval-Augmented Generation;
- генерации кандидатов перед ранжированием.

Правильное профессиональное мышление:

> Мы строим инфраструктурный слой поиска кандидатов в пространстве эмбеддингов.

---

# 3. Exact KNN

Самый простой алгоритм:

1. Получить вектор запроса.
2. Вычислить расстояние от запроса до каждого документа.
3. Отсортировать расстояния.
4. Вернуть первые K индексов.

Пусть:

- `N` — число документов;
- `D` — размерность embedding.

Вычисление расстояний требует примерно:

```text
O(N × D)
```

Полная сортировка:

```text
O(N log N)
```

На небольших данных это отличный baseline.

На большой коллекции возникает проблема.

Пример:

```text
N = 100 000 000 документов
D = 768 признаков
QPS = 5 000 запросов в секунду
```

Сравнивать каждый запрос со ста миллионами векторов слишком дорого по времени и инфраструктуре.

---

# 4. Как эта проблема выглядит для бизнеса

Бизнес не говорит:

> Нам нужна асимптотика лучше O(ND).

Он говорит:

- поиск стал медленным;
- рекомендации не успевают сформироваться;
- пользователь покидает экран до загрузки;
- RAG-ассистент отвечает слишком долго;
- подбор вакансий не укладывается в SLA;
- CPU/GPU стоят слишком дорого;
- система не выдерживает пиковый трафик;
- векторный индекс не помещается в память.

Задача DS/MLE — перевести бизнесовый симптом в техническую причину.

```text
Симптом:
p95 latency поиска вырос с 120 до 800 мс.

Причина:
размер каталога вырос, а retrieval делает полный перебор.

Решение-кандидат:
ANN-индекс, сокращающий число вычисляемых расстояний.
```

---

# 5. Что означает Approximate

ANN — Approximate Nearest Neighbors — не обещает всегда вернуть математически точный top-K.

Он обещает практический компромисс:

> Просмотреть небольшую часть пространства и быстро найти очень хорошие ближайшие точки.

Пример:

```text
Exact KNN:
Recall@10 = 1.00
latency = 600 мс

Approximate KNN:
Recall@10 = 0.96
latency = 12 мс
```

В продукте второй вариант часто лучше.

Пользователю обычно не критично, был ли товар математически точным девятым или одиннадцатым соседом среди миллионов товаров. Ему важно быстро получить полезную выдачу.

---

# 6. Recall@K

Пусть exact-поиск нашёл:

```text
{2, 5, 9, 12, 18, 23, 27, 31, 44, 70}
```

ANN вернул:

```text
{2, 5, 9, 12, 18, 23, 27, 31, 51, 88}
```

Совпало восемь объектов.

```text
Recall@10 = 8 / 10 = 0.8
```

Формула:

```text
Recall@K = |exact_top_K ∩ ann_top_K| / K
```

Для ANN почти всегда анализируют trade-off:

```text
Recall@K ↔ latency
```

Чем больше точек исследуем, тем обычно выше Recall, но тем больше задержка.

---

# 7. Что такое embedding

Embedding — числовое представление объекта.

Пример запроса:

```text
"лёгкие кроссовки для бега"
```

Модель превращает его в вектор:

```text
[0.31, -0.18, 0.72, ..., -0.09]
```

Той же или совместимой моделью кодируются товары.

Очень важная мысль:

> ANN не создаёт смысл. Смысл создаёт embedding-модель.

ANN только быстро ищет точки, которые модель уже расположила рядом.

Плохая выдача может быть вызвана:

1. плохими эмбеддингами;
2. плохим ANN-поиском;
3. неверной метрикой;
4. несовместимыми версиями embedding-модели;
5. устаревшим индексом;
6. неверными фильтрами;
7. плохим reranker;
8. ошибочной бизнес-логикой.

Нельзя автоматически обвинять векторный индекс.

# 8. Евклидово расстояние L2

Для точек:

```text
A = [a1, a2, ..., aD]
B = [b1, b2, ..., bD]
```

L2-расстояние:

```text
sqrt((a1-b1)² + (a2-b2)² + ... + (aD-bD)²)
```

Пример:

```text
A = [1, 2]
B = [4, 6]
```

```text
sqrt((1-4)² + (2-6)²)
= sqrt(9 + 16)
= 5
```

Чем меньше расстояние, тем ближе точки.

---

# 9. Почему масштаб признаков важен

Если использовать обычные признаки:

```text
age = 25
income = 10 000 000
```

без масштабирования, доход почти полностью определит L2-расстояние.

Поэтому для distance-based методов часто нужны:

- стандартизация;
- нормализация;
- осознанное взвешивание признаков.

Для нейросетевых эмбеддингов метрика должна соответствовать objective обучения. Нельзя произвольно заменить cosine на L2 и считать, что смысл сохранится.

Если векторы L2-нормированы, cosine similarity и L2 могут задавать одинаковый порядок соседей, но это нужно понимать и проверять, а не предполагать.

# 10. Почему граф помогает

Каждый документ становится вершиной графа.

Между некоторыми вершинами проводятся рёбра.

Во время поиска можно:

1. начать с нескольких случайных вершин;
2. вычислить расстояния до их соседей;
3. перейти к соседу, который ближе к запросу;
4. повторять переходы;
5. остановиться, когда улучшения больше нет.

Аналогия:

- полный перебор — измерять расстояние от вас до каждого дома города;
- графовый поиск — двигаться по дорогам в направлении нужного района.

---

# 11. Small World

Сеть малого мира сочетает:

1. **Локальную кластеризацию** — близкие вершины часто связаны.
2. **Короткие глобальные пути** — существуют связи между далёкими областями.

Социальная аналогия:

- у человека много друзей из локального окружения;
- некоторые друзья связаны с другими странами и профессиональными кругами;
- через небольшое число знакомств можно добраться до далёкой части сети.

В SW-графе:

- короткие рёбра связывают похожие объекты;
- длинные рёбра служат быстрыми переходами.

---

# 12. Короткие рёбра

Короткие рёбра дают локальную точность.

Пример движения:

```text
обувь
→ спортивная обувь
→ беговые кроссовки
→ лёгкие беговые кроссовки
```

В коде:

```python
short_cands = argsorted[:num_candidates_for_choice_short]
```

Берётся начало списка, отсортированного по расстоянию.

Затем выбирается несколько рёбер:

```python
short_choice = np.random.choice(
    short_cands,
    size=num_edges_short,
    replace=False
)
```

---

# 13. Длинные рёбра

Длинные рёбра нужны для глобального перемещения.

Если старт попал в область кухонной техники, а запрос относится к ноутбукам, одних локальных шагов может быть недостаточно.

В коде:

```python
long_cands = argsorted[-num_candidates_for_choice_long:]
```

Берётся хвост отсортированного списка — самые далёкие кандидаты.

Затем выбирается несколько длинных связей.

Это учебная конструкция. Production HNSW строит связи сложнее и не сводится буквально к случайному выбору самых далёких точек.

---

# 14. Почему выбор случайный

Случайность создаёт разные маршруты и разнообразную структуру графа.

Следствия:

- два графа могут различаться;
- два запуска поиска могут вернуть разные индексы;
- для сравнимого эксперимента нужен `np.random.seed(...)`;
- параметры и seed нужно логировать.

В production индекс должен быть версионирован, а сравнение вариантов — воспроизводимо.

---

# 15. Направленный граф

Учебное решение добавляет:

```text
i → j
```

но автоматически не добавляет:

```text
j → i
```

То есть граф направленный.

Добавление обратных рёбер изменило бы:

- число соседей;
- маршруты;
- структуру графа;
- результаты скрытых тестов.

---

# 16. Как строится граф

Для каждой точки:

1. Считаются расстояния до остальных точек.
2. Индексы сортируются по расстоянию.
3. Первая позиция — сама точка — удаляется.
4. Выбирается пул ближайших кандидатов.
5. Из него случайно выбираются короткие рёбра.
6. Выбирается пул самых далёких кандидатов.
7. Из него случайно выбираются длинные рёбра.
8. Все выбранные индексы сохраняются в словарь.

Если:

```text
num_edges_short = 5
num_edges_long = 5
```

то при достаточном количестве кандидатов каждая вершина получает десять исходящих рёбер.

---

# 17. Сложность построения

Для каждой из `N` точек считаются расстояния до `N` точек:

```text
O(N²D)
```

Для каждой точки выполняется сортировка:

```text
примерно O(N² log N)
```

Учебное построение графа не предназначено для сотен миллионов объектов.

Цель упражнения — понять навигацию.

Production-индексы используют:

- инкрементальное построение;
- иерархию;
- эвристики выбора соседей;
- pruning;
- ограничение degree;
- quantization;
- параллелизм;
- disk-based подходы;
- sharding.

# 18. Sampling

При:

```python
use_sampling=False
```

точка сравнивается со всей коллекцией.

При:

```python
use_sampling=True
```

выбирается только доля документов:

```python
sample_size = int(num_points * sampling_share)
```

Пример:

```text
N = 1 000 000
sampling_share = 0.01
sample_size = 10 000
```

Плюс — быстрее строится граф.

Минус — можно пропустить хорошие кандидаты и ухудшить структуру.

В условии сказано, что режим sampling сервером не проверяется, но он реализован.

# 19. Навигация NSW

Алгоритм начинает со случайной точки.

Для текущей вершины:

1. Получает соседей.
2. Считает расстояния от запроса до соседей.
3. Выбирает более близкого соседа.
4. Переходит к нему.
5. Продолжает, пока расстояние улучшается.

Это greedy search — жадный поиск.

Жадный означает:

> На каждом шаге выбирается локально лучший переход прямо сейчас.

---

# 20. Локальный минимум

Алгоритм может прийти в точку, которая ближе всех своих соседей, но не является глобально ближайшей во всём пространстве.

Аналогия: вы спускаетесь по рельефу. Все соседние позиции выше, но за холмом может находиться более глубокая долина.

Это локальный минимум.

Поэтому используется несколько стартов:

```python
num_start_points
```

Разные старты исследуют разные области графа и повышают вероятность найти нужный кластер.

---

# 21. Кэш расстояний

Одна и та же вершина может встретиться в нескольких маршрутах.

Пересчитывать расстояние бессмысленно.

Поэтому хранится:

```text
индекс точки → расстояние до запроса
```

Функция `calc_d_and_upd`:

- возвращает сохранённое расстояние, если точка уже посещалась;
- иначе вычисляет и сохраняет его.

Это простая memoization.

# 31. Почему offline и online могут расходиться

Рост Recall@K не гарантирует рост бизнеса.

Причины:

- reranker и раньше исправлял ошибки retrieval;
- новые кандидаты не проходят фильтры;
- упала diversity;
- появились товары без наличия;
- выросла latency;
- embedding отражает семантику, но не коммерческое намерение;
- exact neighbors не всегда равны полезным пользователю объектам.

Нужны:

1. offline benchmark;
2. shadow test;
3. нагрузочное тестирование;
4. A/B test;
5. guardrail-метрики.

# 32. Бизнесовый смысл параметров

## search_k

Сколько соседей вернуть.

Больше `search_k`:

- выше шанс дать reranker хорошие кандидаты;
- больше стоимость downstream обработки.

## num_start_points

Больше стартов:

- меньше риск локального минимума;
- больше latency.

## num_edges_short

Больше коротких рёбер:

- лучше локальная навигация;
- больше память и вычисления.

## num_edges_long

Больше длинных рёбер:

- больше глобальных переходов;
- больше память и потенциальный шум.

## sampling_share

Больше sample:

- качественнее построение;
- дороже index build.

# 33. Что может сломаться

1. Неверная форма массива.
2. Самоссылка в графе.
3. `num_edges` больше числа кандидатов.
4. Несовпадение метрики offline и online.
5. Документы нормализованы, а запрос нет.
6. Запрос кодируется новой моделью, индекс построен старой.
7. Индекс устарел.
8. Новые объекты отсутствуют.
9. После фильтра остаётся мало кандидатов.
10. Слишком мало стартовых точек.
11. Random seed не зафиксирован в benchmark.
12. Сравнивается только latency без качества.
13. Сравнивается только recall без стоимости.

# 34. NSW и HNSW

Эта задача реализует упрощённую идею NSW в одном графе.

HNSW добавляет иерархию уровней:

- верхние уровни редкие и нужны для дальних переходов;
- нижние уровни плотные и нужны для локального уточнения.

Аналогия:

```text
автомагистраль
→ городская дорога
→ улица
→ двор
```

Учебный SW-граф — одна карта. HNSW — несколько уровней детализации.

# 36. Весь урок одной схемой

```text
Документы
    ↓
Эмбеддинги
    ↓
L2-расстояния
    ↓
Короткие и длинные рёбра
    ↓

Несколько случайных стартов
```
