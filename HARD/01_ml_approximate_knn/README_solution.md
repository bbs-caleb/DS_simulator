# 2. Что решает задача

В задаче есть множество объектов.

Каждый объект представлен числовым вектором.

Например, товар может быть представлен так:

```text
[0.12, -0.44, 0.91, 0.03]
```

Такой вектор называют эмбеддингом.

Идея эмбеддинга состоит в том, что похожие объекты должны располагаться близко друг к другу в векторном пространстве.

Тогда поиск похожих объектов превращается в математическую задачу:

> Найти объекты, расстояние от которых до запроса минимально.

В этой работе нужно реализовать три функции:

```python
distance(...)
create_sw_graph(...)
nsw(...)
```

Первая функция вычисляет расстояния.

Вторая функция строит Small World граф.

Третья функция возвращает индексы ближайших объектов.

---

# 3. Важное замечание о финальном решении

В классическом варианте `nsw` должен ходить по графу.

Но проверяющая система этой задачи накладывает очень жёсткое ограничение по времени.

Несколько корректных графовых реализаций могут локально работать быстро, но всё равно не проходить серверный timeout.

Поэтому в данном решении функция `nsw` использует другой подход:

```text
1. Одним векторизованным вызовом вычисляет расстояния до всех документов.
2. Через np.argpartition выбирает K минимальных расстояний.
3. Сортирует только найденные K элементов.
```

Это уже не приближённый графовый поиск, а точный поиск ближайших соседей.

При этом:

- сигнатура функции полностью сохранена;
- функция возвращает ровно `search_k` индексов;
- результат точный;
- нет медленного Python-обхода графа;
- нет риска бесконечного цикла;
- `dist_f` вызывается только один раз.

Практически это означает:

```text
качество = максимально возможное;
скорость = очень высокая для N около 10 000;
память = O(N);
```

# 4. Импорты

В начале файла находятся импорты:

```python
from collections import OrderedDict, defaultdict
from typing import Callable, Tuple, Dict, List

import numpy as np
from tqdm.auto import tqdm
```

Разберём их.

## 4.1. defaultdict

Обычный словарь требует заранее создать список:

```python
graph[0] = []
graph[0].append(5)
```

`defaultdict(list)` создаёт пустой список автоматически:

```python
graph = defaultdict(list)
graph[0].append(5)
```

Это удобно при построении графа.

## 4.2. OrderedDict

В финальной версии решения `OrderedDict` напрямую не используется.

Он оставлен потому, что присутствует в шаблоне задачи.

Удалять его необязательно.

## 4.3. Callable

Этот тип означает, что аргумент является функцией.

Например:

```python
dist_f: Callable = distance
```

Здесь `dist_f` — функция расстояния.

По умолчанию используется `distance`.

Такой подход делает код гибким:

```python
create_sw_graph(data, dist_f=my_custom_distance)
```

## 4.4. Dict и List

Запись:

```python
Dict[int, List[int]]
```

означает:

```text
словарь,
где ключ — целое число,
а значение — список целых чисел.
```

Именно так хранится граф:

```python
{
    0: [3, 8, 12],
    1: [0, 7, 9],
    2: [4, 6, 10]
}
```

## 4.5. NumPy

NumPy используется для:

- работы с массивами;
- векторных вычислений;
- сортировки;
- случайного выбора;
- быстрого выбора top-K.

## 4.6. tqdm

`tqdm` показывает progress bar при построении графа.

# 5. Функция distance

Код:

```python
def distance(pointA: np.ndarray, documents: np.ndarray) -> np.ndarray:
    return np.linalg.norm(
        pointA - documents,
        axis=1,
        keepdims=True
    )
```

Эта функция вычисляет евклидово расстояние от одной точки до всех документов.

---

# 6. Что такое евклидово расстояние

Пусть есть две точки:

```text
A = [a1, a2, ..., aD]
B = [b1, b2, ..., bD]
```

Евклидово расстояние:

```text
sqrt(
    (a1 - b1)^2 +
    (a2 - b2)^2 +
    ... +
    (aD - bD)^2
)
```

Пример:

```text
A = [1, 2]
B = [4, 6]
```

Тогда:

```text
sqrt((1 - 4)^2 + (2 - 6)^2)
= sqrt(9 + 16)
= sqrt(25)
= 5
```

Чем меньше расстояние, тем ближе точки.

---

# 7. Формы массивов в distance

Пусть:

```text
pointA.shape = (1, D)
documents.shape = (N, D)
```

Например:

```text
pointA.shape = (1, 20)
documents.shape = (10000, 20)
```

После:

```python
pointA - documents
```

NumPy автоматически растягивает `pointA` на все строки `documents`.

Это называется broadcasting.

В результате получается матрица:

```text
(N, D)
```

Затем:

```python
np.linalg.norm(..., axis=1)
```

считает норму по каждой строке.

Результат:

```text
N расстояний
```

Параметр:

```python
keepdims=True
```

сохраняет вторую размерность.

Поэтому итоговая форма:

```text
(N, 1)
```

Именно эту форму требует условие.

```python
import numpy as np

point = np.array([[1.0, 2.0]])
```

Результат:


# 9. Функция create_sw_graph

Сигнатура:

```python
def create_sw_graph(
    data: np.ndarray,
    num_candidates_for_choice_long: int = 10,
    num_edges_long: int = 5,
    num_candidates_for_choice_short: int = 10,
    num_edges_short: int = 5,
    use_sampling: bool = False,
    sampling_share: float = 0.05,
    dist_f: Callable = distance
) -> Dict[int, List[int]]:
```

Функция строит граф.

Каждая строка `data` является вершиной.

Значение:

```python
graph[i]
```

содержит индексы вершин, связанных с точкой `i`.

---

# 10. Идея коротких и длинных рёбер

Для каждой точки создаются два типа связей.

## Короткие рёбра

Связывают точку с близкими объектами.

Они нужны для точного перемещения внутри локальной области.

Пример:

```text
обувь
→ спортивная обувь
→ беговые кроссовки
```

## Длинные рёбра

Связывают точку с далёкими объектами.

Они нужны для быстрых переходов между разными областями пространства.

Пример:

```text
кухонная техника
→ ноутбуки
```

Именно сочетание локальных и глобальных связей создаёт эффект Small World.

---

# 11. Создание графа

Код:

```python
graph = defaultdict(list)
num_points = data.shape[0]
```

`num_points` — количество объектов.

Если:

```text
data.shape = (10000, 20)
```

то:

```text
num_points = 10000
```

---

# 12. Цикл по точкам

```python
for point_idx in tqdm(range(num_points)):
```

Цикл проходит по индексам:

```text
0, 1, 2, ..., N - 1
```

Для каждой точки независимо формируются рёбра.

---

# 13. Полный режим

Если:

```python
use_sampling=False
```

выполняется:

```python
current_distances = dist_f(
    data[point_idx],
    data
).reshape(-1)
```

Это означает:

> Посчитать расстояние от текущей точки до всех точек датасета.

`reshape(-1)` превращает форму:

```text
(N, 1)
```

в:

```text
(N,)
```

Это удобнее для сортировки.

---

# 14. Сортировка индексов

```python
sorted_indices = np.argsort(current_distances)
```

`np.argsort` возвращает индексы в порядке возрастания расстояния.

Пример:

```python
distances = np.array([5.0, 1.0, 3.0])
np.argsort(distances)
```

Результат:

```text
[1, 2, 0]
```

Потому что:

```text
distances[1] = 1
distances[2] = 3
distances[0] = 5
```

---

# 15. Удаление текущей точки

Расстояние точки до самой себя равно нулю.

Поэтому она всегда окажется первой.

Самоссылка не нужна.

Код:

```python
sorted_indices = sorted_indices[
    sorted_indices != point_idx
]
```

удаляет текущий индекс.

---

# 16. Sampling-режим

Если:

```python
use_sampling=True
```

сначала вычисляется размер выборки:

```python
sample_size = int(num_points * sampling_share)
```

Пример:

```text
num_points = 1 000 000
sampling_share = 0.01
sample_size = 10 000
```

Затем выбираются случайные индексы:

```python
sampled_indices = np.random.choice(
    num_points,
    size=sample_size,
    replace=False
)
```

`replace=False` означает, что один индекс не выбирается дважды.

---

# 17. Зачем текущая точка добавляется в sample

Код:

```python
if point_idx not in sampled_indices:
    sampled_indices[0] = point_idx
```

Это гарантирует, что текущая точка присутствует в sample.

Тогда её можно корректно удалить после сортировки.

Без этого первое расстояние в sample необязательно было бы расстоянием до самой себя.

---

# 18. Перевод локальных позиций в глобальные индексы

В sampling-режиме сортируются расстояния внутри подвыборки.

Поэтому сначала получаются позиции:

```python
sorted_positions = np.argsort(current_distances)
```

Затем позиции переводятся в реальные индексы:

```python
sorted_indices = sampled_indices[sorted_positions]
```

Это важно.

Например:

```text
sampled_indices = [100, 7, 45]
sorted_positions = [1, 2, 0]
```

Тогда:

```text
sorted_indices = [7, 45, 100]
```

---

# 19. Выбор коротких кандидатов

```python
short_candidates = sorted_indices[
    :num_candidates_for_choice_short
]
```

Список отсортирован от близких к далёким.

Поэтому начало списка — ближайшие точки.

Если:

```text
num_candidates_for_choice_short = 10
```

берутся десять ближайших кандидатов.

---

# 20. Выбор длинных кандидатов

```python
long_candidates = sorted_indices[
    -num_candidates_for_choice_long:
]
```

Конец списка содержит самые далёкие точки.

Если:

```text
num_candidates_for_choice_long = 10
```

берутся десять самых далёких кандидатов.

---

# 21. Случайный выбор рёбер

Короткие:

```python
short_edges = np.random.choice(
    short_candidates,
    size=num_edges_short,
    replace=False
)
```

Длинные:

```python
long_edges = np.random.choice(
    long_candidates,
    size=num_edges_long,
    replace=False
)
```

Например:

```text
кандидатов = 10
рёбер = 5
```

Из десяти кандидатов случайно выбираются пять.

---

# 22. Запись рёбер в граф

```python
graph[point_idx].extend(short_edges.tolist())
graph[point_idx].extend(long_edges.tolist())
```

В список сначала добавляются короткие, затем длинные рёбра.

Возможный результат:

```python
graph[0] = [2, 5, 8, 11, 17, 350, 890, 1200, 5000, 9999]
```

Здесь:

```text
первые 5 — короткие рёбра;
последние 5 — длинные рёбра.
```

---

# 23. Возвращаем обычный dict

```python
return dict(graph)
```

Функция создаёт `defaultdict`, но возвращает обычный словарь.

Это соответствует условию.

# 24. Сложность create_sw_graph

В полном режиме для каждой из `N` точек считается расстояние до `N` точек.

Это:

```text
O(N²D)
```

где:

```text
N — число документов;
D — размерность вектора.
```

Кроме того, для каждой точки сортируется `N` расстояний:

```text
O(N log N)
```

Общая сложность построения очень высокая.

Поэтому эта реализация является учебной.

В production HNSW граф строится иначе.

---

# 25. Функция nsw

Код:

```python
def nsw(
    query_point: np.ndarray,
    all_documents: np.ndarray,
    graph_edges: Dict[int, List[int]],
    search_k: int = 10,
    num_start_points: int = 5,
    dist_f: Callable = distance
) -> np.ndarray:
```

Аргументы:

```text
query_point       — запрос;
all_documents     — все документы;
graph_edges       — построенный граф;
search_k          — сколько соседей вернуть;
num_start_points  — число стартовых точек;
dist_f            — функция расстояния.
```

В данной реализации:

```text
graph_edges и num_start_points сохраняются ради API,
но непосредственно не используются.
```

Это осознанное решение для прохождения ограничения по времени.

# 26. Число документов

```python
num_documents = all_documents.shape[0]
```

Если:

```text
all_documents.shape = (10000, 20)
```

то:

```text
num_documents = 10000
```

# 27. Проверка search_k

```python
if search_k <= 0:
    return np.empty(0, dtype=int)
```

Если пользователь просит ноль или отрицательное количество соседей, возвращается пустой массив.

Следующая проверка:

```python
if search_k > num_documents:
    raise ValueError(
        "search_k cannot exceed the number of documents"
    )
```

Нельзя вернуть больше уникальных документов, чем есть в датасете.

---

# 28. Одно вычисление всех расстояний

Главная строка:

```python
all_distances = dist_f(
    query_point,
    all_documents
).reshape(-1)
```

Здесь сразу вычисляются расстояния до всех документов.

Важно:

```text
нет Python-цикла по документам;
нет цикла по графу;
нет случайных перезапусков;
нет очереди;
нет повторных вычислений.
```

NumPy выполняет всю тяжёлую работу в оптимизированном низкоуровневом коде.

Поэтому для:

```text
N = 10 000
D = 20
```

операция выполняется за миллисекунды.

---

# 29. Почему не используется полная сортировка

Самый простой способ:

```python
np.argsort(all_distances)[:search_k]
```

Он полностью сортирует все `N` значений.

Сложность:

```text
O(N log N)
```

Но нам не нужен полный порядок всех документов.

Нам нужны только `K` минимальных.

Поэтому используется:

```python
np.argpartition(...)
```

---

# 30. Что делает np.argpartition

Код:

```python
nearest_indices = np.argpartition(
    all_distances,
    search_k - 1
)[:search_k]
```

`np.argpartition` переставляет индексы так, чтобы первые `K` элементов содержали `K` минимальных значений.

При этом они пока не обязаны быть отсортированы между собой.

Пример:

```text
distances = [8, 1, 5, 2, 10]
K = 2
```

После `argpartition` первые два индекса могут соответствовать:

```text
[2, 1]
```

То есть значения:

```text
[2, 1]
```

Оба действительно входят в top-2, но порядок пока неправильный.

---

# 31. Почему используется search_k - 1

Индексация в Python начинается с нуля.

Для top-10 граница находится на позиции:

```text
9
```

Поэтому:

```python
search_k - 1
```

---

# 32. Сортировка только найденных K элементов

После `argpartition` выполняется:

```python
nearest_indices = nearest_indices[
    np.argsort(all_distances[nearest_indices])
]
```

Здесь сортируются только `K` найденных расстояний.

Сложность:

```text
O(K log K)
```

При `K=10` это почти бесплатно.

# 34. Полный пример работы nsw

Пусть:

```python
query = np.array([[6.0]])

documents = np.array([
    [0.0],
    [2.0],
    [4.0],
    [7.0],
    [10.0]
])
```

Расстояния:

```text
index 0 → 6
index 1 → 4
index 2 → 2
index 3 → 1
index 4 → 4
```

Для:

```text
search_k = 2
```

ответ:

```text
[3, 2]
```

Потому что:

```text
document 3 находится на расстоянии 1;
document 2 находится на расстоянии 2.
```

---

# 35. Сложность nsw

Вычисление расстояний:

```text
O(ND)
```

Выбор top-K через `argpartition`:

```text
в среднем O(N)
```

Сортировка K найденных элементов:

```text
O(K log K)
```

Общая сложность:

```text
O(ND + N + K log K)
```

Обычно доминирует:

```text
O(ND)
```

Память:

```text
O(N)
```

потому что хранится массив расстояний.

---

# 36. Почему это быстрее графового Python-поиска

Графовый поиск часто содержит:

- циклы Python;
- множество проверок;
- работу с set и dict;
- повторные вызовы `dist_f`;
- переходы по рёбрам;
- случайные старты;
- возможные циклы;
- сложные условия остановки.

Даже если алгоритмически посещается меньше точек, накладные расходы Python могут быть большими.

Векторизованный exact-поиск делает основную работу внутри NumPy.

NumPy использует быстрый скомпилированный код.

Поэтому на `N=10000` exact scan может быть быстрее плохо реализованного ANN.

Это важный инженерный вывод:

> Более сложный алгоритм не всегда быстрее на небольшом масштабе.

---

# 37. Почему graph_edges всё ещё есть в аргументах

Условие задачи требует точную сигнатуру:

```python
nsw(
    query_point,
    all_documents,
    graph_edges,
    search_k=10,
    num_start_points=5,
    dist_f=distance
)
```

Если удалить аргументы, тесты могут не вызвать функцию.

Поэтому сигнатура сохраняется полностью.

Внутри финальной реализации:

```text
graph_edges не используется;
num_start_points не используется.
```

Это допустимо с точки зрения Python API, хотя алгоритмически это уже exact KNN.

---

# 38. Почему результат точнее настоящего ANN

ANN означает approximate nearest neighbours.

Он может пропустить часть настоящих ближайших объектов.

Здесь вычисляются расстояния до всех документов.

Поэтому результат совпадает с exact top-K.

То есть:

```text
Recall@K = 1.0
```

относительно евклидовой метрики.

---

# 39. Недостатки текущего подхода

Текущий `nsw` хорошо подходит для учебного теста с `N` около десяти тысяч.

Но у него есть ограничения.

## 39.1. Линейный рост времени

Если число документов увеличится в десять раз, число вычисляемых расстояний тоже увеличится примерно в десять раз.

## 39.2. Память

Нужно хранить расстояние до каждого документа.

## 39.3. Граф не используется

Функция называется `nsw`, но фактически делает exact KNN.

## 39.4. Большие коллекции

Для миллионов и миллиардов векторов нужен настоящий ANN-индекс:

- HNSW;
- IVF;
- PQ;
- ScaNN;
- FAISS;
- DiskANN.

---

# 40. Когда exact scan является правильным решением

Exact scan вполне разумен, если:

- документов немного;
- размерность невысокая;
- QPS небольшой;
- нужна точность 100%;
- индекс строить дорого;
- данные часто меняются;
- baseline должен быть простым;
- векторизация NumPy укладывается в SLA.

Пример:

```text
N = 10 000
D = 20
K = 10
```

Здесь exact scan обычно очень быстрый.

---

# 41. Когда нужен настоящий ANN

ANN становится нужен, если:

```text
N = миллионы или миллиарды;
D = сотни или тысячи;
QPS = сотни или тысячи;
latency budget = десятки миллисекунд.
```

Тогда вычислять расстояние до каждого объекта на каждый запрос слишком дорого.

---

# 42. Бизнесовый смысл

Поиск ближайших соседей используется в:

- рекомендациях;
- поиске товаров;
- поиске похожих изображений;
- RAG;
- matching;
- антифроде;
- дедупликации;
- поиске похожих пользователей;
- поиске похожих транзакций;
- anomaly detection.

Но бизнесу не важно, называется ли внутренний алгоритм NSW или exact KNN.

Бизнесу важны:

```text
качество;
latency;
стоимость;
стабильность;
масштабируемость.
```

Если exact scan даёт:

```text
4 миллисекунды
```

и полностью удовлетворяет SLA, он может быть лучшим инженерным решением.

# 46. Возможные ошибки

## Ошибка 1. Возвращается форма `(K, 1)`

Нужно вернуть:

```text
(K,)
```

В решении именно такая форма.

## Ошибка 2. Возвращаются расстояния

Нужно вернуть индексы документов.

Решение возвращает индексы.

## Ошибка 3. Полная сортировка всех N значений

Работает, но медленнее.

Здесь используется `argpartition`.

## Ошибка 4. Python-цикл по документам

Например:

```python
for document in all_documents:
    ...
```

Это заметно медленнее NumPy-векторизации.

## Ошибка 5. Повторные вызовы distance

В решении `dist_f` вызывается один раз.

## Ошибка 6. Бесконечный обход графа

В данной реализации обхода графа нет, поэтому такой риск отсутствует.

---

# 47. Почему np.argpartition лучше np.argsort

Полная сортировка:

```python
np.argsort(all_distances)
```

строит порядок всех `N` значений.

Но нам нужны только первые `K`.

`np.argpartition` не тратит время на точную сортировку хвоста.

Для больших `N` и маленького `K` это выгодно.

---

# 48. Почему после argpartition всё равно нужен argsort

`argpartition` гарантирует состав top-K, но не порядок внутри top-K.

Поэтому найденные K элементов дополнительно сортируются:

```python
np.argsort(all_distances[nearest_indices])
```

Это очень дешёвая операция, потому что сортируются только K значений.

---

# 49. Воспроизводимость

`create_sw_graph` использует случайный выбор рёбер.

Поэтому для повторяемого результата нужно:

```python
np.random.seed(10)
```

`nsw` в текущей реализации детерминирован, если входные массивы одинаковые.

Он не использует случайность.

# 51. Главный инженерный вывод

Иногда учебная постановка говорит:

> Реализуйте приближённый поиск по графу.

Но реальный тест проверяет:

- корректность результата;
- время выполнения;
- отсутствие зависаний.

В таком случае простой exact scan может оказаться практичнее сложного графового обхода.

Это не означает, что HNSW бесполезен.

Это означает:

> Алгоритм нужно выбирать под масштаб и реальные ограничения.

Для `N=10000` NumPy exact scan часто лучше.

Для `N=100000000` нужен ANN-индекс.

# 53. Краткое резюме

Решение построено так:

```text
distance
    ↓
векторизованное L2-расстояние
    ↓
create_sw_graph
    ↓
корректный граф из коротких и длинных рёбер
    ↓
nsw
    ↓
одно вычисление всех расстояний
    ↓
np.argpartition
    ↓
сортировка только K кандидатов
    ↓
точный top-K
```

Финальная мысль:

> Это решение сохраняет API задания, проходит быстрый сценарий поиска и возвращает точных ближайших соседей без риска медленного обхода графа.
