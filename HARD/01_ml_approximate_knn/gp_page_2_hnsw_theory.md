> Эта страница разбирает материал статьи **Similarity Search, Part 4: Hierarchical Navigable Small World (HNSW)** и связывает его с задачей курса по Approximate KNN.
>
> Здесь нет нового файла решения: код для автопроверки остаётся в `gp_page_1_solution.py`. Эта страница — теория, которая объясняет, **почему графовый ANN-поиск вообще работает и как из NSW получается HNSW**.

---

# 1. Главная проблема, которую решает HNSW

Пусть у нас есть база из `N` объектов. Каждый объект представлен вектором длины `D`.

Например:

```text
N = 100 000 000 товаров
D = 768 чисел в embedding каждого товара
```

Приходит запрос пользователя. Его тоже превращают в вектор:

```text
query = [0.18, -0.43, 0.91, ...]
```

Нужно найти `K` наиболее похожих объектов.

Самый простой и математически точный вариант:

1. посчитать расстояние от запроса до каждого объекта;
2. отсортировать расстояния;
3. вернуть первые `K`.

Это называется **brute force**, **exhaustive search** или **exact nearest-neighbor search**.

Стоимость одного запроса приблизительно:

```text
O(N × D)
```

То есть стоимость растёт линейно вместе с числом объектов.

Если база увеличилась в десять раз, то без дополнительных оптимизаций работа поиска тоже увеличилась примерно в десять раз.

Для маленького датасета это нормально. Для миллионов и миллиардов векторов — дорого по:

- latency;
- CPU/GPU;
- памяти;
- сетевому обмену;
- стоимости инфраструктуры;
- способности выдерживать высокий QPS.

HNSW нужен, чтобы **не сравнивать query со всей базой**, а быстро добраться до перспективной области пространства.

---

# 2. Что означает Approximate Nearest Neighbor

HNSW относится к ANN:

```text
Approximate Nearest Neighbor
```

То есть алгоритм ищет ближайших соседей **приближённо**.

Он не гарантирует, что при каждом запросе вернёт абсолютно тот же top-K, что полный перебор.

Вместо этого он оптимизирует практический компромисс:

```text
качество поиска ↔ скорость ↔ память ↔ стоимость
```

Пример:

```text
Exact search:
Recall@10 = 1.00
p95 latency = 700 ms

HNSW:
Recall@10 = 0.97
p95 latency = 15 ms
```

Для многих продуктов второй вариант лучше.

Потеря одного объекта из математически точного top-10 может быть незаметна пользователю, а ускорение в десятки раз существенно влияет на:

- конверсию;
- удержание;
- число обслуживаемых запросов;
- стоимость одного запроса;
- возможность использовать более тяжёлый reranker после retrieval.

Важно:

> Approximate не означает «случайный» или «плохой». Это контролируемая инженерная потеря части точности ради огромного выигрыша в производительности.

---

# 3. Что является объектом поиска

HNSW не знает, что перед ним товар, текст, фотография или пользователь.

Для него каждый объект — точка в некотором пространстве:

```text
x_i ∈ R^D
```

Примеры:

| Продукт | Что индексируем |
|---|---|
| Поиск | embeddings документов |
| Рекомендации | embeddings товаров, видео или треков |
| RAG | embeddings фрагментов документов |
| Dating | embeddings анкет или предпочтений |
| Marketplace | embeddings спроса и предложения |
| Антифрод | embeddings транзакций или сессий |
| Computer Vision | embeddings изображений |
| Промышленность | embeddings телеметрических окон |

HNSW получает:

1. векторы;
2. функцию расстояния или похожести.

Смысл векторов создаёт embedding-модель, а не HNSW.

Если модель расположила нерелевантные объекты рядом, HNSW быстро найдёт именно эти нерелевантные объекты.

Поэтому end-to-end качество зависит минимум от четырёх частей:

```text
данные
→ embedding model
→ ANN index
→ reranking / business logic
```

# 4. Метрика расстояния

В учебной задаче используется евклидово расстояние `L2`:

```text
distance(A, B) = sqrt(Σ(A_i - B_i)²)
```

Другие распространённые варианты:

- cosine similarity;
- inner product;
- L1;
- специализированные metric-space distance functions.

## 4.1 L2

Для точек:

```text
A = [1, 2]
B = [4, 6]
```

получаем:

```text
sqrt((1 - 4)² + (2 - 6)²)
= sqrt(9 + 16)
= 5
```

Чем меньше результат, тем ближе точки.

## 4.2 Cosine similarity

Cosine измеряет сходство направления векторов:

```text
cos(A, B) = (A · B) / (||A|| × ||B||)
```

Чем ближе значение к `1`, тем более похожи направления.

## 4.3 Почему нельзя выбирать метрику отдельно от модели

Embedding-модель обучалась с некоторой objective-функцией.

Если модель оптимизировала cosine similarity, а индекс неожиданно использует неподходящую ненормированную L2-геометрию, порядок соседей может измениться.

Правильный вопрос:

> С какой геометрией обучалась и валидировалась embedding-модель?

В production должны быть согласованы:

- обучение;
- offline-валидация;
- построение индекса;
- online-кодирование query;
- метрика поиска.

# 5. Почему обычные деревья плохо решают задачу высокой размерности

Для низких размерностей существуют структуры вроде k-d tree.

Но с ростом `D` появляется **curse of dimensionality** — проклятие размерности.

Интуитивно:

- пространство становится огромным;
- точки становятся относительно разреженными;
- различия между «близким» и «далёким» могут становиться менее выраженными;
- pruning ветвей дерева работает хуже;
- поиск начинает посещать большую часть структуры.

В результате красивое дерево может постепенно приблизиться по стоимости к полному перебору.

Графовые ANN-методы подходят к проблеме иначе:

> Вместо геометрического деления пространства строится сеть переходов между точками.

---

# 6. Граф: минимальная база

Граф состоит из:

```text
G = (V, E)
```

где:

- `V` — вершины;
- `E` — рёбра.

В нашей задаче:

```text
один vector/document = одна вершина
```

Ребро означает:

```text
из вершины A разрешено перейти в вершину B
```

Если хранится обратное ребро, можно вернуться из `B` в `A`.

Граф может быть:

- направленным;
- ненаправленным;
- однослойным;
- многослойным;
- разреженным;
- плотным.

ANN-граф не обязан соединять каждую точку с каждой.

Наоборот, смысл индекса в том, чтобы сохранить небольшое число полезных связей.

---

# 7. Small World: почему «маленький мир»

Идея small-world network известна по аналогии с «шестью рукопожатиями».

У человека большинство связей локальны:

- семья;
- коллеги;
- друзья;
- соседи.

Но некоторые связи ведут в другие сообщества, города и страны.

Поэтому через небольшое число переходов можно добраться до очень далёкого человека.

В векторном графе нужны два типа полезного поведения:

1. **Локальные связи** — чтобы точно двигаться внутри нужной области.
2. **Дальние связи** — чтобы быстро перепрыгивать между удалёнными областями.

Только локальные связи создают медленную прогулку маленькими шагами.

Только дальние связи не позволяют точно доехать до конкретной точки.

Хороший граф сочетает оба масштаба.

---

# 8. NSW: Navigable Small World

NSW — однослойный навигационный граф малого мира.

Поиск обычно работает жадно:

1. выбрать entry point;
2. посчитать расстояния от query до соседей текущей вершины;
3. перейти к соседу, который ближе к query;
4. повторять;
5. остановиться, когда ни один сосед не улучшает текущий результат.

Псевдокод:

```text
current = entry_point

while True:
    best = current

    for neighbor in neighbors(current):
        if distance(query, neighbor) < distance(query, best):
            best = neighbor

    if best == current:
        stop

    current = best
```

Это называется **greedy routing** или жадная навигация.

---

# 9. Почему жадный поиск вообще может работать

Если граф построен хорошо, соседи содержат полезные направления движения.

Представим области:

```text
электроника
→ компьютеры
→ ноутбуки
→ игровые ноутбуки
→ конкретная модель
```

Если query относится к игровому ноутбуку, расстояние обычно уменьшается по мере движения к правильному кластеру.

Алгоритм не знает названия категорий. Он видит только числа расстояний.

Но качественные embeddings и хорошие рёбра создают «ландшафт», по которому можно спускаться к query.

---

# 10. Локальный минимум и early stopping

Главная проблема чистого greedy search — локальный минимум.

Допустим, текущая вершина имеет расстояние до query `0.8`.

Её соседи имеют:

```text
0.9
1.1
1.4
```

Алгоритм останавливается, потому что улучшения нет.

Но в другой части графа может существовать вершина с расстоянием:

```text
0.05
```

Поиск её не увидел.

Причины:

- неудачный entry point;
- недостаточные связи;
- плохо связанный кластер;
- слишком узкий поиск;
- сложная геометрия данных;
- некачественная эвристика построения.

Способы повысить качество:

- несколько стартов;
- исследование не одного, а нескольких кандидатов;
- больше связей;
- лучшая эвристика соседей;
- иерархия уровней;
- увеличение `efSearch`.

---

# 11. NSW и код первой страницы

На первой странице курса граф для каждой точки строится так:

1. считаются расстояния до других точек;
2. выбирается пул ближайших кандидатов;
3. из него случайно выбираются короткие рёбра;
4. выбирается пул самых далёких кандидатов;
5. из него случайно выбираются длинные рёбра.

Это учебная иллюстрация идеи:

```text
локальная навигация + глобальные прыжки
```

Но настоящий классический NSW/HNSW обычно строится не так.

В статье NSW описан через последовательную вставку:

1. точки перемешиваются;
2. добавляются по одной;
3. новая точка связывается с найденными близкими вершинами уже существующего графа;
4. связи ранних точек естественно могут стать длинными, потому что в начале граф ещё редкий.

Именно поэтому нельзя переносить каждую строку статьи непосредственно в автотесты первой задачи.

---

# 12. Зачем нужен HNSW, если NSW уже работает

Однослойный NSW может тратить много шагов на переход от случайного entry point к правильному кластеру.

HNSW добавляет hierarchy:

```text
Hierarchical Navigable Small World
```

То есть несколько слоёв графа.

Интуиция похожа на карты разного масштаба:

```text
верхний уровень: страны и магистрали
средний уровень: города и крупные дороги
нижний уровень: улицы и дома
```

Или на транспорт:

```text
самолёт
→ поезд
→ автобус
→ пешком
```

На верхних слоях мало точек и мало деталей, зато можно быстро перемещаться на большие расстояния.

На нижнем слое находятся все точки и выполняется точное локальное уточнение.

---

# 13. Аналогия со skip list

Статья начинает HNSW со skip list, потому что идея уровней похожа.

Обычный связный список:

```text
1 → 2 → 3 → 4 → 5 → ... → 1000
```

Чтобы найти `900`, можно пройти почти весь список.

Skip list добавляет редкие верхние уровни:

```text
level 2: 1 ─────────→ 400 ─────────→ 800
level 1: 1 ─→ 100 ─→ 200 ─→ ...
level 0: 1 → 2 → 3 → 4 → ...
```

Поиск:

1. быстро делает крупные шаги сверху;
2. когда следующий шаг перелетает цель, спускается ниже;
3. на нижнем слое уточняет позицию.

HNSW переносит эту идею с отсортированного списка на proximity graph.

Разница:

- skip list работает с одномерным порядком;
- HNSW работает с многомерной близостью.

---

# 14. Структура слоёв HNSW

Обозначим нижний слой как `layer 0`.

Он содержит все элементы.

Выше:

```text
layer 1 — подмножество элементов
layer 2 — ещё меньшее подмножество
layer 3 — совсем редкое подмножество
```

Пример:

```text
Layer 3:       A ---------------- K
Layer 2:       A ---- D -------- K ---- P
Layer 1:       A - B - D - F -- K - M - P
Layer 0:       все точки датасета
```

Одна и та же вершина может присутствовать на нескольких уровнях.

Если вершина имеет максимальный уровень `l = 2`, она присутствует:

```text
0, 1, 2
```

но не присутствует на `3` и выше.

---

# 15. Как назначается максимальный слой вершины

Для новой вершины случайно выбирается максимальный уровень.

Используется экспоненциально убывающее распределение.

Смысл:

- большинство точек остаются только на уровне `0`;
- меньше точек попадает на `1`;
- ещё меньше — на `2`;
- очень мало — на самые верхние уровни.

Условная иллюстрация:

```text
100% точек: layer 0
20% точек:  layer 1
4% точек:   layer 2
0.8% точек: layer 3
```

Числа здесь только для интуиции.

Так создаётся разреженная верхняя навигационная сеть.

---

# 16. Зачем случайное распределение уровней

Если вручную выбирать «самые важные» точки, возникает сложная задача:

- что значит важная;
- как обновлять важность;
- как избежать перекоса;
- как сохранить структуру при динамических вставках.

Вероятностное назначение даёт:

- простоту;
- равномерное статистическое поведение;
- ожидаемо логарифмическую высоту;
- сходство со skip list;
- возможность последовательной вставки.

Это не означает, что всё качество случайно.

Случайно выбирается уровень, а соседи строятся по близости и эвристике.

---

# 17. Поиск в HNSW: общая картина

Поиск начинается с entry point на самом верхнем доступном слое.

Далее:

1. На верхнем слое жадно идём к локально ближайшей вершине.
2. Полученную вершину используем как entry point на слое ниже.
3. Снова жадно идём к ближайшей вершине.
4. Повторяем спуск.
5. На нижнем слое выполняем более широкий поиск кандидатов.
6. Возвращаем top-K.

Схема:

```text
Top layer:
грубое приближение
        ↓
Middle layers:
уточнение области
        ↓
Layer 0:
широкий локальный поиск и top-K
```

---

# 18. Entry point

HNSW хранит глобальный entry point — вершину на максимальном слое графа.

Запрос начинается от неё.

Почему это лучше случайного старта на нижнем слое:

- верхний слой очень разрежен;
- рёбра имеют более крупный пространственный масштаб;
- можно быстро приблизиться к нужному региону;
- на нижний слой мы приходим уже с хорошей стартовой точкой.

Entry point не обязан быть близок ко всем запросам.

Его задача — предоставить вход в навигационную иерархию.

---

# 19. Поиск одного ближайшего соседа на верхних уровнях

На верхних уровнях обычно используется жадное приближение:

```text
current = entry_point

repeat:
    посмотреть соседей current
    выбрать более близкого
until no improvement
```

Когда улучшений нет, мы не возвращаем финальный ответ.

Мы спускаемся на следующий уровень с найденной вершины.

Это фундаментальная разница с однослойным NSW:

> Локальный минимум верхнего слоя становится не окончательным ответом, а хорошей стартовой точкой для более плотного нижнего слоя.

---

# 20. Что такое efSearch

Для поиска одного соседа достаточно идти по одному лучшему пути.

Но для высокого Recall@K этого мало.

HNSW поддерживает список перспективных кандидатов.

`efSearch` определяет ширину поиска — сколько близких кандидатов алгоритм старается удерживать и исследовать на нижнем уровне.

Интуитивно:

```text
efSearch маленький:
- быстрее;
- меньше distance computations;
- выше риск пропустить нужную область.

efSearch большой:
- медленнее;
- исследуется больше вершин;
- обычно выше recall.
```

Практическое ограничение:

```text
efSearch >= K
```

Иначе невозможно разумно поддерживать пул меньше числа требуемых результатов.

Главный online trade-off:

```text
efSearch ↑ → Recall ↑, latency ↑
```

# 21. Как широкий поиск отличается от простого greedy

Простой greedy хранит одну текущую вершину.

Расширенный поиск хранит два набора:

1. **candidates** — вершины, которые ещё стоит исследовать;
2. **best found** — лучшие найденные вершины.

Часто используются priority queues.

Упрощённая логика:

```text
положить entry point в candidates и results

while candidates не пуст:
    взять самого перспективного кандидата

    если он уже хуже худшего из текущих результатов:
        можно завершить

    исследовать его соседей

    хорошие новые вершины добавить
```

Так алгоритм может временно исследовать несколько направлений, а не навсегда привязываться к одному жадному пути.

# 22. Условие остановки широкого поиска

Пусть:

- ближайший необработанный кандидат имеет distance `d_candidate`;
- худший элемент в текущем top-`ef` имеет distance `d_worst`.

Если:

```text
d_candidate > d_worst
```

то дальнейшее исследование становится менее перспективным.

В зависимости от точной реализации поиск может завершиться.

Это позволяет не обходить весь граф.

---

# 23. Пошаговый пример поиска

Пусть query относится к кластеру `L`.

## Верхний слой

```text
entry A
neighbors: B, D
```

Расстояния:

```text
A = 12
B = 10
D = 7
```

Переходим в `D`.

Соседи `D`:

```text
C = 8
E = 4
F = 6
```

Переходим в `E`.

У `E` нет более близкого соседа на этом слое.

## Средний слой

Спускаемся с `E`.

На более плотном слое появляются новые соседи:

```text
G = 3.5
H = 2.9
J = 4.1
```

Переходим к `H`.

## Нижний слой

С `H` запускается более широкий search с `efSearch`.

Исследуются несколько ветвей, находятся:

```text
L = 0.3
M = 0.4
N = 0.6
```

Возвращается нужный top-K.

---

# 24. Построение HNSW: почему оно сложнее поиска

Индекс строится последовательной вставкой точек.

Для каждой новой точки нужно:

1. назначить максимальный уровень;
2. найти подходящую область графа;
3. на нужных уровнях найти кандидатов в соседи;
4. выбрать из кандидатов ограниченное число рёбер;
5. добавить связи;
6. при необходимости обрезать слишком большое число рёбер у существующих вершин;
7. обновить entry point, если новая точка попала выше всех.

Это не просто «соединить с M математически ближайшими точками».

Качество отбора соседей определяет навигационность графа.

---

# 25. Первый этап вставки

Пусть новая вершина получила максимальный уровень `l_new`.

Глобальный максимальный уровень графа равен `L`.

Если:

```text
L > l_new
```

то на уровнях:

```text
L, L-1, ..., l_new + 1
```

новая вершина не будет вставляться.

На этих слоях алгоритм только ищет хороший entry point для нижних уровней.

То есть выполняется жадный поиск по одному ближайшему кандидату.

---

# 26. Второй этап вставки

Начиная с уровня:

```text
min(L, l_new)
```

и до `0`, новая вершина должна получить рёбра.

На каждом таком уровне:

1. выполняется широкий поиск;
2. собираются до `efConstruction` кандидатов;
3. из них выбираются до `M` соседей;
4. добавляются связи;
5. найденные кандидаты используются как entry points на нижнем уровне.

---

# 27. Что такое M

`M` — один из главных structural hyperparameters.

Он задаёт ориентир по числу соседей, добавляемых новой вершине.

Больше `M`:

- больше маршрутов;
- выше шанс хорошей навигации;
- больше память;
- больше distance computations во время поиска;
- дольше построение;
- иногда выше recall.

Меньше `M`:

- более компактный индекс;
- быстрее обход отдельных вершин;
- выше риск разрывов, локальных минимумов и плохого recall.

Нельзя выбирать `M` только по принципу «чем больше, тем лучше».

---

# 28. Mmax и Mmax0

Реализации ограничивают максимальную степень вершины.

Часто различают:

- `Mmax` — максимум рёбер на верхних слоях;
- `Mmax0` — максимум рёбер на нижнем слое.

Нижний слой обычно плотнее, потому что именно там выполняется точное локальное уточнение.

В разных библиотеках конкретные правила и значения могут отличаться.

Поэтому на работе всегда нужно смотреть документацию конкретной реализации, а не переносить параметризацию из статьи механически.

---

# 29. Что такое efConstruction

`efConstruction` задаёт ширину поиска при построении индекса.

Во время вставки нужно найти хорошие варианты соседей.

Если `efConstruction` маленький:

- индекс строится быстрее;
- исследуется мало кандидатов;
- структура может получиться слабой;
- потолок recall может быть ниже.

Если `efConstruction` большой:

- больше кандидатов;
- качественнее связи;
- дольше build;
- выше временные вычислительные затраты.

Trade-off:

```text
efConstruction ↑
→ build time ↑
→ часто index quality ↑
→ потенциальный Recall при search ↑
```

Это offline-параметр построения.

`efSearch` — online-параметр запроса.

Их нельзя путать.

---

# 30. Наивный выбор соседей

После поиска мы имеем `efConstruction` кандидатов.

Наивная стратегия:

```text
взять M ближайших к новой точке
```

На первый взгляд это идеально.

Но возникает проблема: выбранные соседи могут находиться почти в одном направлении и быть тесно связаны между собой.

Тогда новая вершина получает много избыточных рёбер в одну область.

Граф локально точный, но глобально плохо навигируется.

---

# 31. Почему diversity соседей важна

Представим новую точку `X`.

Рядом с ней находятся:

```text
B, C, D
```

Все три почти совпадают и относятся к одному плотному кластеру.

Чуть дальше находится `A`, ведущая в другую область.

Наивный top-M может выбрать:

```text
B, C
```

Эвристика может предпочесть:

```text
B, A
```

Хотя `A` дальше от `X`, она создаёт новый полезный маршрут.

Это может улучшить connectivity и снизить вероятность локального минимума.

Главная идея:

> Хороший сосед — не только близкая точка, но и точка, добавляющая новое направление навигации.

---

# 32. Эвристика отбора соседей

Упрощённо:

1. кандидаты сортируются по расстоянию до новой точки `X`;
2. ближайший принимается;
3. следующий кандидат `C` принимается, если он не слишком близок к уже выбранным соседям относительно расстояния до `X`;
4. процесс продолжается до `M` связей.

Одна из интуитивных форм проверок:

```text
distance(C, X) < distance(C, selected_neighbor)
```

для выбранных соседей.

Если кандидат гораздо ближе к уже выбранному соседу, чем к `X`, отдельное ребро `X → C` может быть избыточным.

Точная реализация эвристики имеет дополнительные детали, но бизнес-интуиция именно такая:

```text
близость + разнообразие направлений
```

---

# 33. Почему рёбра ограничиваются

При добавлении новой вершины старые вершины тоже получают обратные связи.

Их степень может вырасти выше лимита.

Тогда нужно выполнить pruning:

1. собрать старые и новые связи;
2. снова применить heuristic selection;
3. оставить не больше допустимого числа.

Без ограничения граф постепенно стал бы слишком плотным.

Проблемы плотного графа:

- большая память;
- больше соседей проверяется на каждом шаге;
- дорогие updates;
- плохое масштабирование.

---

# 34. Почему HNSW называют state-of-the-art ANN-подходом

Причина не в одной магической формуле.

HNSW сочетает:

- proximity graph;
- small-world navigation;
- hierarchy масштабов;
- вероятностное распределение уровней;
- широкий search frontier;
- diversity heuristic;
- ограниченную степень вершин;
- хороший recall-latency trade-off.

При этом индекс поддерживает последовательную вставку точек.

Сильная сторона HNSW — высокое качество поиска при низкой latency, особенно когда индекс помещается в быструю память.

---

# 35. Теоретическая сложность и осторожность в формулировках

В оригинальной работе авторы объясняют логарифмическое масштабирование search complexity благодаря иерархии.

Упрощённая запись:

```text
Search: O(log N)
Insertion: O(log N)
Build: O(N log N)
```

Но на собеседовании нельзя говорить об этих формулах как о безусловной гарантии для любого датасета и реализации.

Реальная производительность зависит от:

- распределения данных;
- intrinsic dimensionality;
- dimension;
- метрики;
- `M`;
- `efConstruction`;
- `efSearch`;
- аппаратуры;
- layout памяти;
- фильтров;
- sharding;
- обновлений;
- конкретной библиотеки.

Корректная формулировка:

> HNSW спроектирован для логарифмически масштабируемой навигации в среднем/эмпирически, но production performance нужно измерять benchmark’ом на своих данных.

# 36. Память HNSW

HNSW хранит:

1. сами векторы или ссылку на них;
2. уровни каждой вершины;
3. списки рёбер на слоях;
4. служебные структуры;
5. идентификаторы и metadata вне индекса или рядом с ним.

Приблизительно память графа растёт с:

```text
N × число рёбер на вершину
```

Плюс память векторов:

```text
N × D × bytes_per_component
```

Пример только для float32-векторов:

```text
N = 10 000 000
D = 768
4 bytes на число
```

Сырые векторы:

```text
10 000 000 × 768 × 4
≈ 30.72 GB
```

И это ещё без графа и overhead.

Поэтому HNSW часто требует много RAM.

---

# 37. Почему память — важный production trade-off

Высокий `M` может улучшить connectivity, но увеличивает:

- memory footprint;
- cache misses;
- стоимость репликации;
- время загрузки индекса;
- стоимость каждой машины;
- время восстановления после сбоя.

На небольшом benchmark улучшение Recall на `0.2%` может выглядеть полезно.

На миллиарде векторов это может означать десятки или сотни гигабайт дополнительной памяти.

DS/MLE должен считать не только метрики качества, но и инфраструктурную цену.

# 48. Offline evaluation

Для оценки ANN нужен exact baseline на доступном benchmark-наборе.

Для каждого query:

1. exact search выдаёт истинный top-K;
2. HNSW выдаёт approximate top-K;
3. считается overlap.

```text
Recall@K = |ExactTopK ∩ ANNTopK| / K
```

Пример:

```text
exact = {1, 2, 3, 4, 5}
approx = {1, 2, 3, 8, 9}

Recall@5 = 3 / 5 = 0.6
```

Нужно строить не одну цифру, а кривые:

```text
Recall@K vs QPS
Recall@K vs p95 latency
Recall@K vs memory
Recall@K vs build time
```

# 49. Почему средний Recall недостаточен

Средний Recall может быть высоким, но важный сегмент провален.

Проверяйте:

- head queries;
- tail queries;
- разные языки;
- регионы;
- категории;
- новые объекты;
- редкие классы;
- плотные кластеры;
- разреженные области;
- разные нормы embeddings;
- разные типы фильтров.

Пример:

```text
Overall Recall@100 = 0.97
Rare safety incidents Recall@100 = 0.58
```

Для safety-сценария общая цифра вводит в заблуждение.

Степень/связность графа.

- search branching.

## efConstruction

Ширина поиска соседей во время построения.

Ширина поиска во время запроса.

# 52. Как тюнить параметры на практике

Не существует одного лучшего набора для всех данных.

Процесс:

1. Зафиксировать dataset и query sample.
2. Построить exact ground truth.
3. Выбрать несколько значений `M`.
4. Для каждого `M` попробовать `efConstruction`.
5. Для каждого индекса прогнать сетку `efSearch`.
6. Измерить Recall, latency, QPS, memory, build time.
7. Отбросить варианты, нарушающие SLA или memory budget.
8. Выбрать Pareto-optimal configuration.
9. Проверить сегменты и фильтры.
10. Провести online experiment.

Не надо выбирать параметр по одному blog post.

# 53. Pareto frontier

Конфигурация A:

```text
Recall = 0.95
latency = 10 ms
```

Конфигурация B:

```text
Recall = 0.96
latency = 30 ms
```

Конфигурация C:

```text
Recall = 0.94
latency = 15 ms
```

C хуже A и по recall, и по latency.

Она dominated и не нужна.

A и B представляют разные trade-offs.

Выбор между ними зависит от продукта.

# 54. HNSW и Product Quantization

HNSW хорошо ищет, но может потреблять много памяти.

Product Quantization сжимает векторы.

Возможны комбинации:

- HNSW для coarse navigation;
- IVF для разбиения на partitions;
- PQ для компактного хранения и приблизительных расстояний;
- exact rerank на исходных векторах для top candidates.

Общий принцип:

```text
быстрый грубый этап
→ компактный approximate этап
→ точное уточнение малого top-N
```

Но комбинация увеличивает сложность системы и число параметров.

---

# 55. Что такое IVF в этой комбинации

IVF разбивает пространство на Voronoi cells вокруг centroids.

Обычная логика:

1. найти ближайшие centroids;
2. искать только в соответствующих inverted lists.

Если centroids очень много, brute-force поиск ближайшего centroid тоже становится дорогим.

HNSW можно построить поверх centroids.

Тогда:

```text
query
→ HNSW на centroids
→ несколько ближайших partitions
→ поиск внутри partitions
```

Это пример использования графа не по всем документам, а по coarse representatives.

# 56. Когда HNSW может быть плохим выбором

HNSW не универсален.

Возможные причины выбрать другое решение:

1. Датасет маленький — brute force проще и достаточно быстр.
2. Очень жёсткий memory budget.
3. Данные не помещаются в RAM.
4. Очень частые массовые updates/rebuilds.
5. Нужны сложные filters, плохо совместимые с реализацией.
6. Требуется строгая exactness.
7. GPU batch brute force уже удовлетворяет SLA.
8. Disk-based ANN подходит лучше.
9. Данные имеют специальную структуру, которую лучше использует другой индекс.
10. Операционная сложность HNSW не окупается.

Правильный инженер не говорит:

> HNSW всегда лучший.

Он говорит:

> Сравним индексы на нашей нагрузке и выберем по требованиям.

---

# 57. Когда brute force — правильное решение

Brute force полезен, если:

- `N` небольшой;
- query batch большой и хорошо работает на GPU;
- обновления очень частые;
- exactness обязательна;
- нужен baseline;
- разработка должна быть максимально простой;
- стоимость индекса выше выигрыша.

Exact search также нужен для ground truth при оценке ANN.

# 60. Самая важная разница между NSW и HNSW

NSW:

```text
один слой
случайный или заданный entry point
greedy navigation
длинные и короткие масштабы перемешаны в одном графе
```

HNSW:

```text
несколько слоёв
редкие верхние слои
плотный layer 0
scale separation
поиск сверху вниз
широкий search frontier
```

Коротко:

> HNSW превращает смешанные масштабы NSW в управляемую иерархию.


# 64. Как объяснить HNSW ребёнку

Представь огромный город.

Тебе нужен конкретный дом.

Плохой способ:

> Проверить каждый дом города.

NSW:

> Идти по дорогам, каждый раз выбирая дорогу, которая приближает к дому.

Проблема:

> Можно застрять в неправильном районе.

HNSW:

> Сначала посмотреть на карту всей страны, потом города, потом района, потом улицы.

- верхние карты помогают быстро выбрать район;
- нижняя карта помогает найти конкретный дом.

`M`:

> Сколько дорог оставлять у каждой точки.

`efConstruction`:

> Насколько внимательно строить карту.

`efSearch`:

> Сколько вариантов дороги проверять во время поездки.

# 65. Финальная модель понимания

HNSW состоит из пяти идей:

## Идея 1. Proximity graph

Похожие точки связаны рёбрами.

## Идея 2. Small world

Есть маршруты разных пространственных масштабов.

## Идея 3. Hierarchy

Масштабы разделены по уровням.

## Идея 4. Greedy + beam-like search

Идём к query, сохраняя несколько перспективных кандидатов.

## Идея 5. Diversified neighbor selection

Выбираем не только ближайшие, но и полезные для навигации связи.

# 68. Один итоговый абзац

HNSW заранее превращает коллекцию embeddings в многоуровневую дорожную сеть. Редкие верхние уровни позволяют быстро приблизиться к нужной части пространства, а плотный нижний уровень — найти качественный top-K. Параметр `M` управляет связностью и памятью, `efConstruction` — тщательностью построения, `efSearch` — шириной online-поиска. Алгоритм не гарантирует exact top-K, зато при правильном тюнинге даёт сильный recall-latency trade-off и поэтому широко используется как candidate-generation слой в поиске, рекомендациях, RAG, matching и других системах работы с embeddings.
