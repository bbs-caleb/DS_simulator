# 2. Что требует первая задача

Нужно реализовать:

```python
distance(...)
create_sw_graph(...)
nsw(...)
```

Граф строится сразу для всей матрицы `data`.

Для каждой точки:

1. считаются расстояния;
2. сортируются индексы;
3. берутся ближайшие кандидаты;
4. случайно выбираются короткие рёбра;
5. берутся самые далёкие кандидаты;
6. случайно выбираются длинные рёбра.

Поиск:

1. выбирает случайные старты;
2. жадно переходит к более близкому соседу;
3. сохраняет все посещённые точки;
4. возвращает лучшие из посещённых.

---

# 3. Что описывает статья для NSW

Классический NSW строится последовательной вставкой:

```text
shuffle points
→ insert one by one
→ search existing graph
→ connect new point to neighbours
```

Длинные связи часто появляются естественно у ранних точек.

Когда граф ещё маленький, ближайшие доступные соседи могут быть геометрически далеко.

После роста графа эти ранние рёбра становятся мостами между областями.

---

# 4. Что описывает статья для HNSW

HNSW добавляет:

- случайный максимальный слой каждой вершины;
- многоуровневый граф;
- глобальный entry point;
- top-down search;
- `M`;
- `efConstruction`;
- `efSearch`;
- эвристику diversity при выборе соседей;
- pruning степени вершин.

Ничего этого первая задача напрямую не требует.

# 5. Таблица различий

| Свойство | Задание курса | Классический NSW | HNSW |
|---|---|---|---|
| Число слоёв | 1 | 1 | несколько |
| Построение | полный расчёт расстояний для каждой точки | последовательная вставка | последовательная вставка по слоям |
| Локальные рёбра | случайно из ближайшего пула | к найденным близким точкам | к найденным кандидатам с эвристикой |
| Дальние рёбра | явно выбираются из самых далёких | возникают естественно | представлены верхними слоями/масштабами |
| Entry point | случайные старты | entry point | верхний глобальный entry point |
| Борьба с local minima | несколько стартов | расширенный поиск/старты | hierarchy + ef search |
| Основные параметры | short/long candidates и edges | degree/search settings | M, efConstruction, efSearch |
| Цель | учебное понимание | navigable graph | production-grade ANN идея |

# 6. Почему нельзя заменить create_sw_graph на HNSW builder

Автотесты ожидают словарь:

```python
Dict[int, List[int]]
```

и поведение параметров:

```text
num_candidates_for_choice_long
num_edges_long
num_candidates_for_choice_short
num_edges_short
```

Если реализовать слои, `M`, `efConstruction` и insertion, это нарушит контракт.

Даже если алгоритм будет сильнее, тесты могут проверить:

- число ключей;
- число рёбер;
- использование `dist_f`;
- форму результата;
- логику short/long candidates;
- отсутствие self-edge;
- число возвращаемых индексов.

---

# 7. Почему в задании берутся самые далёкие точки

Это педагогический способ принудительно добавить long-range edges.

В настоящем NSW/HNSW хорошие дальние маршруты возникают более естественно:

- через порядок вставки;
- через иерархию;
- через эвристику связности.

Выбор абсолютно самых далёких точек не является универсальным production-рецептом.

Он нужен, чтобы новичок увидел:

```text
только близкие связи недостаточны
```

---

# 8. Почему курс использует несколько случайных стартов

Один greedy-маршрут может застрять.

Несколько стартов:

- покрывают разные части графа;
- увеличивают число посещённых вершин;
- повышают шанс найти хороший кластер;
- позволяют вернуть `search_k` уникальных объектов.

В HNSW схожая цель достигается иначе:

- верхними слоями;
- хорошим entry point;
- candidate frontier;
- `efSearch`.

---

# 9. Сопоставление переменных курса с идеями HNSW

Это не точное равенство, а смысловая аналогия.

| Код курса | Смысл | Ближайшая идея HNSW |
|---|---|---|
| `num_edges_short` | локальная связность | `M` на плотном слое |
| `num_edges_long` | дальние переходы | верхние уровни |
| `num_start_points` | больше маршрутов | более широкий frontier / efSearch |
| `all_visited_points` | кэш и множество исследованных вершин | visited set |
| greedy переход | локальное улучшение | greedy descent |
| `search_k` | число результатов | `K` |

# 10. Строка distance

Курс:

```python
dist = np.linalg.norm(pointA - documents, axis=1, keepdims=True)
```

Это векторизованный exact distance calculation для переданного набора документов.

HNSW не отменяет эту операцию.

Он уменьшает число документов, для которых она вызывается.

Ключевая мысль:

```text
ускорение ANN не потому, что одно расстояние стало дешевле,
а потому, что мы считаем намного меньше расстояний
```

---

# 11. Что делает граф

Без графа:

```text
query сравнивается с N точками
```

С графом:

```text
query сравнивается только с посещёнными вершинами и их соседями
```

Если:

```text
visited_count << N
```

мы экономим вычисления.

---

# 12. Что означает кэш в calc_d_and_upd

Одна вершина может быть соседом нескольких текущих точек.

Без кэша:

```text
одно и то же расстояние считается повторно
```

С кэшем:

```text
index → already computed distance
```

В настоящих ANN-реализациях visited structures также критичны для:

- избежания циклов;
- сокращения повторных вычислений;
- эффективного traversal.

---

# 13. Почему OrderedDict не является сутью HNSW

`OrderedDict` в решении курса удобен, чтобы синхронно хранить:

```text
keys = индексы
values = расстояния
```

В production используются более эффективные структуры:

- arrays;
- bitsets;
- heaps;
- custom allocators;
- SIMD-friendly layouts.

Нельзя делать вывод, что HNSW концептуально зависит от Python `OrderedDict`.

---

# 14. Почему defaultdict не является сутью графа

`defaultdict(list)` — удобная Python-обёртка.

Настоящий индекс может хранить соседей:

- contiguous arrays;
- compact adjacency lists;
- memory-mapped regions;
- compressed IDs;
- off-heap structures.

Учебный код показывает алгоритм, а не оптимальный memory layout.

---

# 15. Главный недостаток create_sw_graph

Для каждой точки считаются расстояния до всех точек.

Сложность около:

```text
O(N²D)
```

Это невозможно для огромного датасета.

Например:

```text
N = 10 000 000
```

Число пар порядка:

```text
10^14
```

Поэтому функция курса нужна для понимания, а не для production build.

HNSW строится инкрементально и ищет соседей через уже существующий индекс.

---

# 16. Почему use_sampling не превращает код в production HNSW

Sampling уменьшает число сравнений при построении.

Но остаются проблемы:

- случайный sample может не содержать хороших соседей;
- нет hierarchy;
- нет neighbor diversity heuristic;
- нет degree pruning;
- нет incremental insertion logic;
- нет persistence;
- нет concurrency;
- нет updates/deletes architecture.

Sampling — полезная оптимизация учебного builder, но не HNSW.

---

# 17. Как объяснить переход на интервью

Хороший ответ:

> В учебной реализации мы вручную создаём локальные и дальние рёбра и используем несколько случайных стартов. Это показывает основную интуицию NSW. HNSW заменяет смешанные дальние масштабы многоуровневой структурой: верхние слои отвечают за крупные переходы, нижний — за локальный поиск. Кроме того, HNSW строится последовательной вставкой и использует `M`, `efConstruction`, `efSearch` и diversity heuristic.

# 19. Как самостоятельно проверить связь

## Шаг 1

Запустить exact top-K:

```python
exact = np.argsort(distance(query, data).reshape(-1))[:10]
```

## Шаг 2

Запустить учебный NSW:

```python
approx = nsw(query, data, graph, search_k=10)
```

## Шаг 3

Посчитать Recall@10:

```python
recall = len(set(exact) & set(approx)) / 10
```

## Шаг 4

Изменять:

```text
num_edges_short
num_edges_long
num_start_points
```

## Шаг 5

Наблюдать:

```text
Recall ↔ число вычислений ↔ время
```

Это маленькая модель тех же trade-offs, которые в HNSW контролируются через `M` и `ef`.
