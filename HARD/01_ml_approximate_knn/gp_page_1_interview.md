# 9. Почему поиск обычно двухэтапный

## Этап 1 — retrieval

- очень быстрый;
- работает по огромной коллекции;
- возвращает сотни кандидатов;
- оптимизирован на высокий recall;
- допускает небольшую потерю точности.

## Этап 2 — reranking

- работает по небольшому набору;
- использует тяжёлую модель;
- учитывает дополнительные признаки;
- формирует финальный порядок.

Сильный ответ:

> Я не стал бы заставлять ANN решать всю задачу ранжирования. Его роль — быстро дать набор кандидатов с высоким recall. Коммерческие, поведенческие, safety и контекстные признаки лучше использовать на reranking-этапе.

---

# 10. Где применять фильтры

## До ANN

Сначала ограничить пространство, потом искать.

Плюсы:

- меньше кандидатов;
- жёсткие ограничения соблюдаются сразу.

Минусы:

- не каждый индекс эффективно поддерживает произвольные фильтры;
- маленький сегмент может ухудшить навигацию.

## После ANN

Сначала retrieval, затем фильтрация.

Плюсы:

- простая архитектура;
- один общий индекс.

Минусы:

- после фильтра может остаться мало результатов.

Практическая стратегия:

```text
нужно вернуть 50;
извлекаем 500;
фильтруем;
rerank;
возвращаем 50.
```

Если после фильтра осталось мало объектов, можно выполнить повторный поиск с большим candidate pool.

# 14. Freshness

Разные продукты требуют разной свежести.

## Новости

Новый контент должен попадать в индекс очень быстро.

## E-commerce

Цена и остаток могут обновляться как metadata чаще, чем semantic embedding карточки.

## Dating

Профильные признаки меняются умеренно, поведенческие — быстро.

## Fraud

Новые паттерны могут требовать near-real-time обновления.

Варианты архитектуры:

- периодический batch rebuild;
- incremental insert;
- основной индекс плюс delta-index;
- периодическое слияние;
- hot и cold tiers.

# 15. Sharding

Когда индекс не помещается на одну машину, его делят.

## Random sharding

Объекты распределяются случайно.

Каждый запрос отправляется во все shard, а результаты объединяются.

Плюсы:

- простое балансирование.

Минусы:

- fan-out;
- сетевые расходы;
- tail latency определяется медленным shard.

## Semantic sharding

Объекты делятся по темам или кластерам.

Плюсы:

- можно искать только в релевантных shard.

Минусы:

- routing может ошибиться;
- сложнее балансировать;
- распределение меняется со временем.

## Business sharding

Деление по:

- региону;
- языку;
- типу контента;
- tenant;
- домену.

Подходит, если запросы естественно ограничены бизнесовым сегментом.

---

# 16. Replication

Реплики нужны для:

- высокой доступности;
- распределения QPS;
- failover;
- безопасного обновления.

Контролируйте, чтобы реплики использовали одинаковую версию индекса. Иначе одинаковые запросы будут давать разные результаты из-за рассинхронизации.

---

# 17. Cache

Возможные уровни:

- query embedding cache;
- retrieval result cache;
- popular query cache;
- document metadata cache;
- reranker feature cache.

Кэш опасен при:

- персонализации;
- быстро меняющемся наличии;
- access control;
- удалении контента;
- privacy constraints.

Cache key должен включать существенные параметры и версии.

---

# 18. Fallback

Если ANN-сервис недоступен, варианты:

- lexical search;
- cached results;
- category bestsellers;
- popularity list;
- rules-based retrieval;
- меньший backup-index.

Плохой fallback:

> При падении ANN сделаем exact scan по ста миллионам объектов.

Такой fallback может обрушить всю систему.

# 20. Как измерять Recall@K в production

Exact поиск на каждом запросе дорог.

Можно:

1. Семплировать небольшую долю запросов.
2. Асинхронно запускать exact baseline.
3. Сравнивать ANN top-K с exact top-K.
4. Строить Recall@K по сегментам.

Сегменты:

- язык;
- регион;
- категория;
- head/tail query;
- новый/старый пользователь;
- устройство;
- тип фильтра;
- embedding norm.

Средний Recall может скрывать серьёзный провал в одном важном сегменте.

# 22. A/B тест

Даже если новый индекс быстрее и имеет хороший offline Recall, нужен online experiment.

Пример гипотезы:

> Более быстрый retrieval уменьшит время ответа и повысит долю успешных поисковых сессий.

Primary metric:

- conversion;
- successful search rate;
- successful match rate.

Guardrails:

- latency;
- zero-result rate;
- complaint rate;
- diversity;
- safety violations;
- downstream cost.

# 23. Пример ответа для marketplace

> У нас 50 миллионов карточек, и retrieval должен вернуть кандидатов за 30 мс. Я использую двухэтапную систему. Запрос и товары кодируются совместимой embedding-моделью. ANN-индекс возвращает около 500 кандидатов с высоким Recall@500. Затем применяются наличие, регион, политика продавца и другие обязательные фильтры. Reranker использует текстовые, поведенческие, ценовые и коммерческие признаки и формирует top-50. Индекс версионируется вместе с embedding-моделью. Retrieval оценивается по Recall@K относительно exact baseline, а продуктовый эффект — через A/B тест по conversion, CTR и guardrail latency.

---

# 24. Пример ответа для финтех-антифрода

> Я не использовал бы ближайших соседей как единственное основание для блокировки. ANN нужен как быстрый генератор похожих исторических кейсов и признаков. Транзакция кодируется в embedding, индекс возвращает похожие операции, после чего supervised risk model и policy engine принимают решение. Важны event-time корректность, отсутствие leakage, freshness и контроль false positive. Кроме retrieval Recall я измерял бы fraud loss, approval rate, customer friction и latency.

---

# 25. Пример ответа для RAG

> Документы разбиваются на chunks. Каждый chunk получает embedding и metadata: источник, tenant, permissions, дата и тип документа. Query embedding используется для ANN retrieval. Access-control фильтры обязательны: модель не должна увидеть недоступный пользователю chunk. После retrieval применяется hybrid search или reranker. Метрики: Recall@K по размеченным вопросам, groundedness, citation correctness, latency и zero-evidence rate. Индекс и embedding-модель версионируются совместно.

---

# 26. Почему не всегда ANN

Сильный ответ:

> Exact KNN — правильный baseline и может быть лучшим решением при небольшом N, низком QPS или абсолютном требовании к точности. ANN нужен только тогда, когда полный перебор не укладывается в latency или стоимость. Выбор нужно подтверждать benchmark, а не модой.

---

# 27. Почему граф

> Граф сохраняет локальную структуру пространства и позволяет двигаться от текущей вершины к более близким соседям. Дальние связи ускоряют переход между областями, а локальные помогают уточнять результат. Поэтому вычисляются расстояния только для посещённых вершин, а не для всей коллекции.

---

# 28. Почему несколько стартов

> Жадная навигация может остановиться в локальном минимуме. Несколько стартов исследуют разные области и повышают вероятность попасть в правильный кластер. Цена — дополнительные вычисления и latency.

---

# 29. NSW и HNSW

> NSW использует один граф малого мира. HNSW добавляет иерархию слоёв. Верхние разреженные слои нужны для быстрых дальних переходов, нижний плотный слой — для локального уточнения. Это улучшает масштабирование и качество навигации.

---

# 30. Основные trade-offs

1. Recall ↔ latency.
2. Recall ↔ memory.
3. Build time ↔ index quality.
4. Freshness ↔ rebuild cost.
5. Filter flexibility ↔ search efficiency.
6. Replication ↔ infrastructure cost.
7. Embedding complexity ↔ online latency.
8. Candidate count ↔ reranker cost.
