# SKU Embeddings — пошаговое объяснение User-Item Matrix с нуля

# 1. Что лежит во входной таблице

Исходная таблица содержит транзакции:

```text
timestamp | order_id | user_id | item_id | price | units
```

Пример строки:

```text
2020-01-01 00:00:38 | 3132 | 36914 | 13490 | 418.95 | 1
```

Расшифровка:

- `timestamp` — дата и время операции;
- `order_id` — идентификатор заказа;
- `user_id` — идентификатор покупателя;
- `item_id` — идентификатор товара;
- `price` — цена товара в транзакции;
- `units` — количество единиц товара.

Один пользователь может покупать один товар несколько раз. Поэтому перед построением матрицы данные нужно агрегировать.

---

# 2. Какой результат должен вернуть SQL

Требуются колонки:

```text
user_id
item_id
qty
price
```

Одна строка описывает одну пару:

```text
пользователь + товар
```

Но показатели считаются по-разному.

## `qty`

Количество товара, купленное конкретным пользователем:

```text
GROUP BY user_id, item_id
```

## `price`

Средняя цена товара среди всех транзакций выбранного периода:

```text
GROUP BY item_id
```

Это главный нюанс задания.

---

# 3. Почему нельзя считать всё одной группировкой

Допустим, есть данные:

| user_id | item_id | units | price |
|---:|---:|---:|---:|
| 1 | 10 | 1 | 100 |
| 1 | 10 | 2 | 110 |
| 2 | 10 | 1 | 120 |

Для пользователя 1 количество:

```text
1 + 2 = 3
```

Средняя цена товара 10 среди всех транзакций:

```text
(100 + 110 + 120) / 3 = 110
```

Если написать:

```sql
GROUP BY user_id, item_id
```

и там же:

```sql
avg(price)
```

для пользователя 1 получится:

```text
(100 + 110) / 2 = 105
```

Это другая величина.

Поэтому нужны две подвыборки.

---

# 4. Первая подвыборка: количество

```sql
SELECT
    user_id,
    item_id,
    sum(units) AS qty
FROM default.karpov_express_orders
```

`sum(units)` складывает именно единицы товара.

Это не то же самое, что:

```sql
count(units)
```

Пример:

```text
units = 3
units = 2
```

Результаты:

```text
count = 2 строки
sum   = 5 единиц
```

Задаче требуется `sum`.

---

# 5. Фильтр по датам

```sql
WHERE toDate(timestamp) >= toDate(%(start_date)s)
  AND toDate(timestamp) <= toDate(%(end_date)s)
```

## Почему `toDate`

`timestamp` содержит время:

```text
2022-10-20 14:25:30
```

После `toDate`:

```text
2022-10-20
```

## Почему используются `>=` и `<=`

Обе границы периода должны входить в выгрузку.

## Что означает `%(start_date)s`

Это параметр, который подставит проверяющая система.

Пример параметров:

```python
{
    "start_date": "2022-10-20",
    "end_date": "2022-10-23",
}
```

---

# 6. Группировка количества

```sql
GROUP BY
    user_id,
    item_id
```

После неё для каждой пары пользователь–товар остаётся одна строка.

Подвыборка получает имя:

```sql
AS sales
```

Поэтому снаружи можно обратиться:

```sql
sales.user_id
sales.item_id
sales.qty
```

---

# 7. Вторая подвыборка: средняя цена

```sql
SELECT
    item_id,
    round(avg(price), 2) AS price
FROM default.karpov_express_orders
```

Разберём выражение:

```sql
avg(price)
```

считает арифметическое среднее.

```sql
round(..., 2)
```

округляет до двух знаков после запятой.

```sql
AS price
```

даёт колонке требуемое имя.

---

# 8. Почему фильтр дат повторяется

Цена должна быть рассчитана за тот же период, что и количество:

```sql
WHERE toDate(timestamp) >= toDate(%(start_date)s)
  AND toDate(timestamp) <= toDate(%(end_date)s)
```

Если не поставить фильтр, получится средняя цена за всю историю, а не за выбранный период.

---

# 9. Группировка цены

```sql
GROUP BY item_id
```

Здесь специально нет `user_id`.

Один товар получает одну среднюю цену на весь выбранный период.

Подвыборка называется:

```sql
AS prices
```

---

# 10. Соединение результатов

```sql
INNER JOIN (...) AS prices
USING (item_id)
```

`USING (item_id)` означает:

> Соедини строки, у которых одинаковый `item_id`.

Теперь каждая строка `user_id + item_id + qty` получает среднюю цену товара.

---

# 11. Финальная сортировка

```sql
ORDER BY
    user_id,
    item_id
```

Сначала строки сортируются по пользователю, затем внутри пользователя по товару.

---

# 12. Переходим к Python

После SQL приходит DataFrame:

```text
user_id | item_id | qty | price
```

Нужно превратить его в разреженную матрицу.

# 13. Импорты

```python
from typing import Dict
```

```python
import pandas as pd
```

```python
from scipy.sparse import csr_matrix
```

Импортирует формат разреженной матрицы.

---

# 14. Создание класса

```python
class UserItemMatrix:
```

Класс объединяет данные и методы работы с ними.

Создание объекта:

```python
obj = UserItemMatrix(sales_data)
```

автоматически вызывает `__init__`.

---

# 15. Копия входных данных

```python
self._sales_data = sales_data.copy()
```

Класс сохраняет отдельную копию DataFrame.

Префикс `_` означает внутренний атрибут класса.

---

# 16. Уникальные пользователи

```python
user_ids = sorted(self._sales_data["user_id"].unique())
```

Пошагово:

1. выбирается колонка `user_id`;
2. `.unique()` удаляет повторы;
3. `sorted()` сортирует значения.

Пример:

```text
[5, 1, 4, 1, 2]
```

превратится в:

```text
[1, 2, 4, 5]
```

---

# 17. Уникальные товары

```python
item_ids = sorted(self._sales_data["item_id"].unique())
```

Логика полностью аналогична.

---

# 18. Количество пользователей и товаров

```python
self._user_count = len(user_ids)
self._item_count = len(item_ids)
```

Если уникальных пользователей четыре:

```text
_user_count = 4
```

---

# 19. Mapping пользователей

```python
self._user_map = {
    int(user_id): index
    for index, user_id in enumerate(user_ids)
}
```

Для:

```text
user_ids = [1, 2, 4, 5]
```

получается:

```python
{
    1: 0,
    2: 1,
    4: 2,
    5: 3,
}
```

Ключ — исходный ID.

Значение — номер строки матрицы.

---

# 20. Mapping товаров

```python
self._item_map = {
    int(item_id): index
    for index, item_id in enumerate(item_ids)
}
```

Значения словаря — номера столбцов матрицы.

---

# 21. Почему нельзя использовать реальные ID напрямую

Представим пользователей:

```text
1
1000
900000
```

Их всего три.

Если использовать ID как номера строк, понадобится матрица минимум на 900001 строку.

Mapping сжимает их до:

```text
0
1
2
```

---

# 22. Номера строк

```python
row_ind = (
    self._sales_data["user_id"]
    .map(self._user_map)
    .to_numpy(dtype=int)
)
```

`.map` заменяет ID индексом.

Пример:

```text
user_id = [1, 1, 4]
```

при mapping:

```text
1 -> 0
4 -> 1
```

становится:

```text
[0, 0, 1]
```

---

# 23. Номера столбцов

```python
col_ind = (
    self._sales_data["item_id"]
    .map(self._item_map)
    .to_numpy(dtype=int)
)
```

То же действие выполняется для товаров.

---

# 24. Значения матрицы

```python
values = pd.to_numeric(
    self._sales_data["qty"]
).to_numpy()
```

В матрицу записывается `qty`.

`price` в CSR на этом шаге не используется.

---

# 25. Конструктор CSR

```python
self._matrix = csr_matrix(
    (values, (row_ind, col_ind)),
    shape=(self._user_count, self._item_count),
)
```

Формат входа:

```text
(values, (rows, columns))
```

Пример:

```text
values  = [1, 5, 2]
rows    = [0, 0, 1]
columns = [0, 2, 1]
```

Матрица:

```text
[
    [1, 0, 5],
    [0, 2, 0],
]
```

Нули не хранятся явно.

---

# 26. Форма матрицы

```python
shape=(user_count, item_count)
```

Первая размерность — пользователи.

Вторая — товары.

---

# 27. Properties

Декоратор:

```python
@property
```

позволяет обращаться к методу как к атрибуту:

```python
obj.user_count
```

а не:

```python
obj.user_count()
```

Вычисления уже выполнены в `__init__`, поэтому property только возвращает готовое значение.

---

# 29. Почему `.toarray()` опасен в production

`.toarray()` превращает sparse-матрицу в плотную.

На большом каталоге это может привести к:

- переполнению памяти;
- падению процесса;
- долгому выполнению.

Использовать `.toarray()` следует только в маленьких тестах.

---

# 30. Как это связано с эмбеддингами

CSR — исходная матрица взаимодействий.

Дальше можно применить:

- cosine similarity;
- SVD;
- ALS;
- matrix factorization;
- LightFM;
- neural collaborative filtering.

Общая идея факторизации:

```text
User-Item Matrix
≈
User Embeddings × Item Embeddingsᵀ
```

Так большая разреженная матрица превращается в компактные векторы.
