# SKU Embeddings — User-Item Matrix

## 1. Что происходит на этой странице

На этой странице мы переходим от общей идеи товарных эмбеддингов к первой практической структуре данных — **User-Item Matrix**.

У нас есть таблица транзакций:

```text
timestamp | order_id | user_id | item_id | price | units
```

Каждая строка описывает отдельную покупку или товарную позицию внутри заказа.

Для рекомендательных алгоритмов такой формат неудобен. Нам нужна матрица:

```text
строки    = пользователи
столбцы   = товары
значения  = количество купленных единиц
```

Эта матрица позволяет сравнивать:

- пользователей между собой — по строкам;
- товары между собой — по столбцам.

---

# 2. Бизнесовый смысл

## 2.1. Что на самом деле хочет бизнес

Бизнес не просит нас «создать CSR-матрицу». Бизнес хочет решать прикладные задачи:

- находить похожие товары;
- строить персональные рекомендации;
- подбирать замену отсутствующему товару;
- искать товары-комплементы;
- оценивать каннибализацию;
- переносить знания на новые SKU;
- улучшать прогноз спроса;
- оптимизировать ассортимент;
- строить сегменты покупателей;
- повышать конверсию и средний чек.

CSR-матрица — это технический фундамент для этих задач.

---

## 2.2. Как один и тот же объект выглядит с двух сторон

Пользователь характеризуется товарами, которые он покупал.

Товар характеризуется пользователями, которые его покупали.

Пример:

| user_id | item_A | item_B | item_C |
|---|---:|---:|---:|
| 101 | 3 | 0 | 1 |
| 205 | 0 | 4 | 2 |
| 777 | 2 | 0 | 1 |

Строка пользователя 101:

```text
[3, 0, 1]
```

означает:

- 3 единицы товара A;
- 0 единиц товара B;
- 1 единица товара C.

Столбец товара A:

```text
[3, 0, 2]
```

показывает, как этот товар распределён по пользователям.

Если два столбца похожи, у товаров похожая аудитория.

# 3. Коллаборативная фильтрация

Коллаборативная фильтрация строится на предположении:

> Пользователи с похожим поведением выбирают похожие товары, а товары, которые выбирают похожие пользователи, связаны между собой.

Есть два классических взгляда.

## User-based collaborative filtering

Сравниваем строки.

Вопрос:

> Какие пользователи похожи на данного пользователя?

После этого можно рекомендовать товары, популярные у похожих пользователей.

## Item-based collaborative filtering

Сравниваем столбцы.

Вопрос:

> Какие товары покупает похожая аудитория?

Это особенно важно для SKU embeddings.

# 4. Почему сначала нужен SQL

Сырые транзакции могут содержать несколько строк для одной пары:

```text
user_id = 10, item_id = 25, units = 1
user_id = 10, item_id = 25, units = 2
user_id = 10, item_id = 25, units = 1
```

В матрице нужна одна ячейка:

```text
user 10, item 25 -> qty 4
```

Поэтому SQL должен:

1. ограничить данные по датам;
2. сгруппировать продажи по `user_id` и `item_id`;
3. сложить `units`;
4. отдельно рассчитать среднюю цену товара;
5. соединить две агрегации;
6. отсортировать результат.

---

# 5. Ключевая логика цены

Это самый важный и самый коварный момент задачи.

У `qty` и `price` разные уровни агрегации.

## `qty`

Считается для конкретной пары:

```text
user_id + item_id
```

Формула:

```sql
sum(units)
```

## `price`

Считается для товара среди всех транзакций выбранного периода:

```text
item_id
```

Формула:

```sql
round(avg(price), 2)
```

Цена не должна зависеть от конкретного пользователя.

Поэтому нельзя написать один простой запрос:

```sql
GROUP BY user_id, item_id
```

и в том же уровне считать:

```sql
avg(price)
```

Тогда получится средняя цена товара только для конкретного пользователя.

Правильное решение использует две отдельные агрегации:

```text
1. sales  -> GROUP BY user_id, item_id
2. prices -> GROUP BY item_id
```

Затем они соединяются по `item_id`.

---

# 6. Почему фильтр дат должен быть в обеих подвыборках

Задание требует выгружать данные за заданный период.

Если:

```text
start_date = 2022-10-20
end_date   = 2022-10-23
```

то используются транзакции:

```text
2022-10-20
2022-10-21
2022-10-22
2022-10-23
```

И количество, и средняя цена должны быть рассчитаны на одном и том же временном срезе.

Поэтому фильтр дат повторяется:

- в агрегации количества;
- в агрегации цены.

---

# 7. Почему используется `toDate(timestamp)`

`timestamp` содержит дату и время:

```text
2022-10-20 18:35:44
```

Параметры задачи задаются как даты:

```text
2022-10-20
```

Функция:

```sql
toDate(timestamp)
```

оставляет только календарную дату.

---

# 8. Почему обе границы включены

Условие задания требует включать и начальную, и конечную дату.

Используется:

```sql
toDate(timestamp) >= toDate(%(start_date)s)
AND
toDate(timestamp) <= toDate(%(end_date)s)
```

Это эквивалентно включительному `BETWEEN`.

---

# 9. Почему матрица разреженная

В реальном каталоге могут быть:

```text
1 000 000 пользователей
100 000 товаров
```

Полная матрица содержит:

```text
100 000 000 000 ячеек
```

Но один пользователь покупает лишь небольшую долю каталога.

Большинство значений равны нулю.

Хранить все нули бессмысленно и дорого.

---

# 10. Что такое CSR

CSR расшифровывается как:

```text
Compressed Sparse Row
```

Формат хранит:

- ненулевые значения;
- индексы столбцов;
- указатели начала строк.

Вместо плотной матрицы:

```text
[
    [0, 0, 5, 0],
    [2, 0, 0, 0],
    [0, 1, 0, 3],
]
```

хранятся только реальные взаимодействия:

```text
row 0, col 2 -> 5
row 1, col 0 -> 2
row 2, col 1 -> 1
row 2, col 3 -> 3
```

---

# 11. Зачем нужны `user_map` и `item_map`

Реальные ID могут быть большими и иметь пропуски:

```text
user_id: 4, 117, 900000
```

Нельзя использовать их как номера строк.

Создаётся отображение:

```python
{
    4: 0,
    117: 1,
    900000: 2,
}
```

То же самое для товаров.

---

# 12. Почему mappings сортируются

Сортировка делает результат:

- детерминированным;
- воспроизводимым;
- независимым от порядка входных строк;
- удобным для тестов;
- безопасным для сохранения модели.

Если DataFrame перемешать, mapping всё равно останется тем же.

---

# 13. Почему вычисления выполняются в `__init__`

Properties должны только возвращать готовые значения:

```python
matrix.user_count
matrix.item_count
matrix.user_map
matrix.item_map
matrix.csr_matrix
```

Матрица не должна пересчитываться при каждом обращении.

Поэтому всё создаётся один раз в `__init__`.

---

# 14. Почему `price` не входит в CSR

Значением ячейки является:

```text
qty
```

Цена нужна в агрегированном наборе данных, но не используется как сила взаимодействия в этой задаче.

В production цена может использоваться:

- как отдельный признак;
- для расчёта выручки;
- в ranker;
- для фильтрации;
- для price-aware recommendations.

# 16. Ограничения подхода

User-Item Matrix не знает:

- текст товара;
- изображение;
- категорию;
- наличие;
- факт показа;
- позицию;
- контекст;
- причину покупки.

Поэтому возникают:

- cold start;
- popularity bias;
- exposure bias;
- сезонность;
- влияние промо;
- влияние out-of-stock;
- sparsity.

В production чаще используется гибрид:

```text
collaborative signals
+ content
+ context
+ availability
+ price
+ business rules
```

# 17. Итог

Полный пайплайн этого шага:

```text
транзакции
    ↓
агрегация qty по user_id + item_id
    ↓
агрегация price по item_id
    ↓
JOIN по item_id
    ↓
mapping реальных ID
    ↓
CSR User-Item Matrix
    ↓
similarity / factorization / embeddings
```

Главная мысль:

> Смысл будущих эмбеддингов определяется тем, что именно мы положили в ячейки User-Item Matrix.
