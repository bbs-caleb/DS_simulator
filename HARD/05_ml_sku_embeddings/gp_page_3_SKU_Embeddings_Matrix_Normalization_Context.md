# SKU Embeddings — Matrix Normalization

## Бизнесовый и технический контекст задачи

# 1. Где мы находимся в общем пайплайне

На предыдущем шаге мы построили User-Item Matrix:

```text
строки    = пользователи
столбцы   = товары
значения  = количество купленных единиц
```

Пример:

| user_id | item_A | item_B | item_C |
|---|---:|---:|---:|
| 101 | 10 | 0 | 1 |
| 205 | 0 | 2 | 0 |
| 777 | 5 | 1 | 0 |

Такая матрица уже содержит полезный поведенческий сигнал. Но сырые количества покупок часто имеют очень разные масштабы.

Например:

- один пользователь покупает 5–10 единиц в месяц;
- другой закупает 500 единиц;
- один товар продаётся несколько раз;
- другой является массовым товаром ежедневного спроса;
- один пользователь обычный розничный покупатель;
- другой может быть корпоративным клиентом или оптовиком.

Если сразу считать похожесть или обучать модель на сырых значениях, крупные числа могут доминировать.

Поэтому следующий шаг — **нормализация**.

---

# 2. Что означает нормализация

Нормализация — это преобразование чисел, которое меняет масштаб, но старается сохранить полезную структуру данных.

В этой задаче реализуются четыре подхода:

1. нормализация по столбцам;
2. нормализация по строкам;
3. TF-IDF;
4. BM25.

У каждого подхода свой бизнесовый смысл.

---

# 3. Почему сырые продажи могут вводить модель в заблуждение

Пусть два пользователя покупают следующие товары:

```text
Пользователь A: [100, 10, 0]
Пользователь B: [10, 1, 0]
```

По абсолютным значениям первый пользователь намного крупнее.

Но пропорции одинаковые:

```text
100 : 10 = 10 : 1
```

Оба пользователя распределяют покупки одинаково:

```text
90.9% на первый товар
9.1% на второй товар
```

Если задача — понять структуру интересов, абсолютный масштаб мешает.

Нормализация по строкам превращает обе строки примерно в:

```text
[0.909, 0.091, 0]
```

Теперь видно, что профили одинаковые.

---

# 4. Нормализация по строкам

## Формула

Для элемента матрицы \(x_{ij}\):

\[
x'_{ij} =
\frac{x_{ij}}
{\sum_{k=1}^{M} x_{ik}}
\]

Где:

- \(i\) — пользователь;
- \(j\) — товар;
- \(M\) — число товаров;
- знаменатель — общая сумма покупок пользователя.

## Пример

Исходная строка:

```text
[6, 0, 2, 2]
```

Сумма:

```text
6 + 0 + 2 + 2 = 10
```

После нормализации:

```text
[0.6, 0, 0.2, 0.2]
```

Сумма ненулевой строки становится равна 1.

---

# 5. Бизнесовый смысл нормализации по строкам

Нормализация по строкам отвечает на вопрос:

> Как пользователь распределяет свои покупки между товарами?

Она уменьшает влияние общего масштаба пользователя.

До нормализации:

```text
пользователь A купил 100 единиц
пользователь B купил 10 единиц
```

После нормализации важны доли, а не объёмы.

Это полезно для:

- сравнения пользовательских предпочтений;
- сегментации;
- поиска похожих пользователей;
- построения профиля интересов;
- снижения влияния оптовых покупателей;
- сравнения пользователей с разной активностью.

---

# 6. Ограничения нормализации по строкам

Она уничтожает информацию об абсолютном масштабе.

Пользователи:

```text
A: [1000, 0]
B: [1, 0]
```

после нормализации становятся одинаковыми:

```text
[1, 0]
```

Но бизнесово они могут быть очень разными:

- A — крупный клиент;
- B — единичный покупатель.

Поэтому абсолютные объёмы часто сохраняют как отдельные признаки:

```text
total_qty
order_count
revenue
active_days
```

---

# 7. Нормализация по столбцам

## Формула

\[
x'_{ij} =
\frac{x_{ij}}
{\sum_{k=1}^{N} x_{kj}}
\]

Где:

- \(N\) — число пользователей;
- знаменатель — общая сумма продаж товара.

## Пример

Столбец товара:

```text
[6, 3, 1]
```

Сумма:

```text
10
```

После нормализации:

```text
[0.6, 0.3, 0.1]
```

---

# 8. Бизнесовый смысл нормализации по столбцам

Она отвечает на вопрос:

> Как продажи конкретного товара распределены между пользователями?

Это полезно, когда товары имеют очень разную популярность.

Без нормализации:

- массовый товар может доминировать;
- редкие товары выглядят незначительными;
- крупные категории получают чрезмерный вес.

После нормализации каждый товар рассматривается относительно собственного масштаба.

Применения:

- анализ аудитории товара;
- поиск похожих товаров по структуре покупателей;
- сравнение массовых и нишевых SKU;
- выявление концентрации продаж;
- анализ зависимости товара от отдельных клиентов.

---

# 9. Ограничения нормализации по столбцам

Она также теряет абсолютный масштаб.

Товары:

```text
товар A продан 1 000 000 раз
товар B продан 10 раз
```

могут получить похожие распределения после нормализации.

Поэтому полезно отдельно хранить:

```text
item_total_qty
buyer_count
revenue
sales_frequency
```

---

# 10. Параллель с Bag of Words

В NLP существует модель Bag of Words.

Строки:

```text
документы
```

Столбцы:

```text
слова
```

Значения:

```text
сколько раз слово встретилось в документе
```

Пример:

| документ | cat | dog | pen |
|---|---:|---:|---:|
| doc_1 | 1 | 0 | 0 |
| doc_2 | 1 | 1 | 0 |
| doc_3 | 0 | 0 | 2 |

User-Item Matrix устроена аналогично:

| пользователь | item_A | item_B | item_C |
|---|---:|---:|---:|
| user_1 | 1 | 0 | 0 |
| user_2 | 1 | 1 | 0 |
| user_3 | 0 | 0 | 2 |

Аналогия:

```text
документ     ↔ пользователь
слово        ↔ товар
частота слова ↔ количество покупок
```

Эта аналогия позволяет применять TF-IDF и BM25.

---

# 11. TF — Term Frequency

TF показывает, насколько слово важно внутри конкретного документа.

Формула:

\[
TF(t,d) =
\frac{\text{число вхождений слова }t}
{\text{общее число слов в документе }d}
\]

В User-Item Matrix:

\[
TF(item,user) =
\frac{\text{qty конкретного товара}}
{\text{общее qty пользователя}}
\]

То есть TF — это нормализация по строкам.

---

# 12. IDF — Inverse Document Frequency

IDF уменьшает вес слов, которые встречаются почти во всех документах.

Формула:

\[
IDF(t,D) =
\log
\left(
\frac{|D|}
{df(t)}
\right)
\]

Где:

- \(|D|\) — число документов;
- \(df(t)\) — число документов, содержащих слово;
- в нашей задаче \(|D|\) — число пользователей;
- \(df(item)\) — число пользователей, купивших товар.

---

# 13. Интуиция IDF

Представим товар:

```text
пакет
```

Его покупают почти все.

Такой товар мало помогает отличить одного пользователя от другого.

Другой товар:

```text
профессиональный телескопический окуляр
```

покупает небольшая специфическая аудитория.

Он сильнее характеризует пользователя.

IDF делает:

- популярные у всех товары менее весомыми;
- редкие и отличительные товары более весомыми.

---

# 14. Пример IDF

Пусть пользователей 100.

## Товар A

Купили 100 пользователей:

\[
IDF_A = \log(100/100) = \log(1) = 0
\]

Товар не различает пользователей.

## Товар B

Купили 10 пользователей:

\[
IDF_B = \log(100/10) = \log(10)
\]

Вес выше.

## Товар C

Купил 1 пользователь:

\[
IDF_C = \log(100/1) = \log(100)
\]

Вес ещё выше.

---

# 15. TF-IDF

Итоговый вес:

\[
TFIDF(t,d) = TF(t,d) \times IDF(t,D)
\]

В нашей задаче:

\[
TFIDF_{ij}
=
\frac{x_{ij}}
{\sum_k x_{ik}}
\times
\log
\left(
\frac{N}
{df_j}
\right)
\]

Где:

- \(x_{ij}\) — количество покупок товара \(j\) пользователем \(i\);
- \(N\) — число пользователей;
- \(df_j\) — число пользователей с ненулевым значением в столбце \(j\).

---

# 16. Бизнесовый смысл TF-IDF

TF-IDF отвечает на вопрос:

> Какие товары особенно характерны для конкретного пользователя, но не являются массовыми для всех?

Пример:

Пользователь часто покупает:

- воду;
- хлеб;
- редкий профессиональный инструмент.

Вода и хлеб массовые, поэтому их IDF низок.

Редкий инструмент сильнее характеризует пользователя.

Применения:

- user embeddings;
- item embeddings;
- сегментация;
- похожие пользователи;
- персональные рекомендации;
- тематические профили;
- снижение popularity bias.

---

# 17. Почему TF-IDF полезен не всегда

Редкость не всегда означает полезность.

Редкий товар может быть:

- ошибочной карточкой;
- случайной покупкой;
- товаром с плохим наличием;
- новым товаром;
- техническим SKU;
- мошеннической активностью.

Поэтому production-система должна применять:

- фильтрацию;
- минимальный support;
- контроль качества каталога;
- clipping IDF;
- business rules.

---

# 18. BM25

BM25 — развитие TF-IDF.

Главная проблема простого TF-IDF:

> Частота может расти линейно, хотя десятая покупка товара не обязательно в десять раз информативнее первой.

BM25 вводит насыщение.

Рост веса замедляется при увеличении TF.

---

# 19. Формула BM25-веса в этой задаче

Используется часть BM25 внутри суммы:

\[
w_{ij}
=
IDF_j
\cdot
\frac{
TF_{ij}(k_1+1)
}{
TF_{ij}
+
k_1
\left(
1-b+b\frac{|d_i|}{avgdl}
\right)
}
\]

Где:

- \(TF_{ij}\) — строково нормализованная частота;
- \(k_1\) — регулирует насыщение;
- \(b\) — регулирует влияние длины строки;
- \(|d_i|\) — сумма исходных покупок пользователя;
- \(avgdl\) — средняя сумма покупок пользователя;
- \(IDF_j\) — обратная частота товара.

---

# 20. Параметр `k1`

`k1` управляет насыщением TF.

При небольшом `k1`:

- насыщение наступает быстрее;
- большие количества сильнее сжимаются.

При большом `k1`:

- функция ближе к линейной;
- различия в частотах сохраняются сильнее.

В задаче значение по умолчанию:

```python
k1 = 2.0
```

---

# 21. Параметр `b`

`b` регулирует нормализацию по длине строки.

## `b = 0`

Длина пользовательского профиля не влияет.

## `b = 1`

Используется полная нормализация относительно средней длины.

Значение по умолчанию:

```python
b = 0.75
```

---

# 22. Почему BM25 часто лучше TF-IDF

TF-IDF линейно зависит от TF.

BM25 учитывает:

- насыщение частоты;
- длину профиля пользователя;
- среднюю длину профиля;
- редкость товара.

Это делает вес более устойчивым к:

- очень активным пользователям;
- крупным покупкам;
- длинным профилям;
- массовым товарам.

---

# 23. Почему нельзя переводить CSR в плотный NumPy-массив

Пусть:

```text
1 000 000 пользователей
100 000 товаров
```

Полная матрица:

```text
100 000 000 000 элементов
```

Даже `float64` потребует примерно:

```text
800 ГБ
```

Поэтому запрещено делать:

```python
matrix.toarray()
```

или:

```python
np.array(matrix)
```

на больших данных.

---

# 24. Что можно переводить в NumPy

Допустимо создавать небольшие векторы:

```text
суммы строк      — длина N
суммы столбцов   — длина M
document frequency — длина M
IDF              — длина M
delta            — длина N
```

Это память:

```text
O(N + M)
```

а не:

```text
O(N × M)
```

# 25. Диагональные матрицы

Для масштабирования строк и столбцов удобно использовать:

```python
scipy.sparse.diags
```

## Масштабирование строк

\[
D_{row} X
\]

где диагональ содержит коэффициенты для строк.

## Масштабирование столбцов

\[
X D_{col}
\]

где диагональ содержит коэффициенты для столбцов.

Диагональная sparse-матрица хранит только диагональ.

---

# 26. Почему нельзя прибавлять вектор к sparse-матрице

Операция вида:

```text
sparse_matrix + row_vector
```

может превратить множество нулей в ненулевые значения.

Это разрушает разреженность.

В BM25 знаменатель содержит:

\[
TF + \Delta_i
\]

где \(\Delta_i\) зависит от строки.

Прямое прибавление \(\Delta\) ко всей матрице заполнило бы нули.

Поэтому формулу переписывают.

---

# 27. Переписывание формулы BM25

Исходная часть:

\[
\frac{TF(k_1+1)}{TF+\Delta}
\]

Делим числитель и знаменатель на TF:

\[
\frac{k_1+1}{1+\Delta/TF}
\]

Пусть:

\[
Z = TF/\Delta
\]

Тогда:

\[
\Delta/TF = Z^{-1}
\]

И формула:

\[
(k_1+1)
\cdot
(Z^{-1}+1)^{-1}
\]

Теперь:

- деление TF на \(\Delta\) — масштабирование строк;
- `power(-1)` применяется только к хранящимся ненулевым элементам;
- `+1` прибавляется только к `data`;
- структура sparse-матрицы не меняется.

---

# 28. Зачем использовать `.data`

В CSR ненулевые значения хранятся в:

```python
matrix.data
```

Операция:

```python
matrix.data += 1.0
```

прибавляет единицу только к уже существующим ненулевым элементам.

Нули, которых нет в структуре, остаются нулями.

Это и есть ключ к memory-safe BM25.

# 29. Бизнес-применения

## Retail

- снижение влияния товаров ежедневного спроса;
- похожие корзины;
- поиск профильных покупателей;
- customer segmentation;
- рекомендации;
- аналоги для прогноза спроса.

## Marketplace

- candidate generation;
- similar items;
- снижение dominance популярных карточек;
- long-tail discovery;
- seller assortment similarity.

## Fintech

Аналог:

```text
клиент × merchant category
```

TF-IDF выделяет специфичные категории расходов.

Применения:

- клиентские сегменты;
- next best offer;
- behavioural embeddings;
- anomaly detection.

## Two-sided marketplace

Матрица:

```text
заказчик × исполнитель
```

Нормализация уменьшает влияние очень активных участников.

## Dating

Матрица:

```text
пользователь × профиль
```

TF-IDF может уменьшить вес массово популярных профилей и выделить специфичные предпочтения.

Обязательны fairness и safety ограничения.

## iGaming

Матрица:

```text
игрок × игра
```

BM25 уменьшает влияние экстремально активных игроков.

Нельзя использовать рекомендации без responsible gaming guardrails.

## Crypto

Матрица:

```text
wallet × token
```

TF-IDF выделяет токены, характерные для конкретных кошельков.

Применения:

- clustering;
- behavioural similarity;
- fraud detection;
- wallet profiling.

# 30. Как мыслить продуктово

Нормализация — не просто математическая операция.

Нужно спросить:

1. Что означает большая частота?
2. Всегда ли десять покупок в десять раз важнее одной?
3. Нужно ли сохранить масштаб пользователя?
4. Нужно ли снизить популярность массовых товаров?
5. Редкий товар — полезный сигнал или шум?
6. Как нормализация повлияет на long tail?
7. Как она изменит fairness?
8. Какая downstream-метрика должна улучшиться?

---

# 31. Что нормализация не решает

Она не исправляет:

- плохие данные;
- ботов;
- возвраты;
- отсутствие товара;
- exposure bias;
- промо;
- сезонность;
- дубли карточек;
- cold start;
- мошенничество.

Нормализация — только один слой пайплайна.

---

# 32. Итог

Четыре метода отвечают на разные вопросы:

## By row

```text
Как пользователь распределяет свои покупки?
```

## By column

```text
Как аудитория распределяется внутри товара?
```

## TF-IDF

```text
Какие товары особенно характерны для пользователя?
```

## BM25

```text
Какие товары характерны для пользователя с учётом редкости,
насыщения частоты и длины профиля?
```

Главная идея:

> Хорошая нормализация превращает количество событий в более содержательный поведенческий сигнал.
