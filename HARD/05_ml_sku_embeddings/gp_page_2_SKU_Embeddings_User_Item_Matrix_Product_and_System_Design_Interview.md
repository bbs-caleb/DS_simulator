# SKU Embeddings — Product и ML System Design Interview Guide

# 1. Как объяснить решение за одну минуту

> У нас есть история транзакций пользователей по товарам. Я агрегирую количество по паре `user_id + item_id`, а среднюю цену считаю отдельно по `item_id` внутри того же временного окна. После этого реальные идентификаторы отображаю в плотные индексы и создаю CSR-матрицу, потому что большинство возможных взаимодействий отсутствует. Такая матрица становится основой item-based collaborative filtering, ALS или matrix factorization. Для production я отдельно решу temporal split, implicit feedback weighting, cold start, popularity bias, возвраты, наличие, версионирование mappings и online A/B-тест.

- связать offline-метрики с бизнес-эффектом.

# 4. Explicit и implicit feedback

## Explicit feedback

Пользователь явно сообщает отношение:

```text
оценка 1–5
like/dislike
```

## Implicit feedback

Мы наблюдаем действия:

```text
просмотр
клик
корзина
покупка
qty
время просмотра
```

Покупка — положительный сигнал.

Но отсутствие покупки не является надёжным отрицательным сигналом.

---

# 5. Почему ноль неоднозначен

Нулевая ячейка может означать:

- пользователь не заинтересован;
- пользователь не видел товар;
- товар не показывался;
- товара не было в наличии;
- товар недоступен в регионе;
- товар появился недавно;
- цена была слишком высокой;
- пользователь отложил покупку.

Поэтому нельзя автоматически считать все нули одинаковыми негативными примерами.

---

# 6. Как выбирать силу взаимодействия

В учебной задаче:

```text
interaction = qty
```

В production возможны другие варианты.

## Binary

```text
купил -> 1
```

## Log-transform

```text
log(1 + qty)
```

Уменьшает влияние очень больших количеств.

## Revenue

```text
qty × price
```

Подходит, когда важна денежная ценность.

## Confidence weighting

```text
confidence = 1 + alpha × qty
```

Часто применяется в implicit ALS.

## Time decay

Свежие взаимодействия получают больший вес.

# 7. Почему CSR

Сильный ответ:

> Полная матрица `users × items` может содержать миллиарды ячеек, но реальных взаимодействий намного меньше. CSR хранит только ненулевые значения и служебные индексы, поэтому память зависит в основном от числа взаимодействий, а не от полного декартова произведения.

Преимущества:

- экономия памяти;
- быстрый доступ к строкам;
- эффективное умножение;
- поддержка SciPy;
- совместимость с рекомендательными библиотеками.

Недостатки:

- медленное изменение структуры;
- не самый удобный формат для частых точечных вставок;
- для столбцов иногда удобнее CSC.

---

# 8. CSR, CSC и COO

## COO

Хранит:

```text
row
column
value
```

Удобен для первоначальной сборки.

## CSR

Сжатие по строкам.

Удобен для пользовательских профилей и матричных операций.

## CSC

Сжатие по столбцам.

Удобен для операций по товарам.

---

# 9. Как получить похожие товары

Исходная матрица:

```text
users × items
```

Каждый товар — столбец.

Для item-item similarity:

1. транспонировать матрицу;
2. получить `items × users`;
3. нормализовать строки;
4. посчитать cosine similarity;
5. выбрать top-K соседей.

На большом каталоге полная item-item матрица слишком дорога. Тогда используют:

- matrix factorization;
- item embeddings;
- ANN-индекс;
- sparse top-K алгоритмы.

---

# 10. Matrix factorization

Идея:

\[
R \approx U V^T
\]

Где:

- `R` — User-Item Matrix;
- `U` — user embeddings;
- `V` — item embeddings.

Пример:

```text
R: 1 000 000 × 100 000
```

При размерности 64:

```text
U: 1 000 000 × 64
V:   100 000 × 64
```

Так мы получаем компактные представления.

---

# 11. ALS

Alternating Least Squares поочерёдно оптимизирует:

- user factors при фиксированных item factors;
- item factors при фиксированных user factors.

Плюсы:

- хорошо работает со sparse matrices;
- масштабируется;
- является сильным baseline;
- подходит для implicit feedback.

# 13. Temporal split

Правильная схема:

```text
train      = прошлое
validation = более поздний период
test       = будущее
```

Случайный split часто создаёт слишком оптимистичную оценку.

---

# 14. Leakage

Примеры:

- embeddings обучены на тестовом периоде;
- цена из будущего;
- будущие покупки попали в train;
- test interactions используются при построении mapping или признаков;
- исправленная в будущем категория используется как исторический признак.

---

# 15. Popularity bias

Популярные товары получают много взаимодействий и могут доминировать.

Последствия:

- всем рекомендуют одно и то же;
- long tail не получает трафик;
- падает diversity;
- новые товары не развиваются.

Решения:

- BM25 weighting;
- inverse popularity;
- negative sampling;
- diversity reranking;
- exploration;
- отдельная оценка head/mid/tail.

# 18. Почему CTR недостаточно

CTR может вырасти, если карточка:

- яркая;
- дешёвая;
- провокационная;
- нерелевантная, но любопытная.

Нужно смотреть всю воронку:

```text
impression
-> click
-> add to cart
-> purchase
-> non-return
-> repeat purchase
```

# 22. Пример полного ответа на System Design интервью

> Сначала я уточню бизнес-сценарий. Предположим, нужны похожие товары на основе покупок. Я соберу транзакции за фиксированное временное окно, агрегирую количество по `user_id + item_id`, а среднюю цену рассчитаю отдельно по `item_id` внутри того же периода.
>
> Реальные ID переведу в версионируемые плотные индексы. Матрицу построю в CSR, потому что число реальных взаимодействий намного меньше полного пространства users × items.
>
> В качестве baseline возьму item-item cosine similarity или implicit ALS. Для cold start добавлю контентные признаки: название, категорию, бренд, атрибуты и изображение.
>
> Данные разделю по времени. Offline измерю Recall@K, NDCG@K, coverage, diversity и качество отдельно на head, mid, tail и новых товарах.
>
> После обучения построю ANN-индекс по item embeddings. Online сервис получит кандидатов, отфильтрует недоступные товары и применит ranker с учётом цены, наличия, доставки и бизнес-правил.
>
> В A/B-тесте primary metric будет purchase conversion или revenue per session. Guardrails — latency, returns, complaints, out-of-stock rate и diversity.
>
> Mapping, embeddings и ANN index будут версионироваться совместно. При сбое будет fallback на популярные товары категории или предыдущую стабильную версию.
