## Блок 0. Импорты

```python
from torch import nn
from transformers import BertModel
```

**`from torch import nn`** — из библиотеки PyTorch берём модуль `nn` (neural networks). Внутри лежат все «кубики» нейросетей: слои, функции активации. Мы напишем `nn.Linear`, `nn.TransformerDecoder` и т.д.

Почему `from torch import nn`, а не `import torch.nn as nn`? Работают обе, но pylint предпочитает первую (правило R0402 `consider-using-from-import`) — а нам нужны 10/10.

**`from transformers import BertModel`** — из библиотеки HuggingFace Transformers берём класс `BertModel`. Это готовая реализация BERT: 12 слоёв энкодера, обученные веса подтянутся из интернета одной строкой.

Больше НИЧЕГО не импортируем: `torch` целиком не нужен (мы не создаём тензоры руками), лишние импорты — минус баллы за качество кода (pylint W0611 `unused-import`).

## Блок 1. Объявление класса

```python
class TASA_Transformer(nn.Module):  # pylint: disable=invalid-name
```

**`class TASA_Transformer`** — объявляем класс (чертёж модели). Имя задано условием задачи БУКВА В БУКВУ — грейдер будет искать именно `TASA_Transformer`. Изменить нельзя, иначе 0 баллов за нарушение API.

**`(nn.Module)`** — наследование: наш класс «ребёнок» класса `nn.Module`, базового для всех нейросетей PyTorch. Наследование бесплатно даёт: перемещение на GPU (`.to(device)`), режимы `train()/eval()`, автоматический сбор всех обучаемых параметров для оптимизатора, магию `model(x)` → вызов нашего `forward`.

**`# pylint: disable=invalid-name`** — точечное отключение одной проверки pylint. По PEP8 имя класса должно быть `PascalCase` без подчёркиваний (`TasaTransformer`), но API задачи требует `TASA_Transformer`. Конфликт «API vs PEP8» решается ровно так: одним инлайн-комментарием. Это наш стандартный паттерн из прошлых уроков (как с параметром `X` в sklearn-классах).

```python
    """
    Класс модели, использующей BERT-энкодер и трансформер-декодер
    для предсказания следующего элемента в последовательности.
    """
```

Докстринг класса — обязателен (иначе pylint C0115 `missing-class-docstring` и минус к 20% баллов за код).

## Блок 2. Конструктор `__init__` — собираем детали модели

```python
    def __init__(self, vocab_size, hidden_dim, nhead, num_layers):
```

`__init__` вызывается один раз при создании модели: `model = TASA_Transformer(vocab_size=460, hidden_dim=768, nhead=8, num_layers=3)`. Здесь мы **создаём и сохраняем** все слои. Параметры:

- **`vocab_size`** — сколько разных товаров существует (размер «словаря»). В ноутбуке ≈460 (столько уникальных ItemID в урезанном датасете). Нужен для последнего слоя: модель должна выдать оценку КАЖДОМУ товару.
- **`hidden_dim`** — размерность внутренних векторов. В ноутбуке 768 — и это не случайность: **BERT-base выдаёт векторы ровно 768 чисел**, все стыкующиеся с ним слои обязаны совпасть по размерности, иначе тензоры «не влезут» друг в друга.
- **`nhead`** — количество голов внимания в декодере (8). Требование: `hidden_dim % nhead == 0` (768/8 = 96 чисел на голову) — иначе PyTorch выбросит ошибку.
- **`num_layers`** — сколько слоёв декодера ставим друг на друга (3).

```python
        super().__init__()
```

Вызов конструктора родителя `nn.Module`. **Обязательная первая строка** любого `__init__` нейросети: родитель инициализирует внутреннюю машинерию регистрации слоёв. Забыть — модель сломается с загадочной ошибкой. Пишем современно `super().__init__()`, а не `super(TASA_Transformer, self).__init__()` из шаблона — старая форма делает то же самое, но pylint ругается (R1725 `super-with-arguments`).

### 2.1. BERT-энкодер

```python
        self.bert_encoder = BertModel.from_pretrained("bert-base-uncased")
```

**`BertModel.from_pretrained("bert-base-uncased")`** — скачивает (при первом запуске, ~440 МБ) и загружает **предобученный** BERT: архитектура + веса, выученные Google на миллиардах слов. `"bert-base-uncased"` — имя конкретной версии: base (12 слоёв, 768 измерений), uncased (не различает регистр букв).

**`self.bert_encoder = ...`** — сохраняем модель как атрибут. `self` — это «сам объект»; всё, что записано в `self.*` в `__init__`, доступно потом в `forward`. Плюс `nn.Module` автоматически регистрирует слой: его веса попадут в оптимизатор и будут дообучаться (fine-tuning) вместе с остальной моделью.

Это замена LSTM-энкодера из оригинальной TASA.

### 2.2. Трансформер-декодер

```python
        decoder_layer = nn.TransformerDecoderLayer(
            d_model=hidden_dim, nhead=nhead, batch_first=True
        )
```

**`nn.TransformerDecoderLayer`** — ОДИН слой декодера трансформера. Внутри него уже собраны (нам вручную ничего писать не надо): self-attention по целевой последовательности → encoder-decoder attention (подглядывание в выход энкодера) → feedforward-слой → нормализации и残остаточные связи.

Параметры:
- **`d_model=hidden_dim`** — размерность векторов (768), обязана совпасть с выходом BERT.
- **`nhead=nhead`** — 8 голов внимания.
- **`batch_first=True`** — КРИТИЧЕСКИ важный флаг. Он говорит: «тензоры приходят в формате (батч, длина, размерность)». BERT выдаёт именно так: `(B, L, 768)`. Без флага PyTorch по историческим причинам ждёт `(L, B, 768)` — код бы даже не упал, но модель молча перепутала бы батч с последовательностью и училась бы ерунде. Классическая тихая ошибка.

`decoder_layer` — локальная переменная (без `self.`): она нужна только как заготовка для следующей строки.

```python
        self.transformer_decoder = nn.TransformerDecoder(
            decoder_layer, num_layers=num_layers
        )
```

**`nn.TransformerDecoder(decoder_layer, num_layers=3)`** — берёт слой-заготовку и клонирует его 3 раза, ставя друг на друга (выход слоя 1 → вход слоя 2 → ...). Больше слоёв = глубже модель = сложнее закономерности (и дороже вычисления). Это замена LSTM-декодера.

### 2.3. Полносвязный слой

```python
        self.fc = nn.Linear(hidden_dim, vocab_size)
```

**`nn.Linear(768, 460)`** — обычный полносвязный (fully connected, отсюда `fc`) слой: умножение на матрицу весов + сдвиг. Превращает каждый вектор из 768 чисел в вектор из `vocab_size` чисел — по одному числу (логиту) на каждый товар словаря. Чем больше логит товара j — тем увереннее модель, что следующий клик будет по нему. Это «переводчик» с внутреннего языка модели на язык каталога товаров.

## Блок 3. `forward` — прямой проход (как данные текут сквозь модель)

```python
    def forward(self, src_items, src_times, tgt_items, tgt_times):
```

`forward` вызывается на каждом батче: `logits = model(src_items, src_times, tgt_items, tgt_times)` (PyTorch сам перенаправляет вызов `model(...)` в `forward`). Входы:

- **`src_items`** `(B, 50)` — батч сессий: 50 ID товаров на пользователя (короткие сессии дополнены нулями-паддингом справа).
- **`src_times`** `(B, 50)` — нормированные временные метки (0…1) для src.
- **`tgt_items`**, **`tgt_times`** — то же самое для целевой последовательности (в обучении обычно та же сессия — модель учится восстанавливать/продолжать её, это и есть автоэнкодерная идея).

```python
        # pylint: disable=unused-argument
```

Честное признание: в учебной версии временные метки в вычислениях не участвуют (упрощение курса; в полноценной TASA время подмешивалось бы в эмбеддинги). Но сигнатура задана API — выкидывать параметры нельзя. Pylint видит неиспользуемые аргументы и ругается (W0613) — глушим точечно внутри метода.

### 3.1. Кодируем src через BERT

```python
        src_mask = (src_items > 0).long()  # 1 - реальный товар, 0 - паддинг
```

Строим **маску внимания**. `(src_items > 0)` — поэлементное сравнение: тензор `(B, 50)` из True/False (True там, где реальный товар, False — где ноль-паддинг). `.long()` — переводим в числа 1/0 (формат, который ждёт BERT). Зачем: паддинг — это «пустота», BERT не должен тратить на неё внимание и подмешивать её в смысл сессии. Маска говорит: «эти позиции игнорируй».

```python
        src_encoded = self.bert_encoder(
            input_ids=src_items, attention_mask=src_mask
        ).last_hidden_state
```

Прогоняем сессию через BERT:
- **`input_ids=src_items`** — ID товаров подаются туда, куда BERT обычно принимает ID слов. Наши ID (0…459) укладываются в словарь BERT (30 522 позиции) — модель просто возьмёт соответствующие строки своей таблицы эмбеддингов и при дообучении наполнит их «товарным» смыслом.
- **`attention_mask=src_mask`** — та самая маска паддинга.
- **`.last_hidden_state`** — BERT возвращает объект с несколькими полями; нам нужно поле «выход последнего слоя»: тензор **`(B, 50, 768)`** — 768-мерный контекстный вектор для КАЖДОЙ из 50 позиций. Каждый вектор уже «впитал» всю сессию через self-attention.

**Форма: `(B, 50)` → `(B, 50, 768)`.**

### 3.2. Кодируем tgt через BERT

```python
        tgt_mask = (tgt_items > 0).long()
        tgt_encoded = self.bert_encoder(
            input_ids=tgt_items, attention_mask=tgt_mask
        ).last_hidden_state
```

Абсолютно то же самое для целевой последовательности — тем же самым BERT (веса общие: одна модель кодирует и вход, и цель). Декодер трансформера не умеет есть «сырые» ID — ему нужны векторы, поэтому tgt тоже сначала превращаем в эмбеддинги. **Форма: `(B, 50, 768)`.**

### 3.3. Декодер

```python
        decoded_output = self.transformer_decoder(tgt_encoded, src_encoded)
```

Два аргумента по порядку (важно не перепутать!):
1. **`tgt`** = `tgt_encoded` — «что декодируем»;
2. **`memory`** = `src_encoded` — «память», выход энкодера, в который декодер подглядывает через encoder-decoder attention.

Внутри каждого из 3 слоёв: self-attention по tgt («как элементы цели связаны между собой») → cross-attention на память («что во входной сессии важно для этого шага») → feedforward. **Форма сохраняется: `(B, 50, 768)`.**

### 3.4. Логиты

```python
        logits = self.fc(decoded_output)
        return logits
```

Полносвязный слой применяется к каждому из 50 векторов каждого пользователя независимо: **`(B, 50, 768)` → `(B, 50, vocab_size)`**.

Читаем результат: `logits[b, i, j]` — оценка того, что у пользователя `b` на шаге `i` следующим будет товар `j`. Для рекомендации берём последний реальный шаг сессии и топ-K товаров по логитам. `return logits` — отдаём наружу; функция потерь (CrossEntropyLoss в ноутбуке) сравнит их с реальными следующими товарами и накажет модель за ошибки.

## Блок 5. Типичные ошибки, на которых валятся

1. **Переименовать класс/метод/параметры** → грейдер не найдёт API → 0 баллов за всё.
2. **Забыть `batch_first=True`** → тихая перестановка батча и длины, модель учится ерунде.
3. **Перепутать порядок аргументов декодера** (`decoder(src, tgt)` вместо `decoder(tgt, memory)`) → декодер подглядывает не туда.
4. **`hidden_dim ≠ 768` при BERT-base** → ошибка несовпадения размерностей на стыке BERT → декодер.
5. **Взять `.pooler_output` вместо `.last_hidden_state`** → получите один вектор на сессию `(B, 768)` вместо повектора на позицию `(B, 50, 768)` — декодеру нечего декодировать пошагово.
6. **Оставить в solution.py код обучения/вызовов** → грейдер импортирует файл, код выполнится и упадёт (нет GPU/данных).
7. **Файл со суффиксом ` (1)` после скачивания** → сдаётся старая версия. Переименовать!

## Блок 6. Чему научились (в одном абзаце)

Мы собрали seq2seq-модель для рекомендаций из готовых кубиков: предобученный BERT превращает последовательность ID товаров в контекстные векторы (замена LSTM-энкодера), стек трансформер-декодеров сопоставляет целевую последовательность с памятью входной (замена LSTM-декодера), линейный слой переводит внутренние векторы в оценки по каталогу товаров. Ключевые ремесленные навыки: стыковка размерностей (768 у BERT-base), маски паддинга, `batch_first=True`, точечные pylint-disable для конфликтов API с PEP8.
