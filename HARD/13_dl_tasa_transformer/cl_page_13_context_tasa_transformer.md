## Часть 1. Бизнес-контекст: зачем всё это вообще

### 1.1. Проблема, которую мы решаем

Классическая рекомендательная система работает так: раз в сутки (или раз в час) пересчитываются рекомендации на основе **истории** пользователя. Вы месяц покупали корм для кота — вам месяц показывают корм для кота.

Проблема: **пользователь живёт в моменте**. Сценарий из урока:

1. Пользователь зашёл посмотреть робот-пылесос.
2. Отвлёкся, начал листать портативные колонки.
3. Классическая система продолжит пихать пылесосы (они в истории!). А правильно — мгновенно понять: «сейчас его интересуют колонки» и повести его по этому пути.

**Real-Time RecSys** — система, которая перестраивает рекомендации после **каждого действия**: клика, поиска, добавления в корзину, свайпа. Не «кем пользователь был вчера», а «кто он прямо сейчас, в этой сессии».

### 1.2. Почему бизнесу это критично (в деньгах)

- **Конверсия сессии.** Большинство сессий в e-commerce заканчиваются без покупки. Если система угадывает намерение *внутри сессии*, она сокращает путь до покупки. Подъём CTR рекомендательных полок на 1–2 п.п. на маркетплейсе масштаба Ozon/Wildberries — это сотни миллионов рублей GMV в год.
- **Холодные пользователи.** У нового пользователя нет истории — классические модели слепы. Но у него есть **текущая сессия** из 5–10 кликов, и sequence-модель уже может рекомендовать.
- **Скоропортящееся намерение.** Намерение «купить подарок к завтрашнему дню рождения» живёт часы. Батчевая система, обновляющаяся раз в сутки, его просто не успеет отработать.

### 1.3. Как эта задача выглядит в разных индустриях

Это важно для собеседований: одна и та же технология («эмбеддинги последовательностей действий в реальном времени»), но разные бизнес-задачи.

**Маркетплейсы / E-commerce (Ozon, WB, Amazon, KarpovExpress).**
Задача: next-item prediction внутри сессии → полки «Вам может понравиться», «Часто смотрят вместе», ранжирование поисковой выдачи с учётом сессии. Метрики: CTR, CR (conversion rate), GMV per session, add-to-cart rate. Ключевое мышление: у маркетплейса миллионы SKU, длинный хвост товаров — sequence-модель помогает вытаскивать релевантный хвост, который никогда не попал бы в топ по популярности.

**Two-sided marketplaces (Avito, Airbnb, Uber, доставка).**
Специфика: рекомендуешь не товар со склада, а предложение живого продавца/водителя/хоста, у которого ограничен инвентарь (квартира одна, водитель один). Real-time сигналы нужны не только для персонализации, но и для **балансировки маркетплейса**: нельзя рекомендовать всем одну и ту же квартиру. Эмбеддинг сессии подаётся в ранкер вместе с сигналами доступности supply.

**Финтех / Банки (SigmaBank, Тинькофф, Revolut).**
Последовательность действий = транзакции + клики в приложении. Применения: (а) next-best-offer — какой продукт предложить (кредитка, вклад, страховка) исходя из свежего поведения; (б) **антифрод** — эмбеддинг сессии сравнивается с типичным «профилем поведения» клиента, аномальное отклонение → повод для проверки. Именно автоэнкодерная природа TASA здесь золото: модель училась восстанавливать нормальные последовательности, фрод она восстанавливает плохо → большая ошибка реконструкции → алерт.

**iGaming / Casino / Betting.**
Последовательность ставок/игр в реальном времени. Применения: рекомендация следующей игры, персональные бонусы в момент, когда игрок готов уйти (churn-in-session prediction), и — важная этическая/регуляторная сторона — **responsible gaming**: детекция паттернов проблемной игры (лудомании) по последовательности действий и принудительное охлаждение. На собеседовании упоминание RG-применения показывает зрелость.

**Crypto / Trading-платформы.**
Последовательности сделок и просмотров активов. Real-time рекомендации инструментов, детекция wash-trading и манипуляций (снова аномалии в последовательностях), персонализация лент новостей по текущему интересу к активу.

**Dating (Tinder-подобные).**
Последовательность свайпов — чистейший real-time сигнал. Вкус пользователя дрейфует внутри сессии; sequence-модель ловит «сегодняшнее настроение» лучше, чем усреднённый профиль. Метрики: match rate, длина сессии, retention.

**Бигтех (Google, Meta, Netflix, Anthropic).**
- Google/YouTube: сессионное ранжирование выдачи и видео (модели типа BERT4Rec/SASRec выросли ровно из этой парадигмы).
- Meta: лента и реклама — предсказание следующего вовлекающего поста по последовательности взаимодействий в текущей сессии.
- Netflix: «продолжить смотреть» и порядок строк на главной перестраиваются от сессии.
- Anthropic/LLM-компании: та же математика — LLM и есть модель «предскажи следующий токен в последовательности». Рекомендации next-item и языковое моделирование — близнецы: словарь товаров вместо словаря слов.

**Промышленность / Энергетика (BP, SpaceX).**
Здесь «пользователь» — датчик или система. Последовательность телеметрии → автоэнкодер учится восстанавливать норму → аномалия реконструкции = предвестник отказа оборудования (predictive maintenance). Архитектурно это та же TASA.

### 1.4. Что просит тимлид (перевод с менеджерского на технический)

«Хочу рекомендации, подстраивающиеся под каждый клик» =

1. Собрать **последовательности действий** пользователя (user trail): клики, поиски, корзина, с временными метками.
2. Обучить модель, которая сжимает последовательность в **эмбеддинг** (вектор чисел) — «отпечаток текущего намерения».
3. Модель должна уметь **предсказывать следующее действие** — это и есть рекомендация.
4. Вместо LSTM (как в оригинальной TASA) использовать **трансформер** — лучше контекст, параллелизм, масштабируемость.

## Часть 2. Техническая теория с нуля

### 2.1. Напоминание: что такое последовательность и почему она сложная

Последовательность — упорядоченный набор элементов: `[товар_5, товар_17, товар_5, товар_233]`. Порядок несёт смысл: «посмотрел телефон → посмотрел чехол» ≠ «посмотрел чехол → посмотрел телефон».

Проблема: обычные модели (линейная регрессия, бустинг) едят таблицу фиксированной ширины и не понимают порядок. Нужны специальные архитектуры.

### 2.2. Как это решали раньше: RNN и LSTM

**RNN** читает последовательность слева направо, по одному элементу, таща за собой «память» (скрытое состояние). Проблема: на длинных последовательностях память «выцветает» — к 50-му элементу модель забыла первый (затухающие градиенты).

**LSTM** — усовершенствованная RNN с «клапанами» (gates), которые решают, что запомнить надолго, что забыть, что отдать наружу. Работает лучше, но:

- всё равно читает **последовательно** → нельзя распараллелить на GPU → медленное обучение;
- очень длинные зависимости всё равно теряются;
- информация о начале последовательности доходит до конца через десятки промежуточных шагов, как в испорченном телефоне.

### 2.3. Transformer: «Attention is All You Need» (2017)

Ключевая идея: **выкинуть рекуррентность вообще**. Пусть каждый элемент последовательности смотрит на ВСЕ остальные элементы напрямую, за один шаг. Этот механизм называется **Self-Attention** (само-внимание).

#### Self-Attention на пальцах

Пример из урока: «Собака не перебежала дорогу, потому что **она** была слишком уставшей». К чему относится «она» — к собаке или к дороге? Человек понимает из контекста. Self-attention делает то же: слово «она» «смотрит» на все слова предложения и вычисляет, насколько каждое из них релевантно, — и связь «она ↔ собака» получает большой вес. Если фраза «...потому что она была слишком широкой» — вес перетечёт к «дороге».

Механика (упрощённо, без формул):

1. Каждый элемент превращается в три вектора: **Query** (что я ищу), **Key** (что во мне можно найти), **Value** (что я отдам, если меня выберут).
2. Query каждого элемента сравнивается с Key всех элементов → получаются «оценки внимания» (насколько элемент i важен для элемента j).
3. Оценки превращаются в веса (softmax: неотрицательные, в сумме 1).
4. Новое представление элемента = взвешенная сумма Value всех элементов.

Итог: каждый элемент обновил своё представление, «впитав» контекст всей последовательности за один параллельный шаг. Никакого испорченного телефона.

В рекомендациях: товар «чехол для iPhone» в сессии впитывает контекст «до этого смотрели iPhone 15» — и модель понимает, что это чехол именно к нему.

#### Multi-Head Attention

Одна «голова» внимания ловит один тип связей. Но связи бывают разные: «товар-аксессуар», «товар-конкурент», «товар той же категории», «товар в том же ценовом сегменте». **Multi-Head** = несколько параллельных механизмов внимания (голов), каждая учится ловить свой тип зависимостей, потом их результаты склеиваются. В нашей задаче `nhead=8` — 8 голов.

Важное ограничение: `hidden_dim` должен делиться на `nhead` нацело (768 / 8 = 96 — размерность на голову).

#### Positional Encoding

Внимание смотрит на все элементы **одновременно** → само по себе оно не знает порядок («мешок элементов»). Чтобы вернуть порядок, к эмбеддингу каждого элемента добавляется вектор позиции — **Positional Encoding**. У BERT это обучаемые позиционные эмбеддинги, встроенные внутрь модели, — нам их вручную добавлять не надо.

#### Feedforward Layers

После внимания каждый элемент пропускается через обычный полносвязный мини-слой (одинаковый для всех позиций). Внимание «собирает информацию», feedforward — «переваривает» её.

#### Сборка: энкодер и декодер

Трансформер из статьи = **стек энкодеров** + **стек декодеров** (в статье по 6 слоёв, у нас будет 3 в декодере).

- **Слой энкодера** = Self-Attention + Feedforward. Задача: построить богатое представление входа.
- **Слой декодера** = Self-Attention (по своей последовательности) + **Encoder-Decoder Attention** (внимание на выходы энкодера) + Feedforward. Encoder-Decoder Attention — это когда декодер, генерируя очередной элемент, «подглядывает» в закодированный вход: «что там было в исходной последовательности, что мне сейчас важно?».

В PyTorch это готовые кубики: `nn.TransformerDecoderLayer` (один слой) и `nn.TransformerDecoder` (стек из N слоёв). Мы будем использовать именно их.

### 2.4. BERT

**BERT** (Bidirectional Encoder Representations from Transformers, Google, 2018) — это **только стек энкодеров** трансформера (без декодера), предобученный на гигантском корпусе текста.

Ключевые факты:

- **Двунаправленность**: каждый токен видит контекст и слева, и справа (LSTM обычно читает только слева направо).
- **Предобучение**: BERT уже обучен на миллиардах слов. Мы берём готовые веса (`bert-base-uncased`) и дообучаем под свою задачу (fine-tuning). Это как нанять сотрудника с высшим образованием вместо обучения с детского сада.
- **Размеры bert-base**: 12 слоёв энкодера, скрытая размерность **768**, 12 голов внимания, словарь ~30 522 токена. Отсюда в ноутбуке `hidden_dim = 768` — мы обязаны совпасть с BERT.
- **Выход**: для входа из L токенов BERT возвращает L векторов по 768 чисел (`last_hidden_state`) — контекстуализированное представление каждого токена.

Почему BERT для рекомендаций (из урока): справляется и с короткими, и с длинными сессиями; легко дообучается на доменных данных; двунаправленный контекст → лучше персонализация. Индустриальный аналог — модель **BERT4Rec**.

Хитрость нашей задачи: BERT ждёт на вход ID **слов**, а мы подаём ID **товаров**. Это легально: для BERT input_ids — просто индексы в таблице эмбеддингов. Наши ID товаров (0…459) попадают в диапазон словаря BERT (0…30521), модель дообучится придавать этим индексам «товарный» смысл. В проде словарь товаров обучали бы с нуля, но для учебной задачи трюк стандартный.

### 2.5. TASA: временно-осознанный автоэнкодер

Разложим аббревиатуру идеи (Time-Aware Sequential Autoencoder):

- **Автоэнкодер** — сеть из двух половин: **энкодер** сжимает вход в компактный вектор (эмбеддинг), **декодер** пытается восстановить вход из этого вектора. Обучение без разметки (unsupervised): «ошибка = насколько плохо восстановил». Чтобы восстановление удалось, эмбеддинг вынужден вместить всю суть входа — так мы получаем **осмысленное сжатое представление**.
- **Sequential** — вход не картинка и не таблица, а последовательность действий (user trail).
- **Time-Aware** — каждый элемент это пара (действие, время): `(a1, t1), (a2, t2)...`. Паузы между действиями несут сигнал: 2 секунды между кликами ≠ 2 часа.

Оригинальная TASA: пары (действие, время) → эмбеддинг-слой → **LSTM-энкодер** → скрытое состояние h_L → **LSTM-декодер** восстанавливает последовательность → полносвязный слой выдаёт логиты (оценки вероятностей действий на каждом шаге).

Конвейер системы вокруг модели (вторая картинка урока):

1. **Data Sources** — клики, поиски, письма, любые цифровые следы.
2. **Sequence Generation** — склейка следов в хронологический user trail.
3. **TASA Training** — обучение автоэнкодера восстанавливать последовательности.
4. **Centralized Pool of Embeddings** — готовые эмбеддинги пользователей складываются в хранилище, откуда их берут ЛЮБЫЕ downstream-задачи: сегментация, предсказание кликов, рекомендации, антифрод. Один эмбеддинг — много применений. Это главная продуктовая ценность: команда обучает модель один раз, а фичами пользуется вся компания.
5. **Apply TASA** — новая сессия нового пользователя → прогон через энкодер → свежий эмбеддинг → рекомендации в реальном времени.

### 2.6. TASA Transformer: что меняем и зачем

Заменяем обе LSTM-половины:

| Было (TASA) | Стало (TASA Transformer) |
|---|---|
| LSTM-энкодер | **BERT** (предобученный стек энкодеров) |
| LSTM-декодер | **nn.TransformerDecoder** (стек трансформер-декодеров) |
| Полносвязный слой → логиты | Тот же полносвязный слой |

Выгоды (из урока, с пояснениями):

1. **Внимание к контексту.** Self-attention напрямую связывает отдалённые действия: клик в начале сессии и клик в конце «видят» друг друга без испорченного телефона.
2. **Гибкость.** Двунаправленность + параллельность → быстрее адаптация к real-time потоку.
3. **Масштабируемость.** Параллельная обработка на GPU → обучение на миллиардах сессий реально; LSTM бы захлебнулась.
4. **Богатство представлений.** Многослойное внимание строит более глубокие эмбеддинги → точнее рекомендации.

Цена (важно проговаривать на собеседовании): внимание стоит O(L²) по длине последовательности — на очень длинных историях дорого; трансформеры прожорливее к данным; инференс BERT тяжелее, чем LSTM, — для real-time нужны дистилляция/квантизация/кэширование эмбеддингов.

### 2.7. Как течёт информация в нашей модели (главная схема)

```
src_items (сессия, batch×50 ID товаров) ──► BERT ──► src_encoded (batch×50×768)
                                                           │ (память для декодера)
                                                           ▼
tgt_items (целевая посл-ть) ──► BERT ──► tgt_encoded ──► TransformerDecoder ──► (batch×50×768)
                                                           │
                                                           ▼
                                                    Linear(768 → vocab_size)
                                                           │
                                                           ▼
                                              logits (batch×50×460) — оценка
                                              каждого товара на каждом шаге
```

`logits[b, i, j]` = «насколько модель уверена, что на шаге i пользователь b выберет товар j». Softmax поверх — вероятности; argmax — рекомендация.

## Часть 3. Мышление для собеседований

### 3.1. Product / бизнес-интервью: как рассуждать

Вопрос-классика: «Как бы вы улучшили рекомендации на маркетплейсе?». Каркас ответа:

1. **Сегментируй проблему**: новые пользователи (нет истории) / активные (богатая история) / вернувшиеся (история устарела). Real-time сессионные модели больше всего дают первым и третьим.
2. **Назови метрики**: онлайн — CTR, CR, GMV/session, add-to-cart; офлайн — Recall@K, NDCG@K, MRR для next-item prediction.
3. **Предложи A/B**: контроль (батчевые рекомендации) vs тест (сессионные), guardrail-метрики (латентность страницы, разнообразие выдачи, доля рекомендованного длинного хвоста).
4. **Проговори trade-off'ы**: точность vs латентность инференса; свежесть vs стоимость пересчёта; персонализация vs privacy.

### 3.2. System design интервью: real-time RecSys

Скелет ответа «спроектируйте real-time рекомендации»:

1. **Ingestion**: события кликстрима → Kafka.
2. **Feature/Sequence store**: последние N действий пользователя в быстром KV-хранилище (Redis) — это user trail.
3. **Модель**: двухстадийная архитектура — (а) **retrieval**: эмбеддинг сессии (наша TASA Transformer) → ANN-поиск ближайших товаров (FAISS/ScaNN) → сотни кандидатов; (б) **ranking**: тяжёлый ранкер (бустинг/нейросеть) дособирает топ-10.
4. **Пул эмбеддингов**: обновляется после каждого события (инференс энкодера) либо микробатчами раз в секунды.
5. **Латентность**: бюджет ~100 мс на всё → BERT дистиллируют/квантизируют, эмбеддинги товаров предрассчитаны, онлайн считается только эмбеддинг сессии.
6. **Обучение**: офлайн, на исторических сессиях, переобучение ежедневно/еженедельно; мониторинг дрейфа.

### 3.3. Вопросы, которые могут задать по этой модели

- «Почему автоэнкодер, а не просто классификатор next-item?» → Эмбеддинг переиспользуется во многих задачах (пул эмбеддингов), обучение не требует разметки, ошибка реконструкции — бесплатный детектор аномалий.
- «Почему BERT лучше LSTM тут?» → параллелизм, длинные зависимости, предобучение, двунаправленность (см. 2.6) — и честно назвать цену O(L²).
- «Как подавать время в модель?» → бакеты интервалов как токены, добавка к позиционным эмбеддингам, или конкатенация нормированного времени к эмбеддингу действия (в учебной версии время упрощено).

## Часть 4. Словарик урока

| Термин | Простыми словами |
|---|---|
| Эмбеддинг | Вектор чисел, кодирующий смысл объекта (товара, сессии, пользователя) |
| Токен | Один элемент последовательности (у нас — ID товара) |
| Логиты | «Сырые» оценки модели до превращения в вероятности |
| Self-Attention | Каждый элемент смотрит на все остальные и впитывает контекст |
| Multi-Head | Несколько параллельных вниманий, каждое ловит свой тип связей |
| Positional Encoding | Добавка, сообщающая модели порядок элементов |
| Энкодер | Сжимает вход в представление |
| Декодер | Разворачивает представление обратно в последовательность |
| Автоэнкодер | Энкодер+декодер, учится восстанавливать вход; эмбеддинг — побочный продукт |
| Fine-tuning | Дообучение предобученной модели под свою задачу |
| last_hidden_state | Выход BERT: вектор 768 чисел на каждый входной токен |
| User trail | Хронологическая последовательность действий пользователя |
