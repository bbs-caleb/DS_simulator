# Страница 2. GMROI Optimization — продуктовое и system design интервью

# 1. Как кратко рассказать решение

> Я объединил недельные характеристики товара, продажи и остатки по составному ключу `week`, `seller_id`, `product_id`. На уровне товара рассчитал маржу, стоимость текущего запаса и расходы хранения. Затем агрегировал компоненты по продавцу или категории и только после этого рассчитал GMROI. Для закупок отфильтровал собственный ассортимент `seller_id = 0`, присоединил категорию и себестоимость, рассчитал стоимость закупки и развернул категории в столбцы.

---

# 2. Почему нельзя merge только по товару

Один товар повторяется:

- по неделям;
- по продавцам.

Merge только по `product_id` создаст many-to-many соединение и завысит суммы.

---

# 3. Почему нельзя брать среднее GMROI

У товаров разные размеры запасов.

Обычное среднее даст одинаковый вес:

- маленькому SKU;
- крупному SKU.

Правильно:

\[
\frac{\sum profit}{\sum inventory}
\]

---

# 4. Почему purchases соединяется только с seller 0

`purchases.csv` описывает закупки KarpovExpress.

Если присоединить все строки `products.csv`, одна покупка размножится на несколько продавцов.

---

# 5. Почему логистика не умножается на qty

По документации `logistic_costs` — уже общая недельная сумма для товара.

Повторное умножение завысит расходы.

---

# 6. Почему хранение умножается на остаток

`unit_storage_cost` — цена хранения одной единицы.

Полная стоимость:

\[
stock\_qty \times unit\_storage\_cost
\]

---

# 7. Продуктовый вопрос: кого продвигать

Недостаточно выбрать максимальный GMROI.

Нужно проверить:

- абсолютную прибыль;
- объём заказов;
- стабильность по неделям;
- возвраты;
- наличие;
- SLA;
- комиссию платформы;
- способность масштабировать запас.

---

# 8. Продуктовый вопрос: какую категорию закупать больше

Высокий GMROI может быть вызван дефицитом.

Нужно дополнительно смотреть:

- OOS;
- lost sales;
- прогноз спроса;
- lead time;
- MOQ;
- сезонность;
- срок годности;
- стабильность метрики.

---

# 11. Data quality проверки

- уникальность ключа;
- отсутствие пропусков `cost`;
- отсутствие отрицательного запаса;
- отсутствие дубликатов;

- экстремальные GMROI;
- сверка с финансами.

---

# 12. Главный ответ для интервью

> Ключевой момент решения — считать GMROI не как среднее товарных коэффициентов, а как отношение агрегированных компонентов. Второй критический момент — корректные ключи merge. Третий — использование только продавца `0` при расчёте закупок. После расчёта я бы не применял GMROI как единственный критерий бизнес-решения, а дополнил его прибылью, объёмом, возвратами, наличием и экспериментальной проверкой.
