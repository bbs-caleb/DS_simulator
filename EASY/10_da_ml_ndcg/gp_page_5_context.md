# gp_page_5 — Бизнесовый и технический контекст Average nDCG

# 1. Что изменилось на этой странице

До этого момента мы оценивали ранжирование для одного запроса.

Для одного вопроса студента мы могли получить:

```python
relevance = [0.99, 0.94, 0.74, 0.88, 0.71, 0.68]
```

и вычислить:

\[
nDCG@k
\]

Это позволяло ответить:

> Насколько хорошо модель расположила видео для одного конкретного запроса?

Но production-система обслуживает не один запрос.

Она получает:

- сотни запросов;
- тысячи запросов;
- миллионы пользовательских сессий;
- новые темы;
- редкие формулировки;
- разные языки;
- разные сегменты пользователей.

Поэтому возникает следующий вопрос:

> Как оценить качество модели в целом, а не только на одном запросе?

Для этого используется **Average nDCG**.

---

# 2. Что такое Average nDCG

Average nDCG — среднее значение nDCG по множеству запросов.

Для каждого запроса:

1. берётся отдельный список релевантностей;
2. вычисляется nDCG@k;
3. полученные значения складываются;
4. сумма делится на число запросов.

Формула:

\[
Average\ nDCG@k
=
\frac{1}{n}
\sum_{i=1}^{n}
nDCG@k(q_i)
\]

Где:

- `n` — количество запросов;
- `q_i` — запрос с номером `i`;
- `nDCG@k(q_i)` — nDCG для конкретного запроса.

---

# 3. Главная бизнесовая идея

Один запрос не показывает качество всей системы.

Предположим, мы проверили только запрос:

```text
Собеседование System Design
```

и получили:

```text
nDCG@5 = 0.99
```

Это отличный результат.

Но по другим запросам модель может работать плохо:

```text
Как выбрать статистический тест?      nDCG@5 = 0.48
Что такое оконная функция SQL?        nDCG@5 = 0.35
Как устроен CatBoost?                 nDCG@5 = 0.92
Что такое доверительный интервал?     nDCG@5 = 0.41
```

Если смотреть только на System Design, можно ошибочно решить, что модель готова к запуску.

Average nDCG позволяет посмотреть на качество по набору запросов.

---

# 4. Почему сначала считается nDCG для каждого запроса

Очень важно соблюдать порядок вычислений.

Правильный алгоритм:

1. посчитать nDCG отдельно для каждого запроса;
2. затем усреднить значения.

Неправильный алгоритм:

1. объединить все релевантности в один длинный список;
2. посчитать один nDCG.

Почему второй вариант неправильный?

Потому что запросы независимы.

Позиция 1 одного запроса не должна сравниваться с позицией 1 другого запроса как части одной общей выдачи.

У каждого запроса есть:

- собственная фактическая выдача;
- собственная идеальная выдача;
- собственный DCG;
- собственный IDCG;
- собственный nDCG.

---

# 5. Почему нельзя усреднять сырые DCG

Рассмотрим два запроса.

## Запрос A

```python
[3, 2, 1]
```

## Запрос B

```python
[0.3, 0.2, 0.1]
```

Даже если оба списка идеально отсортированы, DCG первого будет намного больше.

Причина не в том, что модель лучше отработала запрос A.

Причина в масштабе relevance labels.

nDCG нормализует каждый запрос относительно его собственного идеала.

Поэтому после нормализации оба идеальных запроса получают:

```text
1.0
```

И только после этого их можно осмысленно усреднить.

---

# 6. Macro-average

Функция из задания считает обычное арифметическое среднее:

\[
\frac{x_1+x_2+\dots+x_n}{n}
\]

Каждый запрос имеет одинаковый вес.

Такой подход называется **macro-average**.

Пример:

```text
Запрос 1: nDCG = 0.90
Запрос 2: nDCG = 0.60
Запрос 3: nDCG = 0.30
```

Среднее:

\[
\frac{0.90+0.60+0.30}{3}
=
0.60
\]

Каждый запрос повлиял на результат одинаково.

---

# 7. Бизнесовый смысл macro-average

Macro-average отвечает на вопрос:

> Каково среднее качество ранжирования для случайно выбранного запроса из evaluation-набора?

Это полезно, если:

- каждый запрос считается одинаково важным;
- evaluation set репрезентативен;
- мы не хотим, чтобы популярные запросы полностью подавили редкие;
- нас интересует качество long tail.

Но macro-average имеет ограничение.

Запрос, который встречается один раз, получает такой же вес, как запрос с миллионом показов.

---

# 8. Weighted average

В production иногда используют взвешенное среднее:

\[
Weighted\ nDCG
=
\frac{\sum_i w_i \cdot nDCG_i}
{\sum_i w_i}
\]

Весом может быть:

- частота запроса;
- число показов;
- число пользователей;
- выручка;
- важность сегмента;
- риск;
- бизнес-приоритет.

Но текущая задача требует именно обычное среднее.

Нельзя самостоятельно добавлять веса.

---

# 9. Почему Average nDCG не равно отношению сумм DCG и IDCG

Есть два разных способа агрегации.

## Macro-average

\[
\frac{1}{n}
\sum_i
\frac{DCG_i}{IDCG_i}
\]

## Отношение сумм

\[
\frac{\sum_i DCG_i}{\sum_i IDCG_i}
\]

Это не одно и то же.

Пример:

```text
Запрос A: DCG = 9, IDCG = 10, nDCG = 0.9
Запрос B: DCG = 1, IDCG = 2, nDCG = 0.5
```

Macro-average:

\[
\frac{0.9+0.5}{2}
=
0.7
\]

Отношение сумм:

\[
\frac{9+1}{10+2}
=
\frac{10}{12}
\approx
0.833
\]

Вторая формула сильнее взвешивает запрос A, потому что его IDCG больше.

В задании требуется именно среднее nDCG по запросам:

\[
\frac{1}{n}\sum_i nDCG_i
\]

# 16. Среднее значение

Пусть индивидуальные значения условно равны:

```text
q1: 0.999...
q2: 0.990...
q3: 0.999...
```

Тогда:

\[
Average\ nDCG
=
\frac{nDCG_1+nDCG_2+nDCG_3}{3}
\]

Получается:

```text
0.99613...
```

# 17. Что должна делать функция

API:

```python
def avg_ndcg(
    list_relevances: List[List[float]],
    k: int,
    method: str = "standard",
) -> float:
```

Алгоритм:

1. проверить `method`;
2. создать список результатов;
3. пройти циклом по запросам;
4. для каждого запроса построить actual top-k;
5. построить ideal top-k;
6. вычислить discounts;
7. применить standard или industry gain;
8. посчитать DCG;
9. посчитать IDCG;
10. получить nDCG;
11. добавить его в список;
12. найти среднее;
13. вернуть `float`.

# 19. Почему метод проверяется один раз

`method` один и тот же для всех запросов.

Поэтому его достаточно проверить до цикла.

Это:

- не повторяет одну и ту же проверку;
- делает код проще;
- быстрее завершает функцию при ошибке;
- соответствует контракту.

---

# 20. Почему используется цикл по запросам

Каждая строка — отдельная задача ранжирования.

Цикл:

```python
for relevance in list_relevances:
```

означает:

> Последовательно взять релевантности каждого запроса.

Внутри цикла полностью вычисляется nDCG текущего запроса.

---

# 21. Почему запросы могут иметь разную длину

В реальном наборе:

```python
[
    [0.9, 0.8, 0.7],
    [0.7, 0.4],
    [1.0, 0.8, 0.5, 0.3, 0.1],
]
```

Длины отличаются.

Это может происходить, если:

- retrieval вернул разное число кандидатов;
- часть документов отфильтрована;
- каталог ограничен;
- запрос имеет мало доступных результатов.

Решение работает с каждой строкой отдельно и строит позиции по фактическому числу элементов.

---

# 22. Почему нельзя преобразовать всю матрицу через обычный `np.asarray(..., dtype=float)`

Если все внутренние списки имеют одинаковую длину, можно получить прямоугольную матрицу.

Но если длины разные, NumPy не сможет создать обычный двумерный float-массив без дополнительного заполнения.

Поэтому безопаснее преобразовывать каждую строку отдельно:

```python
for relevance in list_relevances:
    relevance_array = np.asarray(relevance, dtype=float)
```

Это поддерживает ragged lists — списки разной длины.

# 23. Построение actual и ideal

Для каждого запроса:

```python
actual_relevance = relevance_array[:k]
```

Фактический порядок сохраняется.

Идеальный порядок:

```python
ideal_relevance = np.sort(relevance_array)[::-1][:k]
```

Критически важно:

- сортируется весь запрос;
- потом берётся top-k.

---

# 24. Позиционные discounts

```python
positions = np.arange(1, actual_relevance.size + 1)
discounts = np.log2(positions + 1)
```

Если у текущего запроса только 3 доступных объекта, позиции будут:

```python
[1, 2, 3]
```

Даже если `k = 5`.

Это предотвращает ошибку несовпадения размеров.

# 25. Standard gain

Для метода:

```python
"standard"
```

используется:

```python
actual_gain = actual_relevance
ideal_gain = ideal_relevance
```

Формула:

\[
gain(rel)=rel
\]

---

# 26. Industry gain

Для метода:

```python
"industry"
```

используется:

```python
actual_gain = np.power(2, actual_relevance) - 1
ideal_gain = np.power(2, ideal_relevance) - 1
```

Формула:

\[
gain(rel)=2^{rel}-1
\]

Actual и ideal обязательно преобразуются одинаково.

---

# 27. Вычисление nDCG одного запроса

Actual DCG:

```python
actual_dcg = np.sum(actual_gain / discounts)
```

Ideal DCG:

```python
ideal_dcg = np.sum(ideal_gain / discounts)
```

nDCG:

```python
actual_dcg / ideal_dcg
```

---

# 28. Запросы с IDCG = 0

Если все релевантности запроса равны нулю:

```python
[0.0, 0.0, 0.0]
```

то:

```text
DCG = 0
IDCG = 0
```

Деление невозможно.

В решении такой запрос получает:

```python
0.0
```

и это значение участвует в среднем.

Это означает:

> Запрос без релевантного результата считается запросом с нулевым качеством.

В production это решение должно быть согласовано отдельно.

Иногда такие запросы исключают из nDCG и измеряют coverage отдельной метрикой.

# 30. Почему используется `np.mean`

После цикла список `scores` содержит:

```python
[
    ndcg_query_1,
    ndcg_query_2,
    ndcg_query_3,
]
```

Тогда:

```python
np.mean(scores)
```

вычисляет обычное арифметическое среднее.

Это точно соответствует формуле Average nDCG.

---

# 31. Почему результат преобразуется в `float`

`np.mean` может вернуть:

```python
numpy.float64
```

Функция обещает:

```python
-> float
```

Поэтому используется:

```python
return float(np.mean(scores))
```

---

# 32. Что Average nDCG измеряет хорошо

Метрика подходит, когда:

- есть множество запросов;
- у каждого запроса есть градуированные relevance labels;
- верх выдачи важнее низа;
- нужно сравнить модели offline;
- запросы нужно оценить на общей нормированной шкале;
- каждому запросу даётся одинаковый вес.

---

# 33. Что среднее может скрыть

Одна средняя цифра может скрыть серьёзные проблемы.

Пример:

```text
Head queries:      nDCG = 0.99
Long-tail queries: nDCG = 0.55
Общее среднее:     nDCG = 0.88
```

Общее значение выглядит хорошим.

Но long tail работает плохо.

Поэтому в production Average nDCG нужно смотреть по сегментам:

- популярные и редкие запросы;
- новые и старые пользователи;
- языки;
- страны;
- категории;
- длина запроса;
- тип intent;
- наличие опечаток;
- новые документы;
- устройства.

---

# 34. Среднее без доверительного интервала

Две модели могут иметь:

```text
Model A: 0.812
Model B: 0.815
```

Разница:

```text
0.003
```

Она может быть случайным шумом evaluation-набора.

Поэтому зрелая оценка включает:

- bootstrap confidence interval;
- paired bootstrap по запросам;
- permutation test;
- анализ распределения разницы;
- долю запросов, где новая модель лучше.

Текущая задача требует только среднее, но в работе одной цифры недостаточно.

---

# 35. Почему сравнение должно быть парным

Если две модели оцениваются на одном и том же наборе запросов, для каждого запроса можно вычислить:

\[
\Delta_i =
nDCG_i^{new} - nDCG_i^{old}
\]

Это парное сравнение.

Оно лучше учитывает сложность конкретных запросов.

Можно анализировать:

- среднюю разницу;
- медиану;
- доверительный интервал;
- win rate;
- worst regressions.

---

# 36. Offline Average nDCG и бизнес

Рост Average nDCG не гарантирует:

- рост CTR;
- рост конверсии;
- рост выручки;
- снижение времени до ответа;
- улучшение удовлетворённости;
- повышение retention.

Причины:

- relevance labels могут быть несовершенным proxy;
- модель может стать медленнее;
- результаты могут стать однообразнее;
- может ухудшиться важный сегмент;
- клики могут быть смещены позициями;
- offline dataset может не соответствовать текущему трафику.

Поэтому после offline-оценки нужен A/B-тест.

Срезы:

# 38. Применение в финтехе

Можно оценивать среднее ранжирование:

- банковских предложений;

# 40. Применение в dating

Можно усреднять качество ранжирования профилей по пользователям.

Но relevance должна учитывать не только односторонний лайк.

Важны:

- взаимный match;
- meaningful conversation;
- безопасность;
- fairness;
- fatigue;
- долгосрочное удовлетворение.

Среднее может скрыть плохое качество для отдельных групп пользователей.

# 41. Применение в iGaming

Average nDCG может оценивать порядок игр или контента.

Но engagement нельзя делать единственной целью.

Обязательны:

- responsible-gaming guardrails;
- лимиты;
- self-exclusion;
- возрастные ограничения;
- harm indicators;
- регуляторные требования.

# 42. Применение в RAG и LLM

Для набора вопросов можно усреднять nDCG retrieved chunks.

Но нужно отдельно измерять:

- Recall@k;
- answer correctness;
- faithfulness;
- citation accuracy;
- latency;
- hallucination rate.

Хорошее среднее retrieval-качество не гарантирует хороший финальный ответ.

Можно усреднять nDCG по:

- high-severity incidents;

# 44. Типичные ошибки реализации

## Ошибка 1. Объединить все запросы

Нельзя превращать матрицу в один длинный список.

## Ошибка 2. Усреднить DCG

Нужно усреднять nDCG каждого запроса.

## Ошибка 3. Посчитать отношение сумм

Это не macro-average.

## Ошибка 4. Сортировать actual

Тогда каждый nDCG станет 1.

## Ошибка 5. Сортировать только actual top-k для IDCG

Лучшие документы за пределами top-k потеряются.

## Ошибка 6. Не обработать IDCG = 0

Появится `NaN`.

## Ошибка 7. Не обработать пустой внешний список

`np.mean([])` даст `NaN`.

## Ошибка 8. Использовать разные методы для actual и ideal

Нормализация станет некорректной.

## Ошибка 9. Добавить текст в ValueError

Условие требует пустое исключение.

## Ошибка 10. Округлить ответ

Тест может ожидать полную точность.

# 45. Сложность алгоритма

Пусть:

- `n` — число запросов;
- `m_i` — число документов для запроса `i`.

Для каждого запроса выполняется сортировка:

\[
O(m_i \log m_i)
\]

Общая сложность:

\[
O\left(
\sum_{i=1}^{n}
m_i \log m_i
\right)
\]

Дополнительная память зависит от размера текущего запроса и списка scores.

---

# 46. Почему не нужна сложная векторизация

Можно пытаться представить все запросы одной матрицей и вычислять всё одновременно.

Но это усложняется, если:

- строки имеют разную длину;
- есть padding;
- padding нельзя считать relevance;
- `k` больше длины части строк;
- нужно корректно обрабатывать пустые строки.

Цикл по запросам:

- проще;
- понятнее;
- надёжнее;
- соответствует учебной задаче;
- поддерживает разные длины.

# 47. Что проверят автотесты

С высокой вероятностью:

1. точное имя `avg_ndcg`;
2. точный порядок аргументов;
3. default method;
4. standard;
5. industry;
6. invalid method;
7. `raise ValueError` без текста;
8. корректный nDCG для каждого запроса;
9. обычное среднее;
10. сортировку всего списка для IDCG;
11. возвращаемый float;
12. PEP8/Pylint.

Могут быть проверки:

- один запрос;
- несколько запросов;
- разные длины;
- пустая строка;
- все нули;
- пустой внешний список;
- `k = 0`;
- `k` больше длины.

# 48. Итоговая логика

Для каждого запроса:

```text
actual top-k
        ↓
actual DCG

весь список → сортировка ↓ → ideal top-k
                               ↓
                             IDCG

nDCG = actual DCG / IDCG
```

После этого:

```text
Average nDCG = среднее всех nDCG
```

Главный бизнес-смысл:

> Average nDCG показывает среднюю долю от идеального качества ранжирования по набору запросов.
