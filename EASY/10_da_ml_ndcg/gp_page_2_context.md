# gp_page_2 — Бизнесовый и технический контекст Cumulative Gain

## 1. Что происходит в задаче

У нас есть две версии Telegram-бота:

- CyberTolya v1.0;
- CyberTolya v2.0.

Пользователь задаёт вопрос, а каждая версия бота возвращает ранжированный список видео. Для каждого видео есть численная оценка релевантности: насколько хорошо видео отвечает на вопрос пользователя.

Нужно получить метрику, которая позволит сравнить две выдачи численно.

На этой странице вводится первая метрика:

\[
CG@k = \sum_{i=1}^{k} rel_i
\]

`CG` означает **Cumulative Gain**, то есть накопленная или суммарная полезность.

---

# 2. Бизнесовая постановка

## 2.1. Зачем бизнесу численная метрика

Без метрики обсуждение качества выглядит так:

- «мне кажется, вторая версия лучше»;
- «первое видео хорошее»;
- «выдача вроде стала точнее».

Так невозможно надёжно:

- сравнить модели;
- выбрать версию для production;
- поймать регрессию;
- повторить эксперимент;
- установить критерий запуска;
- договориться между аналитиком, ML-инженером и продуктом.

Метрика переводит субъективное впечатление в воспроизводимое число.

---

## 2.2. Что является объектом, запросом и полезностью

В CyberTolya:

- **пользователь** — студент;
- **запрос** — вопрос студента;
- **объект** — видео;
- **ранжирование** — порядок видео;
- **релевантность** — насколько видео отвечает вопросу;
- **gain** — численная полезность видео.

Пример:

| Позиция | Релевантность | Интерпретация |
|---:|---:|---|
| 1 | 0.99 | Почти идеальное соответствие |
| 2 | 0.94 | Очень хорошее соответствие |
| 3 | 0.88 | Хорошее соответствие |
| 4 | 0.20 | Слабая связь |
| 5 | 0.00 | Не относится к вопросу |

---

# 3. Что измеряет Cumulative Gain

CG складывает релевантности первых `k` объектов.

Для трёх объектов:

\[
CG@3 = rel_1 + rel_2 + rel_3
\]

Для пяти объектов:

\[
CG@5 = rel_1 + rel_2 + rel_3 + rel_4 + rel_5
\]

Метрика отвечает на вопрос:

> Сколько суммарной релевантности содержится в первых k результатах выдачи?

---

# 4. Почему используется постфикс `@k`

В реальной системе может быть очень много кандидатов:

- тысячи видео;
- миллионы товаров;
- сотни тысяч вакансий;
- миллионы постов;
- тысячи документов базы знаний.

Но пользователь обычно видит и рассматривает только верхнюю часть выдачи.

Примеры:

- бот отправляет 3 ссылки;
- поисковик показывает 10 результатов;
- маркетплейс показывает 20 товаров на странице;
- рекомендательная карусель содержит 10 карточек;
- retrieval-этап передаёт ranker-у top-100 кандидатов.

Поэтому нас интересуют первые `k` позиций.

Примеры:

- `CG@1` — только первый объект;
- `CG@3` — первые три;
- `CG@5` — первые пять;
- `CG@10` — первые десять.

`k` нужно выбирать по реальному продуктовому интерфейсу или по этапу ML-пайплайна.

---

# 5. Ручной расчёт CyberTolya v1.0

Релевантности:

\[
[0.99,\ 0.91,\ 0.83]
\]

Считаем:

\[
CG@3_{v1} = 0.99 + 0.91 + 0.83
\]

По шагам:

\[
0.99 + 0.91 = 1.90
\]

\[
1.90 + 0.83 = 2.73
\]

Результат:

\[
CG@3_{v1} = 2.73
\]

---

# 6. Ручной расчёт CyberTolya v2.0

Релевантности:

\[
[0.99,\ 0.94,\ 0.88]
\]

Считаем:

\[
CG@3_{v2} = 0.99 + 0.94 + 0.88
\]

По шагам:

\[
0.99 + 0.94 = 1.93
\]

\[
1.93 + 0.88 = 2.81
\]

Результат:

\[
CG@3_{v2} = 2.81
\]

---

# 7. Бизнес-интерпретация сравнения

Получили:

\[
2.81 > 2.73
\]

По `CG@3` вторая версия лучше.

Абсолютное улучшение:

\[
2.81 - 2.73 = 0.08
\]

Относительное улучшение:

\[
\frac{2.81 - 2.73}{2.73} \times 100\%
\approx 2.93\%
\]

В коде задания относительное улучшение считать не нужно. Оно приведено, чтобы показать, как продукт может интерпретировать разницу моделей.

---

# 8. Что должна делать функция

Функция имеет точный API:

```python
def cumulative_gain(relevance: List[float], k: int) -> float:
```

Она должна:

1. получить список релевантностей;
2. взять первые `k` значений;
3. сложить их;
4. вернуть сумму как число.

Пример входа:

```python
relevance = [0.99, 0.94, 0.88, 0.74, 0.71, 0.68]
k = 5
```

Первые пять значений:

```python
[0.99, 0.94, 0.88, 0.74, 0.71]
```

Сумма:

\[
0.99 + 0.94 + 0.88 + 0.74 + 0.71 = 4.26
\]

---

# 9. Почему используется срез `relevance[:k]`

В Python индексы начинаются с нуля.

Для списка:

```python
[0.99, 0.94, 0.88, 0.74, 0.71, 0.68]
```

индексы такие:

| Индекс Python | Позиция выдачи | Значение |
|---:|---:|---:|
| 0 | 1 | 0.99 |
| 1 | 2 | 0.94 |
| 2 | 3 | 0.88 |
| 3 | 4 | 0.74 |
| 4 | 5 | 0.71 |
| 5 | 6 | 0.68 |

Срез:

```python
relevance[:5]
```

берёт индексы `0, 1, 2, 3, 4`, то есть ровно первые пять элементов.

Не нужно писать:

```python
relevance[:k + 1]
```

Это взяло бы лишний элемент.

---

# 10. Почему решение минимальное

Математическая задача состоит всего из двух операций:

1. выбрать первые `k` элементов;
2. посчитать сумму.

Поэтому правильное решение не должно включать:

- сортировку;
- логарифм;
- нормализацию;
- среднее;
- округление;
- сложную валидацию;
- изменение исходного списка.

Чем меньше лишней логики, тем меньше риск сломать скрытые тесты.

---

# 11. Почему используется NumPy

В шаблоне уже указан импорт:

```python
import numpy as np
```

NumPy предоставляет функцию:

```python
np.sum(...)
```

Она складывает числа.

Пример:

```python
np.sum([1.0, 2.0, 3.0])
```

возвращает:

```python
6.0
```

Можно было бы использовать встроенный `sum`, но `np.sum`:

- соответствует шаблону;
- использует предоставленный импорт;
- привычен в DA/ML;
- не оставляет неиспользуемый импорт, что полезно для Pylint.

---

# 12. Почему результат преобразуется в `float`

`np.sum` часто возвращает тип:

```python
numpy.float64
```

В API функции указан:

```python
-> float
```

Поэтому возвращается:

```python
float(score)
```

Это стандартный тип Python.

Так контракт функции становится яснее и стабильнее.

---

# 13. Граничные случаи

Условие не требует собственной валидации. Базовая реализация естественно обрабатывает типичные случаи.

## `k = 0`

```python
relevance[:0]
```

даёт пустой список, а его сумма равна `0.0`.

## Пустой список

```python
cumulative_gain([], 5)
```

вернёт `0.0`.

## `k` больше длины списка

```python
[0.8, 0.7][:10]
```

вернёт все доступные элементы. Ошибки не будет.

## Отрицательный `k`

В математическом смысле `k` должно быть неотрицательным. Но условие не задаёт, что делать с отрицательным значением. Поэтому мы не придумываем собственное исключение и строго сохраняем минимальный API.

---

# 14. Главное ограничение CG

CG не учитывает порядок объектов внутри рассматриваемой части выдачи.

Сравним:

```python
A = [1.0, 0.5, 0.0]
B = [0.0, 0.5, 1.0]
```

Для A:

\[
CG@3 = 1.5
\]

Для B:

\[
CG@3 = 1.5
\]

Но для пользователя A лучше: самый полезный объект находится первым.

Это не ошибка функции. Это ограничение самой метрики.

Именно поэтому дальше появляется DCG — Discounted Cumulative Gain, где нижние позиции штрафуются.

---

# 15. Что CG измеряет и чего не измеряет

CG измеряет:

- суммарную релевантность top-k;
- наличие полезных объектов в выбранной глубине.

CG не измеряет:

- правильность порядка;
- вероятность клика;
- удовлетворённость;
- время до ответа;
- разнообразие;
- новизну;
- fairness;
- безопасность;
- прибыль;
- долгосрочное удержание.

CG — базовый учебный строительный блок.

# 16. Применение в индустриях

## Образование

Объекты:

- видео;
- статьи;
- уроки;
- упражнения.

Gain:

- экспертная полезность;
- вероятность решить вопрос;
- релевантность учебной цели.

## E-commerce

Объекты:

- товары.

Gain:

- соответствие поисковому запросу;
- вероятность покупки;
- ожидаемая полезность.

Но позицию учитывать всё равно нужно, поэтому одного CG недостаточно.

## Финтех

Объекты:

- финансовые продукты;
- лиды;
- обращения;
- AML-кейсы.

Gain:

- ожидаемая ценность;
- приоритет;
- вероятность отклика;
- предотвращённый риск.

Нужно дополнительно учитывать compliance, fairness и suitability.

## Two-sided marketplace

Объекты:

- продавцы;
- товары;
- исполнители;
- заказы.

Gain должен учитывать обе стороны рынка, а не только пользователя.

## Dating

Объекты:

- профили.

Gain может быть вероятностью взаимного match, но нужны безопасность, fairness и долгосрочное качество.

## iGaming

Gain нельзя сводить только к вовлечению. Необходимы responsible-gaming ограничения, лимиты и защита пользователя.

## Big Tech и промышленность

Ранжироваться могут:

- документы;
- посты;
- рекламные объявления;
- фрагменты для RAG;
- алерты;
- инциденты;
- задачи обслуживания.

Gain означает полезность или критичность объекта.

# 17. Типичные ошибки в решении

## Ошибка 1. Суммировать весь список

```python
np.sum(relevance)
```

Тогда `k` игнорируется.

## Ошибка 2. Использовать `k + 1`

```python
np.sum(relevance[:k + 1])
```

Берётся лишний элемент.

## Ошибка 3. Сортировать список

```python
np.sum(sorted(relevance, reverse=True)[:k])
```

Это меняет фактическую выдачу.

## Ошибка 4. Считать среднее

```python
np.mean(relevance[:k])
```

Это другая метрика.

## Ошибка 5. Печатать вместо возврата

```python
print(np.sum(relevance[:k]))
```

Функция вернёт `None`, и тест упадёт.

## Ошибка 6. Округлять

```python
round(score, 2)
```

Условие округления не требует, а точность теряется.

## Ошибка 7. Менять API

Нельзя переименовывать функцию или добавлять обязательные аргументы.

# 18. Что проверяют тесты

С высокой вероятностью проверяется:

- точное имя `cumulative_gain`;
- два аргумента `relevance` и `k`;
- использование только первых `k` элементов;
- корректная сумма;
- числовой возврат;
- отсутствие нарушения API;
- PEP8/Pylint.

---

# 19. Сложность

Если учитывается `k` объектов, время:

\[
O(\min(k, n))
\]

где `n` — длина списка.

Срез списка создаёт копию первых элементов, поэтому дополнительная память:

\[
O(\min(k, n))
\]

Для учебной задачи это полностью нормально.

---

# 20. Итог

Формула:

\[
CG@k = \sum_{i=1}^{k} rel_i
\]

Кодовая логика:

```python
score = np.sum(relevance[:k])
return float(score)
```

Бизнесовый смысл:

> CG@k показывает суммарную полезность первых k объектов выдачи.

Ограничение:

> CG не учитывает, насколько правильно полезные объекты расположены внутри top-k.
