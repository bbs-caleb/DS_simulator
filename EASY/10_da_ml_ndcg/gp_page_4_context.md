# gp_page_4 — Бизнесовый и технический контекст nDCG

# 1. Где мы находимся в развитии метрики

В этом уроке метрика строится поэтапно.

Мы уже прошли два шага.

## Шаг 1. Cumulative Gain

\[
CG@k = \sum_{i=1}^{k} rel_i
\]

CG отвечает на вопрос:

> Сколько суммарной релевантности находится в первых `k` объектах?

Главный недостаток:

> CG не учитывает порядок элементов внутри top-k.

---

## Шаг 2. Discounted Cumulative Gain

Standard-вариант:

\[
DCG@k =
\sum_{i=1}^{k}
\frac{rel_i}{\log_2(i+1)}
\]

Industry-вариант:

\[
DCG@k =
\sum_{i=1}^{k}
\frac{2^{rel_i}-1}{\log_2(i+1)}
\]

DCG отвечает на вопрос:

> Насколько много релевантности находится в top-k с учётом того, что верхние позиции важнее нижних?

DCG исправляет главный недостаток CG: релевантный объект на первой позиции даёт больший вклад, чем такой же объект на пятой позиции.

Но у DCG остаётся другая проблема.

---

# 2. Главная проблема ненормированного DCG

Сырое значение DCG зависит не только от качества порядка.

Оно также зависит от:

- количества релевантных объектов;
- абсолютного уровня релевантностей;
- выбранного `k`;
- шкалы разметки;
- сложности запроса;
- наличия подходящего контента в каталоге.

Из-за этого одинаково хорошие ранжирования для двух разных запросов могут иметь совершенно разные значения DCG.

---

# 3. Бизнес-пример: лёгкий и сложный запрос

Представим два вопроса.

## Запрос A

```text
Собеседование System Design
```

На канале есть несколько видео, которые идеально соответствуют запросу.

Релевантности могут выглядеть так:

```python
[0.99, 0.94, 0.88, 0.74, 0.71]
```

DCG будет относительно большим.

---

## Запрос B

```text
Как красиво писать художественные тексты?
```

На канале почти нет подходящих материалов.

Даже лучшие найденные видео имеют слабую связь:

```python
[0.12, 0.08, 0.05, 0.03, 0.01]
```

DCG будет маленьким.

---

## В чём проблема сравнения

Предположим, что обе выдачи расположены идеально:

- для запроса A все доступные релевантности стоят в правильном порядке;
- для запроса B тоже выбран лучший возможный порядок.

Система сделала максимум возможного в обоих случаях.

Но сырой DCG запроса A намного больше DCG запроса B.

Если просто усреднить эти числа, запрос A будет иметь намного больший численный вклад только потому, что в каталоге для него есть хорошие документы.

Получается, мы смешиваем две вещи:

1. **качество порядка модели**;
2. **насколько вообще богат каталог подходящими объектами**.

Для оценки ранжирования желательно отделить первое от второго.

---

# 4. Продуктовая интерпретация проблемы

Представим двух пользователей.

Пользователь 1 задаёт популярный вопрос, для которого в базе есть 20 идеальных видео.

Пользователь 2 задаёт редкий вопрос, для которого существует только одно частично подходящее видео.

Если система в обоих случаях поставила лучший доступный результат на первое место, то с точки зрения ранжирования она отработала хорошо.

Но абсолютные значения DCG будут несопоставимы.

Поэтому продукту нужна метрика, которая отвечает не на вопрос:

> Сколько абсолютной релевантности система набрала?

а на вопрос:

> Какую долю от максимально возможного качества для этого конкретного запроса система достигла?

Именно это делает nDCG.

---

# 5. Идеальная выдача

Чтобы понять качество фактической выдачи, нужно сравнить её с лучшим возможным порядком.

Пусть фактические релевантности:

```python
[0.99, 0.94, 0.74, 0.88, 0.71, 0.68]
```

Видно, что значение `0.88` стоит ниже значения `0.74`.

Идеальный порядок получается сортировкой по убыванию:

```python
[0.99, 0.94, 0.88, 0.74, 0.71, 0.68]
```

В нём самый релевантный объект стоит первым, следующий по релевантности — вторым и так далее.

---

# 6. Что такое IDCG

**IDCG** расшифровывается как:

**Ideal Discounted Cumulative Gain**

То есть идеальный DCG.

Чтобы получить IDCG:

1. взять все истинные оценки релевантности для текущей выдачи;
2. отсортировать их по убыванию;
3. взять первые `k`;
4. применить ту же формулу DCG.

Standard IDCG:

\[
IDCG@k =
\sum_{i=1}^{k}
\frac{rel_i^{ideal}}{\log_2(i+1)}
\]

Industry IDCG:

\[
IDCG@k =
\sum_{i=1}^{k}
\frac{2^{rel_i^{ideal}}-1}{\log_2(i+1)}
\]

IDCG — максимально возможный DCG для этого набора релелевантностей и выбранного `k`.

---

# 7. Очень важная тонкость: сортируется весь список

Предположим:

```python
relevance = [0.9, 0.8, 0.1, 1.0]
k = 3
```

Фактический top-3:

```python
[0.9, 0.8, 0.1]
```

Но в полном списке на четвёртой позиции находится лучший объект:

```python
1.0
```

Правильная идеальная выдача:

```python
[1.0, 0.9, 0.8]
```

Для этого нужно:

1. отсортировать **весь** список;
2. затем взять первые `k`.

Правильно:

```python
sorted(relevance, reverse=True)[:k]
```

Неправильно:

```python
sorted(relevance[:k], reverse=True)
```

Во втором случае объект `1.0`, который модель ошибочно поставила за пределы top-3, вообще не попадёт в идеальную выдачу.

Тогда метрика не накажет модель за важную ошибку.

---

# 8. Формула nDCG

\[
nDCG@k =
\frac{DCG@k}{IDCG@k}
\]

Где:

- `DCG@k` — значение фактического порядка;
- `IDCG@k` — значение идеального порядка.

Название:

**Normalized Discounted Cumulative Gain**

То есть нормированный дисконтированный накопленный выигрыш.

---

# 9. Интуиция деления

Предположим:

```text
Фактический DCG = 8
Идеальный DCG = 10
```

Тогда:

\[
nDCG = \frac{8}{10} = 0.8
\]

Интерпретация:

> Текущая выдача набрала 80% от максимально возможного качества для этого запроса.

---

# 10. Почему идеальная выдача имеет nDCG = 1

Если фактический порядок уже идеален, то:

\[
DCG@k = IDCG@k
\]

Следовательно:

\[
nDCG@k =
\frac{DCG@k}{IDCG@k}
=
\frac{IDCG@k}{IDCG@k}
=
1
\]

Значение `1.0` означает идеальный порядок при выбранных:

- релевантностях;
- `k`;
- методе DCG.

---

# 11. Почему nDCG обычно находится от 0 до 1

При стандартных предпосылках:

- релевантности неотрицательные;
- gain неотрицательный;
- IDCG посчитан по тому же набору;
- используется одинаковый `k`;
- используется одинаковый метод;
- идеальный порядок действительно максимизирует DCG.

Тогда:

\[
0 \le DCG@k \le IDCG@k
\]

После деления:

\[
0 \le nDCG@k \le 1
\]

Чем ближе значение к 1, тем ближе фактический порядок к идеальному.

---

# 12. Ручной пример из задания

Дано:

```python
relevance = [0.99, 0.94, 0.74, 0.88, 0.71, 0.68]
k = 5
method = "standard"
```

Фактический top-5:

```python
[0.99, 0.94, 0.74, 0.88, 0.71]
```

Идеальный top-5:

```python
[0.99, 0.94, 0.88, 0.74, 0.71]
```

---

# 13. Расчёт фактического DCG

\[
DCG@5 =
\frac{0.99}{\log_2(2)}
+
\frac{0.94}{\log_2(3)}
+
\frac{0.74}{\log_2(4)}
+
\frac{0.88}{\log_2(5)}
+
\frac{0.71}{\log_2(6)}
\]

Приближённые вклады:

| Позиция | rel | discount | Вклад |
|---:|---:|---:|---:|
| 1 | 0.99 | 1.0000 | 0.9900 |
| 2 | 0.94 | 1.5850 | 0.5931 |
| 3 | 0.74 | 2.0000 | 0.3700 |
| 4 | 0.88 | 2.3219 | 0.3790 |
| 5 | 0.71 | 2.5850 | 0.2747 |

Сумма:

\[
DCG@5 \approx 2.6067348326
\]

---

# 14. Расчёт идеального DCG

Идеальный порядок:

```python
[0.99, 0.94, 0.88, 0.74, 0.71]
```

\[
IDCG@5 =
\frac{0.99}{\log_2(2)}
+
\frac{0.94}{\log_2(3)}
+
\frac{0.88}{\log_2(4)}
+
\frac{0.74}{\log_2(5)}
+
\frac{0.71}{\log_2(6)}
\]

Приблизительно:

\[
IDCG@5 \approx 2.6164401145
\]

---

# 15. Расчёт nDCG

\[
nDCG@5 =
\frac{2.6067348326}{2.6164401145}
\]

\[
nDCG@5 \approx 0.9962906539
\]

Поэтому в условии показано:

```text
0.9962...
```

Выдача очень близка к идеальной.

Единственная заметная ошибка — перестановка значений `0.74` и `0.88`.

---

# 16. Почему значение настолько высокое

Ошибка произошла не на первой позиции.

Первое и второе места уже правильные:

```python
0.99
0.94
```

Переставлены только третья и четвёртая релевантности:

```python
0.74
0.88
```

Логарифмические веса третьей и четвёртой позиций отличаются, но не радикально.

Поэтому потеря относительно идеала небольшая.

---

# 17. Standard и Industry в nDCG

Функция должна поддерживать те же два метода, что и DCG.

## Standard

Actual:

\[
DCG =
\sum
\frac{rel_i}{\log_2(i+1)}
\]

Ideal:

\[
IDCG =
\sum
\frac{rel_i^{ideal}}{\log_2(i+1)}
\]

## Industry

Actual:

\[
DCG =
\sum
\frac{2^{rel_i}-1}{\log_2(i+1)}
\]

Ideal:

\[
IDCG =
\sum
\frac{2^{rel_i^{ideal}}-1}{\log_2(i+1)}
\]

Критически важно:

> В числителе и знаменателе должен использоваться один и тот же метод.

Нельзя считать actual через standard, а ideal через industry.

---

# 18. Почему метод может менять nDCG

Standard и industry по-разному преобразуют релевантность.

Industry сильнее меняет относительные различия высоких значений.

Поэтому одна и та же перестановка объектов может иметь разную стоимость.

Если модель опустила очень релевантный объект вниз, industry-вариант может наказать её сильнее, особенно при многоуровневой шкале вроде `0, 1, 2, 3`.

---

# 19. Зачем нормализация позволяет усреднять запросы

После нормализации каждый запрос оценивается относительно собственного идеала.

Пример:

| Запрос | DCG | IDCG | nDCG |
|---|---:|---:|---:|
| Популярный | 8.0 | 10.0 | 0.80 |
| Редкий | 0.8 | 1.0 | 0.80 |

Сырые DCG отличаются в десять раз.

Но обе системы достигли 80% от идеала.

Поэтому nDCG лучше подходит для macro-average по запросам.

---

# 20. Средний nDCG

Для множества запросов:

\[
Mean\ nDCG@k =
\frac{1}{N}
\sum_{q=1}^{N}
nDCG_q@k
\]

Каждый запрос сначала нормализуется отдельно.

Только после этого значения усредняются.

Неправильно:

1. сложить все DCG;
2. сложить все IDCG;
3. поделить общие суммы, если нужна именно macro-average оценка.

Это будет другая схема взвешивания.

---

# 21. Macro-average и weighted average

## Macro-average

Каждый запрос имеет одинаковый вес.

Плюс:

- редкие запросы не исчезают в среднем.

Минус:

- запрос с одним показом весит столько же, сколько запрос с миллионом показов.

## Weighted average

Запросам задаются веса:

- по числу показов;
- по бизнес-ценности;
- по сегменту;
- по выручке;
- по важности.

Формула:

\[
Weighted\ nDCG =
\frac{\sum_q w_q \cdot nDCG_q}
{\sum_q w_q}
\]

В production полезно смотреть оба варианта и сегменты.

---

# 22. Случай IDCG = 0

Если все релевантности равны нулю:

```python
[0.0, 0.0, 0.0]
```

Тогда:

```text
DCG = 0
IDCG = 0
```

Формально возникает:

\[
\frac{0}{0}
\]

Это неопределённость.

В разных системах возможны правила:

- вернуть `0.0`;
- вернуть `1.0`;
- исключить запрос из среднего;
- вернуть `NaN`.

В этом решении возвращается:

```python
0.0
```

Почему это практично:

- нет деления на ноль;
- нет `NaN`;
- результат остаётся числом;
- запрос без релевантных объектов не даёт искусственного идеального балла.

В реальном проекте это правило нужно явно зафиксировать в контракте метрики.

---

# 23. Почему нельзя автоматически считать zero-IDCG запрос идеальным

Можно рассуждать так:

> Модель ничего не могла найти, поэтому она ничего не испортила.

Тогда кто-то захочет вернуть `1.0`.

Но это может искусственно повысить среднее качество на запросах, где каталог вообще ничего не умеет покрывать.

Возврат `0.0` делает отсутствие полезных объектов заметным.

Другой зрелый вариант — отдельно считать:

- nDCG только на запросах с релевантными документами;
- долю запросов с `IDCG = 0`;
- coverage каталога.

---

# 24. Что именно должна делать функция

API:

```python
def normalized_dcg(
    relevance: List[float],
    k: int,
    method: str = "standard",
) -> float:
```

Алгоритм:

1. проверить `method`;
2. преобразовать релевантности в NumPy-массив;
3. получить фактический top-k;
4. отсортировать весь массив по убыванию;
5. получить идеальный top-k;
6. построить позиционные discounts;
7. получить actual gain;
8. получить ideal gain;
9. посчитать DCG;
10. посчитать IDCG;
11. обработать `IDCG = 0`;
12. вернуть `DCG / IDCG`.

---

# 25. Почему invalid method должен вызвать ValueError

Допустимы только:

```python
"standard"
```

и:

```python
"industry"
```

Для любого другого значения:

```python
raise ValueError
```

Условие отдельно требует исключение без текста.

Правильно:

```python
raise ValueError
```

Неправильно:

```python
raise ValueError("Unknown method")
```

Даже полезное сообщение может нарушить строгий автотест.

---

# 26. Почему actual и ideal должны иметь одинаковые позиции

Оба списка содержат первые `k` элементов.

Для каждого строятся одинаковые позиции:

```text
1, 2, 3, ..., k
```

Различается только порядок gain.

Это обеспечивает честное сравнение:

- одна глубина;
- одна функция discount;
- один метод;
- один набор исходных релевантностей.

---

# 27. Почему нельзя сортировать фактическую выдачу

Фактический DCG должен отражать реальный порядок модели.

Если отсортировать actual:

```python
actual_relevance = np.sort(relevance_array)[::-1][:k]
```

то actual и ideal станут одинаковыми.

nDCG всегда будет равно `1.0`.

Метрика перестанет измерять качество модели.

---

# 28. Почему нельзя строить IDCG по model scores

Для nDCG нужны истинные relevance labels.

Model score определяет порядок, но не является эталонной полезностью.

IDCG строится по истинной релевантности документов, расположенных в максимально хорошем порядке.

Если сортировать по model score, получится снова фактический порядок модели, а не идеал.

---

# 29. Почему нельзя использовать разные k

Неправильно:

```text
DCG@5 / IDCG@10
```

Числитель и знаменатель должны относиться к одной глубине.

Правильно:

```text
DCG@5 / IDCG@5
```

---

# 30. Почему нельзя сравнивать nDCG без указания k

`nDCG@1` и `nDCG@10` измеряют разные продуктовые сценарии.

Модель может:

- отлично выбирать первый объект;
- плохо упорядочивать следующие девять.

Тогда:

```text
nDCG@1 — высокий
nDCG@10 — ниже
```

Поэтому всегда нужно указывать глубину.

---

# 31. Как выбирать k в продукте

`k` должен соответствовать реальному использованию.

Примеры:

- бот отправляет 3 видео → `nDCG@3`;
- поисковик показывает 10 ссылок → `nDCG@10`;
- карусель содержит 20 карточек → `nDCG@20`;
- retrieval передаёт ranker-у 100 кандидатов → чаще `Recall@100`, а для порядка можно смотреть `nDCG@100`.

Полезно считать несколько глубин:

```text
nDCG@1
nDCG@3
nDCG@5
nDCG@10
```

---

# 32. Что nDCG измеряет хорошо

nDCG хорошо подходит, если:

- порядок объектов важен;
- полезных объектов может быть несколько;
- релевантность градуированная;
- верхние позиции важнее нижних;
- нужно сравнивать разные запросы;
- есть эталонные labels;
- продукт использует top-k.

---

# 33. Что nDCG не измеряет

Обычный nDCG не учитывает автоматически:

- latency;
- diversity;
- novelty;
- fairness;
- безопасность;
- прибыль;
- возвраты;
- жалобы;
- долгосрочное удержание;
- position bias в самих labels;
- отсутствие релевантного контента в каталоге;
- пользовательскую персонализацию, если labels общие.

Поэтому nDCG — сильная offline-метрика, но не полная бизнес-цель.

---

# 34. Offline и online качество

Высокий nDCG может не привести к росту бизнеса.

Причины:

- labels не отражают реальную ценность;
- пользователь не доверяет интерфейсу;
- модель стала медленнее;
- результаты однообразные;
- вырос CTR, но упала конверсия;
- произошёл data drift;
- повысилось качество на лёгких запросах и снизилось на важных;
- были использованы смещённые клики.

Правильная схема:

1. offline nDCG;
2. error analysis;
3. latency и нагрузочное тестирование;
4. A/B-тест;
5. guardrails;
6. мониторинг после запуска.

# 35. Связь с retrieval и ranking

В двухэтапной системе:

1. retrieval находит кандидатов;
2. ranker сортирует кандидатов.

Если релевантный документ не попал в candidates, ranker его уже не спасёт.

Поэтому:

- retrieval часто оценивается через `Recall@k`;
- финальное ранжирование — через `nDCG@k`, MRR, MAP и продуктовые метрики.

Диагностика:

| Recall кандидатов | nDCG | Возможная проблема |
|---|---|---|
| Низкий | Низкий | Retrieval не находит нужное |
| Высокий | Низкий | Ranker плохо сортирует |
| Высокий | Высокий | Offline pipeline выглядит хорошо |
| Низкий | Высокий | Возможно, оценка построена на смещённом candidate set |

Частая практика:

# 44. Типичные ошибки реализации

## Ошибка 1. Сортировать только первые k

```python
np.sort(relevance_array[:k])[::-1]
```

Правильно сортировать весь список, затем брать top-k.

## Ошибка 2. Сортировать actual

Тогда nDCG всегда станет 1.

## Ошибка 3. Использовать разные методы

Actual standard и ideal industry несовместимы.

## Ошибка 4. Использовать разные discounts

Позиционные веса должны совпадать.

## Ошибка 5. Делить IDCG на DCG

Правильная формула:

\[
DCG / IDCG
\]

а не наоборот.

## Ошибка 6. Не обработать IDCG = 0

Появится `NaN` или warning о делении на ноль.

## Ошибка 7. Добавить текст в ValueError

Условие требует исключение без сообщения.

## Ошибка 8. Округлять результат

Автотест может ожидать полную точность.

## Ошибка 9. Использовать model scores вместо labels

Метрика потеряет смысл эталонного сравнения.

## Ошибка 10. Забыть ограничение k

Тогда будет рассчитана метрика по всей выдаче.

# 45. Сложность алгоритма

Пусть длина списка — `n`.

## Сортировка

\[
O(n \log n)
\]

## Расчёт actual и ideal DCG

\[
O(\min(k, n))
\]

## Общая временная сложность

\[
O(n \log n)
\]

## Дополнительная память

NumPy создаёт массивы для:

- relevance;
- ideal order;
- positions;
- discounts;
- gains.

Примерно:

\[
O(n)
\]

Для учебной задачи это корректно и достаточно эффективно.

---

# 46. Production-оптимизация top-k

Если список огромный, полная сортировка может быть дорогой.

Вместо полного `O(n log n)` можно найти top-k через partition:

```python
np.partition
```

или специализированную структуру.

Тогда можно приблизиться к:

\[
O(n + k \log k)
\]

Но в текущем задании это не нужно.

Полная сортировка:

- проще;
- надёжнее;
- понятнее;
- меньше риск ошибок;
- соответствует учебной цели.

# 47. Что проверяют автотесты

С высокой вероятностью тесты проверят:

1. точное имя `normalized_dcg`;
2. три аргумента;
3. default `"standard"`;
4. правильный actual DCG;
5. правильный IDCG;
6. сортировку по убыванию;
7. standard;
8. industry;
9. неправильный method;
10. `raise ValueError` без текста;
11. числовой возврат;
12. PEP8/Pylint.

Могут проверяться:

- пустой список;
- `k = 0`;
- `k > len(relevance)`;
- идеальный порядок;
- объект высокой релевантности за пределами top-k;
- все нулевые labels.

# 48. Почему решение минимальное, но надёжное

Решение не добавляет:

- необязательные исключения;
- округление;
- сортировку фактической выдачи;
- сторонние зависимости;
- классы;
- сложную архитектуру.

Но обрабатывает критические случаи:

- два метода;
- invalid method;
- `IDCG = 0`;
- `k > len(relevance)`;
- пустой список.

# 49. Итоговая бизнес-интерпретация

CG:

> Сколько релевантности находится в top-k?

DCG:

> Насколько хорошо релевантность расположена с учётом позиций?

IDCG:

> Какой DCG был бы у идеального порядка?

nDCG:

> Какую долю от идеального качества набрала текущая выдача?

Главная формула:

\[
nDCG@k =
\frac{DCG@k}{IDCG@k}
\]

Главный смысл:

> nDCG позволяет сравнивать качество порядка между запросами с разным абсолютным уровнем релевантности.
