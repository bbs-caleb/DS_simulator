# gp_page_1 — Подробный контекст и теория nDCG

# 1. Зачем вообще нужна эта задача

Представим, что студент пишет в чат вопрос:

> Как интерпретировать p-value в A/B-тесте?

В базе есть сотни видео: про p-value, статистическую мощность, доверительные интервалы, SQL, Python, машинное обучение, ClickHouse и карьеру аналитика.

CyberTolya должен выбрать несколько видео и расположить их в правильном порядке.

Пользователю не нужен случайный набор роликов. Ему нужен короткий список, в котором:

1. первое видео почти наверняка отвечает на вопрос;
2. второе полезно, если первое оказалось недостаточным;
3. третье раскрывает соседнюю тему;
4. нерелевантные видео не занимают верхние позиции.

Это задача ранжирования.

---

# 2. Бизнес-контекст

## 2.1. Что бизнес получает от хорошего ранжирования

Хорошая выдача может повысить:

- долю пользователей, которые находят ответ;
- скорость решения вопроса;
- удовлетворённость;
- удержание;
- число просмотренных полезных материалов;
- конверсию в обучение;
- доверие к продукту;
- снижение нагрузки на поддержку и преподавателей.

Плохая выдача приводит к:

- раздражению;
- потере времени;
- отказу от продукта;
- повторным вопросам;
- росту нагрузки на поддержку;
- недоверию к автоматизации.

Ранжирование — не абстрактная математика. Оно напрямую влияет на пользовательский опыт и экономику продукта.

## 2.2. Почему недостаточно просто найти все релевантные видео

Допустим, система вернула десять видео. Среди них есть три полезных.

Система A:

1. полезное;
2. полезное;
3. полезное;
4–10. нерелевантные.

Система B:

1–7. нерелевантные;
8. полезное;
9. полезное;
10. полезное.

Количество полезных видео одинаковое: три. Но пользовательский опыт радикально отличается.

В системе A человек почти сразу получает ответ. В системе B он, скорее всего, уйдёт раньше, чем дойдёт до полезных результатов.

Следовательно, метрика должна учитывать позицию.

---

# 3. Основные сущности

## 3.1. Запрос

Запрос — выражение потребности пользователя.

Это может быть:

- текстовый вопрос;
- поисковая фраза;
- история просмотров;
- текущая карточка товара;
- профиль пользователя;
- контекст сессии.

В CyberTolya запрос — вопрос студента.

## 3.2. Документ

Документ — объект, который система может показать пользователю.

Документом может быть:

- видео;
- товар;
- ресторан;
- вакансия;
- кандидат;
- трек;
- пост;
- реклама;
- потенциальный партнёр;
- статья базы знаний;
- chunk для RAG;
- ответ модели.

В CyberTolya документ — YouTube-видео.

## 3.3. Кандидаты

Обычно система не ранжирует всю базу сразу. Сначала быстро выбирается набор потенциально подходящих объектов, затем более точная модель сортирует их.

Типичная архитектура:

1. **retrieval** — быстро получить кандидатов;
2. **ranking** — точно упорядочить кандидатов.

В больших системах часто есть:

1. candidate generation;
2. pre-ranking;
3. ranking;
4. re-ranking;
5. business rules;
6. diversity;
7. safety filtering.

## 3.4. Релевантность

Релевантность показывает, насколько объект отвечает потребности пользователя.

В условии курса это экспертная оценка от `0.0` до `1.0`.

Пример:

- `1.0` — полностью отвечает;
- `0.8` — очень полезно;
- `0.5` — частично связано;
- `0.2` — слабая связь;
- `0.0` — не относится к запросу.

Релевантность бывает:

### Бинарной

- `1` — релевантно;
- `0` — нерелевантно.

### Градуированной

- `0` — не релевантно;
- `1` — слабо релевантно;
- `2` — релевантно;
- `3` — очень релевантно.

### Непрерывной

Любое число на шкале, например от `0` до `1`.

nDCG особенно полезна при градуированной релевантности.

---

# 4. Ранжирование и классификация — не одно и то же

Классификация отвечает:

> К какому классу относится объект?

Например:

- спам или не спам;
- дефолт или не дефолт;
- мошенничество или не мошенничество.

Ранжирование отвечает:

> Как упорядочить множество объектов для конкретного запроса?

| Аспект | Классификация | Ранжирование |
|---|---|---|
| Выход | класс или вероятность | порядок объектов |
| Сравнение | объект часто рассматривается отдельно | объекты сравниваются между собой |
| Важность позиции | обычно нет | критически важна |
| Типичные метрики | accuracy, ROC-AUC, F1 | nDCG, MAP, MRR, Recall@k |
| Контекст запроса | может отсутствовать | почти всегда есть |
| Цель | определить класс | поставить лучшие объекты выше |

Модель ранжирования обычно выдаёт score для пары `(query, document)`, после чего документы сортируются по score.

---

# 5. Что означает top-k

`k` — число первых позиций, которые оцениваются.

Примеры:

- `nDCG@1` — только первое место;
- `nDCG@3` — первые три;
- `nDCG@10` — первые десять;
- `nDCG@100` — первые сто.

Почему это важно:

- пользователь может видеть только 3 карточки;
- на экране помещается 10 товаров;
- реклама оплачивается только в верхних позициях;
- downstream-система использует top-5.

Одна и та же модель может иметь высокий `nDCG@100`, но низкий `nDCG@3`. Значит, хорошие объекты в целом находятся, но поднимаются недостаточно высоко.

Говорить «nDCG равен 0.82» без `k` — неполно.

---

# 6. Cumulative Gain — CG

## 6.1. Формула

\[
CG@k = \sum_{i=1}^{k} rel_i
\]

где:

- `k` — глубина выдачи;
- `i` — позиция;
- `rel_i` — релевантность объекта на позиции `i`.

CG просто складывает полезность первых `k` результатов.

## 6.2. Пример

Пусть релевантности первых трёх видео:

\[
[1.0,\ 0.7,\ 0.2]
\]

Тогда:

\[
CG@3 = 1.0 + 0.7 + 0.2 = 1.9
\]

## 6.3. Главный недостаток CG

Выдача A:

\[
[1.0,\ 0.7,\ 0.2]
\]

Выдача B:

\[
[0.2,\ 0.7,\ 1.0]
\]

Для обеих:

\[
CG@3 = 1.9
\]

Но A лучше: самое релевантное видео стоит первым.

CG не видит порядок, только сумму.

## 6.4. Когда CG полезен

CG полезен как учебный шаг и как простая мера суммарной полезности в top-k.

Он отвечает:

> Сколько общей релевантности находится в top-k?

Но не отвечает:

> Насколько хорошо эта релевантность распределена по позициям?

---

# 7. Discounted Cumulative Gain — DCG

Слово **discounted** означает: вклад результата уменьшается с ухудшением позиции.

Идея:

- первое место почти не штрафуется;
- второе получает небольшой штраф;
- десятое — заметный;
- сотое — ещё больший.

---

# 8. Стандартная формула DCG

\[
DCG@k =
\sum_{i=1}^{k}
\frac{rel_i}{\log_2(i+1)}
\]

## 8.1. Числитель `rel_i`

Чем релевантнее объект, тем больший вклад он даёт.

## 8.2. Знаменатель `log2(i+1)`

Чем больше позиция `i`, тем больше знаменатель и тем меньше вклад.

## 8.3. Почему `i + 1`

При `i = 1`:

\[
\log_2(1)=0
\]

Делить на ноль нельзя. Поэтому используется:

\[
\log_2(i+1)
\]

Для первой позиции:

\[
\log_2(2)=1
\]

Первый результат не штрафуется.

---

# 9. Значения дисконта

| Позиция `i` | `log2(i+1)` | `1/log2(i+1)` |
|---:|---:|---:|
| 1 | 1.000 | 1.000 |
| 2 | 1.585 | 0.631 |
| 3 | 2.000 | 0.500 |
| 4 | 2.322 | 0.431 |
| 5 | 2.585 | 0.387 |
| 10 | 3.459 | 0.289 |

Интерпретация:

- первая позиция учитывается полностью;
- вторая — примерно на 63%;
- третья — на 50%;
- десятая — примерно на 29%.

Логарифм снижает вклад плавно.

---

# 10. Ручной расчёт DCG

Пусть:

\[
rel = [1.0,\ 0.7,\ 0.2]
\]

Тогда:

\[
DCG@3 =
\frac{1.0}{\log_2(2)} +
\frac{0.7}{\log_2(3)} +
\frac{0.2}{\log_2(4)}
\]

Значения:

\[
\log_2(2)=1
\]

\[
\log_2(3)\approx1.585
\]

\[
\log_2(4)=2
\]

Подставляем:

\[
DCG@3 = 1.0 + \frac{0.7}{1.585} + \frac{0.2}{2}
\]

\[
DCG@3 \approx 1.0 + 0.442 + 0.1 = 1.542
\]

---

# 11. Сравнение двух порядков через DCG

Выдача A:

\[
[1.0,\ 0.7,\ 0.2]
\]

Выдача B:

\[
[0.2,\ 0.7,\ 1.0]
\]

Для A:

\[
DCG_A@3 \approx 1.542
\]

Для B:

\[
DCG_B@3 = 0.2 + \frac{0.7}{1.585} + \frac{1.0}{2}
\]

\[
DCG_B@3 \approx 0.2 + 0.442 + 0.5 = 1.142
\]

Теперь метрика видит, что A лучше.

---

# 12. Индустриальная формула DCG

Часто используют:

\[
DCG@k =
\sum_{i=1}^{k}
\frac{2^{rel_i}-1}{\log_2(i+1)}
\]

Здесь gain:

\[
gain(rel_i)=2^{rel_i}-1
\]

## 12.1. Зачем экспоненциальный gain

Он сильнее подчёркивает разницу между высокими уровнями релевантности.

| rel | Линейный gain | `2^rel - 1` |
|---:|---:|---:|
| 0 | 0 | 0 |
| 1 | 1 | 1 |
| 2 | 2 | 3 |
| 3 | 3 | 7 |

Разница между уровнями `2` и `3` становится гораздо важнее.

Это полезно, если «идеально подходящий» объект намного ценнее, чем просто «хороший».

## 12.2. Как выбрать gain

Линейный gain подходит, если соседние уровни шкалы примерно равноудалены по смыслу.

Экспоненциальный gain подходит, если высокий уровень релевантности должен иметь непропорционально большой вес.

Метрика должна соответствовать бизнес-смыслу разметки.

---

# 13. Почему одного DCG недостаточно

DCG зависит от:

- числа релевантных объектов;
- максимальных оценок;
- длины списка;
- конкретного запроса;
- выбранного `k`.

Запрос A может иметь много хороших кандидатов:

\[
[3,3,2,2,1]
\]

Запрос B — только один умеренно полезный:

\[
[1,0,0,0,0]
\]

Даже идеальный DCG для B будет намного меньше, чем для A. Сырые значения неудобно сравнивать.

Нужна нормализация.

---

# 14. Ideal Discounted Cumulative Gain — IDCG

IDCG — максимальный возможный DCG для данного набора релевантностей и данного `k`.

Чтобы получить IDCG:

1. взять истинные оценки релевантности;
2. отсортировать по убыванию;
3. посчитать DCG.

Пример.

Фактическая выдача:

\[
[0.2,\ 1.0,\ 0.7]
\]

Идеальный порядок:

\[
[1.0,\ 0.7,\ 0.2]
\]

Тогда:

\[
IDCG@3 = DCG([1.0,\ 0.7,\ 0.2]) \approx 1.542
\]

---

# 15. Normalized Discounted Cumulative Gain — nDCG

\[
nDCG@k = \frac{DCG@k}{IDCG@k}
\]

Для фактической выдачи `[0.2, 1.0, 0.7]`:

\[
DCG@3 = 0.2 + \frac{1.0}{1.585} + \frac{0.7}{2}
\]

\[
DCG@3 \approx 0.2 + 0.631 + 0.35 = 1.181
\]

Тогда:

\[
nDCG@3 = \frac{1.181}{1.542} \approx 0.766
\]

То есть фактическая выдача набрала около 76.6% от максимально возможного качества.

---

# 16. Интерпретация nDCG

Типично:

- `1.0` — идеальный порядок;
- ближе к `0` — плохой порядок;
- `0.8` — 80% от идеального DCG для данного запроса и `k`.

Но нельзя механически переносить пороги между продуктами.

`0.75` может быть:

- отличным для сложного открытого поиска;
- плохим для маленького каталога;
- нормальным baseline;
- недостаточным для production.

Нужно сравнивать:

- с текущей моделью;
- с baseline;
- с историей;
- с human ranking;
- с online-метриками.

---

# 17. Почему nDCG обычно от 0 до 1

IDCG — лучший возможный DCG для того же набора релевантностей.

Следовательно:

\[
DCG \le IDCG
\]

и:

\[
0 \le \frac{DCG}{IDCG} \le 1
\]

Это верно при стандартных предпосылках:

- релевантности неотрицательны;
- DCG и IDCG используют одну формулу;
- одинаковое `k`;
- одинаковый gain;
- IDCG посчитан корректно.

---

# 18. Случай `IDCG = 0`

Если все истинные релевантности нулевые, то:

\[
IDCG=0
\]

И формально:

\[
nDCG=\frac{0}{0}
\]

Возможные правила:

1. вернуть `0.0`;
2. вернуть `1.0`;
3. исключить запрос из среднего;
4. вернуть `NaN`;
5. выбросить исключение.

Единственного универсального решения нет. В учебной задаче поведение задаётся условием или тестами. В production это часть контракта метрики.

---

# 19. Средний nDCG по запросам

Для одного запроса получаем один `nDCG@k`.

Для датасета:

\[
Mean\ nDCG@k = \frac{1}{N}\sum_{q=1}^{N} nDCG_q@k
\]

Это macro-average: каждый запрос имеет одинаковый вес.

Но можно использовать веса:

- число показов;
- выручка;
- сегмент;
- важность запроса;
- сложность;
- страна;
- язык.

Тогда:

\[
Weighted\ nDCG =
\frac{\sum_q w_q \cdot nDCG_q}{\sum_q w_q}
\]

Хорошая практика — смотреть несколько срезов:

- overall;
- head;
- torso;
- tail;
- новые пользователи;
- постоянные;
- разные языки;
- разные категории.

---

# 20. Полный пример с экспоненциальным gain

Пусть top-5:

\[
[3,\ 1,\ 2,\ 0,\ 1]
\]

Используем:

\[
gain=2^{rel}-1
\]

## Шаг 1. Gain

| Позиция | rel | gain |
|---:|---:|---:|
| 1 | 3 | 7 |
| 2 | 1 | 1 |
| 3 | 2 | 3 |
| 4 | 0 | 0 |
| 5 | 1 | 1 |

## Шаг 2. DCG

\[
DCG@5 =
\frac{7}{\log_2(2)} +
\frac{1}{\log_2(3)} +
\frac{3}{\log_2(4)} +
\frac{0}{\log_2(5)} +
\frac{1}{\log_2(6)}
\]

Приближённо:

\[
DCG@5 = 7 + 0.631 + 1.5 + 0 + 0.387 = 9.518
\]

## Шаг 3. Идеальный порядок

\[
[3,\ 2,\ 1,\ 1,\ 0]
\]

## Шаг 4. IDCG

\[
IDCG@5 = 7 + \frac{3}{1.585} + \frac{1}{2} + \frac{1}{2.322}
\]

\[
IDCG@5 \approx 7 + 1.893 + 0.5 + 0.431 = 9.824
\]

## Шаг 5. Нормализация

\[
nDCG@5 = \frac{9.518}{9.824} \approx 0.969
\]

Выдача почти идеальна. Главная ошибка: объект с релевантностью 2 стоит ниже объекта с релевантностью 1.

---

# 21. Почему логарифмический дисконт

Логарифм — разумный компромисс:

- сильно отличает первые позиции;
- затем снижает вклад плавно;
- не делает дальние позиции полностью бесполезными;
- приблизительно соответствует убывающему вниманию пользователя.

Но это не закон природы.

В конкретном продукте можно использовать:

- `1/i`;
- `1/log2(i+1)`;
- эмпирическую вероятность просмотра позиции;
- position bias из click model;
- learned discount;
- бизнес-вес позиции.

Если интерфейс показывает только три карточки без скролла, позиции 4+ можно вообще не учитывать.

---

# 22. Связь nDCG с поведением пользователя

nDCG предполагает:

1. верхние позиции важнее;
2. разница между позициями убывает логарифмически;
3. relevance score отражает ценность;
4. пользователь получает суммарную полезность от нескольких объектов.

Это модель поведения, а не абсолютная истина.

Пользователь может:

- кликнуть только первый результат;
- сравнить несколько карточек;
- скроллить глубоко;
- хотеть один точный ответ;
- хотеть разнообразие;
- доверять знакомому бренду;
- игнорировать рекламу.

Поэтому nDCG — proxy-метрика.

# 23. nDCG и другие метрики

## Precision@k

\[
Precision@k = \frac{\text{релевантных объектов в top-k}}{k}
\]

Подходит при бинарной релевантности. Обычно не различает порядок внутри top-k.

## Recall@k

\[
Recall@k = \frac{\text{найденных релевантных}}{\text{всех релевантных}}
\]

Полезен, когда важно не пропустить кандидата.

## MRR

\[
RR = \frac{1}{rank_{first\ relevant}}
\]

Хорош, когда нужен один правильный ответ.

## MAP

Учитывает позиции всех релевантных объектов, обычно при бинарной релевантности.

## HitRate@k

Проверяет, попал ли хотя бы один релевантный объект в top-k.

## ROC-AUC

Оценивает, насколько положительный объект в среднем выше отрицательного, но не фокусируется на top-k и позиции.

Высокий ROC-AUC не гарантирует хорошую верхушку выдачи.

---

# 24. Когда nDCG подходит

nDCG подходит, когда:

- объектов несколько;
- порядок важен;
- верхние позиции ценнее;
- есть несколько уровней релевантности;
- есть labels;
- нужно сравнивать запросы;
- оценивается top-k;
- полезно несколько результатов.

---

# 25. Когда nDCG может быть плохим выбором

## Один правильный ответ

Лучше MRR, Success@1 или Accuracy@1.

## Важно найти все релевантные объекты

Лучше Recall@k.

## Бизнес-ценность не совпадает с relevance

Нужна utility- или revenue-weighted метрика.

## Важны diversity, novelty, fairness

Обычный nDCG этого не учитывает.

## Клики сильно смещены позицией

Нельзя считать сырые клики истинной релевантностью.

## Пользователь выбирает набор, а не последовательность

Может потребоваться set-based метрика.

---

# 26. Откуда берутся relevance labels

## Экспертная разметка

Плюсы:

- понятная шкала;
- меньше position bias;
- можно оценивать новую модель офлайн.

Минусы:

- дорого;
- медленно;
- субъективно;
- возможны разногласия.

## Клики

Плюсы:

- масштаб;
- реальное поведение;
- собираются автоматически.

Минусы:

- position bias;
- trust bias;
- presentation bias;
- клик не всегда означает удовлетворение;
- отсутствие клика не равно нерелевантности.

## Конверсии

Примеры:

- покупка;
- досмотр;
- сохранение;
- заявка;
- найм;
- матч;
- депозит;
- решение вопроса.

Они ближе к бизнесу, но редкие и запаздывающие.

## Смешанная разметка

Можно объединить экспертизу и поведение, но тогда нужно аккуратно определить gain.

---

# 27. Position bias

Пользователь чаще кликает верхний результат просто потому, что он верхний.

Замкнутый круг:

1. старая модель ставит объект наверх;
2. объект получает больше кликов;
3. клики принимаются за relevance;
4. новая модель снова ставит объект наверх.

Для борьбы используют:

- randomization buckets;
- interleaving;
- propensity weighting;
- click models;
- exploration;
- counterfactual evaluation.

---

# 28. Leakage при оценке

Утечка возникает, если:

- один пользователь есть в train и test;
- почти одинаковые запросы попали в разные части;
- будущие документы попали в обучение;
- label использует событие, недоступное на момент предсказания;
- признаки используют итоговую позицию;
- клики из оцениваемой выдачи используются без коррекции bias.

Для временных систем часто нужен time-based split.

---

# 29. Offline и online качество

Высокий offline nDCG не гарантирует бизнес-рост.

Причины:

- labels не отражают реальную цель;
- dataset shift;
- интерфейс;
- latency;
- diversity;
- novelty;
- trust;
- inventory constraints;
- улучшение лёгких, но ухудшение важных запросов;
- оптимизация proxy вместо результата.

Нормальная цепочка:

1. offline validation;
2. shadow testing;
3. canary;
4. A/B test;
5. guardrails;
6. rollout;
7. monitoring.

# 33. Типичные ошибки новичков

1. Не указывать `k`.

# 34. Проверка понимания

## Почему `[1,0,0]` лучше `[0,0,1]`, хотя CG одинаков?

Релевантный объект стоит выше, поэтому пользователь увидит его раньше.

## Зачем логарифм?

Плавно уменьшать вклад нижних позиций.

## Зачем IDCG?

Определить максимально возможное качество для конкретного запроса.

## Зачем делить на IDCG?

Получить относительную шкалу и сделать запросы более сопоставимыми.

## Почему `nDCG@3` и `nDCG@10` — разные метрики?

Они оценивают разную глубину выдачи.

## Когда MRR лучше?

Когда пользователю нужен один правильный результат.

## Почему высокий offline nDCG не гарантирует рост выручки?

Потому что labels могут быть несовершенным proxy, а реальное поведение зависит от интерфейса, latency, trust, diversity и downstream-качества.

# 36. Как это будет выглядеть в коде позже

Общая структура DCG:

```python
import math


def dcg(relevance, k):
    score = 0.0

    for position, rel in enumerate(relevance[:k], start=1):
        score += rel / math.log2(position + 1)

    return score
```

Общая структура nDCG:

```python
def ndcg(relevance, k):
    actual_dcg = dcg(relevance, k)
    ideal_relevance = sorted(relevance, reverse=True)
    ideal_dcg = dcg(ideal_relevance, k)

    if ideal_dcg == 0:
        return 0.0

    return actual_dcg / ideal_dcg
```

Это не решение конкретной задачи курса, потому что API ещё не дан. Это только иллюстрация логики.

В реальном задании нужно строго соблюдать:

- имя функции;
- типы аргументов;
- возвращаемый тип;
- формулу gain;
- обработку `k`;
- edge cases;
- style checks.

# 37. Production-нюансы

## Vectorization

Для больших датасетов расчёт делают батчами через NumPy, PyTorch, Spark или SQL.

## Ties

Если несколько объектов имеют одинаковую релевантность, идеальных порядков может быть несколько. IDCG обычно не меняется.

## Truncation

Если `k` больше длины списка, обычно используются все доступные элементы.

## Negative labels

Чаще запрещаются или преобразуются, потому что стандартная интерпретация предполагает неотрицательную полезность.

## Missing labels

Нужно определить: считать нулём, исключать, доразмечать или использовать partial judgments.

## Query grouping

Метрика считается внутри каждого query group, затем агрегируется.

## Reproducibility

Фиксируются:

- версия датасета;
- формула;
- `k`;
- gain;
- правило zero-IDCG;
- веса запросов;
- набор срезов.

# 39. Итоговая модель понимания

Вся логика укладывается в пять шагов:

1. **CG** — сколько суммарной релевантности есть в выдаче.
2. **DCG** — насколько высоко расположена эта релевантность.
3. **IDCG** — сколько могла бы набрать идеальная выдача.
4. **nDCG** — какую долю от идеала набрала текущая выдача.
5. **Mean nDCG** — среднее качество по запросам или пользователям.

Главная бизнес-интерпретация:

> nDCG показывает, насколько хорошо система размещает наиболее полезные результаты в верхней части выдачи по сравнению с идеальным порядком.
