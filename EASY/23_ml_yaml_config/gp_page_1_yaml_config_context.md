# ML: YAML CONFIG — бизнесовый и технический контекст

## 1. Суть задачи простыми словами

Команда запускает Text-to-Speech модель для клонирования голоса. У модели есть много настроек:

- пути к весам;
- путь к ASR-конфигу;
- частота дискретизации;
- параметры спектрограммы;
- размерности нейросети;
- коэффициенты функции потерь;
- learning rate;
- количество эпох;
- размер batch;
- устройство `cpu` или `cuda`;
- режим mixed precision.

Человеку удобно хранить их в иерархическом YAML:

```yaml
preprocess_params:
  sr: 24000
  spect_params:
    n_fft: 2048
    win_length: 1200
    hop_length: 300
```

Инфраструктура запуска часто ожидает плоские переменные окружения:

```text
preprocess_params.sr=24000
preprocess_params.spect_params.n_fft=2048
preprocess_params.spect_params.win_length=1200
preprocess_params.spect_params.hop_length=300
```

Наша утилита — двунаправленный переводчик между этими двумя представлениями одной конфигурации.

# 2. Бизнесовый смысл

## 2.1. Почему это не просто упражнение по рекурсии

Конфиг управляет поведением реальной ML-системы. Например:

```yaml
device: cuda
batch_size: 5
fp16_run: true
```

Эти параметры влияют на:

- стоимость GPU;
- скорость обучения;
- вероятность Out Of Memory;
- время эксперимента;
- пропускную способность inference;
- качество модели;
- срок выхода функции в production.

Ошибка в конфиге может привести к бизнесовым последствиям:

- модель обучалась несколько часов с неправильными параметрами;
- сервис загрузил не те веса;
- контейнер не запустился;
- выросла latency;
- пользователи получили деградацию качества;
- команда потеряла вычислительный бюджет;
- эксперимент невозможно воспроизвести.

Поэтому конфигурация — часть контракта production-системы.

## 2.2. Почему нужны и YAML, и ENV

У команды есть разные среды:

- ноутбук разработчика;
- unit-тесты;
- CI/CD;
- Docker;
- Kubernetes;
- staging;
- production;
- облачная ML-платформа.

YAML удобен человеку: структура видна глазами.

ENV удобен инфраструктуре: отдельный параметр можно изменить без правки кода и пересборки контейнера.

Польза утилиты:

1. Хранить настройки в читаемом виде.
2. Переопределять отдельные параметры при деплое.
3. Переносить конфиги между средами.
4. Восстанавливать YAML для повторного запуска.
5. Снижать количество ручных ошибок.
6. Разделять код и настройки.

# 3. YAML как дерево

После:

```python
config = yaml.safe_load(config_file)
```

YAML:

```yaml
loss_params:
  g_loss:
    lambda_sty: 1.0
    lambda_cyc: 5.0
```

становится словарём:

```python
{
    "loss_params": {
        "g_loss": {
            "lambda_sty": 1.0,
            "lambda_cyc": 5.0,
        }
    }
}
```

Это дерево:

```text
loss_params
└── g_loss
    ├── lambda_sty → 1.0
    └── lambda_cyc → 5.0
```

Внутренние узлы:

```text
loss_params
g_loss
```

Листья:

```text
lambda_sty → 1.0
lambda_cyc → 5.0
```

Переменные окружения создаются для листьев:

```text
loss_params.g_loss.lambda_sty=1.0
loss_params.g_loss.lambda_cyc=5.0
```

Полный путь от корня до листа становится ключом.

# 4. Что такое рекурсия

Рекурсивная функция вызывает саму себя на меньшей части исходной задачи.

Бытовая аналогия: поиск файла в папках.

1. Смотрим содержимое текущей папки.
2. Если видим файл — обрабатываем его.
3. Если видим вложенную папку — применяем к ней тот же алгоритм.
4. Останавливаемся, когда вложенных папок больше нет.

Для YAML:

1. Берём текущий словарь.
2. Проходим по его ключам.
3. Если значение — словарь, спускаемся внутрь.
4. Если значение — не словарь, нашли лист.
5. Формируем `полный.путь=значение`.

## 4.1. Базовый случай

Базовый случай останавливает рекурсию.

```python
"sr": 24000
```

Значение `24000` не является словарём. Дальше спускаться некуда. Формируем:

```text
preprocess_params.sr=24000
```

## 4.2. Рекурсивный случай

```python
"spect_params": {
    "n_fft": 2048,
    "win_length": 1200,
}
```

Значение — словарь. Вызываем функцию снова и передаём накопленный путь:

```text
preprocess_params.spect_params
```

## 4.3. Главный вопрос при рекурсии

> Как задача становится меньше на каждом вызове и когда функция остановится?

В нашей задаче новый вызов получает более маленький вложенный словарь. На листе новый вызов не выполняется.

# 5. Пошаговый проход YAML → ENV

Исходный YAML:

```yaml
preprocess_params:
  sr: 24000
  spect_params:
    n_fft: 2048
```

После `safe_load`:

```python
{
    "preprocess_params": {
        "sr": 24000,
        "spect_params": {
            "n_fft": 2048,
        },
    }
}
```

Первый вызов:

```python
flatten(config, "")
```

Встречаем `preprocess_params`. Значение — словарь. Новый вызов:

```python
flatten(config["preprocess_params"], "preprocess_params")
```

Внутри встречаем `sr=24000`. Это лист:

```text
preprocess_params.sr=24000
```

Затем встречаем `spect_params`. Это словарь. Новый путь:

```text
preprocess_params.spect_params
```

Внутри лист `n_fft=2048`:

```text
preprocess_params.spect_params.n_fft=2048
```

Итог:

```text
preprocess_params.sr=24000
preprocess_params.spect_params.n_fft=2048
```

# 6. Обратная задача ENV → YAML

Вход:

```text
preprocess_params.spect_params.n_fft=2048
```

Ключ делится по точке:

```python
[
    "preprocess_params",
    "spect_params",
    "n_fft",
]
```

Нужно построить:

```python
{
    "preprocess_params": {
        "spect_params": {
            "n_fft": 2048,
        }
    }
}
```

Движемся слева направо:

1. Создаём `preprocess_params: {}`.
2. Внутри создаём `spect_params: {}`.
3. Последнему ключу `n_fft` присваиваем `2048`.

Метод:

```python
current_level.setdefault(path_part, {})
```

означает:

- если ключ есть — вернуть существующий словарь;
- если ключа нет — создать пустой словарь и вернуть его.

# 7. Почему критично сохранять типы

> Важно для этого задания: при преобразовании YAML → ENV автотест ожидает Python-представление логических значений — `True` и `False` с заглавной буквы. В результирующем YAML PyYAML может снова вывести `true` и `false` в нижнем регистре — это корректный YAML.

ENV физически является текстом:

```text
epochs=150
fp16_run=True
optimizer_params.lr=0.0001
```

Но YAML должен содержать разные типы:

```yaml
epochs: 150
fp16_run: true
optimizer_params:
  lr: 0.0001
```

После загрузки:

```python
{
    "epochs": 150,          # int
    "fp16_run": True,       # bool
    "optimizer_params": {
        "lr": 0.0001,       # float
    },
}
```

Если всё оставить строками:

```yaml
epochs: "150"
fp16_run: "true"
```

ML-код может сломаться:

```python
for epoch in range(config["epochs"]):
    ...
```

`range(150)` работает, а `range("150")` — нет.

Для восстановления типов используется:

```python
yaml.safe_load(value_text)
```

Примеры:

```text
150     → int 150
0.0001  → float 0.0001
True    → bool True
cuda    → str "cuda"
```

# 8. Критический edge case: пустая строка

В полном конфиге:

```yaml
pretrained_model: ""
```

После flattening:

```text
pretrained_model=
```

Но:

```python
yaml.safe_load("")
```

возвращает:

```python
None
```

Пустая строка `""` и `None` — разные значения.

Поэтому в решении:

```python
if value_text == "":
    value = ""
else:
    value = yaml.safe_load(value_text)
```

Без этой проверки полный пример не восстановится корректно.

# 9. `safe_load` и `safe_dump`

## `safe_load`

Преобразует YAML в стандартные безопасные Python-типы:

- словарь;
- список;
- строка;
- число;
- `bool`;
- `None`.

Для внешнего конфигурационного текста предпочтительнее использовать безопасный парсер.

## `safe_dump`

Преобразует Python-словарь обратно в YAML:

```python
yaml.safe_dump(
    config,
    sort_keys=False,
    allow_unicode=True,
)
```

`sort_keys=False` не заставляет сортировать ключи.

`allow_unicode=True` нормально выводит кириллицу.

# 10. Применение в индустриях

## 10.1. Финтех

```yaml

  threshold: 0.73
  version: risk_v15
service:
  timeout_ms: 80
```

Бизнесовые последствия ошибки:

- меняется approval rate;
- растёт default rate;
- увеличивается expected loss;
- нарушается SLA ответа;
- используется неправильная версия модели.

## 10.2. Marketplace и e-commerce

```yaml
ranking:
  top_k: 50
  diversity_weight: 0.2

  candidate_limit: 1000
```

Связанные метрики:

- CTR;
- conversion rate;
- GMV;
- average order value;

- latency.

## 10.3. Two-sided marketplace

```yaml
matching:
  max_distance_km: 15
```

Нужно балансировать интересы двух сторон: покупателей и продавцов/исполнителей.

## 10.4. iGaming и casino


Особенно важны аудит, контроль доступа, безопасный rollout и соблюдение ограничений.

Критичны точность типов, audit trail, secret management и возможность отката.

## 10.7. Anthropic, Google, Meta и другой Big Tech

Конфиги управляют:

- model routing;

Главная идея:

# 11. Учебное решение против production

Учебное решение делает только преобразование. В production потребуются дополнительные слои.

## 11.1. Схема и валидация

Проверять:

- обязательные ключи;
- допустимые типы;
- диапазоны;
- неизвестные параметры;
- несовместимые комбинации.

Инструменты: Pydantic, JSON Schema, Marshmallow.

## 11.2. Секреты

Пароли и API-ключи нельзя хранить в обычном YAML в репозитории. Нужны Vault, AWS Secrets Manager, Google Secret Manager, Azure Key Vault или Kubernetes Secrets.

## 11.3. Коллизии структуры

```text
a=1
a.b=2
```

Первая строка утверждает, что `a` — число. Вторая — что `a` является словарём. Production-версия должна отклонять такой ввод.

## 11.4. Точки в настоящем ключе

```yaml
"model.version": v2
```

Запись `model.version=v2` неоднозначна. Нужен escaping или другой разделитель.

## 11.5. Списки

Нужно заранее определить контракт:

- хранить список как одно значение;
- или разворачивать `devices.0`, `devices.1`.

## 11.6. Комментарии

PyYAML не сохраняет комментарии после `safe_load`/`safe_dump`. Для точного round-trip с комментариями можно использовать `ruamel.yaml`.

## 11.7. Наблюдаемость

Полезно измерять:

- число успешных преобразований;
- ошибки парсинга;
- глубину дерева;
- число ключей;
- конфликты;
- неизвестные параметры.

## 11.8. Версионирование

Конфиг стоит связывать с:

- версией кода;
- версией модели;
- версией данных;
- experiment ID;
- временем запуска;
- автором изменения.

# 12. Правильное инженерное мышление

1. Зафиксировать вход и выход функций.
2. Представить YAML как дерево.
3. Определить базовый и рекурсивный случаи.
4. Для прямого преобразования накапливать путь.
5. Для обратного преобразования строить словари по частям пути.
6. Восстанавливать типы.
7. Проверить пустую строку.
8. Проверить глубокую вложенность.
9. Сравнивать разобранные структуры, а не форматирование YAML.
10. Не менять обязательный API.

Это и есть подход DS/MLE: сначала контракт и риски, затем минимальный корректный код.

# 16. Что мы узнали из реального автотеста

Первая версия решения получила 60/100:

```text
env_to_yaml — 40/40
pylint      — 20/20
yaml_to_env — 0/40
```

Сравнение `Expected` и `Got` показало, что рекурсия, пути, числа, строки и глубина вложенности уже были правильными. Отличались только логические значения:

```text
Expected: use_gpu=False
Got:      use_gpu=false
```

Поэтому правильное исправление — не переписывать алгоритм, а удалить `.lower()`:

```python
env_value = str(value)
```

Теперь:

```python
str(True)   # "True"
str(False)  # "False"
```

Это важный инженерный урок: сначала локализуем минимальное расхождение, а затем меняем только неисправный участок, сохраняя уже прошедшие тесты.
