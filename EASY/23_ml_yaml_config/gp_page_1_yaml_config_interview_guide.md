> Мы строим двунаправленный адаптер между человекочитаемым иерархическим YAML-конфигом и плоским представлением переменных окружения, сохраняя структуру и основные типы данных.

# 3. Продуктовые вопросы

## Зачем бизнесу такая утилита?

> Она отделяет настройки от кода. Команда может менять параметры модели, сервиса и инфраструктуры без пересборки приложения. Это ускоряет эксперименты и деплой, улучшает воспроизводимость и снижает риск ручных ошибок.

## Какая продуктовая метрика может пострадать?

Для TTS:

- latency;
- error rate;
- доля успешной генерации;
- пользовательская оценка качества;
- стоимость inference;
- GPU utilization.

Для рекомендаций:

- CTR;
- conversion;
- GMV;
- retention;
- latency.

Для скоринга:

- approval rate;
- default rate;
- expected loss;
- decision latency.

## Почему не зашить параметры в код?

> Любая смена настройки потребует правки кода, review, сборки и деплоя. Смешиваются бизнесовые настройки и логика программы. Сложнее запускать разные среды и воспроизводить эксперименты.

## Почему не использовать только YAML?

> YAML удобен в разработке, но CI/CD, Docker и Kubernetes часто переопределяют отдельные значения через environment variables. Поэтому полезно поддерживать оба представления.

## Главная продуктовая ценность

> Быстрое и безопасное изменение поведения системы при сохранении контроля и воспроизводимости.

# 10. Edge cases

## Пустая строка

```text
model_path=
```

Не должна превращаться в `null`.

## Значение с `=`

```text
token=abc=def
```

Разделяем только по первому `=`.

## Конфликт структуры

```text
a=1
a.b=2
```

Следует отклонить как противоречивый ввод.

## Точка внутри настоящего ключа

```yaml
"model.version": v2
```

Нужен escaping или другой delimiter.

## Очень глубокая структура

Рекурсия может достичь лимита Python. Для экстремальной глубины можно использовать собственный стек.

## Дубликаты

```text
a.b=1
a.b=2
```

Нужно определить политику:

- последнее значение побеждает;
- первое побеждает;
- ошибка.

## Строка, похожая на число

```text
code=00123
```

Нужно определить, это строка или число. Автоматический вывод типа может изменить смысл.

## Комментарии

PyYAML теряет комментарии. Для их сохранения можно использовать `ruamel.yaml`.

# 11. Как рассказать алгоритм

## YAML → ENV

> Использую DFS. Для каждого ключа строю полный путь. Если значение — словарь, рекурсивно продолжаю обход. Если leaf — сериализую значение и добавляю `path=value`.

Псевдокод:

```text
flatten(node, prefix):
    for key, value in node:
        path = join(prefix, key)

        if value is dictionary:
            flatten(value, path)
        else:
            result.append(path + "=" + value)
```

## ENV → YAML

> Каждую строку делю по первому `=`, путь — по точкам. Все части пути, кроме последней, создаю как словари. Последнему ключу присваиваю распарсенное значение.

Псевдокод:

```text
for line in env:
    key, value = split_once(line, "=")
    path = split(key, ".")
    current = root

    for part in path except last:
        current = get_or_create_dictionary(current, part)

    current[last] = parse_type(value)
```

# 12. Сложность

YAML → ENV:

```text
O(N)
```

ENV → YAML:

```text
O(L × D)
```

где `L` — число строк, `D` — средняя глубина.

Память:

```text
O(N)
```

# 13. Вопросы интервьюера

## Почему рекурсия?

> Глубина заранее неизвестна, а каждый вложенный словарь имеет ту же форму, что и корневой.

## Можно без рекурсии?

> Да. Можно использовать явный стек пар `(node, prefix)`. Это полезно при экстремальной глубине.

## Почему `safe_load`?

> Он создаёт стандартные безопасные Python-типы и не предназначен для произвольных пользовательских объектов.

## Как сохранить комментарии?

> PyYAML их теряет. Для round-trip preservation можно использовать `ruamel.yaml`.

## Как хранить секреты?

> Не в обычном YAML-репозитории. Использовать secret manager, а в логах редактировать значения.

## Как rollout-ить новую версию?

> Dry-run или shadow, затем canary на малой доле, проверка ошибок и продуктовых метрик, постепенное расширение, быстрый rollback.

## Что мониторить?

> Ошибки парсинга, schema errors, неизвестные ключи, latency, hash применённого конфига, версию схемы и rollback count.

# 14. STAR-ответ

## Situation

> Команда запускала ML-сервис в нескольких окружениях. Настройки хранились в YAML, а deployment-платформа переопределяла параметры через ENV.

## Task

> Нужно было обеспечить двунаправленное преобразование без потери структуры и основных типов.

## Action

> Я описал контракт, представил YAML как дерево, реализовал рекурсивный DFS для flattening и итеративное восстановление словаря по пути. Для типов использовал safe YAML parsing, отдельно обработал пустые строки и разделение по первому `=`. Добавил round-trip тесты.

## Result

> Команда получила единый механизм конфигурации для локальной разработки и деплоя, уменьшила ручные ошибки и сохранила воспроизводимость запусков.
