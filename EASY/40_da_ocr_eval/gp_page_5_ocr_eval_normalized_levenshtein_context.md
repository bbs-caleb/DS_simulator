# Страница 5. OCR Eval: Normalized Levenshtein Distance

## 1. Зачем понадобилась нормализация

На предыдущем шаге мы научились считать расстояние Левенштейна — минимальное
число вставок, удалений и замен, необходимых для преобразования эталонного
текста в результат OCR.

Это уже намного лучше прямого сравнения по индексам, потому что одна вставка
или одно удаление не разрушает сопоставление всех последующих символов.

Но абсолютное расстояние создаёт новую проблему.

Представим два документа:

```text
Документ A:
длина = 20 символов
расстояние Левенштейна = 2

Документ B:
длина = 200 символов
расстояние Левенштейна = 2
```

Абсолютная метрика говорит, что документы одинаково плохи.

Но доля испорченного текста различается:

```text
Документ A: 2 / 20 = 10%
Документ B: 2 / 200 = 1%
```

Для пользователя и бизнеса первый документ распознан существенно хуже.

Поэтому расстояние нужно привести к единому масштабу.

---

## 2. Цель нормализации

Нормализованная метрика должна:

- находиться примерно в диапазоне от `0` до `1`;
- учитывать длину сравниваемых строк;
- позволять сопоставлять короткие и длинные документы;
- сохранять смысл «чем больше score, тем хуже OCR»;
- учитывать пользовательские стоимости операций.

Интерпретация:

```text
0.0 — строки идентичны;
значение ближе к 0 — небольшая доля ошибок;
значение ближе к 1 — очень большая доля различий.
```

---

## 3. Почему нормализуем на максимальную длину

В условии сказано учитывать максимальную длину пары текстов:

```python
max(len(text), len(ocr_text))
```

Почему не минимальную длину?

Рассмотрим:

```text
text = "a"
ocr_text = "abcdefghij"
```

Если делить на минимальную длину `1`, результат легко станет намного больше
единицы.

Максимальная длина лучше отражает размер более длинной последовательности и
позволяет построить естественную верхнюю границу.

---

## 4. Почему нужно учитывать максимальную стоимость операции

Если все стоимости равны единице, простая нормализация может выглядеть так:

```python
distance / max(len(text), len(ocr_text))
```

Но в этой задаче веса могут быть другими:

```python
insert_cost = 0.5
delete_cost = 1.5
substitute_cost = 2
```

Тогда абсолютное расстояние измеряется уже не в количестве операций, а в
суммарной стоимости.

Делить такую сумму только на длину некорректно: верхняя граница изменилась.

Поэтому используется:

```python
max_operation_cost = max(
    insert_cost,
    delete_cost,
    substitute_cost,
)
```

А максимальная нормализующая величина:

```python
max_possible_distance = (
    max(len(text), len(ocr_text))
    * max_operation_cost
)
```

Итоговая формула:

\[
NormalizedDistance =
\frac{WeightedLevenshteinDistance}
{\max(|text|, |ocr\_text|) \cdot
 \max(c_{insert}, c_{delete}, c_{substitute})}
\]

---

## 5. Что означают веса операций

### `insert_cost`

Стоимость вставки символа в эталонную строку.

Пример:

```text
cat → cart
```

Нужно вставить `r`.

### `delete_cost`

Стоимость удаления символа из эталонной строки.

Пример:

```text
house → hose
```

Нужно удалить `u`.

### `substitute_cost`

Стоимость замены одного символа другим.

Пример:

```text
cat → cut
```

Нужно заменить `a` на `u`.

---

## 6. Как веса меняют поведение алгоритма

Алгоритм Вагнера–Фишера всегда выбирает минимальную стоимость.

Пусть:

```python
insert_cost = 1
delete_cost = 1
substitute_cost = 3
```

Замена стоит `3`, но удалить символ и вставить новый стоит:

```text
1 + 1 = 2
```

Поэтому алгоритм может отказаться от прямой замены и выбрать комбинацию
удаления и вставки.

Это важный момент:

> Вес операции не гарантирует, что именно эта операция будет использована.

Алгоритм ищет самый дешёвый путь среди всех допустимых преобразований.

---

## 7. Почему веса ограничены диапазоном [0, 2]

Шаблон требует:

```text
0 <= cost <= 2
```

Это ограничение:

- не позволяет передать отрицательную стоимость;
- защищает от бессмысленных параметров;
- удерживает шкалу в контролируемом диапазоне;
- упрощает тестирование;
- снижает риск чрезмерно больших значений.

Нулевая стоимость разрешена.

Например:

```python
substitute_cost = 0
```

означает, что замены не штрафуются.

Это необычно для production-системы, но допустимо по контракту задачи.

---

## 8. Как инициализируется взвешенная матрица

В обычном расстоянии Левенштейна первый столбец заполнялся:

```text
0, 1, 2, 3, ...
```

Теперь удаление может стоить не `1`.

Поэтому:

```python
distances[row][0] = row * self.delete_cost
```

А первая строка:

```python
distances[0][column] = column * self.insert_cost
```

Например, если вставка стоит `0.5`, то:

```text
0, 0.5, 1.0, 1.5, ...
```

---

## 9. Взвешенная рекуррентная формула

Для каждой ячейки считаются три варианта.

### Вставка

```python
distances[row][column - 1] + self.insert_cost
```

### Удаление

```python
distances[row - 1][column] + self.delete_cost
```

### Замена или совпадение

```python
distances[row - 1][column - 1] + replacement_penalty
```

где:

```python
replacement_penalty = 0
```

если символы совпадают, и:

```python
replacement_penalty = self.substitute_cost
```

если различаются.

---

## 10. Почему score остаётся в диапазоне 0–1

Для каждой позиции максимальная единичная стоимость ограничена:

```python
max_operation_cost
```

А число необходимых «шагов» для прямого преобразования не превышает длину
более длинной строки, если использовать замены для общей части и вставки или
удаления для остатка.

Поэтому нормирующий знаменатель:

```python
max_length * max_cost
```

задаёт безопасную верхнюю границу.

---

## 11. Особый случай: нулевой знаменатель

Знаменатель равен нулю в двух типичных ситуациях.

### Обе строки пустые

```python
text = ""
ocr_text = ""
```

Тогда максимальная длина равна `0`.

### Все стоимости равны нулю

```python
insert_cost = 0
delete_cost = 0
substitute_cost = 0
```

Тогда максимальная стоимость равна `0`.

Делить на ноль нельзя.

Поэтому решение возвращает:

```python
0.0
```

Это логично: при нулевой длине или нулевой стоимости всех преобразований
штраф отсутствует.

---

## 12. Бизнесовый смысл взвешивания

Одинаковая редакторская операция не всегда имеет одинаковый риск.

### EdTech

Замена:

```text
O → 0
```

может повредить число или формулу.

Замена:

```text
е → e
```

может быть менее критичной визуально, хотя технически символы разные.

### Финтех

Ошибка в одной цифре номера счёта критичнее нескольких опечаток в адресе.

### Маркетплейсы

```text
500 мл → 5000 мл
```

меняет товарную характеристику.

### Медицина

```text
1.0 мг → 10 мг
```

может быть опасной ошибкой.

### Crypto

Один неверный символ адреса кошелька меняет адрес полностью.

В production обычно используют не только глобальные веса операции, но и
веса конкретных пар символов, типов полей и зон документа.

---

## 13. Почему глобального substitute_cost недостаточно

В текущей задаче все замены имеют одну стоимость:

```python
substitute_cost = 1.2
```

Но реальная OCR-система часто путает конкретные пары:

```text
O ↔ 0
l ↔ 1
S ↔ 5
B ↔ 8
rn ↔ m
```

Более развитая система может использовать confusion matrix:

```text
стоимость O → 0 = 2.0
стоимость е → e = 0.5
стоимость а → a = 0.5
```

Это уже называется контекстным или матричным взвешиванием замен.

В текущем задании этого делать нельзя: нужно строго реализовать заданный API.

## 14. Применение в разных индустриях

### EdTech

Метрика помогает:

- ранжировать плохо распознанные задания;
- находить сложные шрифты;
- сравнивать OCR-модели;
- выявлять проблемы с формулами;
- отбирать примеры для ручной разметки.

### Финтех

Применения:

- паспорта;
- договоры;
- выписки;
- платёжные реквизиты;
- KYC;
- чеки.

Нормализованная метрика удобна для сравнения документов разной длины, но
критические поля всё равно оцениваются отдельно.

### E-commerce

Применения:

- карточки товаров;
- упаковка;
- цены;
- объёмы;
- артикулы;
- бренды.

### Two-sided marketplaces

Можно проверять:

- сертификаты исполнителей;
- лицензии;
- документы;
- адреса;
- чеки.

### iGaming и casino

OCR участвует в KYC, проверке возраста и платёжных документов.

### Crypto

OCR может читать документы и скриншоты транзакций, но адреса кошельков требуют
отдельной строгой проверки.

### Dating

OCR используется в age verification, document verification и модерации текста
на изображениях.

### Big Tech

В Google, Meta, Anthropic и других компаниях OCR-метрики могут быть частью
мультимодальных систем, поиска, accessibility и модерации.

### SpaceX и BP

В технической документации одна ошибка в числе, давлении, коде детали или
единице измерения может иметь высокий риск. Нужна сегментация ошибок по
критичности.

## 15. Как мыслить на продуктовом интервью

Хороший ответ строится не вокруг одной формулы, а вокруг системы оценки.

### Шаг 1. Определить пользовательский сценарий

Что человек загружает?

- фотографию задания;
- паспорт;
- чек;
- техническую схему;
- скриншот;
- рукописный текст.

### Шаг 2. Определить критические символы

- числа;
- отрицания;
- валюты;
- единицы измерения;
- даты;
- идентификаторы;
- математические знаки.

### Шаг 3. Выбрать offline-метрики

- normalized Levenshtein distance;
- Character Error Rate;
- Word Error Rate;
- exact match;
- field accuracy;
- weighted critical-token error rate.

### Шаг 4. Сегментировать качество

- язык;
- фон;
- размер шрифта;
- освещение;
- угол;
- устройство;
- тип документа;
- печатный или рукописный текст.

### Шаг 5. Связать с продуктом

- доля успешных ответов;
- повторная загрузка;
- ручное исправление;
- время до результата;
- отказ пользователя;
- жалобы;

### Шаг 6. Настроить fallback

- попросить переснять фото;

- дать пользователю исправить текст;
- отправить на более дорогую модель;
- вызвать ручную проверку.

## 16. Итог

`NormLevEval` решает две проблемы обычного расстояния Левенштейна:

1. делает score сопоставимым для текстов разной длины;
2. позволяет управлять стоимостью вставки, удаления и замены.

Формула нормализации:

```python
weighted_distance / (
    max(len(text), len(ocr_text))
    * max(insert_cost, delete_cost, substitute_cost)
)
```

Это хороший учебный baseline для OCR Evaluation, хотя production-система
обычно дополняет его полевыми, семантическими и бизнес-метриками.
