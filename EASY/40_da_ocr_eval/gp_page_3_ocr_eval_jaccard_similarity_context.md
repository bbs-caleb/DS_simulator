# Страница 3. OCR Eval: Jaccard Similarity

## 1. Что происходит на этой странице

На предыдущем шаге мы реализовали самое простое посимвольное сравнение двух строк. Для каждой позиции мы проверяли, совпадает ли символ эталонного текста с символом OCR-текста, а затем добавляли разницу длин.

Такой baseline полезен, потому что он:

- очень простой;
- быстро работает;
- легко объясняется;
- позволяет проверить весь pipeline загрузки и сортировки данных.

Но у него есть фундаментальная проблема: одна вставка или одно удаление символа сдвигает индексы всех следующих символов.

Пример:

```text
Эталон: abcdef
OCR:    abdef
```

Фактически OCR пропустил только символ `c`. Но прямое сравнение по индексам после этой позиции начинает сопоставлять:

```text
c с d
d с e
e с f
```

В результате одна ошибка выглядит как несколько ошибок.

На текущем шаге предлагается другой baseline: не сравнивать символы по позициям, а:

1. разбить каждый текст на слова;
2. превратить слова в множества;
3. сравнить множества через коэффициент Жаккара;
4. превратить меру сходства в меру различия.

---

# 2. Бизнесовый смысл задачи

## 2.1. OCR — это начало продуктовой цепочки

В Smart Tutor OCR не является конечным продуктом. Он является входным компонентом следующего процесса:

```text
изображение задания
        ↓
OCR
        ↓
распознанный текст
        ↓
понимание запроса
        ↓
построение решения
        ↓
ответ школьнику
```

Если OCR ошибается, ошибка попадает в контекст диалогового помощника. Даже сильная языковая модель может уверенно решить не ту задачу, если получила искажённые входные данные.

Например:

```text
5x + 3 = 18
```

может быть распознано как:

```text
Sx + 3 = 18
```

Или:

```text
ускорение свободного падения 9.8 м/с²
```

может превратиться в:

```text
ускорение свободного падения 98 м/с²
```

Второй пример особенно важен: технически потерян только один символ — точка, но смысл числового значения изменился в десять раз.

Поэтому цель оценки OCR не должна сводиться к абстрактному вопросу «насколько тексты похожи». Практическая цель — понять:

- где OCR ошибается чаще всего;
- какие ошибки наиболее опасны для конечного ответа;
- какие сегменты изображений проблемны;
- стала ли новая версия лучше старой;
- можно ли выпускать новую модель;
- где нужен fallback или повторное распознавание;
- какие примеры необходимо добавить в датасет дообучения.

## 2.2. Почему нужно ранжирование документов

Допустим, у команды 100 000 пар:

```text
эталонный текст ↔ OCR-текст
```

Человек не может вручную просмотреть их все. Нужна метрика, которая присвоит каждой паре `score`, после чего документы можно отсортировать.

В этой задаче принято соглашение:

```text
чем больше score, тем сильнее ошибка
```

Поэтому после вычисления метрики результаты сортируются по убыванию.

Команда качества сначала смотрит верх списка — самые подозрительные OCR-примеры.

# 3. Почему векторный поиск не является основным решением

Эмбеддинги и семантические модели отвечают на вопрос:

> Насколько похож смысл двух текстов?

Это полезно в поиске, рекомендациях, intent classification и RAG. Но OCR Evaluation задаёт другой вопрос:

> Насколько точно система воспроизвела исходную последовательность символов или слов?

Например, фразы:

```text
Рассчитайте площадь круга.
Найдите площадь окружности.
```

могут быть семантически близки. Но если второй текст появился как OCR-результат первого, это уже не точное распознавание.

Особенно опасны пары, которые семантически похожи, но отличаются критическим токеном:

```text
Доза: 1.0 мг
Доза: 10 мг
```

```text
Ответьте «да».
Ответьте «нет».
```

```text
x < 5
x > 5
```

Семантическая близость может остаться высокой, хотя бизнес-риск огромен.

Поэтому на этой странице применяется детерминированная строковая метрика, а не семантическая модель.

# 4. Коэффициент Жаккара

Коэффициент Жаккара измеряет сходство двух множеств.

Правильная формула:

\[
J(A, B) = \frac{|A \cap B|}{|A \cup B|}
\]

где:

- `A` — множество слов эталонного текста;
- `B` — множество слов OCR-текста;
- `A ∩ B` — пересечение множеств;
- `A ∪ B` — объединение множеств;
- `|...|` — количество элементов.

Важно: числитель — размер пересечения, знаменатель — размер объединения.

## 4.1. Интерпретация

Если множества полностью совпадают:

```text
A = {apple, orange}
B = {apple, orange}
```

то:

```text
|A ∩ B| = 2
|A ∪ B| = 2
J = 2 / 2 = 1
```

Если общих слов нет:

```text
A = {apple, orange}
B = {pear, banana}
```

то:

```text
|A ∩ B| = 0
|A ∪ B| = 4
J = 0 / 4 = 0
```

Следовательно:

- `1` — максимальное сходство;
- `0` — отсутствие общих элементов.

# 5. Подробный пример

Пусть:

```python
text = "apple orange"
ocr_text = "apple banana"
```

После разбиения по пробелу:

```python
text.split(" ")
# ["apple", "orange"]

ocr_text.split(" ")
# ["apple", "banana"]
```

После преобразования к множествам:

```text
A = {apple, orange}
B = {apple, banana}
```

Пересечение:

```text
A ∩ B = {apple}
```

Размер пересечения:

```text
1
```

Объединение:

```text
A ∪ B = {apple, orange, banana}
```

Размер объединения:

```text
3
```

Jaccard Similarity:

\[
J(A, B) = \frac{1}{3} \approx 0.33
\]

Но это сходство. В задаче нужен `score` ошибки, который растёт при ухудшении качества.

Поэтому применяется Jaccard distance:

\[
D(A, B) = 1 - J(A, B)
\]

Для примера:

\[
D(A, B) = 1 - \frac{1}{3}
= \frac{2}{3}
\approx 0.67
\]

---

# 6. Почему используются множества

Множество `set` хранит только уникальные элементы.

```python
set(["home", "work", "home"])
```

Результат:

```python
{"home", "work"}
```

Множества удобны, потому что Python уже умеет считать:

```python
first_set.intersection(second_set)
first_set.union(second_set)
```

То есть не нужно вручную писать вложенные циклы и проверять каждое слово.

Но использование множества одновременно создаёт главные ограничения метрики: теряются порядок и частота слов.

---

# 7. Почему JaccardEval наследуется от BaseEval

В `BaseEval` уже реализована общая инфраструктура:

- проверка существования файла;
- чтение JSON;
- обработка повреждённого JSON;
- проверка корневого списка;
- проверка структуры каждого документа;
- создание пар `(text, ocr_text)`;
- расчёт score по всем документам;
- стабильная сортировка;
- применение `limit`.

Для Jaccard-метрики всё это остаётся прежним. Меняется только одна операция:

```text
как посчитать score для одной пары строк
```

Поэтому создаётся дочерний класс:

```python
class JaccardEval(BaseEval):
```

и переопределяется только:

```python
def _eval_func(...):
```

Это хороший инженерный дизайн: общая логика не дублируется.

Такой подход близок к паттерну Template Method:

1. базовый класс задаёт общий процесс;
2. дочерний класс подставляет конкретную стратегию оценки.

---

# 8. Как Python выбирает нужный _eval_func

Когда выполняется:

```python
evaluation = JaccardEval("data.json")
result = evaluation.evaluate(limit=3)
```

метод `evaluate` берётся из `BaseEval`, потому что `JaccardEval` его не переопределяет.

Внутри `evaluate` есть:

```python
self._eval_func(text, ocr_text)
```

Но `self` является объектом `JaccardEval`, поэтому Python вызывает:

```python
JaccardEval._eval_func
```

а не базовую посимвольную версию.

Это полиморфизм: один и тот же общий код может использовать разные реализации метрики.

---

# 9. Сильные стороны Jaccard-подхода

## 9.1. Меньшая чувствительность к сдвигу индексов

Одна пропущенная буква внутри слова больше не заставляет сравнивать все последующие символы по неправильным позициям.

Повреждённое слово в общем случае становится одним новым элементом множества, а не цепочкой несовпадений до конца строки.

## 9.2. Нормированный диапазон

Jaccard distance находится в диапазоне от `0` до `1`:

- `0` — множества слов совпадают;
- `1` — общих слов нет.

Это удобнее абсолютного количества ошибок, особенно при первичном сравнении документов разной длины.

## 9.3. Понятная интерпретация

Метрику легко объяснить менеджеру или аналитику:

> Мы сравниваем долю общих уникальных слов относительно всех уникальных слов двух текстов.

## 9.4. Быстрое вычисление

Создание множеств и операции пересечения/объединения обычно работают примерно линейно по числу слов.

# 10. Ограничения Jaccard Similarity

## 10.1. Не учитывается порядок слов

```text
home work
work home
```

После преобразования к множествам оба текста равны:

```text
{home, work}
```

Следовательно:

```text
Jaccard Similarity = 1
Jaccard Distance = 0
```

Но порядок может менять смысл:

```text
собака укусила человека
человек укусил собаку
```

Набор слов одинаков, сообщение — разное.

## 10.2. Не учитывается частота слов

```text
home work home
work home
```

Оба текста превращаются в:

```text
{home, work}
```

Повторное слово исчезает, поэтому score равен `0`.

## 10.3. Похожесть слов не учитывается

Для множества:

```text
растение
растенuе
```

— два полностью разных элемента, хотя они отличаются одной буквой.

## 10.4. Пунктуация остаётся частью токена

При разбиении только по пробелу:

```text
слово
слово,
```

будут разными элементами.

В учебном задании это правильно, потому что требуется использовать пробел как разделитель. В production-системе можно было бы отдельно нормализовать пунктуацию, но это уже другая метрика.

## 10.5. Регистр учитывается

```text
Москва
москва
```

будут разными словами.

Нельзя самовольно применять `.lower()`, потому что условие этого не требует, а скрытые тесты могут ожидать точное поведение.

## 10.6. Короткие тексты получают более резкие оценки

Если в тексте из пяти слов повреждено одно слово, относительная доля изменения большая.

Если в тексте из ста слов повреждено одно слово, доля изменения мала.

Поэтому Jaccard distance склонен ставить короткие тексты выше длинных при одинаковом абсолютном количестве ошибок.

Это не ошибка кода, а свойство нормированной метрики.

---

# 11. Почему нельзя добавлять улучшения в учебное решение

В реальном проекте можно было бы:

- привести текст к нижнему регистру;
- удалить пунктуацию;
- нормализовать Unicode;
- схлопнуть повторные пробелы;
- использовать символьные n-граммы;
- использовать мультимножества;
- отдельно взвешивать числа и формулы;
- считать Levenshtein distance;
- считать Character Error Rate;
- считать Word Error Rate.

Но автотест проверяет конкретный контракт:

1. класс `JaccardEval` наследуется от `BaseEval`;
2. `_eval_func` переопределён;
3. строки разбиваются по пробелу;
4. создаются множества;
5. возвращается `1 - Jaccard Similarity`.

Хороший инженер не добавляет функциональность, которая меняет ожидаемое поведение без требования.

# 12. Бизнес-применение в разных индустриях

## 12.1. EdTech

OCR распознаёт:

- условия задач;
- формулы;
- страницы учебников;
- рукописные ответы;
- тесты;
- таблицы.

Особенно критичны:

- числа;
- знаки операций;
- степени;
- единицы измерения;
- отрицания;
- имена и даты.

Jaccard полезен как грубый baseline для ранжирования, но не заменяет метрики по символам и специализированную проверку формул.

## 12.2. Финтех и банки

OCR используется для:

- паспортов;
- справок о доходах;
- выписок;
- договоров;
- платёжных реквизитов;
- заявлений.

Jaccard может оценивать текстовые поля, но для номера документа, суммы или счёта изменение одного символа критично. Для таких полей нужны отдельные проверки формата и точные посимвольные метрики.

## 12.3. Маркетплейсы и e-commerce

OCR извлекает:

- название товара;
- бренд;
- цену;
- состав;
- характеристики;
- серийные номера;
- маркировку.

Ошибки влияют на поиск, дедупликацию, модерацию и сопоставление SKU.

## 12.4. Two-sided marketplaces

OCR может читать:

- лицензии исполнителей;
- сертификаты;
- чеки;
- документы;
- адреса.

Ошибка может повлиять на допуск исполнителя или доверие к платформе.

## 12.5. iGaming и casino

Основные применения:

- KYC;
- подтверждение адреса;
- проверка платёжных документов;
- антифрод.

Jaccard здесь может быть только дополнительным сигналом. Критические поля должны проверяться отдельно.

## 12.6. Crypto

OCR встречается в KYC, подтверждении адреса и обработке скриншотов транзакций.

Для адресов кошельков Jaccard по словам практически бесполезен: изменение одного символа полностью меняет адрес. Нужны посимвольные проверки и валидация формата.

## 12.7. Dating

Применения:

- проверка документов;
- модерация изображений с текстом;
- поиск контактов и спама;
- проверка возраста.

Здесь строковая и семантическая точность решают разные задачи.

## 12.8. Big Tech

В компаниях уровня Google и Meta OCR может быть частью:

- поиска по изображениям;
- accessibility;
- модерации;
- рекламы;
- перевода;
- визуальных ассистентов.

Jaccard пригоден как простой baseline, но production evaluation обычно сочетает несколько метрик и сегментный анализ.

## 12.9. LLM-продукты и Anthropic

OCR становится входом мультимодальной языковой модели. Ошибка OCR превращается в ошибочный контекст.

Здесь нужно оценивать минимум два уровня:

1. насколько точно распознан текст;
2. насколько ошибка повлияла на ответ модели.

Высокое качество ответа на части примеров не должно маскировать системные ошибки OCR.

## 12.10. SpaceX, промышленность и BP

OCR может читать:

- приборные панели;
- технические схемы;
- серийные номера;
- маркировку деталей;
- журналы;
- инструкции.

В safety-critical среде одна неверная цифра может быть опаснее десятка ошибок в описательном абзаце. Поэтому все токены нельзя считать одинаково важными.

# 13. Как мыслить о метрике как аналитик

Перед выбором метрики нужно задать вопросы:

1. Что является единицей ошибки: символ, слово, строка, поле или документ?
2. Важен ли порядок слов?
3. Важна ли частота?
4. Нужно ли учитывать регистр?
5. Нужно ли учитывать пунктуацию?
6. Что происходит при вставке и удалении?
7. Как метрика ведёт себя на коротких текстах?
8. Как метрика ведёт себя на длинных текстах?
9. Все ли токены одинаково важны?
10. Нужно ли отдельно оценивать числа и формулы?
11. Коррелирует ли offline score с качеством ответа пользователю?
12. Какой порог запускает fallback?
13. Какие сегменты нужно мониторить отдельно?
14. Можно ли по этой метрике принимать решение о релизе?

Это переход от механического подсчёта к продуктово-инженерному мышлению.

# 14. Итог страницы

Jaccard distance решает проблему с каскадным сдвигом символов лучше прямого сравнения по индексам, но создаёт новые ограничения:

- теряет порядок;
- теряет частоту;
- воспринимает повреждённое слово как полностью новое;
- зависит от длины текста;
- не отражает бизнес-критичность токенов.

Поэтому это полезный учебный baseline, но не финальная OCR-метрика.

Для отправки в проверяющую систему используется только файл:

```text
gp_page_3_ocr_eval_jaccard_similarity_solution.py
```

`README.md` не создаётся, потому что он добавляется только на первой странице урока.
