# Страница 5. Сравнение с `difflib.SequenceMatcher`

## 1. Зачем в уроке приводится difflib

После собственной реализации полезно сравнить результат с готовым
инструментом стандартной библиотеки Python.

Используется:

```python
from difflib import SequenceMatcher
```

Затем:

```python
score = SequenceMatcher(None, text, ocr_text).ratio()
```

`ratio()` возвращает меру сходства:

```text
1.0 — полное совпадение;
0.0 — отсутствие сходства.
```

Чтобы получить меру различия:

```python
return 1 - score
```

---

## 2. Важная опечатка в формулировке

В описании иногда встречается название `rank()`.

У `SequenceMatcher` используется метод:

```python
ratio()
```

Именно он присутствует в приведённом коде.

---

## 3. Это не тот же алгоритм, что Levenshtein Distance

`SequenceMatcher` не является обычной реализацией расстояния Левенштейна.

Он ищет совпадающие непрерывные блоки и строит коэффициент сходства на их
основе.

Упрощённо ratio вычисляется через количество совпавших элементов:

\[
ratio = \frac{2M}{T}
\]

где:

- `M` — число элементов в совпадающих блоках;
- `T` — суммарная длина обеих последовательностей.

Поэтому близкие результаты двух подходов на текущем датасете не означают
математическую эквивалентность.

---

## 4. Почему результаты могут быть похожи

OCR-ошибки в наборе данных в основном локальные:

- замена одной буквы;
- замена кириллицы латиницей;
- буква превращается в цифру;
- пропуск небольшого числа символов;
- добавление лишнего символа.

Обе метрики воспринимают такие тексты как преимущественно совпадающие с
небольшой долей различий.

Поэтому ранжирование наиболее проблемных документов может оказаться близким.

---

## 5. Почему результаты могут различаться

Различия особенно заметны при:

- повторяющихся фрагментах;
- перестановках;
- длинных совпадающих блоках;
- неоднозначном выравнивании;
- большом количестве одинаковых символов;
- различных настройках весов Levenshtein.

`SequenceMatcher` не предоставляет те же параметры:

```python
insert_cost
delete_cost
substitute_cost
```

Поэтому он не заменяет `NormLevEval`, когда бизнес хочет управлять типами
ошибок.

---

## 6. Преимущества `SequenceMatcher`

- входит в стандартную библиотеку;
- не требует внешней установки;
- короткий код;
- удобен для быстрого baseline;
- умеет находить совпадающие блоки;
- полезен для diff-инструментов.

---

## 7. Ограничения `SequenceMatcher`

- это не классический Levenshtein distance;
- нельзя независимо задавать стоимость вставки, удаления и замены;
- сложнее интерпретировать как число редакторских операций;
- поведение зависит от структуры совпадающих блоков;
- встроенная эвристика популярных элементов может влиять на длинные
  последовательности;
- результаты не всегда симметричны в практическом использовании разных
  настроек и данных.

---

## 8. `autojunk`

У `SequenceMatcher` есть эвристика:

```python
autojunk=True
```

по умолчанию.

Она может считать слишком часто встречающиеся элементы «мусорными» в длинных
последовательностях.

Для текста это иногда полезно, но иногда создаёт неожиданные результаты.

Можно отключить:

```python
SequenceMatcher(
    None,
    text,
    ocr_text,
    autojunk=False,
)
```

В учебном примере используется стандартное поведение.

---

## 9. Когда выбрать NormLevEval

`NormLevEval` удобнее, когда нужно:

- объяснять score через редакторские операции;
- задавать веса;
- контролировать нормализацию;
- строить собственную бизнес-логику;
- расширять стоимость конкретных ошибок;
- иметь прозрачный алгоритм.

---

## 10. Когда выбрать SequenceMatcher

`SequenceMatcher` удобен, когда нужен:

- быстрый встроенный baseline;
- визуальный diff;
- поиск совпадающих блоков;
- прототип без внешних зависимостей;
- приблизительное сравнение последовательностей.

---

## 11. Что лучше для production OCR Evaluation

Обычно не выбирают только одну метрику.

Возможный набор:

```text
Normalized Levenshtein Distance
Character Error Rate
Word Error Rate
Exact Match
SequenceMatcher ratio
Critical-token accuracy
Field-level accuracy
Downstream answer quality
```

Каждая отвечает на свой вопрос.

### Normalized Levenshtein

Какова нормированная стоимость редакторских исправлений?

### CER

Какова доля символьных ошибок относительно эталона?

### WER

Какова доля ошибок на уровне слов?

### Exact Match

Какова доля полностью безошибочных результатов?

### Downstream quality

Смогла ли конечная система правильно решить задачу?

---

## 12. Почему downstream-метрика обязательна

Два OCR-результата могут иметь одинаковый normalized distance.

Первый:

```text
Опишите процесс фотосинтеза.
```

содержит небольшую опечатку в обычном слове.

Второй:

```text
Рассчитайте 5 × 8.
```

теряет знак умножения.

Редакторская доля может быть похожей, но влияние на ответ различается.

Поэтому offline-метрики нужно связывать с продуктовым результатом.

---

## 13. Практический эксперимент

Можно построить таблицу:

| document_id | norm_lev | difflib_error | answer_correct |
|---|---:|---:|---:|
| 1 | 0.02 | 0.02 | 1 |
| 2 | 0.10 | 0.09 | 1 |
| 3 | 0.15 | 0.14 | 0 |
| 4 | 0.25 | 0.22 | 0 |

Затем проверить:

- корреляцию метрик;
- качество ранжирования;
- порог для fallback;
- связь с ошибкой конечного ответа;
- различия по сегментам.

---

## 14. Итог

Похожий список ошибок у `NormLevEval` и `SequenceMatcher` подтверждает, что
собственная реализация работает разумно на текущих данных.

Но эти алгоритмы не идентичны.

`NormLevEval` ценен прозрачностью, нормализацией и управляемыми весами.
`SequenceMatcher` ценен как быстрый встроенный baseline и инструмент поиска
совпадающих блоков.
