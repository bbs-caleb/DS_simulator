# Страница 5. Нормализация и стоимости операций: глубокий разбор

## 1. Две разные задачи внутри одной метрики

`NormLevEval` решает сразу две независимые проблемы:

1. **взвешивание операций**;
2. **нормализация итогового расстояния**.

Их нельзя путать.

### Взвешивание

Отвечает на вопрос:

> Какие типы редакторских ошибок мы считаем более дорогими?

### Нормализация

Отвечает на вопрос:

> Как сравнивать итоговую ошибку между текстами разной длины?

---

## 2. Взвешенное расстояние

Обычный Левенштейн использует:

```text
insert = 1
delete = 1
substitute = 1
```

Взвешенная версия разрешает:

```text
insert = 0.8
delete = 1.0
substitute = 1.2
```

Тогда один и тот же набор операций получает другую суммарную стоимость.

---

## 3. Почему увеличение substitute_cost не означает простое умножение замен

Алгоритм минимизирует общую стоимость.

Пусть:

```text
insert = 1
delete = 1
substitute = 2
```

Прямая замена стоит `2`.

Удаление плюс вставка тоже стоит:

```text
1 + 1 = 2
```

Оба пути равноправны.

Если:

```text
substitute = 2
insert = 0.4
delete = 0.4
```

то удаление плюс вставка стоит `0.8`, поэтому алгоритм не выберет дорогую
замену.

Следствие:

> Изменение веса влияет не только на стоимость, но и на сам оптимальный путь.

---

## 4. Направленность метрики при разных insert/delete cost

Если стоимости вставки и удаления различаются, расстояние может стать
направленным.

Например:

```python
insert_cost = 0.5
delete_cost = 1.5
```

Преобразование:

```text
"cat" → "cart"
```

требует дешёвой вставки.

Обратное преобразование:

```text
"cart" → "cat"
```

требует дорогого удаления.

Поэтому:

```text
distance("cat", "cart")
```

может отличаться от:

```text
distance("cart", "cat")
```

Для OCR это допустимо, потому что направления имеют смысл:

```text
reference → OCR output
```

Эталон и результат модели не являются взаимозаменяемыми ролями.

---

## 5. Почему в BaseEval порядок `(text, ocr_text)` важен

Класс хранит:

```python
(text, ocr_text)
```

И передаёт именно в таком порядке:

```python
self._eval_func(text, ocr_text)
```

При одинаковых весах направление обычно не влияет на классический Levenshtein
distance.

При разных `insert_cost` и `delete_cost` направление уже важно.

Поэтому менять местами строки нельзя.

---

## 6. Выбор знаменателя

Требуемый знаменатель:

```python
max(len(text), len(ocr_text)) * max_cost
```

где:

```python
max_cost = max(
    insert_cost,
    delete_cost,
    substitute_cost,
)
```

Он выполняет две функции:

1. учитывает масштаб текста;
2. учитывает масштаб пользовательских весов.

---

## 7. Почему не делим на длину эталона

В OCR часто встречаются лишние символы.

Пример:

```text
text = "abc"
ocr_text = "abcdefgh"
```

Если делить только на:

```python
len(text)
```

знаменатель будет слишком мал.

Использование более длинной строки лучше учитывает вставленные символы.

---

## 8. Почему не делим на длину OCR

Может произойти обратная ситуация:

```text
text = "abcdefgh"
ocr_text = "abc"
```

OCR пропустил большую часть строки.

Если делить только на длину OCR, знаменатель снова будет слишком мал.

Поэтому берётся максимум двух длин.

---

## 9. Почему не делим на среднюю длину

Средняя длина:

```python
(len(text) + len(ocr_text)) / 2
```

может использоваться в других метриках, но это не контракт текущего задания.

Кроме того, при сильно различающихся длинах средняя длина меньше максимальной и
даёт более агрессивный score.

---

## 10. Почему не делим на сумму длин

Знаменатель:

```python
len(text) + len(ocr_text)
```

обычно делает score заметно меньше.

Такой подход используется в некоторых коэффициентах сходства, но не
соответствует заданному docstring.

---

## 11. Почему используется максимальный вес

Пусть:

```text
insert = 0.5
delete = 1.0
substitute = 1.5
```

Если не учитывать веса, score зависит от выбранной шкалы.

Например, можно умножить все веса на два:

```text
1.0, 2.0, 3.0
```

Смысл относительных предпочтений почти не изменился, но абсолютное расстояние
удвоилось.

Нормирование на максимальный вес компенсирует изменение общего масштаба.

---

## 12. Инвариантность к общему масштабированию весов

Пусть все веса умножаются на коэффициент `k`.

Тогда:

```text
weighted_distance
```

тоже умножается на `k`.

Знаменатель:

```text
max_length × max_cost
```

также умножается на `k`.

Коэффициент сокращается.

Это важное свойство:

> Если сохранить отношения между весами, но изменить их общий масштаб,
> нормализованный score в идеальном случае остаётся тем же.

Ограничение `[0, 2]` не позволяет произвольно масштабировать веса, но
математический принцип остаётся полезным.

---

## 13. Что означает нулевая стоимость

### Нулевая вставка

```python
insert_cost = 0
```

Лишние символы OCR могут добавляться бесплатно.

### Нулевое удаление

```python
delete_cost = 0
```

Пропуски символов не штрафуются.

### Нулевая замена

```python
substitute_cost = 0
```

Любые замены бесплатны.

### Все стоимости нулевые

Любую строку можно преобразовать в любую другую с нулевой стоимостью.

Поэтому score равен `0.0`.

Это математически согласованно, хотя редко полезно в production.

---

## 14. Пример настройки под OCR

Допустим, бизнес особенно хочет подсветить замены символов:

```python
insert_cost = 1
delete_cost = 1
substitute_cost = 1.2
```

Тогда замена получает дополнительный штраф.

Но нужно помнить: если замена станет дороже суммы удаления и вставки, алгоритм
может выбрать две операции вместо одной.

При диапазоне `[0, 2]` и стандартных вставке/удалении по `1` максимальная
замена `2` как раз равна стоимости удаления плюс вставки.

---

## 15. Почему веса — это продуктовая гипотеза

Число `1.2` не является универсально правильным.

Оно кодирует предположение:

> Замена в нашем продукте примерно на 20% важнее стандартной вставки или
> удаления.

Такую гипотезу нужно проверять.

Возможные источники выбора весов:

- экспертная оценка;
- частота ошибок;
- влияние на downstream-задачу;
- стоимость ручного исправления;
- жалобы пользователей;
- риск ошибки;
- A/B-тест;
- оптимизация корреляции с human rating.

---

## 16. Как обучить веса по данным

В production можно иметь датасет:

```text
OCR-пара
оценка человека
критичность ошибки
успех конечного ответа
```

Далее подбирать веса так, чтобы метрика лучше коррелировала с:

- экспертной оценкой;
- вероятностью неправильного ответа;
- отказом пользователя;
- ручным исправлением;
- бизнес-ущербом.

Это превращает ручные коэффициенты в data-driven параметры.

---

## 17. Глобальные и локальные веса

Текущая задача использует глобальные веса:

```text
одна цена для любой вставки
одна цена для любого удаления
одна цена для любой замены
```

Более развитая система может использовать:

### Веса пар символов

```text
O → 0 = 2.0
е → e = 0.5
```

### Веса классов символов

```text
ошибка в цифре = 2.0
ошибка в пунктуации = 0.5
```

### Веса полей

```text
номер документа = 3.0
адрес = 1.0
описание = 0.5
```

### Веса позиций

```text
ответ задачи = 2.0
поясняющий текст = 1.0
```

---

## 18. Нормализация не делает метрику бизнес-идеальной

Score `0.1` не означает автоматически:

```text
10% бизнес-ошибки
```

Он означает нормированную редакторскую стоимость по конкретной формуле.

Для бизнес-интерпретации нужно проверить связь между score и конечным
качеством.

Например:

```text
score < 0.03
```

может соответствовать 98% успешных ответов.

А:

```text
score > 0.15
```

может резко повышать вероятность неправильного ответа.

Такие пороги определяются по данным.

---

## 19. Как построить мониторинг

Полезно отслеживать:

- средний normalized distance;
- медиану;
- p90;
- p95;
- долю документов выше порога;
- score по типу документа;
- score по языку;
- score по устройству;
- score по версии модели;
- score по длине текста;
- score по типу ошибки.

Одно среднее значение может скрыть серьёзные проблемы в отдельных сегментах.

---

## 20. Итог

Нормализация отвечает за сопоставимость.

Веса отвечают за приоритеты.

Вместе они превращают обычное число редакторских операций в более гибкую
метрику, которую можно адаптировать под свойства конкретной OCR-системы.
