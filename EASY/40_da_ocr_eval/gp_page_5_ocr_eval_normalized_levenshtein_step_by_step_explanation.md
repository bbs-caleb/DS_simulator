# Страница 5. Пошаговое объяснение кода с абсолютного нуля

## 1. Что изменилось относительно обычного LevenshteinEval

Раньше каждая операция стоила единицу:

```text
вставка = 1
удаление = 1
замена = 1
```

Теперь пользователь может передать другие значения:

```python
NormLevEval(
    "data.json",
    insert_cost=0.8,
    delete_cost=1.0,
    substitute_cost=1.2,
)
```

Кроме того, итоговое расстояние делится на нормирующий знаменатель, чтобы
короткие и длинные тексты можно было сравнивать в одном масштабе.

---

## 2. Конструктор класса

```python
def __init__(
    self,
    documents_path: str,
    insert_cost: float = 1,
    delete_cost: float = 1,
    substitute_cost: float = 1,
):
```

### `documents_path`

Путь к JSON-файлу.

### `insert_cost`

Цена вставки.

### `delete_cost`

Цена удаления.

### `substitute_cost`

Цена замены.

Значения по умолчанию равны `1`, поэтому без дополнительных аргументов класс
ведёт себя как обычное расстояние Левенштейна с последующей нормализацией.

---

## 3. Кортеж стоимостей

```python
costs = (insert_cost, delete_cost, substitute_cost)
```

Создаётся кортеж из трёх значений.

Пример:

```python
costs = (1, 1, 1.2)
```

Это позволяет проверить все параметры одним выражением.

---

## 4. Проверка диапазона

```python
if not all(0 <= cost <= 2 for cost in costs):
    raise ValueError(...)
```

Разберём по частям.

### Генератор

```python
0 <= cost <= 2 for cost in costs
```

Для каждого значения создаётся `True` или `False`.

Для:

```python
costs = (1, 1, 1.2)
```

получаем логически:

```python
True
True
True
```

### `all`

```python
all(...)
```

возвращает `True`, только если все элементы истинны.

### `not`

Если хотя бы один параметр неправильный, `all` вернёт `False`, а `not`
превратит его в `True`.

Тогда выполняется:

```python
raise ValueError(...)
```

---

## 5. Почему проверка включает границы

Условие:

```python
0 <= cost <= 2
```

разрешает:

```text
0
2
```

Но запрещает:

```text
-0.1
2.1
```

Это называется закрытым диапазоном `[0, 2]`.

Квадратные скобки означают, что границы входят в диапазон.

---

## 6. Сохранение параметров в объекте

```python
self.insert_cost = insert_cost
self.delete_cost = delete_cost
self.substitute_cost = substitute_cost
```

Теперь веса можно использовать в других методах объекта.

Например:

```python
self.insert_cost
```

означает стоимость вставки именно для этого экземпляра `NormLevEval`.

Можно создать два объекта с разными настройками:

```python
first = NormLevEval(
    "data.json",
    substitute_cost=1,
)

second = NormLevEval(
    "data.json",
    substitute_cost=1.5,
)
```

У них будут разные оценки одних и тех же документов.

---

## 7. Вызов родительского конструктора

```python
super().__init__(documents_path)
```

`super()` предоставляет доступ к родительскому классу `BaseEval`.

Фактически вызывается:

```python
BaseEval.__init__(self, documents_path)
```

Родительский класс:

1. проверяет файл;
2. читает JSON;
3. проверяет документы;
4. сохраняет их в `self.documents`;
5. проверяет, что список не пустой.

---

## 8. Размеры матрицы

```python
rows_count = len(text) + 1
columns_count = len(ocr_text) + 1
```

Если:

```python
text = "cat"
ocr_text = "cut"
```

то:

```text
rows_count = 4
columns_count = 4
```

Дополнительные строка и столбец представляют пустые префиксы.

---

## 9. Создание матрицы float

```python
distances = [
    [0.0] * columns_count
    for _ in range(rows_count)
]
```

Используется `0.0`, а не `0`, потому что веса могут быть дробными:

```python
0.5
1.2
1.75
```

Python и с целыми нулями корректно перешёл бы к float, но `0.0` сразу показывает
смысл матрицы: она может содержать дробные стоимости.

---

## 10. Первый столбец — удаления

```python
for row in range(rows_count):
    distances[row][0] = row * self.delete_cost
```

Пусть:

```python
delete_cost = 1.5
text = "cat"
```

Первый столбец:

```text
0.0
1.5
3.0
4.5
```

Почему?

```text
""    → "" = 0 удалений
"c"   → "" = 1 удаление × 1.5
"ca"  → "" = 2 удаления × 1.5
"cat" → "" = 3 удаления × 1.5
```

---

## 11. Первая строка — вставки

```python
for column in range(columns_count):
    distances[0][column] = column * self.insert_cost
```

Пусть:

```python
insert_cost = 0.5
ocr_text = "cut"
```

Первая строка:

```text
0.0  0.5  1.0  1.5
```

Чтобы из пустой строки получить `"cut"`, нужны три вставки.

---

## 12. Циклы по внутренним ячейкам

```python
for row in range(1, rows_count):
    for column in range(1, columns_count):
```

Циклы начинаются с `1`, потому что:

- строка `0` уже заполнена;
- столбец `0` уже заполнен.

---

## 13. Взвешенная вставка

```python
insertion_cost = (
    distances[row][column - 1]
    + self.insert_cost
)
```

Берётся стоимость из левой ячейки и прибавляется стоимость одной вставки.

Если слева:

```text
2.4
```

а вставка стоит:

```text
0.8
```

новый вариант:

```text
3.2
```

---

## 14. Взвешенное удаление

```python
deletion_cost = (
    distances[row - 1][column]
    + self.delete_cost
)
```

Берётся значение сверху и прибавляется стоимость удаления.

---

## 15. Проверка равенства символов

```python
characters_are_different = (
    text[row - 1] != ocr_text[column - 1]
)
```

Результат — логическое значение.

Если символы равны:

```python
False
```

Если различаются:

```python
True
```

---

## 16. Почему bool можно умножать на float

В Python:

```python
False == 0
True == 1
```

Поэтому:

```python
False * 1.2
```

даёт:

```text
0.0
```

А:

```python
True * 1.2
```

даёт:

```text
1.2
```

Это позволяет компактно записать:

```python
characters_are_different * self.substitute_cost
```

---

## 17. Стоимость замены или совпадения

```python
replacement_cost = (
    distances[row - 1][column - 1]
    + characters_are_different * self.substitute_cost
)
```

### Символы совпадают

Добавляется `0`.

Это означает переход по диагонали без редакторской операции.

### Символы различаются

Добавляется `substitute_cost`.

---

## 18. Выбор самого дешёвого пути

```python
distances[row][column] = min(
    insertion_cost,
    deletion_cost,
    replacement_cost,
)
```

Алгоритм рассматривает три способа и выбирает минимальный.

Предположим:

```text
вставка = 2.0
удаление = 2.0
замена = 2.5
```

Тогда сохраняется:

```text
2.0
```

Даже если символы отличаются, алгоритм может выбрать другой путь.

---

## 19. Почему дорогая замена может превратиться в удаление и вставку

Пусть:

```python
insert_cost = 0.5
delete_cost = 0.5
substitute_cost = 2
```

Прямая замена стоит:

```text
2
```

Удаление плюс вставка:

```text
0.5 + 0.5 = 1
```

Алгоритм выберет более дешёвую комбинацию.

Это правильное поведение задачи поиска минимальной стоимости.

---

## 20. Получение длины более длинной строки

```python
max_text_length = max(len(text), len(ocr_text))
```

Пример:

```python
text = "cat"
ocr_text = "cart"
```

Длины:

```text
3
4
```

Максимальная длина:

```text
4
```

---

## 21. Максимальная стоимость операции

```python
max_operation_cost = max(
    self.insert_cost,
    self.delete_cost,
    self.substitute_cost,
)
```

Для:

```text
insert = 0.8
delete = 1.0
substitute = 1.2
```

получаем:

```text
1.2
```

---

## 22. Максимально возможная нормирующая величина

```python
max_possible_distance = (
    max_text_length * max_operation_cost
)
```

Если:

```text
max_text_length = 50
max_operation_cost = 1.2
```

то:

```text
max_possible_distance = 60
```

---

## 23. Почему это не обязательно фактическое расстояние

Название `max_possible_distance` используется как нормирующая верхняя граница.

Для конкретной пары строк реальное минимальное расстояние может быть намного
меньше.

Нам не нужно, чтобы алгоритм обязательно достигал знаменателя.

Нужно, чтобы знаменатель задавал общий масштаб.

---

## 24. Защита от деления на ноль

```python
if max_possible_distance == 0:
    return 0.0
```

### Случай 1

Обе строки пустые:

```text
max_text_length = 0
```

### Случай 2

Все стоимости нулевые:

```text
max_operation_cost = 0
```

В обоих случаях произведение равно нулю.

---

## 25. Финальная нормализация

```python
return distances[-1][-1] / max_possible_distance
```

`distances[-1][-1]` — итоговая минимальная взвешенная стоимость.

Она делится на нормирующую величину.

---

## 29. Как работает унаследованный `evaluate`

```python
results = [
    (self._eval_func(text, ocr_text), text, ocr_text)
    for text, ocr_text in self.documents
]
```

`self` является объектом `NormLevEval`.

Поэтому вызывается:

```python
NormLevEval._eval_func
```

---

## 30. Сортировка

```python
results.sort(key=lambda result: result[0], reverse=True)
```

Большой normalized distance означает высокую долю ошибок.

Поэтому наиболее проблемные документы выводятся первыми.

---

## 31. Стабильность

Если два документа имеют одинаковый score, Python сохраняет их исходный
относительный порядок.

Это важно для результатов:

```text
0.125
0.125
```

Второй и третий элементы кортежа не используются как дополнительные ключи.

---

## 32. Почему не нужно писать `evaluate` заново

Вся логика `evaluate` уже подходит для любой метрики.

Он не знает, как именно считается score.

Он просто вызывает:

```python
self._eval_func(...)
```

Это пример полиморфизма и шаблонного метода.

---

## 33. Почему нельзя округлять внутри `_eval_func`

Предположим:

```text
score A = 0.1249
score B = 0.1241
```

Если округлить до двух знаков, оба станут:

```text
0.12
```

Это может изменить порядок документов.

Поэтому метод возвращает полную точность.

Округление применяется только при отображении:

```python
print(f"{score:.2f}")
```

---

## 34. Сложность алгоритма

Пусть:

```text
n = len(text)
m = len(ocr_text)
```

Время:

\[
O(nm)
\]

Память:

\[
O(nm)
\]

Потому что создаётся матрица `(n + 1) × (m + 1)` и заполняется каждая ячейка.

---

## 35. Почему полная матрица допустима

Можно оптимизировать память до:

\[
O(\min(n,m))
\]

Но полная матрица:

- легче для понимания;
- соответствует уроку;
- проще отлаживается;
- минимизирует риск ошибки;
- подходит для размеров учебного датасета.

---

## 36. Частые ошибки в реализации

### Перепутаны веса границ

Первый столбец использует `delete_cost`.

Первая строка использует `insert_cost`.

### Использован минимум длины

Нужно:

```python
max(...)
```

### В знаменателе нет максимальной стоимости

Тогда веса ломают единый масштаб.

### Неверно проверен диапазон

Нужно разрешить границы:

```python
0 <= cost <= 2
```

### Не обработаны нулевые веса

При всех нулях знаменатель равен нулю.

### Использована сумма стоимостей

Условие требует максимальную стоимость.

### `super().__init__` не вызван

Тогда нет `self.documents`.

### Добавлена нормализация текста

`.lower()`, удаление пунктуации или пробелов меняют контракт задачи и могут
сломать скрытые тесты.

---

## 37. Что именно проверено в готовом файле

Проверены:

- наследование;
- синтаксис;
- PEP8-длина строк;
- отрицательная стоимость;
- стоимость больше `2`;
- допустимые значения `0` и `2`;
- одинаковые строки;
- пустые строки;
- все веса равны нулю;
- сортировка;
- `limit`;
- результаты на `data.json`;
- порядок первых четырёх документов.

---

## 38. Финальная схема

```text
NormLevEval.__init__
    ↓
проверка весов
    ↓
сохранение весов
    ↓
BaseEval.__init__
    ↓
загрузка документов
    ↓
evaluate
    ↓
NormLevEval._eval_func
    ↓
weighted Levenshtein matrix
    ↓
нормализация
    ↓
стабильная сортировка
    ↓
limit
```

## 39. Главная идея страницы

Обычное расстояние отвечает:

> Сколько стоит исправить текст?

Нормализованное расстояние отвечает:

> Насколько велика стоимость исправления относительно размера текста и
> допустимого масштаба операций?

Именно второй вопрос позволяет честнее сравнивать документы разной длины.
