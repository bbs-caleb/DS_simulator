# DA: OCR EVAL — прямое посимвольное сравнение

## 1. Что происходит в этой задаче

В продукте **Smart Tutor** пользователь может не печатать учебное задание вручную, а отправить фотографию или скриншот. OCR-система должна извлечь текст с изображения и передать его диалоговому помощнику.

Упрощённый путь данных выглядит так:

```text
Фотография задания
        ↓
OCR-модель распознаёт символы
        ↓
Получается строка ocr_text
        ↓
Диалоговый помощник анализирует строку
        ↓
Пользователь получает ответ
```

Если OCR ошибается, следующая модель получает уже искажённый запрос. Даже сильная языковая модель не всегда может восстановить исходный смысл, особенно если OCR повредил:

- число;
- знак математической операции;
- единицу измерения;
- степень;
- отрицание;
- имя собственного объекта;
- химическую формулу;
- важное слово в условии задачи.

Например:

```text
Эталон:  Решите уравнение: 5x + 3 = 18
OCR:     Решите уравнение: 5x - 3 = 18
```

Здесь отличается всего один символ, однако математическая задача стала другой. Поэтому качество OCR нельзя оценивать только по тому, «похожи ли тексты по смыслу».

---

## 2. Бизнес-проблема

### 2.1. Почему ошибка OCR является продуктовой проблемой

OCR — не самостоятельная функция ради распознавания текста. Это один из этапов пользовательского сценария.

Пользователь ожидает:

1. сфотографировать задание;
2. не перепечатывать его вручную;
3. сразу получить корректную помощь.

Если OCR распознаёт текст неверно, возникают следующие последствия:

- помощник решает не ту задачу;
- пользователь вынужден исправлять распознанный текст вручную;
- пользователь повторно загружает фотографию;
- увеличивается время до полезного ответа;
- снижается доверие к продукту;
- растёт доля незавершённых сессий;
- растёт нагрузка на поддержку;
- ухудшается удержание пользователей;
- снижается конверсия в платную подписку.

Поэтому задача аналитика состоит не просто в подсчёте несовпавших символов. Нужно построить измеримый процесс, который отвечает на вопросы:

- насколько хорошо работает OCR;
- на каких документах он ошибается сильнее всего;
- какие типы ошибок встречаются чаще;
- стала ли новая версия OCR лучше;
- какие ошибки критичны для бизнеса;
- что следует исправлять в первую очередь.

## 3. Что содержится в данных

Файл `data.json` содержит список документов. Каждый документ представлен словарём с двумя обязательными полями:

```json
{
    "text": "эталонный текст",
    "ocr_text": "текст, распознанный OCR"
}
```

### `text`

Это эталонный, правильный текст. Его часто называют:

- ground truth;
- reference text;
- target text;
- gold standard;
- разметка;
- истинное значение.

Именно с ним сравнивается результат OCR.

### `ocr_text`

Это строка, которую вернула система распознавания текста.

Одна запись после загрузки должна быть преобразована в кортеж:

```python
(text, ocr_text)
```

А весь файл — в список таких кортежей:

```python
[
    (text_1, ocr_text_1),
    (text_2, ocr_text_2),
    ...
]
```

---

## 4. Почему семантическое сравнение здесь не является главным решением

Векторные представления, эмбеддинги и semantic similarity отвечают примерно на вопрос:

> Передают ли два текста похожий смысл?

Но OCR evaluation на этом этапе отвечает на другой вопрос:

> Какие конкретно символы были распознаны неверно?

Пример:

```text
Эталон: Температура равна 10°C
OCR:    Температура равна 100°C
```

Семантически предложения почти идентичны. Однако с точки зрения точности распознавания и конечного ответа ошибка очень серьёзная.

Другой пример:

```text
Эталон: y = x + 1
OCR:    y = x - 1
```

Смысловая близость высокая, но задача изменилась.

Поэтому для OCR обычно важны метрики, работающие на уровне:

- символов;
- слов;
- вставок;
- удалений;
- замен;
- порядка символов.

На текущем шаге курса рассматривается самый простой вариант — прямое сравнение символов, стоящих на одинаковых позициях.

---

## 5. Что такое прямое посимвольное сравнение

Пусть есть две строки:

```python
text = "hello world!"
ocr_text = "hel1o world1 "
```

Мы сравниваем:

```text
позиция 0: h == h
позиция 1: e == e
позиция 2: l == l
позиция 3: l != 1
позиция 4: o == o
...
```

За каждую несовпавшую позицию добавляется одна ошибка.

Если длины строк различаются, символы, для которых во второй строке нет пары, тоже должны быть учтены. Поэтому итоговая формула имеет вид:

```text
score = несовпадения среди попарно сравниваемых символов
        + абсолютная разница длин строк
```

В Python попарное сравнение удобно выполнять через `zip`:

```python
zip(text, ocr_text)
```

Важно понимать поведение `zip`: он останавливается на самой короткой последовательности.

Пример:

```python
list(zip("abc", "ab"))
```

Результат:

```python
[("a", "a"), ("b", "b")]
```

Символ `c` не попал в пары. Поэтому отдельно добавляется:

```python
abs(len(text) - len(ocr_text))
```

## 6. Почему этот подход даёт завышенные оценки

Рассмотрим строки:

```text
Эталон:  abcdef
OCR:     abdef
```

OCR пропустил один символ `c`.

Человек видит одну ошибку — удаление символа.

Но прямое сравнение по индексам работает так:

```text
Эталон: a b c d e f
OCR:    a b d e f

0: a == a
1: b == b
2: c != d
3: d != e
4: e != f
```

После пропуска все последующие символы сдвинулись. Алгоритм считает несколько несовпадений вместо одного удаления.

То же происходит при добавлении лишнего символа:

```text
Эталон: abcdef
OCR:    abcXdef
```

После вставки индексы расходятся, и последующие символы ошибочно считаются неправильными.

Именно поэтому приведённые в уроке почти одинаковые тексты получают огромные score: 67, 57 и 52.

Это не ошибка реализации. Это ожидаемое ограничение выбранного простого метода.

---

## 7. Зачем тогда реализовывать заведомо слабый метод

Это важный аналитический и инженерный приём: сначала построить простой baseline.

Baseline нужен, чтобы:

- проверить загрузку данных;
- определить формат результата;
- сформировать единый API класса;
- получить первую измеримую метрику;
- увидеть недостатки на реальных примерах;
- сравнить будущие методы с простым подходом;
- не начинать сразу со сложной реализации без контрольной точки.

В промышленной разработке часто двигаются так:

```text
простой baseline
        ↓
анализ ошибок baseline
        ↓
улучшенный алгоритм
        ↓
сравнение результатов
        ↓
выбор решения
```

Таким образом, задача не утверждает, что прямое сравнение является хорошей OCR-метрикой. Наоборот, она позволяет практически увидеть, почему оно недостаточно.

## 8. Техническая цель задания

Нужно реализовать класс `BaseEval`, который выполняет четыре функции:

1. загружает JSON-файл;
2. проверяет структуру каждого документа;
3. считает число различий между двумя строками;
4. сортирует документы от самых ошибочных к наименее ошибочным.

Контракт класса:

```python
BaseEval(documents_path)
```

После создания объекта в нём должно появиться поле:

```python
self.documents
```

Содержащее список кортежей:

```python
[(text, ocr_text), ...]
```

Метод:

```python
evaluate(limit=3)
```

должен вернуть:

```python
[
    (score, text, ocr_text),
    ...
]
```

---

## 9. Требования к каждому методу

### 9.1. `__init__`

Конструктор получает путь к файлу и вызывает `load_documents`.

Если после загрузки список пустой, требуется:

```python
raise ValueError("The documents list is empty.")
```

Это важно: пустой JSON-список синтаксически корректен, но оценивать в нём нечего.

### 9.2. `load_documents`

Метод должен:

- проверить существование файла;
- открыть файл в кодировке UTF-8;
- прочитать JSON;
- убедиться, что корневой объект является списком;
- провалидировать каждый элемент;
- вернуть список кортежей.

### 9.3. `validate_document`

Метод должен проверить:

- документ является словарём;
- присутствует ключ `text`;
- присутствует ключ `ocr_text`.

После этого он возвращает:

```python
(doc["text"], doc["ocr_text"])
```

### 9.4. `_eval_func`

Метод должен посчитать:

```python
sum(source_char != ocr_char for ...)
```

и прибавить абсолютную разницу длин.

### 9.5. `evaluate`

Метод должен:

- посчитать score для каждого документа;
- сформировать кортеж `(score, text, ocr_text)`;
- отсортировать по `score` по убыванию;
- сохранить исходный порядок записей при одинаковых score;
- применить `limit` после сортировки.

---

## 10. Что означает стабильная сортировка

Предположим, исходные результаты имеют вид:

```python
[
    (5, "document A", "ocr A"),
    (8, "document B", "ocr B"),
    (5, "document C", "ocr C"),
]
```

После сортировки по score по убыванию должно получиться:

```python
[
    (8, "document B", "ocr B"),
    (5, "document A", "ocr A"),
    (5, "document C", "ocr C"),
]
```

Документ A раньше документа C в исходном списке. У них одинаковый score, поэтому этот относительный порядок должен сохраниться.

Нельзя сортировать весь кортеж целиком:

```python
sorted(results, reverse=True)
```

Такой вариант при равенстве score начнёт сравнивать `text`, а затем `ocr_text`. Это нарушит требование.

Правильная идея:

```python
results.sort(key=lambda result: result[0], reverse=True)
```

Python использует стабильную сортировку. При равенстве ключей порядок сохраняется.

## 11. Бизнес-применение OCR evaluation в разных индустриях

### EdTech

- распознавание домашних заданий;
- формулы и уравнения;
- рукописные ответы;
- проверка школьных работ;
- перевод учебников в цифровой формат.

Критические сущности: числа, знаки, степени, единицы измерения.

### Финтех и банки

- паспорта;
- банковские выписки;
- чеки;
- кредитные анкеты;
- договоры;
- ИНН, номера документов и суммы.

Критические сущности: идентификаторы, даты, суммы, валюты, персональные данные.

### Маркетплейсы и e-commerce

- распознавание товарных этикеток;
- штрихкоды и SKU;
- характеристики товара;
- чеки поставщиков;
- накладные;
- документы продавцов.

Критические сущности: артикул, цена, количество, бренд, размер.

### Two-sided marketplaces

- документы исполнителей;
- номера автомобилей;
- лицензии;
- чеки;
- доказательства выполнения услуги.

Ошибка OCR может привести к неправильной верификации участника платформы.

### iGaming и casino

- KYC-документы;
- платёжные подтверждения;
- проверка возраста;
- документы для AML-процедур.

Критические последствия: юридические риски, мошенничество, блокировка легитимных клиентов.

### Crypto

- документы KYC;
- адреса кошельков с изображений;
- платёжные реквизиты;
- данные транзакций.

Даже один неверный символ в адресе может полностью изменить адрес получателя.

### Dating

- age verification;
- распознавание документов;
- модерация текста на изображениях;
- обнаружение контактных данных и мошеннических схем.

### Big Tech

Компании уровня Google, Meta, Anthropic и других используют OCR как часть более крупных multimodal-систем:

- поиск по изображениям;
- понимание документов;
- извлечение таблиц;
- обработка скриншотов;
- accessibility;
- document question answering;
- обучение мультимодальных моделей.

На таком масштабе недостаточно одной средней метрики. Качество анализируют по языкам, типам документов, устройствам, разрешению, шрифтам, доменам и критичности символов.

## 12. Продуктовое мышление аналитика

Сильный аналитик не останавливается на вопросе «какой средний score?». Он задаёт дополнительные вопросы:

1. Какие ошибки портят конечный ответ пользователю?
2. Одинаково ли важны ошибка в запятой и ошибка в числе?
3. На каких устройствах OCR работает хуже?
4. Зависит ли качество от освещения и фона?
5. Есть ли разница между печатным и рукописным текстом?
6. Какие школьные предметы страдают сильнее?
7. Что происходит с формулами и специальными символами?
8. Сколько пользователей исправляют текст вручную?
9. Уменьшает ли новая OCR-модель количество повторных загрузок?
10. Улучшается ли конечная корректность ответа помощника?

Это разделяет техническую метрику и бизнес-результат:

```text
качество OCR
    → качество распознанного запроса
    → качество ответа помощника
    → удовлетворённость пользователя
    → удержание и выручка
```

## 13. Главный вывод страницы

Прямое посимвольное сравнение:

- очень простое;
- быстрое;
- прозрачное;
- полезно как baseline;
- корректно замечает замены на одинаковых позициях;
- плохо обрабатывает вставки и удаления;
- может сильно завышать количество ошибок после первого сдвига.

На этой странице важно не только написать код, но и понять причину провала метода. Следующий логичный шаг — использовать алгоритм выравнивания строк, который умеет отдельно учитывать вставки, удаления и замены.
