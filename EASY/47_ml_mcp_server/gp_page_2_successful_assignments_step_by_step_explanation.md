# SQL-решение: пошаговое объяснение с нуля

## 1. Готовый запрос

```sql
SELECT COUNT(*) AS users_count
FROM (
    SELECT user_id
    FROM completions
    GROUP BY user_id
    HAVING AVG(
        CASE
            WHEN score >= 90 THEN 1.0
            ELSE 0.0
        END
    ) >= 0.9
) AS qualified_users;
```

На приложенных CSV запрос возвращает:

```text
2
```

## 2. Что именно нужно посчитать

Нельзя просто посчитать строки со `score >= 90`.

Нужно:

1. отдельно рассмотреть каждого пользователя;
2. среди выполненных им заданий определить успешные;
3. посчитать долю успешных;
4. оставить пользователей с долей не меньше 90%;
5. посчитать количество таких пользователей.

Формула:

```text
успешные выполненные задания
-----------------------------  >= 0.9
все выполненные задания
```

## 3. Почему запрос состоит из двух уровней

Внутренний запрос возвращает подходящих пользователей:

```sql
SELECT user_id
FROM completions
GROUP BY user_id
HAVING ...
```

Внешний запрос считает их:

```sql
SELECT COUNT(*)
FROM (...)
```

После внутреннего запроса результат похож на временную таблицу:

| user_id |
|---:|
| 63 |
| 87 |

Внешний `COUNT(*)` возвращает `2`.

## 4. SELECT COUNT(*)

```sql
SELECT COUNT(*) AS users_count
```

`COUNT(*)` считает строки результата подзапроса.

Поскольку внутренний запрос формирует по одной строке на подходящего пользователя, количество строк равно количеству пользователей.

`AS users_count` задаёт понятное имя итоговой колонке.

## 5. FROM completions

```sql
FROM completions
```

Таблица `completions` содержит только факты выполнения:

- кто выполнил;
- какое задание;
- какой получен score.

Именно она нужна для знаменателя «только выполненные задания».

## 6. GROUP BY user_id

```sql
GROUP BY user_id
```

До группировки каждая строка — отдельное выполнение.

После группировки PostgreSQL рассматривает все строки одного пользователя вместе.

Например:

```text
user_id = 2:
92, 87, 94
```

Теперь агрегатные функции можно вычислить отдельно для пользователя 2.

Без `GROUP BY` среднее считалось бы по всей таблице сразу.

## 7. CASE WHEN

```sql
CASE
    WHEN score >= 90 THEN 1.0
    ELSE 0.0
END
```

Конструкция превращает score в бинарный индикатор:

| score | результат CASE |
|---:|---:|
| 100 | 1.0 |
| 92 | 1.0 |
| 90 | 1.0 |
| 89 | 0.0 |
| 10 | 0.0 |

Успех — `1.0`, неуспех — `0.0`.

Важно использовать `>= 90`, потому что оценка ровно 90 считается успешной.

## 8. Почему AVG даёт долю

Рассмотрим оценки:

```text
95, 93, 80
```

После `CASE`:

```text
1.0, 1.0, 0.0
```

Среднее:

```text
(1 + 1 + 0) / 3 = 0.6667
```

Это 66.67% успешных заданий.

Для оценок:

```text
91, 99
```

получаем:

```text
(1 + 1) / 2 = 1.0
```

То есть 100%.

Поэтому шаблон

```sql
AVG(CASE WHEN условие THEN 1.0 ELSE 0.0 END)
```

удобно использовать для расчёта доли строк, удовлетворяющих условию.

## 9. HAVING

```sql
HAVING AVG(...) >= 0.9
```

`HAVING` фильтрует уже сформированные группы.

После расчёта доли остаются только пользователи с результатом 90% или выше.

## 10. WHERE и HAVING

`WHERE` фильтрует отдельные строки до группировки.

`HAVING` фильтрует группы после расчёта агрегатов.

Неверно:

```sql
WHERE score >= 90
```

Если заранее удалить все неуспешные строки, они исчезнут из знаменателя. Тогда пользователь с оценками `100, 10, 10` будет выглядеть как пользователь со 100% успешностью, потому что останется только строка `100`.

Правильно сохранить все выполнения и учесть неуспешные как нули.

## 11. Почему не COUNT(DISTINCT user_id)

Запрос:

```sql
SELECT COUNT(DISTINCT user_id)
FROM completions
WHERE score >= 90;
```

отвечает на другой вопрос:

```text
Сколько пользователей имеют хотя бы одно успешное задание?
```

Пользователь с оценками:

```text
100, 10, 10, 10
```

будет посчитан, хотя его доля успеха равна 25%.

## 12. Почему не AVG(score)

Запрос:

```sql
HAVING AVG(score) >= 90
```

считает средний балл, а не долю успешных заданий.

Пример:

```text
100, 100, 70
```

Средний score:

```text
90
```

Но успешны только 2 из 3:

```text
66.67%
```

Пользователь не должен проходить исходное условие.

## 13. Почему не нужна таблица assignments

Подсказка говорит учитывать только выполненные задания.

Если делить успешные задания пользователя на общее число строк в `assignments`, мы начнём учитывать задания, которые пользователь вообще не выполнял.

Это изменит смысл метрики.

## 14. Почему не нужна таблица users

В `completions` уже есть `user_id`.

Пользователь без выполненных заданий не должен участвовать в расчёте, поэтому `INNER JOIN users` ничего полезного не добавляет.

Таблица `users` понадобилась бы, если бы нужно было вывести `username`.

## 15. Проверка на данных

В `completions.csv`:

- 100 выполнений;
- 62 пользователя имеют хотя бы одно выполнение;
- 12 строк имеют `score >= 90`;
- только 2 пользователя имеют долю успеха не меньше 90%.

Подходящие:

| user_id | выполнено | успешно | доля |
|---:|---:|---:|---:|
| 63 | 1 | 1 | 1.0 |
| 87 | 1 | 1 | 1.0 |

## 16. Диагностический запрос

Для проверки логики можно выполнить:

```sql
SELECT
    user_id,
    COUNT(*) AS completed_assignments,
    SUM(
        CASE
            WHEN score >= 90 THEN 1
            ELSE 0
        END
    ) AS successful_assignments,
    AVG(
        CASE
            WHEN score >= 90 THEN 1.0
            ELSE 0.0
        END
    ) AS success_share
FROM completions
GROUP BY user_id
ORDER BY user_id;
```

Он покажет расчёт по всем пользователям.

После проверки можно добавить `HAVING` и увидеть только квалифицировавшихся.

## 17. Альтернатива через FILTER

PostgreSQL поддерживает:

```sql
SELECT COUNT(*) AS users_count
FROM (
    SELECT user_id
    FROM completions
    GROUP BY user_id
    HAVING COUNT(*) FILTER (WHERE score >= 90) * 1.0
           / COUNT(*) >= 0.9
) AS qualified_users;
```

Она тоже правильная, но основной вариант через `AVG(CASE ...)` проще для новичка и не требует ручного деления.

## 18. Альтернатива через SUM и COUNT

```sql
SELECT COUNT(*) AS users_count
FROM (
    SELECT user_id
    FROM completions
    GROUP BY user_id
    HAVING SUM(
        CASE
            WHEN score >= 90 THEN 1
            ELSE 0
        END
    ) >= COUNT(*) * 0.9
) AS qualified_users;
```

Логика та же:

```text
число успешных >= число выполненных × 90%
```

## 19. Граничные случаи

### score = 90

Считается успешным.

### Один результат 100

Доля равна 100%. По условию пользователь подходит.

### Пользователь без выполнений

Не присутствует в `completions` и не учитывается.

### score IS NULL

В текущем `CASE` будет считаться неуспехом. Если `NULL` означает «ещё не оценено», в реальном проекте нужно уточнить правило. В учебное решение лишний фильтр добавлять нельзя.

### Повторные попытки

Запрос считает каждую строку выполнением. Если нужна только последняя попытка, необходимы timestamp и дополнительная логика. В условии этого нет.

## 20. Типичные ошибки

1. Использовать `WHERE score >= 90`.
2. Посчитать `COUNT(DISTINCT user_id)` после фильтра.
3. Использовать `AVG(score)`.
4. Написать `score > 90`.
5. Написать долю `> 0.9`.
6. Делить на все задания платформы.
7. Добавить минимальное число попыток, которого нет в задании.
8. Выполнить лишние `JOIN`.
9. Вернуть список пользователей вместо одного числа.
10. Загрузить диагностический запрос вместо финального.
