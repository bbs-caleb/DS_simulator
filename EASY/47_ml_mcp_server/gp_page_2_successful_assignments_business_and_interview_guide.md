# Бизнес-мышление и интервью: Supabase, MCP и SQL

## 1. Что на самом деле проверяет задача

Маленький запрос проверяет не только синтаксис SQL. Он проверяет:

- понимание бизнес-метрики;
- выбор правильного знаменателя;
- группировку по сущности;
- условную агрегацию;
- различие `WHERE` и `HAVING`;
- различие «хотя бы один успех» и «90% успехов»;
- способность проверить результат;
- умение не добавлять лишние условия.

## 2. Бизнес-определение метрики

Корректная формулировка:

> Для каждого пользователя рассматриваются только выполненные им задания. Задание считается успешным при `score >= 90`. Пользователь относится к целевому сегменту, если успешными являются не менее 90% его выполненных заданий.

До написания SQL нужно согласовать именно это определение.

## 3. Какие вопросы задать в реальном проекте

1. Что означает «выполнил»?
2. Может ли `score` быть `NULL`?
3. Учитываются ли повторные попытки?
4. Нужна первая, последняя или лучшая попытка?
5. Достаточно ли одного выполненного задания?
6. Нужен ли минимальный объём наблюдений?
7. Все задания имеют одинаковый вес?
8. Есть ли временной период?
9. Исключаются ли тестовые пользователи?
10. Как обрабатывать удалённые задания?
11. Нужно ли считать отдельно по курсам?
12. Является ли порог 90 включительным?
13. Нужно ли учитывать только опубликованные задания?
14. Что является источником истины?
15. Как часто обновляется показатель?

В учебном задании нельзя самостоятельно добавлять ответы, отсутствующие в условии.

## 4. Почему знаменатель — ключевой продуктовый выбор

Возможные метрики:

```text
успешные / выполненные
```

показывает качество среди попыток;

```text
успешные / все доступные
```

смешивает качество и прогресс;

```text
успешные / начатые
```

может показывать успешность завершения;

```text
пользователи хотя бы с одним успехом
```

показывает наличие единичного сильного результата.

Один и тот же набор данных даёт разные ответы. SQL лишь реализует согласованную бизнес-логику.

## 5. Применение в финтехе

Аналогичная задача:

```text
Сколько клиентов имеют не менее 90%
платежей без просрочки?
```

Строка платежа превращается в:

```text
1 — вовремя
0 — просрочен
```

Среднее по клиенту даёт долю своевременных платежей.

Бизнес-применение:

- risk segmentation;
- предварительный лимит;
- коллекшн;
- retention;
- мониторинг качества портфеля.

Ограничение: клиент с одним своевременным платежом имеет 100%, но это слабое доказательство надёжности.

## 6. Marketplace

Задача:

```text
Сколько продавцов имеют не менее 90%
заказов без отмены по своей вине?
```

Применение:

- seller quality;
- ranking;
- badges;
- buyer protection;
- SLA.

Необходимо учитывать минимальный объём заказов и причину отмены.

## 7. Retail и e-commerce

Задача:

```text
Сколько SKU были в наличии не менее 90%
дней активного ассортимента?
```

Критически важно делить только на дни, когда SKU действительно находился в ассортименте. Дни до листинга нельзя включать в знаменатель.

Это прямой аналог подсказки «учитывайте только выполненные задания».

## 8. Two-sided marketplace

Для курьеров:

```text
Сколько курьеров доставили вовремя
не менее 90% завершённых заказов?
```

Нужно определить:

- что считать вовремя;
- учитывать ли вину ресторана;
- исключать ли отменённые заказы;
- как учитывать сложность маршрутов.

## 9. iGaming и casino

Возможный compliance-пример:

```text
Сколько аккаунтов прошли не менее 90%
применимых KYC-проверок?
```

Знаменатель должен включать только проверки, применимые конкретному пользователю.

MCP-агент не должен получать избыточный доступ к платёжным данным или обходить responsible-gaming ограничения.

## 10. Crypto

Пример:

```text
Сколько валидаторов успешно участвовали
не менее чем в 90% назначенных epochs?
```

Применение:

- reliability ranking;
- delegation;
- SLA;
- monitoring.

## 11. Dating

Пример:

```text
Сколько аккаунтов имеют не менее 90%
взаимодействий без подтверждённых safety incidents?
```

Такие метрики требуют осторожности:

- reports могут быть ошибочными;
- exposure у пользователей разный;
- данные чувствительны;
- автоматическая блокировка опасна.

## 12. Big Tech и ML Platform

Пример:

```text
Сколько ML pipelines имеют не менее 90%
успешных запусков?
```

Тот же SQL-паттерн применяется к:

- CI jobs;
- deployments;
- ETL;
- model training;
- scheduled reports;
- API requests.

## 13. Как связать SQL с MCP

Без MCP:

```text
человек открывает SQL Editor
→ пишет SQL
→ запускает
→ интерпретирует
```

С MCP:

```text
человек задаёт вопрос Cursor
→ модель выбирает Supabase tool
→ MCP server выполняет разрешённую операцию
→ результат возвращается агенту
```

MCP сокращает путь к данным, но не гарантирует правильность метрики.

## 14. Универсальный и доменный tool

### Универсальный

```text
execute_sql(sql)
```

Плюсы:

- гибкий;
- быстро сделать demo.

Минусы:

- произвольные запросы;
- риск утечки;
- тяжёлые операции;
- сложная авторизация;
- prompt injection;
- возможные write-команды.

### Доменный

```text
count_high_success_users(
    score_threshold,
    minimum_success_share
)
```

Плюсы:

- строгая schema;
- безопаснее;
- легче тестировать;
- понятный бизнес-контракт;
- проще вести audit.

Минус — меньшая гибкость.

Для production часто лучше доменные tools или semantic layer.

## 15. Пример контракта tool

```json
{
  "name": "count_high_success_users",
  "inputSchema": {
    "type": "object",
    "properties": {
      "score_threshold": {
        "type": "number",
        "minimum": 0,
        "maximum": 100
      },
      "minimum_success_share": {
        "type": "number",
        "minimum": 0,
        "maximum": 1
      }
    },
    "required": [
      "score_threshold",
      "minimum_success_share"
    ]
  }
}
```

Вызов:

```json
{
  "score_threshold": 90,
  "minimum_success_share": 0.9
}
```

Ответ:

```json
{
  "users_count": 2
}
```

## 16. Безопасность Supabase-интеграции

Нужно обеспечить:

- минимальные права;
- read-only роль;
- отдельный dev-проект;
- RLS;
- ограничения таблиц;
- statement timeout;
- row limit;
- audit;
- redaction секретов;
- ротацию токенов.

Токен нельзя помещать в prompt или репозиторий.

## 17. Что такое RLS

Row Level Security ограничивает строки, доступные конкретному пользователю или роли.

Например:

```text
преподаватель курса A видит completions только курса A
```

Даже корректный SQL не должен обходить tenant или course boundaries.

## 18. Сильный ответ на интервью

> Я сначала формализую метрику. Знаменатель — выполненные задания из `completions`. Для каждой строки строю бинарный индикатор успеха: `1`, если `score >= 90`, иначе `0`. Среднее индикатора по `user_id` даёт долю успехов. Группы фильтрую через `HAVING >= 0.9`, затем внешним `COUNT` считаю пользователей. `WHERE score >= 90` использовать нельзя, потому что он удалит неуспешные строки из знаменателя.

## 19. Типовые вопросы интервьюера

### Почему не нужен JOIN с users?

`user_id` уже есть в `completions`, а пользователи без выполнений не учитываются.

### Почему не нужен assignments?

Подсказка запрещает использовать все задания как знаменатель.

### Почему HAVING?

Фильтруется результат агрегатной функции после группировки.

### Почему не AVG(score)?

Средняя оценка и доля оценок выше порога — разные показатели.

### Что делать с повторными попытками?

Уточнить правило. Для последней попытки нужны время и оконная функция.

### Что делать с одним выполнением?

По исходному условию пользователь проходит. В production можно добавить minimum sample size после согласования с продуктом.
