# Урок 47 — ML: MCP-SERVER
## Страница 2. Интеграция Supabase с Cursor через MCP

# 1. Что происходит на странице

Здесь теория MCP связывается с практикой:

```text
Пользователь
→ Cursor
→ AI-агент
→ MCP-клиент
→ MCP-сервер Supabase
→ PostgreSQL
→ таблицы users, assignments, completions
```

Пользователь может задать вопрос естественным языком, а агент — выбрать инструмент, сформировать SQL, выполнить его через MCP и вернуть результат.

В задании запрос запускается вручную в SQL Editor. Это полезно: прежде чем доверять SQL агенту, специалист должен сам понимать правильную логику запроса.

# 2. Роли компонентов

## Cursor

Cursor выступает AI-host:

- принимает запрос пользователя;
- передаёт задачу модели;
- хранит настройки MCP;
- создаёт MCP-клиент;
- показывает вызовы инструментов;
- возвращает результат.

## MCP-клиент

MCP-клиент внутри Cursor:

- подключается к MCP-серверу;
- узнаёт доступные инструменты;
- отправляет вызов;
- получает ответ.

## MCP-сервер Supabase

Сервер выступает адаптером между AI-приложением и Supabase. Он может:

- получать metadata проекта;
- читать схему;
- выполнять разрешённые SQL-операции;
- возвращать результат агенту.

## Supabase

Supabase предоставляет PostgreSQL, API, аутентификацию и административный интерфейс.

# 3. Структура базы

## users

```text
user_id
username
```

Одна строка — один пользователь.

## assignments

```text
assignment_id
assignment_name
```

Одна строка — одно задание.

## completions

```text
user_id
assignment_id
score
```

Одна строка показывает, что пользователь выполнил конкретное задание и получил оценку.

`completions` — таблица связи между пользователями и заданиями.

# 4. Что означает условие

Есть два порога:

1. Задание успешно, если `score >= 90`.
2. Пользователь подходит, если не менее 90% выполненных им заданий успешны.

Формула:

```text
successful_assignments / completed_assignments >= 0.9
```

Подсказка требует учитывать только выполненные задания. Поэтому знаменатель берётся из `completions`, а не из всей таблицы `assignments`.

# 5. Почему достаточно таблицы completions

Для расчёта нужны:

- идентификатор пользователя;
- факт выполнения;
- score.

Все эти данные есть в `completions`. Таблицы `users` и `assignments` понадобились бы для вывода имени пользователя или названия задания, но не для итогового количества.

# 6. Пример вручную

Пользователь имеет оценки:

```text
92, 87, 94
```

Успешны две оценки: 92 и 94.

```text
2 / 3 = 66.67%
```

Пользователь не проходит порог 90%.

Другой пользователь имеет:

```text
100
```

```text
1 / 1 = 100%
```

Он проходит.

# 7. Бизнесовый смысл

Метрика может использоваться для выделения:

- сильных студентов;
- кандидатов на сертификат;
- пользователей для продвинутого курса;
- участников программы наставничества.

Но в реальном продукте нужно уточнить минимальное число выполненных заданий. Пользователь с одним результатом 100 имеет 100% успешности, хотя данных о нём мало. В учебное решение такой дополнительный фильтр добавлять нельзя, потому что его нет в условии.

# 8. Безопасность интеграции

Supabase access token является секретом. Его нельзя:

- публиковать;
- коммитить в Git;
- вставлять в SQL-файл;
- передавать неизвестным сервисам;
- показывать на скриншотах.

Для аналитического агента предпочтителен read-only доступ и минимальные права.

`SELECT` читает данные. `INSERT`, `UPDATE`, `DELETE`, `DROP`, `TRUNCATE` изменяют или удаляют данные и требуют отдельного контроля.

# 9. Техническая схема решения

```text
completions
→ GROUP BY user_id
→ превратить score >= 90 в 1, остальные в 0
→ AVG получить долю успехов
→ HAVING оставить долю >= 0.9
→ COUNT(*) посчитать пользователей
```

На приложенном наборе данных результат равен:

```text
2
```

Подходят `user_id = 63` и `user_id = 87`.
