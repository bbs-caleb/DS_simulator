# DA: SESSION ANALYSIS

# STEP 4. Средняя длина сессии

## Готовое решение

Готовый ClickHouse-запрос находится в файле:

```text
gp_step_4_avg_session_duration.sql
```

Он возвращает ровно два требуемых поля:

```text
day
avg_session_duration_seconds
```

Где:

- `day` — дата начала сессии с типом `Date`;
- `avg_session_duration_seconds` — средняя продолжительность ненулевых сессий, начавшихся в этот день;
- сессии из одного события, у которых продолжительность равна `0`, исключаются из среднего.

---

# 1. Что требуется посчитать

В исходной таблице:

```text
default.karpovexpress_session_log
```

одна строка соответствует одному событию.

Пример исходных данных:

| timestamp | cookie_id | event_type |
|---|---|---|
| 2024-02-01 10:00:00 | A | view_main |
| 2024-02-01 10:05:00 | A | search |
| 2024-02-01 10:20:00 | A | view_product |
| 2024-02-01 11:10:00 | A | view_cart |

Но нужная метрика считается не на уровне отдельных событий.

Сначала необходимо восстановить сессии.

Для каждой сессии нужно найти:

```text
время начала;
время окончания;
продолжительность в секундах.
```

После этого необходимо:

1. исключить сессии с продолжительностью `0`;
2. объединить остальные сессии по дню начала;
3. вычислить среднюю продолжительность.

---

# 2. Бизнесовое определение сессии

События относятся к одной сессии, если:

1. у них одинаковый `cookie_id`;
2. между двумя соседними событиями прошло не более 30 минут.

Новая сессия начинается, если:

```text
это первое событие данного cookie_id
```

или:

```text
после предыдущего события прошло строго больше 30 минут.
```

Критически важная граница:

| Временной разрыв | Результат |
|---:|---|
| 29 минут 59 секунд | та же сессия |
| ровно 30 минут | та же сессия |
| 30 минут 1 секунда | новая сессия |
| 31 минута | новая сессия |

В секундах:

```text
30 минут = 30 × 60 = 1800 секунд
```

Поэтому в запросе используется:

```sql
> 1800
```

а не:

```sql
>= 1800
```

---

# 3. Что такое продолжительность сессии

Для одной сессии:

```text
session_duration =
    timestamp последнего события
    -
    timestamp первого события
```

Пример:

```text
начало: 10:42:00
конец: 11:15:00
```

Разница:

```text
33 минуты
```

В секундах:

```text
33 × 60 = 1980 секунд
```

---

# 4. Почему сессия из одного события имеет длину 0

Рассмотрим сессию:

```text
09:53 — view_product
```

В ней только одно событие.

Следовательно:

```text
первое событие = 09:53
последнее событие = 09:53
```

Продолжительность:

```text
09:53 - 09:53 = 0 секунд
```

Это не обязательно означает, что человек действительно провёл на странице ноль секунд.

Он мог открыть карточку товара и читать её несколько минут.

Однако в логах нет следующего события, по которому можно было бы оценить время.

Поэтому технически наблюдаемая продолжительность равна нулю.

По условию задачи такие сессии не должны участвовать в среднем.

В запросе это реализуется условием:

```sql
WHERE session_duration_seconds > 0
```

---

# 5. Главная архитектура запроса

Запрос разделён на несколько понятных уровней:

```text
events_with_previous
    ↓
events_with_session_start
    ↓
events_with_session_number
    ↓
sessions
    ↓
финальная дневная метрика
```

Это можно прочитать так:

1. Для каждого события найти предыдущее событие.
2. Определить, начинается ли новая сессия.
3. Присвоить каждому событию номер сессии.
4. Собрать события до уровня одной строки на сессию.
5. Исключить нулевые сессии и посчитать среднее по дням.

---

# 6. Полный SQL-запрос

```sql
WITH events_with_previous AS (
    SELECT
        timestamp,
        cookie_id,
        lagInFrame(timestamp, 1) OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS previous_timestamp,
        row_number() OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
        ) AS event_number
    FROM default.karpovexpress_session_log
),
events_with_session_start AS (
    SELECT
        timestamp,
        cookie_id,
        if(
            event_number = 1
            OR dateDiff('second', previous_timestamp, timestamp) > 1800,
            1,
            0
        ) AS is_new_session
    FROM events_with_previous
),
events_with_session_number AS (
    SELECT
        timestamp,
        cookie_id,
        sum(is_new_session) OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS session_number
    FROM events_with_session_start
),
sessions AS (
    SELECT
        cookie_id,
        session_number,
        min(timestamp) AS session_start,
        max(timestamp) AS session_end,
        dateDiff(
            'second',
            min(timestamp),
            max(timestamp)
        ) AS session_duration_seconds
    FROM events_with_session_number
    GROUP BY
        cookie_id,
        session_number
)
SELECT
    toDate(session_start) AS day,
    avg(session_duration_seconds) AS avg_session_duration_seconds
FROM sessions
WHERE session_duration_seconds > 0
GROUP BY day
ORDER BY day;
```

---

# 7. Первый CTE: events_with_previous

```sql
WITH events_with_previous AS (
```

CTE — это именованный промежуточный результат.

Название можно прочитать как:

> События, к которым добавлено время предыдущего события.

После этого CTE одна строка всё ещё соответствует одному исходному событию.

---

## 7.1. Поле timestamp

```sql
timestamp,
```

Это время текущего события.

Оно понадобится для:

- сортировки;
- сравнения с предыдущим событием;
- определения начала и конца сессии;
- назначения дня сессии.

---

## 7.2. Поле cookie_id

```sql
cookie_id,
```

Сессии строятся отдельно для каждого cookie.

Например:

| timestamp | cookie_id |
|---|---|
| 10:00 | A |
| 10:05 | B |
| 10:10 | A |

Для события cookie `A` в `10:10` предыдущим событием должно быть событие cookie `A` в `10:00`.

Событие cookie `B` в `10:05` не должно влиять на сессию cookie `A`.

---

# 8. lagInFrame

```sql
lagInFrame(timestamp, 1) OVER (
```

Функция `lagInFrame` возвращает значение из предыдущей строки оконной последовательности.

Выражение:

```sql
lagInFrame(timestamp, 1)
```

означает:

```text
возьми timestamp на одну строку назад.
```

---

## 8.1. PARTITION BY cookie_id

```sql
PARTITION BY cookie_id
```

Разделяет все строки на независимые группы:

```text
cookie A;
cookie B;
cookie C;
...
```

Внутри каждой группы строится отдельная временная последовательность.

Без этого события разных пользователей могли бы сравниваться между собой.

---

## 8.2. ORDER BY timestamp

```sql
ORDER BY timestamp
```

Располагает события каждого cookie по времени.

Например, физически строки могли храниться так:

```text
10:20
10:00
10:10
```

Для анализа они должны быть логически упорядочены:

```text
10:00
10:10
10:20
```

Только тогда можно правильно определить предыдущее событие.

---

## 8.3. Оконная рамка

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
```

Она означает:

```text
использовать все строки cookie:
от первой до последней.
```

`lagInFrame` в ClickHouse работает с учётом оконной рамки.

Здесь явно задана полная рамка, чтобы предыдущая строка искалась по всей последовательности cookie.

---

## 8.4. previous_timestamp

```sql
AS previous_timestamp
```

Создаётся поле со временем предыдущего события.

Пример:

| timestamp | previous_timestamp |
|---|---|
| 10:00 | нет реального предыдущего события |
| 10:05 | 10:00 |
| 10:20 | 10:05 |
| 11:10 | 10:20 |

Для первой строки реального предыдущего события нет.

Поэтому дополнительно используется `row_number()`.

---

# 9. row_number

```sql
row_number() OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
) AS event_number
```

Функция нумерует события отдельно для каждого cookie.

Пример:

| cookie_id | timestamp | event_number |
|---|---|---:|
| A | 10:00 | 1 |
| A | 10:05 | 2 |
| A | 10:20 | 3 |
| B | 09:00 | 1 |
| B | 09:15 | 2 |

Первое событие каждого cookie получает:

```text
event_number = 1
```

Оно обязательно начинает первую сессию.

---

# 10. Второй CTE: events_with_session_start

```sql
events_with_session_start AS (
```

На этом уровне для каждого события создаётся флаг:

```text
1 — начинается новая сессия;
0 — продолжается текущая сессия.
```

---

## 10.1. Первое событие cookie

```sql
event_number = 1
```

У первого события нет предыдущей активности.

Следовательно, оно всегда начинает первую сессию.

---

## 10.2. Разрыв больше 30 минут

```sql
dateDiff('second', previous_timestamp, timestamp) > 1800
```

Разбираем по частям.

### dateDiff

Функция вычисляет временную разницу.

### 'second'

Разница считается в секундах.

### previous_timestamp

Время предыдущего события.

### timestamp

Время текущего события.

### > 1800

Новая сессия создаётся только после разрыва строго больше 30 минут.

---

## 10.3. Функция if

```sql
if(
    условие,
    1,
    0
)
```

Если условие истинно:

```text
is_new_session = 1
```

Иначе:

```text
is_new_session = 0
```

Полное условие:

```sql
event_number = 1
OR dateDiff('second', previous_timestamp, timestamp) > 1800
```

То есть новая сессия начинается, если:

- событие первое;
- либо пользователь отсутствовал более 30 минут.

---

# 11. Пример флагов

События:

```text
09:53
10:42
10:53
11:02
11:15
```

Получаем:

| timestamp | previous_timestamp | gap | is_new_session |
|---|---|---:|---:|
| 09:53 | — | — | 1 |
| 10:42 | 09:53 | 49 минут | 1 |
| 10:53 | 10:42 | 11 минут | 0 |
| 11:02 | 10:53 | 9 минут | 0 |
| 11:15 | 11:02 | 13 минут | 0 |

Последовательность флагов:

```text
1, 1, 0, 0, 0
```

---

# 12. Третий CTE: events_with_session_number

```sql
events_with_session_number AS (
```

Теперь каждому событию нужно присвоить номер сессии.

Для этого используется накопительная сумма.

---

# 13. Накопительная сумма

```sql
sum(is_new_session) OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
```

Если флаги равны:

```text
1, 1, 0, 0, 0
```

то накопительная сумма будет:

```text
1, 2, 2, 2, 2
```

Получаем:

| timestamp | is_new_session | session_number |
|---|---:|---:|
| 09:53 | 1 | 1 |
| 10:42 | 1 | 2 |
| 10:53 | 0 | 2 |
| 11:02 | 0 | 2 |
| 11:15 | 0 | 2 |

---

## 13.1. Почему это работает

Каждое начало сессии добавляет единицу.

Пока идут события той же сессии:

```text
is_new_session = 0
```

Номер не меняется.

Когда начинается следующая сессия:

```text
is_new_session = 1
```

Номер увеличивается.

---

## 13.2. Рамка до CURRENT ROW

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

Означает:

```text
суммировать флаги от первого события cookie
до текущего события включительно.
```

Именно это создаёт накопительную, а не общую сумму.

---

# 14. Почему session_number не уникален сам по себе

У cookie `A` может быть:

```text
session_number = 1
```

У cookie `B` тоже может быть:

```text
session_number = 1
```

Поэтому уникальная сессия определяется парой:

```text
(cookie_id, session_number)
```

Именно по этим двум полям выполняется группировка.

---

# 15. Четвёртый CTE: sessions

```sql
sessions AS (
```

На этом этапе мы переходим:

```text
с уровня событий
к уровню сессий.
```

До этого одна строка была одним событием.

После `GROUP BY` одна строка будет одной сессией.

---

# 16. Группировка сессии

```sql
GROUP BY
    cookie_id,
    session_number
```

Все события одного cookie с одинаковым номером сессии объединяются.

---

# 17. Начало сессии

```sql
min(timestamp) AS session_start
```

Минимальное время внутри сессии — это её первое событие.

Пример:

```text
10:42
10:53
11:02
11:15
```

Минимум:

```text
10:42
```

---

# 18. Конец сессии

```sql
max(timestamp) AS session_end
```

Максимальное время внутри сессии — это последнее зафиксированное событие.

Для примера:

```text
11:15
```

Важно:

Это последнее наблюдаемое событие, а не гарантированный момент физического закрытия сайта.

---

# 19. Расчёт продолжительности

```sql
dateDiff(
    'second',
    min(timestamp),
    max(timestamp)
) AS session_duration_seconds
```

Разница считается:

```text
от первого события
до последнего события.
```

Единица:

```text
секунды.
```

Пример:

```text
10:42 → 11:15
```

Получаем:

```text
1980 секунд.
```

---

# 20. Почему нельзя считать среднее раньше

Нельзя вычислять среднюю длительность до формирования сессий.

На event-level нет отдельной продолжительности сессии.

Неправильно усреднять интервалы между событиями.

Пример:

```text
10:00
10:05
10:20
```

Разрывы:

```text
5 минут;
15 минут.
```

Средний разрыв:

```text
10 минут.
```

Но продолжительность всей сессии:

```text
20 минут.
```

Это разные показатели.

---

# 21. Финальный SELECT

```sql
SELECT
    toDate(session_start) AS day,
    avg(session_duration_seconds) AS avg_session_duration_seconds
```

Здесь рассчитывается итоговая дневная метрика.

---

# 22. Почему day определяется по session_start

```sql
toDate(session_start) AS day
```

Сессия относится ко дню своего начала.

Рассмотрим сессию:

```text
2024-02-01 23:55
2024-02-02 00:10
```

Разрыв равен 15 минутам, поэтому это одна сессия.

Её начало:

```text
2024-02-01 23:55
```

Следовательно:

```text
day = 2024-02-01
```

Мы не разрываем сессию в полночь.

---

# 23. Почему day остаётся Date

Функция:

```sql
toDate(session_start)
```

возвращает тип:

```text
Date
```

Мы не используем:

```sql
formatDateTime(...)
```

потому что это могло бы превратить дату в строку.

Условие прямо требует оставить поле с типом `Date`.

---

# 24. Исключение нулевых сессий

```sql
WHERE session_duration_seconds > 0
```

В финальную среднюю попадают только сессии, где:

```text
последнее событие произошло позже первого.
```

Исключаются:

```text
session_duration_seconds = 0
```

То есть сессии из одного события, а также редкие случаи нескольких событий с полностью одинаковым timestamp.

---

## 24.1. Почему фильтр находится после построения сессий

До агрегации невозможно надёжно определить длительность сессии.

Сначала нужно:

1. собрать все события одной сессии;
2. найти минимум и максимум;
3. вычислить duration;
4. затем применить фильтр.

---

# 25. Вычисление среднего

```sql
avg(session_duration_seconds)
```

Пример ненулевых сессий одного дня:

```text
600 секунд;
900 секунд;
1200 секунд.
```

Среднее:

```text
(600 + 900 + 1200) / 3 = 900 секунд.
```

Функция `avg` возвращает вещественное число.

Это соответствует условию:

```text
avg_session_duration_seconds — вещественное число.
```

---

# 26. GROUP BY day

```sql
GROUP BY day
```

Все ненулевые сессии объединяются по дню начала.

Результат:

| day | avg_session_duration_seconds |
|---|---:|
| 2024-02-01 | 806.61 |
| 2024-02-02 | 765.58 |
| 2024-02-03 | 795.09 |

---

# 27. ORDER BY day

```sql
ORDER BY day
```

Возвращает дни в хронологическом порядке.

SQL без `ORDER BY` не гарантирует порядок строк.

---

# 28. Полный ручной пример

Исходные события:

| cookie | timestamp |
|---|---|
| A | 2024-02-01 09:00 |
| A | 2024-02-01 09:10 |
| A | 2024-02-01 10:00 |
| B | 2024-02-01 11:00 |
| B | 2024-02-01 11:20 |
| C | 2024-02-01 12:00 |

Разберём cookie `A`.

```text
09:00 — начало session A1;
09:10 — gap 10 минут, остаётся A1;
10:00 — gap 50 минут, начинается A2.
```

Сессии:

```text
A1: 09:00–09:10 = 600 секунд;
A2: 10:00–10:00 = 0 секунд.
```

Cookie `B`:

```text
11:00–11:20 = 1200 секунд.
```

Cookie `C`:

```text
12:00–12:00 = 0 секунд.
```

До фильтра:

```text
600;
0;
1200;
0.
```

После исключения нулевых:

```text
600;
1200.
```

Среднее:

```text
(600 + 1200) / 2 = 900 секунд.
```

Итог:

| day | avg_session_duration_seconds |
|---|---:|
| 2024-02-01 | 900.0 |

# 29. Граничный случай: ровно 30 минут

События:

```text
10:00
10:30
```

Разрыв:

```text
1800 секунд.
```

Условие новой сессии:

```sql
> 1800
```

Ложно.

Следовательно:

```text
это одна сессия;
duration = 1800 секунд.
```

Она участвует в среднем.

---

# 30. Граничный случай: 30 минут 1 секунда

События:

```text
10:00:00
10:30:01
```

Разрыв:

```text
1801 секунда.
```

Это две отдельные сессии.

Каждая содержит одно событие.

Продолжительности:

```text
0;
0.
```

Обе исключаются из средней длительности.

---

# 31. Длинная активная сессия

События:

```text
10:00
10:25
10:50
11:15
```

Между соседними событиями каждый раз проходит 25 минут.

Поэтому это одна сессия.

Общая длительность:

```text
11:15 - 10:00 = 75 минут = 4500 секунд.
```

Порог в 30 минут ограничивает бездействие между соседними событиями, а не максимальную общую продолжительность сессии.

---

# 32. Сессия через полночь

События:

```text
2024-02-01 23:50
2024-02-02 00:05
```

Gap:

```text
15 минут.
```

Это одна сессия.

Продолжительность:

```text
900 секунд.
```

День:

```text
2024-02-01
```

потому что сессия началась 1 февраля.

---

# 33. Почему нельзя PARTITION BY cookie_id, day

Неправильный вариант:

```sql
PARTITION BY cookie_id, toDate(timestamp)
```

Он принудительно разрывает последовательность в полночь.

События:

```text
23:59
00:01
```

имеют разрыв две минуты, но попадут в разные partitions.

Это нарушает бизнес-правило.

Правильно:

```sql
PARTITION BY cookie_id
```

А день вычислять только после формирования сессии.

---

# 34. Почему нельзя группировать только по cookie_id

Один cookie может иметь много сессий.

Пример:

```text
09:00
09:10
15:00
15:20
```

Это две сессии одного cookie.

Если сделать:

```sql
GROUP BY cookie_id
```

получится:

```text
09:00–15:20
```

То есть ложная продолжительность более шести часов.

Поэтому сначала нужен `session_number`.

---

# 35. Почему нельзя использовать user_id

`user_id` может быть `NULL`.

Пользователь может авторизоваться внутри сессии.

Пример:

| timestamp | cookie_id | user_id |
|---|---|---:|
| 10:00 | A | NULL |
| 10:10 | A | 123 |

По `cookie_id` это одна сессия.

По `user_id` данные разорвутся.

Условие задачи прямо определяет сессии через `cookie_id`.

---

# 36. Почему нельзя считать duration как 30 минут для одиночной сессии

Иногда аналитики искусственно присваивают последнему событию таймаут:

```text
одно событие → 30 минут.
```

Но это другая методика.

В условии KarpovExpress сказано:

```text
сессия из одного события имеет длину 0
и исключается из среднего.
```

Поэтому мы не добавляем 30 минут искусственно.

---

# 37. Почему не используется COUNT событий

Количество событий помогает определить одно-событийную сессию, но для текущего фильтра достаточно duration.

Если:

```text
duration > 0
```

то в сессии есть временной интервал между первым и последним событием.

Можно было бы также считать:

```sql
count() AS events_count
```

и фильтровать:

```text
events_count > 1
```

Но это не всегда абсолютно то же самое.

Например, два события могут иметь одинаковый timestamp:

```text
10:00:00
10:00:00
```

Тогда:

```text
events_count = 2;
duration = 0.
```

Условие задачи говорит исключать именно сессии с нулевой длительностью.

Поэтому фильтр:

```sql
session_duration_seconds > 0
```

точнее соответствует формулировке.

---

# 38. Почему avg возвращает вещественное число

Даже если каждая продолжительность является целым числом секунд, среднее может быть дробным.

Пример:

```text
800;
813.
```

Среднее:

```text
806.5.
```

Поэтому результат должен быть вещественным.

`avg()` в ClickHouse возвращает подходящий числовой тип.

---

# 39. Почему не нужно ROUND

В условии не сказано округлять результат до двух знаков.

Пример показывает два знака, но это может быть формат отображения в Redash.

Добавление:

```sql
round(avg(...), 2)
```

может изменить точное значение и потенциально помешать автоматической проверке.

Поэтому безопаснее вернуть настоящий результат `avg()` без искусственного округления.

---

# 40. Типичные ошибки

## Ошибка 1. Усреднить события, а не сессии

Среднее должно считаться после формирования одной строки на сессию.

## Ошибка 2. Использовать >= 1800

Ровно 30 минут ошибочно создадут новую сессию.

## Ошибка 3. Удалить одиночные события до sessionization

Нельзя заранее понять принадлежность строки к сессии без анализа соседей.

## Ошибка 4. Группировать по cookie_id без session_number

Несколько сессий cookie склеятся.

## Ошибка 5. Разрывать по дате

Сессии через полночь будут рассчитаны неправильно.

## Ошибка 6. Включать duration = 0

Среднее будет занижено и нарушит условие.

## Ошибка 7. Форматировать day строкой

Нужен тип `Date`, а не текст.

## Ошибка 8. Округлять без требования

Проверка может ожидать исходное вещественное значение.

## Ошибка 9. Считать продолжительность как сумму gaps без формирования сессии

Хотя сумма gaps внутри корректной сессии математически совпадает с `max-min`, такой код сложнее и легче сломать.

## Ошибка 10. Считать от первого события до первого разрыва неправильно

Порог применяется к каждой паре соседних событий.

# 42. Почему запрос минимальный

Используются только необходимые столбцы:

```text
timestamp;
cookie_id.
```

Для расчёта не нужны:

- `user_id`;
- `page_url`;
- `page_title`;
- `browser`;
- `country`;
- `search_keyword`;
- `traffic_source`;
- `actions`.

Используемые операции:

- `lagInFrame`;
- `row_number`;
- `dateDiff`;
- оконная `sum`;
- `min`;
- `max`;
- `avg`;
- `toDate`;
- `GROUP BY`;
- `ORDER BY`.

Нет лишних:

- JOIN;
- массивов;
- строкового форматирования;
- искусственных session IDs;
- дополнительных таблиц.

# 43. Производительность

Самая тяжёлая часть — сортировка событий:

```sql
PARTITION BY cookie_id
ORDER BY timestamp
```

Она нужна для восстановления временной последовательности.

Запрос не выбирает лишние поля, что уменьшает объём обрабатываемых данных.

В production сессии часто заранее материализуют в отдельную таблицу, но для учебной задачи корректный оконный запрос является естественным решением.

# 44. Проверки результата

После выполнения полезно проверить:

## Продолжительность неотрицательна

```text
session_duration_seconds >= 0
```

## В финальной средней нет нулевых сессий

Фильтр:

```sql
WHERE session_duration_seconds > 0
```

## День имеет тип Date

Используется:

```sql
toDate(session_start)
```

## Каждая сессия принадлежит одному cookie

Группировка включает `cookie_id`.

## Ровно 30 минут не создают новую сессию

Используется:

```sql
> 1800
```

## Результат отсортирован

Используется:

```sql
ORDER BY day
```

# 45. Бизнесовая интерпретация

После получения временного ряда можно сравнить среднюю продолжительность:

```text
до 17 февраля;
после 17 февраля.
```

Но рост длительности не всегда означает улучшение.

Рост может означать:

- пользователи глубже изучают товары;
- выдача стала интереснее;
- больше сравнивают варианты.

Но также:

- нужный товар найти сложнее;
- приходится открывать больше страниц;
- поисковая выдача менее релевантна;
- интерфейс создаёт лишнее усилие.

Снижение длительности тоже неоднозначно.

Положительная причина:

```text
пользователь быстрее нашёл товар и совершил целевое действие.
```

Негативная причина:

```text
пользователь сразу ушёл, потому что выдача стала хуже.
```

Поэтому среднюю длительность нужно анализировать вместе с:

- search conversion;
- добавлениями в корзину;
- покупками;
- числом поисковых reformulations;
- CTR выдачи;
- долей пустых результатов;
- временем до покупки.

---

# 46. Контрольное объяснение совсем простыми словами

Представим, что для каждого cookie мы выписали все клики по времени.

Если между кликами прошло не больше 30 минут, они относятся к одному посещению.

Если прошло больше 30 минут, начинается новое посещение.

Затем для каждого посещения:

```text
берём время последнего клика;
вычитаем время первого клика.
```

Если получилось `0`, такое посещение не берём.

Остальные продолжительности складываем по каждому дню и делим на число подходящих сессий.

# 47. Итоговая логика

```text
1. Разделить события по cookie_id.
2. Отсортировать их по timestamp.
3. Найти предыдущее событие.
4. Отметить начало новой сессии.
5. Накопительной суммой создать номер сессии.
6. Сгруппировать события по cookie_id и номеру сессии.
7. Найти MIN и MAX timestamp.
8. Вычислить разницу в секундах.
9. Исключить duration = 0.
10. Посчитать AVG по дню начала сессии.
```

Итоговые поля:

```text
day
avg_session_duration_seconds
```

Ключевое условие новой сессии:

```sql
dateDiff('second', previous_timestamp, timestamp) > 1800
```

Ключевой фильтр:

```sql
WHERE session_duration_seconds > 0
```

Ключевая итоговая агрегация:

```sql
avg(session_duration_seconds)
```
