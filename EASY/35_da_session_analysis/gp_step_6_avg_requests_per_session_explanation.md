# DA: SESSION ANALYSIS

# STEP 6. Среднее количество запросов за сессию

## Готовое решение

Готовый ClickHouse-запрос находится в файле:

```text
gp_step_6_avg_requests_per_session.sql
```

Он возвращает ровно два требуемых поля:

```text
day
avg_requests_per_session
```

Где:

- `day` — день начала сессии с типом `Date`;
- `avg_requests_per_session` — среднее количество событий в одной сессии;
- одно-событийные сессии учитываются;
- результат `avg()` является вещественным числом.

---

# 1. Что требуется посчитать

В исходной таблице:

```text
default.karpovexpress_session_log
```

одна строка — одно событие пользователя.

Например:

| timestamp | cookie_id | event_type |
|---|---|---|
| 10:00 | A | view_main |
| 10:05 | A | search |
| 10:08 | A | view_product |
| 11:00 | A | view_cart |

Нужно:

1. восстановить сессии;
2. посчитать число событий в каждой сессии;
3. определить день начала каждой сессии;
4. вычислить среднее число событий по дням.

---

# 2. Почему «запросы» здесь означают события

В условии написано:

```text
среднее количество запросов (событий) за одну сессию
```

Следовательно, считается количество строк лога внутри сессии.

Мы не считаем только заполненные `search_keyword`.

Мы не считаем только события типа `search`.

Мы считаем все события, попавшие в сессию:

```text
view_main;
search;
view_product;
add_to_cart;
view_cart;
и другие.
```

Поэтому используется:

```sql
count()
```

---

# 3. Нужно ли исключать сессии из одного события

Нет.

В предыдущем задании сессии длительностью `0` исключались только из расчёта средней продолжительности.

В текущем задании такого требования нет.

Сессия из одного события имеет:

```text
requests_per_session = 1
```

Она является настоящей сессией по правилам задания и должна участвовать в среднем.

Пример:

```text
сессия 1 — 1 событие;
сессия 2 — 3 события;
сессия 3 — 5 событий.
```

Среднее:

```text
(1 + 3 + 5) / 3 = 3.
```

Если удалить одно-событийную сессию, получится:

```text
(3 + 5) / 2 = 4.
```

Это уже другая метрика.

---

# 4. Правило формирования сессий

События относятся к одной сессии, если:

1. у них одинаковый `cookie_id`;
2. между соседними событиями прошло не более 30 минут.

Новая сессия начинается, когда:

```text
событие является первым для cookie_id
```

или:

```text
между предыдущим и текущим событием прошло больше 30 минут.
```

В секундах:

```text
30 минут = 1800 секунд.
```

Поэтому условие новой сессии:

```sql
dateDiff('second', previous_timestamp, timestamp) > 1800
```

Ровно 30 минут остаются в одной сессии.

---

# 5. Общая архитектура решения

Запрос состоит из четырёх логических уровней:

```text
events_with_previous
    ↓
events_with_session_start
    ↓
events_with_session_number
    ↓
sessions
    ↓
среднее по дням
```

Расшифровка:

1. Для каждого события найти предыдущее.
2. Отметить начало новой сессии.
3. Присвоить номер сессии.
4. Посчитать количество событий внутри каждой сессии.
5. Усреднить количества по дню начала сессии.

---

# 6. Полный запрос

```sql
WITH events_with_previous AS (
    SELECT
        timestamp,
        cookie_id,
        lagInFrame(timestamp, 1) OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS previous_timestamp,
        row_number() OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
        ) AS event_number
    FROM default.karpovexpress_session_log
),
events_with_session_start AS (
    SELECT
        timestamp,
        cookie_id,
        if(
            event_number = 1
            OR dateDiff('second', previous_timestamp, timestamp) > 1800,
            1,
            0
        ) AS is_new_session
    FROM events_with_previous
),
events_with_session_number AS (
    SELECT
        timestamp,
        cookie_id,
        sum(is_new_session) OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS session_number
    FROM events_with_session_start
),
sessions AS (
    SELECT
        cookie_id,
        session_number,
        toDate(min(timestamp)) AS day,
        count() AS requests_per_session
    FROM events_with_session_number
    GROUP BY
        cookie_id,
        session_number
)
SELECT
    day,
    avg(requests_per_session) AS avg_requests_per_session
FROM sessions
GROUP BY day
ORDER BY day;
```

# 7. Первый CTE: events_with_previous

```sql
WITH events_with_previous AS (
```

CTE — это именованный промежуточный результат.

Название означает:

```text
события с добавленным временем предыдущего события.
```

На этом этапе:

```text
одна строка = одно событие.
```

---

## 7.1. timestamp

```sql
timestamp,
```

Нужен для:

- сортировки событий;
- расчёта временного разрыва;
- определения начала сессии;
- назначения дня.

---

## 7.2. cookie_id

```sql
cookie_id,
```

Сессии формируются отдельно для каждого cookie.

События разных cookie нельзя объединять.

Пример:

| timestamp | cookie_id |
|---|---|
| 10:00 | A |
| 10:05 | B |
| 10:10 | A |

Для события cookie `A` в `10:10` предыдущим событием является `10:00`, а не событие cookie `B` в `10:05`.

---

# 8. lagInFrame

```sql
lagInFrame(timestamp, 1) OVER (
```

Функция возвращает значение `timestamp` из предыдущей строки.

Аргументы:

```sql
timestamp
```

— столбец, значение которого нужно получить;

```sql
1
```

— сдвиг на одну строку назад.

---

## 8.1. PARTITION BY cookie_id

```sql
PARTITION BY cookie_id
```

Разделяет таблицу на независимые последовательности:

```text
все события cookie A;
все события cookie B;
все события cookie C.
```

---

## 8.2. ORDER BY timestamp

```sql
ORDER BY timestamp
```

Сортирует события каждого cookie по времени.

Без сортировки невозможно правильно определить предыдущее событие.

---

## 8.3. Оконная рамка

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
```

Означает:

```text
рассматривать все строки cookie
от первой до последней.
```

Это важно для корректной работы `lagInFrame` в ClickHouse.

---

## 8.4. previous_timestamp

```sql
AS previous_timestamp
```

Создаётся новое поле со временем предыдущего события.

Пример:

| timestamp | previous_timestamp |
|---|---|
| 10:00 | нет предыдущего |
| 10:05 | 10:00 |
| 10:08 | 10:05 |
| 11:00 | 10:08 |

---

# 9. row_number

```sql
row_number() OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
) AS event_number
```

Нумерует события каждого cookie:

| cookie_id | timestamp | event_number |
|---|---|---:|
| A | 10:00 | 1 |
| A | 10:05 | 2 |
| A | 10:08 | 3 |
| B | 09:00 | 1 |

Первое событие получает:

```text
event_number = 1.
```

Оно всегда начинает первую сессию.

```sql

```

Здесь для каждого события создаётся флаг:

---

## 10.1. Первое условие

```sql
event_number = 1
```

Первое событие каждого cookie обязательно является началом сессии.

---

## 10.2. Второе условие

```sql
dateDiff('second', previous_timestamp, timestamp) > 1800
```

Новая сессия создаётся, если разрыв больше 1800 секунд.

Граница:

| Разрыв | Новая сессия |
|---:|---|
| 1799 секунд | нет |
| 1800 секунд | нет |
| 1801 секунда | да |

---

## 10.3. OR

```sql
event_number = 1
OR gap > 1800
```

Достаточно выполнения одного условия.

---

## 10.4. if

```sql
if(
    условие,
    1,
    0
)
```

Если условие истинно:

```text
is_new_session = 1.
```

Иначе:

```text
is_new_session = 0.
```

# 12. Третий CTE: events_with_session_number

```sql
events_with_session_number AS (
```

Здесь каждому событию присваивается номер сессии.

Для этого используется накопительная сумма.

---

# 13. Накопительная сумма

```sql
sum(is_new_session) OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
```

Для флагов:

```text
1, 1, 0, 0, 0
```

накопительная сумма:

```text
1, 2, 2, 2, 2
```

Получаем:

| timestamp | is_new_session | session_number |
|---|---:|---:|
| 09:53 | 1 | 1 |
| 10:42 | 1 | 2 |
| 10:53 | 0 | 2 |
| 11:02 | 0 | 2 |
| 11:15 | 0 | 2 |

---

## 13.1. Почему это работает

Каждое начало новой сессии добавляет единицу.

События внутри текущей сессии имеют флаг `0`, поэтому номер не меняется.

---

## 13.2. Оконная рамка

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

означает:

```text
суммировать от первого события cookie
до текущей строки включительно.
```

Именно поэтому сумма является накопительной.

---

# 14. Почему номер сессии нужно использовать вместе с cookie_id

У разных cookie могут быть одинаковые номера:

```text
cookie A, session 1;
cookie B, session 1.
```

Поэтому уникальная сессия определяется парой:

```text
(cookie_id, session_number).
```

# 16. Группировка


# 17. Определение дня

```sql
toDate(min(timestamp)) AS day
```

`min(timestamp)` — первое событие сессии.

`toDate(...)` оставляет только дату.

Пример:

```text
2024-02-01 23:55:00
```

становится:

```text
2024-02-01
```

Тип остаётся `Date`.

---

## 17.1. Почему используется день начала

Сессия может перейти через полночь:

```text
23:55
00:10
```

Разрыв равен 15 минутам, значит это одна сессия.

Она относится ко дню начала:

```text
день = дата 23:55.
```

---

# 18. Подсчёт событий в сессии

```sql
count() AS requests_per_session
```

`count()` считает количество строк внутри группы.

Поскольку одна строка исходной таблицы — одно событие, результат равен количеству событий в сессии.

Пример:

```text
10:00 view_main;
10:05 search;
10:08 view_product;
10:12 add_to_cart.
```

В сессии:

```text
4 события.
```

Значит:

```text
requests_per_session = 4.
```

---

# 19. Почему используется count(), а не count(search_keyword)

`search_keyword` может быть `NULL`.

Если использовать:

```sql
count(search_keyword)
```

будут посчитаны только строки, где поисковая фраза заполнена.

Но задача требует количество событий.

Поэтому используется:

```sql
count()
```

---

# 20. Финальный SELECT

```sql
SELECT
    day,
    avg(requests_per_session) AS avg_requests_per_session
```

После CTE `sessions` каждая строка соответствует одной сессии.

Теперь можно безопасно усреднить число событий.

---

# 21. Почему avg возвращает вещественное число

Пример сессий одного дня:

```text
1 событие;
2 события;
4 события.
```

Среднее:

```text
(1 + 2 + 4) / 3 = 2.333...
```

Поэтому итог не обязан быть целым.

`avg()` возвращает вещественное число, что соответствует условию.

---

```sql
GROUP BY day
```

Все сессии объединяются по дате начала.

Результат:

| day | avg_requests_per_session |
|---|---:|
| 2024-02-01 | 2.03 |
| 2024-02-02 | 1.98 |
| 2024-02-03 | 2.01 |

Сортирует результат по календарю.

# 24. Полный ручной пример

Исходные события:

| cookie | timestamp |
|---|---|
| A | 2024-02-01 09:00 |
| A | 2024-02-01 09:10 |
| A | 2024-02-01 10:00 |
| B | 2024-02-01 11:00 |
| B | 2024-02-01 11:20 |
| B | 2024-02-01 11:25 |
| C | 2024-02-01 12:00 |

Cookie `A`:

```text
09:00 → новая сессия;
09:10 → та же сессия;
10:00 → новая сессия, потому что gap 50 минут.
```

Получаем:

```text
A1 = 2 события;
A2 = 1 событие.
```

Cookie `B`:

```text
11:00;
11:20;
11:25.
```

Все gaps не превышают 30 минут:

```text
B1 = 3 события.
```

Cookie `C`:

```text
C1 = 1 событие.
```

Все сессии дня:

```text
2;
1;
3;
1.
```

Среднее:

```text
(2 + 1 + 3 + 1) / 4 = 1.75.
```

Результат:

| day | avg_requests_per_session |
|---|---:|
| 2024-02-01 | 1.75 |


```text

```


# 29. Почему нельзя разбивать по дню до sessionization

Неправильно:

```sql
PARTITION BY cookie_id, toDate(timestamp)
```

Так события `23:55` и `00:10` попадут в разные группы, хотя между ними только 15 минут.

Правильно:

```sql
PARTITION BY cookie_id
```

А день определять после построения сессии.

---

# 30. Почему нельзя считать DISTINCT cookie_id

Один cookie может иметь несколько сессий.

Пример:

```text
09:00;
09:10;
15:00;
15:05.
```

Это две сессии одного cookie.

`count(DISTINCT cookie_id)` вернёт `1`, хотя сессий `2`.

---

# 31. Почему нельзя группировать только по cookie_id

Если сгруппировать все события cookie за весь период, разные сессии склеятся.

В предыдущем примере получилось бы:

```text
4 события в одной ложной сессии
```

вместо:

```text
2 события;
2 события.
```

---

# 32. Почему нельзя исключать duration = 0

В этой задаче duration вообще не является фильтром.

Одно-событийная сессия:

```text
requests_per_session = 1
```

и должна участвовать в среднем.

Исключение нулевой длительности было специальным правилом только для другой метрики.

---

# 33. Почему не нужно ROUND

Пример результата показывает два знака после запятой.

Но условие не требует выполнять округление.

Redash может визуально отобразить:

```text
2.03
```

даже если внутри хранится более точное значение.

Добавление:

```sql
round(avg(...), 2)
```

может изменить результат, ожидаемый автоматической проверкой.

Поэтому запрос возвращает чистый:

```sql
avg(requests_per_session)
```

# 34. Типичные ошибки

## Ошибка 1

Считать `count(search_keyword)`.

Это не все события.

## Ошибка 2

Исключать одно-событийные сессии.

Условие этого не требует.

## Ошибка 3

Использовать `>= 1800`.

Ровно 30 минут должны оставаться одной сессией.

## Ошибка 4

Группировать только по cookie_id.

Несколько сессий склеятся.

## Ошибка 5

Считать среднее до формирования session-level таблицы.

Среднее должно считаться по количествам событий отдельных сессий.

## Ошибка 6

Добавлять день в оконный partition.

Cross-midnight session разорвётся.

## Ошибка 7

Использовать `user_id`.

Анонимные пользователи имеют `NULL`, а авторизация может произойти посреди сессии.

## Ошибка 8

Форматировать дату строкой.

Требуется тип `Date`.

## Ошибка 9

Считать среднее событий на cookie.

Это другая метрика.

## Ошибка 10

Округлять без требования.

Можно нарушить проверку точного результата.

# 36. Почему решение минимальное

Для расчёта действительно нужны только:

```text
timestamp;
cookie_id.
```

Не нужны:

- `user_id`;
- `page_url`;
- `page_title`;
- `browser`;
- `country`;
- `search_keyword`;
- `actions`;
- `traffic_source`.

Используются базовые операции:

- `lagInFrame`;
- `row_number`;
- `dateDiff`;
- оконная `sum`;
- `min`;
- `count`;
- `avg`;
- `toDate`;
- `GROUP BY`;
- `ORDER BY`.

# 37. Проверки результата

Полезно проверить:

## Каждая сессия имеет минимум одно событие

```text
requests_per_session >= 1.
```

## Одно-событийные сессии присутствуют

Не должно быть фильтра:

```sql
WHERE requests_per_session > 1
```

## Ровно 30 минут не создают новую сессию

Используется:

```sql
> 1800.
```

## День остаётся Date

Используется:

```sql
toDate(min(timestamp)).
```

## Результат является вещественным

Используется:

```sql
avg(requests_per_session).
```

# 38. Бизнесовая интерпретация

Рост среднего числа событий за сессию может означать:

- пользователи активнее изучают ассортимент;
- открывают больше товаров;
- используют фильтры;
- новый поиск повышает вовлечённость.

Но может означать:

- пользователю приходится повторять запрос;
- выдача хуже;
- нужно открывать больше неподходящих товаров;
- путь до покупки стал длиннее.

Снижение может означать:

- пользователь быстрее находит нужный товар;
- меньше лишних действий.

Но также:

- пользователь быстрее уходит;
- выдача не заинтересовала.

Поэтому метрику нужно анализировать вместе с:

- conversion to product click;
- add-to-cart rate;
- purchase conversion;
- reformulation rate;
- zero-results rate;
- time to success;
- exit rate.

# 39. Контрольное объяснение простыми словами

Для каждого cookie выписываем действия по времени.

Если между действиями прошло не больше 30 минут, они остаются в одном посещении.

Если прошло больше 30 минут, начинается новое посещение.

После этого считаем:

```text
сколько строк было в каждом посещении.
```

Например:

```text
сессия 1 — 2 события;
сессия 2 — 1 событие;
сессия 3 — 3 события.
```

Среднее:

```text
(2 + 1 + 3) / 3 = 2.
```

# 40. Итоговая логика

```text
1. Разделить события по cookie_id.
2. Отсортировать по timestamp.
3. Получить previous_timestamp.
4. Отметить начало новой сессии.
5. Накопительной суммой получить session_number.
6. Сгруппировать по cookie_id и session_number.
7. Посчитать count() событий.
8. Назначить дату первого события.
9. Посчитать avg() по каждому дню.
```

Ключевое условие новой сессии:

```sql
dateDiff('second', previous_timestamp, timestamp) > 1800
```

Ключевой подсчёт внутри сессии:

```sql
count() AS requests_per_session
```

Ключевое итоговое среднее:

```sql
avg(requests_per_session) AS avg_requests_per_session
```
