# DA: SESSION ANALYSIS

# STEP 2. Количество сессий в день

## Готовое решение

SQL-запрос находится в отдельном файле:

```text
gp_step_2_sessions_per_day.sql
```

Запрос возвращает ровно два требуемых поля:

```text
day
sessions_per_day
```

- `day` имеет тип `Date`;
- `sessions_per_day` содержит количество сессий, начавшихся в этот день.

---

# 1. Что именно требуется посчитать

В исходной таблице одна строка — это одно событие.

Пример:

| timestamp | cookie_id | event_type |
|---|---|---|
| 2024-02-01 10:00:00 | A | view_main |
| 2024-02-01 10:10:00 | A | search |
| 2024-02-01 11:00:00 | A | view_product |

Нам нужно определить, сколько отдельных сессий было начато в каждый календарный день.

Правило:

> События одного `cookie_id` относятся к одной сессии, если разрыв между соседними событиями не превышает 30 минут.

Следовательно, новая сессия начинается, если:

1. это первое событие данного `cookie_id`;
2. с момента предыдущего события этого же `cookie_id` прошло строго больше 30 минут.

---

# 2. Почему мы считаем именно начала сессий

Каждая сессия имеет ровно одно первое событие.

Следовательно:

```text
число сессий = число событий, являющихся началом сессии
```

Нам необязательно сначала создавать текстовый `session_id`, затем группировать все события до уровня сессий и только после этого считать строки.

Для этой конкретной метрики достаточно:

1. найти начало каждой сессии;
2. назначить началу календарный день;
3. посчитать число начал по дням.

Это делает запрос проще и быстрее.

---

# 3. Полный запрос

```sql
WITH events_with_previous AS (
    SELECT
        timestamp,
        cookie_id,
        lagInFrame(timestamp, 1) OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS previous_timestamp,
        row_number() OVER (
            PARTITION BY cookie_id
            ORDER BY timestamp
        ) AS event_number
    FROM default.karpovexpress_session_log
),
session_starts AS (
    SELECT
        toDate(timestamp) AS day,
        if(
            event_number = 1
            OR dateDiff('second', previous_timestamp, timestamp) > 1800,
            1,
            0
        ) AS is_new_session
    FROM events_with_previous
)
SELECT
    day,
    sum(is_new_session) AS sessions_per_day
FROM session_starts
GROUP BY day
ORDER BY day;
```

# 4. Первый CTE: events_with_previous

```sql
WITH events_with_previous AS (
```

`WITH` позволяет создать промежуточный результат.

Название:

```text
events_with_previous
```

можно прочитать как:

> События, к которым добавлено предыдущее событие.

Внутри этого CTE по-прежнему остаётся одна строка на одно событие.

---

## 4.1. Выбор timestamp

```sql
timestamp,
```

Нам нужно время текущего события, чтобы:

- расположить события по порядку;
- определить календарный день;
- вычислить разницу с предыдущим событием.

---

## 4.2. Выбор cookie_id

```sql
cookie_id,
```

Сессии строятся отдельно для каждого cookie.

Нельзя сравнивать событие одного cookie с событием другого cookie.

Пример:

| timestamp | cookie_id |
|---|---|
| 10:00 | A |
| 10:05 | B |
| 10:10 | A |

Для cookie `A` предыдущим событием перед `10:10` является `10:00`, а не событие cookie `B` в `10:05`.

---

# 5. Функция lagInFrame

```sql
lagInFrame(timestamp, 1) OVER (
```

`lagInFrame` возвращает значение из предыдущей строки оконной последовательности.

Аргументы:

```sql
lagInFrame(timestamp, 1)
```

означают:

- взять столбец `timestamp`;
- сдвинуться назад на одну строку.

Для события в `10:20` функция может вернуть время предыдущего события `10:05`.

---

## 5.1. PARTITION BY cookie_id

```sql
PARTITION BY cookie_id
```

`PARTITION BY` создаёт отдельную последовательность для каждого cookie.

Упрощённо система делает так:

```text
cookie A:
    все его события

cookie B:
    все его события

cookie C:
    все его события
```

После этого `lagInFrame` ищет предыдущую строку только внутри соответствующего cookie.

Без `PARTITION BY cookie_id` события всех пользователей смешались бы.

---

## 5.2. ORDER BY timestamp

```sql
ORDER BY timestamp
```

События каждого cookie располагаются по времени.

Например, сырые строки могли физически лежать так:

```text
10:20
10:00
10:10
```

После логической сортировки:

```text
10:00
10:10
10:20
```

Только после сортировки можно правильно определить предыдущее событие.

---

## 5.3. Оконная рамка

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
```

Эта запись определяет доступную оконную рамку:

```text
от первой строки cookie
до последней строки cookie
```

`lagInFrame` в ClickHouse учитывает рамку окна.

Полная рамка позволяет функции работать как обычный поиск предыдущей строки во всей последовательности данного cookie.

Расшифровка:

- `UNBOUNDED PRECEDING` — от самого начала;
- `UNBOUNDED FOLLOWING` — до самого конца;
- `ROWS` — рамка задаётся строками.

---

## 5.4. Результат previous_timestamp

```sql
AS previous_timestamp
```

Получаем новый столбец:

```text
previous_timestamp
```

Пример:

| timestamp | previous_timestamp |
|---|---|
| 10:00 | служебное значение для первой строки |
| 10:10 | 10:00 |
| 11:00 | 10:10 |

Для первой строки реального предыдущего события нет.

Именно поэтому дополнительно используется `row_number()`.

---

# 6. Функция row_number

```sql
row_number() OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
) AS event_number
```

`row_number()` нумерует события каждого cookie после сортировки по времени.

Пример:

| cookie_id | timestamp | event_number |
|---|---|---:|
| A | 10:00 | 1 |
| A | 10:10 | 2 |
| A | 11:00 | 3 |
| B | 09:00 | 1 |
| B | 09:15 | 2 |

Для каждого нового cookie нумерация начинается заново.

---

## 6.1. Зачем нужен event_number

Первое событие любого cookie всегда является началом первой сессии.

У него нет настоящего предыдущего события.

Проверка:

```sql
event_number = 1
```

надёжно определяет первое событие.

Мы не зависим от того, какое служебное значение `lagInFrame` возвращает за пределами окна.

---

# 7. Почему не используется user_id

В условии сессии определяются по:

```text
cookie_id
```

`user_id` может быть:

```text
NULL
```

Например, пользователь сначала работает анонимно, а затем авторизуется:

| timestamp | cookie_id | user_id |
|---|---|---:|
| 10:00 | A | NULL |
| 10:10 | A | 1001 |

Это всё ещё одна браузерная сессия.

Если разделять события по `user_id`, взаимодействие будет ошибочно разорвано.

---

# 8. Второй CTE: session_starts

```sql
session_starts AS (
```

В этом промежуточном результате мы определяем:

- календарный день события;
- является ли событие началом новой сессии.

---

# 9. Получение поля day

```sql
toDate(timestamp) AS day
```

`timestamp` содержит дату и время:

```text
2024-02-01 10:42:00
```

`toDate(timestamp)` оставляет только дату:

```text
2024-02-01
```

Тип результата — `Date`.

Это полностью соответствует условию:

> Поле `day` не нужно специально форматировать. Оставьте с типом Date.

Мы не используем:

```sql
formatDateTime(...)
```

потому что это превратило бы дату в строку.

---

# 10. Определение начала новой сессии

```sql
if(
    event_number = 1
    OR dateDiff('second', previous_timestamp, timestamp) > 1800,
    1,
    0
) AS is_new_session
```

Функция `if` имеет три части:

```sql
if(условие, значение_если_истина, значение_если_ложь)
```

В нашем случае:

- если событие начинает новую сессию — вернуть `1`;
- иначе — вернуть `0`.

---

# 11. Первое условие

```sql
event_number = 1
```

Если событие первое для cookie, оно обязательно открывает первую сессию.

Пример:

```text
cookie A, первое событие → новая сессия
cookie B, первое событие → новая сессия
cookie C, первое событие → новая сессия
```

---

# 12. Оператор OR

```sql
OR
```

Новая сессия начинается, если выполняется хотя бы одно условие:

1. событие первое;
2. разрыв больше 30 минут.

---

# 13. Точное сравнение времени

```sql
dateDiff('second', previous_timestamp, timestamp) > 1800
```

`dateDiff` считает разницу между временем предыдущего и текущего события.

Мы специально используем секунды.

Почему:

```text
30 минут = 30 × 60 = 1800 секунд
```

Условие задания:

```text
разрыв <= 30 минут → та же сессия
разрыв > 30 минут → новая сессия
```

Поэтому написано:

```sql
> 1800
```

---

## 13.1. Граничные случаи

| Разрыв | Секунды | Новая сессия |
|---|---:|---|
| 29 минут 59 секунд | 1799 | нет |
| ровно 30 минут | 1800 | нет |
| 30 минут 1 секунда | 1801 | да |
| 31 минута | 1860 | да |

---

## 13.2. Почему не используется >= 1800

Если написать:

```sql
>= 1800
```

то ровно 30 минут создадут новую сессию.

Это противоречит условию:

> Время между событиями меньше либо равно 30 минут — одна сессия.

Правильно:

```sql
> 1800
```

---

## 13.3. Почему лучше считать секунды, а не минуты

Наивный вариант:

```sql
dateDiff('minute', previous_timestamp, timestamp) > 30
```

может быть менее наглядным на пограничных значениях.

Например, нам важно отличать:

```text
30 минут 0 секунд
```

от:

```text
30 минут 1 секунда
```

Секунды позволяют реализовать правило буквально:

```text
> 1800
```

---

# 14. Поле is_new_session

Результат:

```text
1 — начало сессии
0 — продолжение сессии
```

Пример:

| timestamp | previous_timestamp | event_number | gap | is_new_session |
|---|---|---:|---:|---:|
| 09:53 | — | 1 | — | 1 |
| 10:42 | 09:53 | 2 | 49 мин | 1 |
| 10:53 | 10:42 | 3 | 11 мин | 0 |
| 11:02 | 10:53 | 4 | 9 мин | 0 |
| 11:15 | 11:02 | 5 | 13 мин | 0 |

Сумма:

```text
1 + 1 + 0 + 0 + 0 = 2 сессии
```

---

# 15. Финальный SELECT

```sql
SELECT
    day,
    sum(is_new_session) AS sessions_per_day
```

Для каждого дня суммируются флаги начала сессии.

Почему сумма равна числу сессий:

- у каждой сессии ровно одно начало;
- начало помечено единицей;
- остальные события помечены нулём.

Пример:

```text
1, 0, 0, 1, 0, 1
```

Сумма:

```text
3
```

Значит, в этот день начались три сессии.

---

# 16. GROUP BY day

```sql
GROUP BY day
```

Все начала сессий объединяются по календарной дате.

Результат имеет одну строку на один день.

Пример:

| day | sessions_per_day |
|---|---:|
| 2024-02-01 | 5524 |
| 2024-02-02 | 5499 |
| 2024-02-03 | 5454 |

---

# 17. ORDER BY day

```sql
ORDER BY day
```

Располагает дни хронологически:

```text
1 февраля
2 февраля
3 февраля
...
```

Без `ORDER BY` база данных не обязана возвращать строки в календарном порядке.

---

# 18. Почему sessions_per_day может отображаться с .00

В примере задания показано:

```text
5524.00
```

Но число сессий по смыслу является целым.

Запрос возвращает сумму целочисленных флагов:

```text
5524
```

Redash или проверяющая система могут визуально показать:

```text
5524.00
```

Формат отображения не меняет значение.

Мы не выполняем искусственное округление и не превращаем число в строку.

---

# 19. Сессия, переходящая через полночь

Рассмотрим:

```text
2024-02-01 23:55
2024-02-02 00:10
```

Разрыв:

```text
15 минут
```

Это одна сессия.

Первое событие получает:

```text
day = 2024-02-01
is_new_session = 1
```

Второе событие:

```text
day = 2024-02-02
is_new_session = 0
```

В результате сессия считается 1 февраля — в день своего начала.

Это правильнее, чем сначала делить события по дням.

---

# 20. Почему нельзя добавлять day в PARTITION BY

Неправильно:

```sql
PARTITION BY cookie_id, toDate(timestamp)
```

Такой код искусственно начнёт новую последовательность в полночь.

Даже если между событиями прошла одна минута, они попадут в разные дневные partition.

В нашем запросе используется только:

```sql
PARTITION BY cookie_id
```

Поэтому сессия может корректно перейти на следующий календарный день.

---

# 21. Почему не нужен полный session_id

Для средней длительности в будущем понадобится собрать все события каждой сессии и вычислить:

```text
MIN(timestamp)
MAX(timestamp)
```

Но для текущей задачи достаточно считать начала.

Каждая новая сессия создаёт ровно одну единицу.

Это минимальное решение без лишних вычислений.

---

# 22. Ручная проверка

Исходные события:

| cookie | timestamp |
|---|---|
| A | 2024-02-01 09:00 |
| A | 2024-02-01 09:20 |
| A | 2024-02-01 10:00 |
| B | 2024-02-01 11:00 |
| B | 2024-02-02 00:10 |
| B | 2024-02-02 00:25 |

Разбор cookie A:

```text
09:00 — первое событие → новая сессия
09:20 — gap 20 минут → та же сессия
10:00 — gap 40 минут → новая сессия
```

Разбор cookie B:

```text
11:00 — первое событие → новая сессия
00:10 следующего дня — gap 13 часов 10 минут → новая сессия
00:25 — gap 15 минут → та же сессия
```

Результат:

```text
2024-02-01 → 3 сессии
2024-02-02 → 1 сессия
```

# 24. Типичные неправильные решения

## Ошибка 1. COUNT DISTINCT cookie_id

```sql
count(DISTINCT cookie_id)
```

Это считает cookie, а не сессии.

Один cookie может создать несколько сессий за день.

---

## Ошибка 2. GROUP BY cookie_id и day

Такой подход не учитывает несколько сессий одного cookie в один день.

---

## Ошибка 3. Условие >= 30 минут

Ровно 30 минут должны оставаться одной сессией.

---

## Ошибка 4. Сравнение разных cookie

Без `PARTITION BY cookie_id` пользовательские последовательности смешиваются.

---

## Ошибка 5. Разбиение по календарному дню до LAG

Сессия через полночь ошибочно станет двумя сессиями.

---

## Ошибка 6. Использование user_id

Анонимные события имеют `NULL`, а пользователь может авторизоваться посреди визита.

---

## Ошибка 7. Форматирование day строкой

Условие требует тип `Date`.

Поэтому нужен:

```sql
toDate(timestamp)
```

а не текстовое форматирование.

---

одна строка = одно событие

# 26. Сложность запроса

Самая тяжёлая операция:

```text
сортировка событий внутри каждого cookie_id
```

Она нужна оконным функциям.

В запросе выбираются только необходимые поля:

```text
timestamp
cookie_id
```

Мы не читаем в расчёт:

- URL;
- браузер;
- страну;
- заголовок;
- search_keyword;
- user_agent.

Это соответствует подсказке:

> Подумайте, какие поля действительно нужны для идентификации сессий.

---

# 27. Почему решение минимальное

Для задачи используются только необходимые элементы:

- `timestamp`;
- `cookie_id`;
- `lagInFrame`;
- `row_number`;
- `dateDiff`;
- `toDate`;
- `sum`;
- `GROUP BY`;
- `ORDER BY`.

Мы не создаём лишние:

- текстовые session IDs;
- массивы;
- JOIN;
- вложенные подзапросы;
- форматирование;
- дополнительные агрегации.

# 28. Итог

Логика решения:

```text
для каждого cookie
    отсортировать события
    найти предыдущее событие

если событие первое
    это новая сессия

если разрыв > 1800 секунд
    это новая сессия

иначе
    это продолжение

затем
    сложить флаги по дню начала
```

Главная формула:

```sql
event_number = 1
OR dateDiff('second', previous_timestamp, timestamp) > 1800
```

Главная агрегация:

```sql
sum(is_new_session)
```

Итоговые поля:

```text
day
sessions_per_day
```
