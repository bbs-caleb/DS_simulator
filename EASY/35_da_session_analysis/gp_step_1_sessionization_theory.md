# DA: SESSION ANALYSIS

# STEP 1. Максимально подробная теория sessionization

## 1. Что находится в исходной таблице

Таблица:

```text
default.karpovexpress_session_log
```

— это журнал событий.

Каждая строка описывает отдельное действие:

| timestamp | cookie_id | user_id | event_type |
|---|---|---:|---|
| 10:00 | A | NULL | view_main |
| 10:02 | A | NULL | search |
| 10:05 | A | 101 | view_product |

Один пользовательский визит может занимать много строк.

В таблице нет готовых полей:

```text
session_id
session_start
session_end
session_duration
session_event_count
```

Их нужно восстановить. Этот процесс называется **sessionization**.

---

# 2. Главная логика sessionization

Для каждого `cookie_id`:

1. расположить события по времени;
2. найти предыдущее событие;
3. вычислить временной разрыв;
4. первое событие считать началом сессии;
5. событие после разрыва больше 30 минут считать началом новой сессии;
6. накопительно пронумеровать сессии;
7. объединить события с одинаковым номером;
8. посчитать начало, конец, длительность и количество событий.

Схема:

```text
events
  ↓
LAG(timestamp)
  ↓
gap
  ↓
new_session_flag
  ↓
cumulative SUM
  ↓
session_number
  ↓
GROUP BY cookie_id, session_number
  ↓
session-level table
```

---

# 3. Почему cookie_id не равен session_id

`cookie_id` идентифицирует браузерный контекст.

Один cookie может посещать сайт много раз:

```text
1 февраля утром
1 февраля вечером
5 февраля
12 февраля
```

Если сделать:

```sql
GROUP BY cookie_id
```

все посещения склеятся в одну огромную сессию.

Поэтому уникальная сессия определяется парой:

```text
(cookie_id, session_number)
```

Пример:

```text
(A, 1)
(A, 2)
(B, 1)
(B, 2)
```

---

# 4. Почему нельзя использовать user_id

`user_id` может быть `NULL`.

Пользователь может войти в аккаунт посреди посещения:

```text
10:00 cookie A, user_id NULL
10:05 cookie A, user_id 123
```

Это одна сессия.

Если разделять по `user_id`, анонимная и авторизованная части разорвутся.

В условии явно задан `cookie_id`, поэтому именно он является основой sessionization.

---

# 5. Подробный разбор полей

## timestamp

Используется для:

- сортировки;
- расчёта разрыва;
- определения начала;
- определения конца;
- длительности;
- назначения даты.

Нужно помнить о:

- timezone;
- NULL;
- одинаковом времени;
- поздних событиях;
- времени клиента и сервера.

## cookie_id

Определяет независимую последовательность событий.

События разных cookie никогда не сравниваются между собой.

## user_id

Полезен для user-level аналитики, но не является session key в этой задаче.

## event_type

Тип действия. Для общего `event_count` каждая строка обычно считается событием.

## page_url и page_title

Помогают понять путь пользователя и отладить результат.

## referrer_url

Источник перехода на текущую страницу. Для timeout sessionization не обязателен.

## browser, operating_system, device_type

Нужны для сегментации и поиска технических проблем.

## screen_resolution

Дополнительный признак устройства, обычно хранится строкой.

## ip_address

Полезен для диагностики и bot/fraud анализа, но не равен пользователю.

## country и language

Нужны для географического и языкового анализа.

## user_agent

Может помочь распознать браузер, устройство и ботов.

## search_keyword

Поисковая фраза. Может быть `NULL`. Нельзя автоматически использовать `COUNT(search_keyword)` как число всех событий.

## actions

Дополнительное действие в строке, например `add_to_cart`.

## traffic_source

Канал трафика. Изменение его структуры способно изменить session metrics без изменения поиска.

# 6. Учебный пример из задания

События cookie:

```text
09:53 — посмотрел страницу 1
10:42 — посмотрел страницу 2
10:53 — посмотрел страницу 3
11:02 — добавил в корзину
11:15 — открыл корзину
```

Считаем разрывы.

| current_time | previous_time | gap | новая сессия |
|---|---|---:|---:|
| 09:53 | NULL | NULL | 1 |
| 10:42 | 09:53 | 49 минут | 1 |
| 10:53 | 10:42 | 11 минут | 0 |
| 11:02 | 10:53 | 9 минут | 0 |
| 11:15 | 11:02 | 13 минут | 0 |

Получается:

```text
сессия 1: 09:53
сессия 2: 10:42–11:15
```

---

# 7. Почему первое событие всегда начинает сессию

У первого события cookie нет предыдущего события.

Оконная функция вернёт:

```text
previous_timestamp = NULL
```

Следовательно:

```text
new_session_flag = 1
```

Иначе первая сессия вообще не получит номер.

---

# 8. Оконная функция LAG

`LAG` возвращает значение из предыдущей строки.

Концептуальная запись:

```sql
LAG(timestamp) OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
)
```

Читаем:

> Внутри каждого cookie отсортируй события по времени и покажи timestamp предыдущего события.

---

## 8.1. PARTITION BY

```sql
PARTITION BY cookie_id
```

Создаёт отдельное окно для каждого cookie.

Истории A и B не смешиваются.

---

## 8.2. ORDER BY

```sql
ORDER BY timestamp
```

Определяет последовательность.

Без сортировки понятие «предыдущее событие» не имеет смысла.

---

## 8.3. Что происходит на первой строке

Предыдущей строки нет.

Результат:

```text
NULL
```

---

# 9. Флаг новой сессии

Логика:

```text
если previous_timestamp IS NULL → 1
если gap > 30 минут → 1
иначе → 0
```

Пример:

```text
1, 1, 0, 0, 0
```

---

# 10. Почему условие строгое

В условии написано:

> больше 30 минут.

Значит:

```text
gap > 30
```

Ровно 30 минут не создают новую сессию.

Граница:

| gap | результат |
|---:|---|
| 29:59 | та же сессия |
| 30:00 | та же сессия |
| 30:01 | новая сессия |

Использование `>= 30` изменит бизнес-логику и может привести к провалу тестов.

---

# 11. Накопительная сумма

Пусть флаги:

```text
1, 1, 0, 0, 0
```

Накопительная сумма:

```text
1, 2, 2, 2, 2
```

Это номер сессии внутри cookie.

Концептуально:

```sql
SUM(new_session_flag) OVER (
    PARTITION BY cookie_id
    ORDER BY timestamp
)
```

Каждая единица увеличивает номер.

Нули сохраняют текущий номер.

---

# 12. Почему номер недостаточно использовать один

У cookie A есть session 1.

У cookie B тоже есть session 1.

Поэтому группировка только по `session_number` смешает разных пользователей.

Правильно:

```text
GROUP BY cookie_id, session_number
```

---

# 13. Переход к session-level

После разметки события агрегируются.

Для каждой пары:

```text
cookie_id + session_number
```

считаются:

```text
session_start = MIN(timestamp)
session_end = MAX(timestamp)
event_count = COUNT(rows)
duration = session_end - session_start
```

---

# 14. Начало и конец

## Начало

```text
MIN(timestamp)
```

Первое наблюдаемое событие.

## Конец

```text
MAX(timestamp)
```

Последнее наблюдаемое событие.

Это не обязательно физический момент закрытия вкладки.

---

# 15. Длительность

Определение:

```text
session_end - session_start
```

Пример:

```text
10:42 → 11:15 = 33 минуты = 1980 секунд
```

В ClickHouse длительность часто получают через функции типа `dateDiff`.

Единица измерения должна соответствовать заданию.

---

# 16. Сессия из одного события

Если строка одна:

```text
MIN(timestamp) = MAX(timestamp)
```

Следовательно:

```text
duration = 0
event_count = 1
```

Это не ошибка расчёта. Это ограничение наблюдения.

Пользователь мог читать страницу, но других событий нет.

---

# 17. Как обрабатывать нулевые сессии

Правило KarpovExpress:

> При средней длительности не учитывать сессии с duration = 0.

Правильный порядок:

1. построить все сессии;
2. посчитать duration;
3. в запросе средней длительности оставить `duration > 0`.

Нельзя удалять одно-событийные строки до sessionization.

---

# 18. Нулевые сессии в других метриках

Условие не говорит удалять их из:

- числа сессий;
- среднего event count;
- доли одно-событийных сессий.

Если удалить их везде, получится другая бизнес-метрика.

---

# 19. Цепочечная природа timeout

События:

```text
10:00
10:25
10:50
11:15
```

Между соседними событиями по 25 минут.

Это одна сессия, хотя от начала до конца прошло 75 минут.

Порог применяется к соседним событиям, а не к расстоянию от начала сессии.

---

# 20. Критическая ошибка: сравнение с началом сессии

Неверная логика:

> Если текущее событие более чем на 30 минут позже первого, начать новую сессию.

Это раздробит длинный активный визит.

Правильная логика:

> Сравнивать текущее событие только с непосредственно предыдущим событием того же cookie.

---

# 21. Сессия через полночь

Пример:

```text
23:55
00:10
```

Разрыв 15 минут.

По timeout это одна сессия.

Нельзя заранее делать:

```text
PARTITION BY cookie_id, date
```

Иначе любая полночь принудительно разорвёт сессию.

Правильный порядок:

1. sessionize полную временную последовательность;
2. затем назначить сессии дату.

Обычно используют дату старта, но точное правило должно задаваться требованиями.

---

# 22. Уровень дня

Фраза «количество сессий в день» может означать:

1. временной ряд — одно значение на каждый день;
2. одно среднее по дневным значениям;
3. среднее число сессий на пользователя в день.

Это разные формулы. Поэтому до следующего шага нельзя угадывать окончательный формат.

---

# 23. Среднее средних

День 1:

```text
1 сессия длиной 100 секунд
```

День 2:

```text
100 сессий длиной 10 секунд
```

Общее среднее по сессиям:

```text
(100 + 100 × 10) / 101 ≈ 10.89
```

Среднее двух дневных средних:

```text
(100 + 10) / 2 = 55
```

Результаты не равны.

Поэтому аналитик обязан понимать denominator.

---

# 24. Denominator

Для средней длительности:

```text
число сессий с duration > 0
```

Для среднего количества событий:

```text
число сессий
```

если не задано другое правило.

Для среднего дневного количества:

```text
число дней
```

Но нужно понимать, входят ли дни без событий.

---

# 25. Дни без событий

Если анализируется десять календарных дней, но в один день нет строк:

- можно усреднить только дни с событиями;
- можно включить отсутствующий день как ноль.

Результат будет разным.

В production обычно используют календарную таблицу. В учебном задании нужно следовать точному требованию.

---

# 26. Почему нельзя усреднять gaps вместо duration

Сессия:

```text
10:00
10:05
10:20
```

Разрывы:

```text
5 минут
15 минут
```

Средний gap:

```text
10 минут
```

Duration:

```text
20 минут
```

Это разные метрики.

---

# 27. Почему нельзя считать event_count по cookie

Один cookie имеет несколько сессий.

Если считать все события за период:

```text
COUNT(*) GROUP BY cookie_id
```

получится активность cookie, а не события за сессию.

Сначала нужен session number.

---

# 28. Логический CTE pipeline

Удобная структура:

```text
1. base_events
2. events_with_previous
3. events_with_flag
4. events_with_session_number
5. sessions
6. final_metric
```

## base_events

Берём необходимые поля.

## events_with_previous

Добавляем предыдущий timestamp.

## events_with_flag

Помечаем начало новой сессии.

## events_with_session_number

Считаем накопительную сумму.

## sessions

Переходим к одной строке на сессию.

## final_metric

Считаем нужную метрику.

---

# 29. Оконные функции против GROUP BY

## Оконная функция

Не уменьшает число строк.

Было 100 событий — остаётся 100 событий.

Добавляется previous timestamp или номер сессии.

## GROUP BY

Схлопывает строки.

100 событий могут превратиться в 20 сессий.

Sessionization требует сначала оконных функций, затем агрегации.

---

# 30. Одинаковые timestamp

События:

```text
10:00:00 search
10:00:00 view_product
```

По одному времени точный порядок неизвестен.

Для session boundary это обычно не проблема, потому что gap = 0.

Для точной последовательности нужен tie-breaker:

- event_id;
- sequence number;
- ingestion time;
- microseconds.

Если такого поля нет, порядок равных timestamp недетерминирован.

---

# 31. Дубликаты

Причины:

- retry;
- повторная отправка;
- at-least-once delivery;
- повторная загрузка ETL.

Влияние:

- завышается event_count;
- меняется воронка;
- engagement выглядит выше.

Нужно проверить наличие event_id и правила дедупликации.

Но нельзя самовольно удалять данные, если условие этого не требует.

---

# 32. NULL в cookie_id

Если все NULL попадут в один partition, события разных людей склеятся.

В реальном проекте нужно:

- проверить количество NULL;
- исключить;
- использовать другой ключ;
- отправить проблему data engineering.

---

# 33. NULL в timestamp

Событие без времени нельзя корректно расположить.

В production его:

- исключают из session analysis;
- отправляют в quarantine;
- восстанавливают по техническим метаданным.

---

# 34. Поздно пришедшие события

Сначала пришли:

```text
10:00
10:50
```

Получились две сессии.

Позже пришло:

```text
10:25
```

Теперь gaps:

```text
25 и 25 минут
```

Это одна сессия.

Следовательно, при late events прошлые session boundaries могут измениться.

В production применяют:

- watermark;
- период пересчёта;
- backfill;
- версионирование.

---

# 35. Граница периода

Есть события:

```text
31 января 23:50
1 февраля 00:10
```

Если отфильтровать только февраль до `LAG`, событие 00:10 станет ложным началом.

Для строгого production-расчёта нужен lookback до начала периода.

Аналогично конец периода может обрезать сессию.

---

# 36. Timezone

UTC и локальная дата могут отличаться.

```text
2024-02-01 22:30 UTC
```

в UTC+5:

```text
2024-02-02 03:30
```

Число сессий по дням зависит от выбранной timezone.

---

# 37. Боты

Бот может генерировать:

- тысячи событий;
- равные интервалы;
- очень длинные сессии;
- невозможную скорость переходов;
- отсутствие конверсии.

Session metrics нужно проверять на bot contamination.

---

# 38. Активное время против elapsed time

Наша формула:

```text
последнее событие - первое событие
```

считает elapsed time.

Она не показывает, был ли пользователь активен всё время.

Для active time нужны:

- heartbeat;
- focus;
- visibility;
- mouse/keyboard activity;
- playback events.

---

# 39. Правое цензурирование

После последнего события мы не видим дальнейшее присутствие.

Сессия:

```text
10:00
10:10
```

получает duration 10 минут.

Но пользователь мог оставаться до 10:25.

Поэтому duration часто недооценивает реальное время.

---

# 40. Левое цензурирование

Если данные начинаются посреди уже активной сессии, наблюдаемое начало будет позже реального.

Это ещё одна причина осторожно работать с границами периода.

---

# 41. Альтернативные определения сессии

## Явный start/end

Надёжность `end` низкая: приложение может закрыться без события.

## Login/logout

Не подходит анонимным и пользователям, которые долго не выходят.

## Task-based

Одна заявка, один checkout, один игровой раунд.

## ML-based

Модель определяет смену намерения. Это сложнее и редко нужно для базовой аналитики.

---

# 42. Session не всегда равна intent

В одной timeout-сессии пользователь может:

- искать чайник;
- затем телевизор;
- затем открыть профиль.

Сессия — техническая группировка активности, а не идеальная единица намерения.

Для более глубокого анализа строят:

- query journey;
- shopping mission;
- funnel;
- multi-session journey.

---

# 43. Session analysis для поиска

После создания session table можно считать:

- наличие поиска;
- поиски за сессию;
- уникальные запросы;
- reformulations;
- просмотренные товары;
- клики по выдаче;
- add-to-cart;
- purchase;
- time to first click;
- time to success.

---

# 44. Reformulation

Пример:

```text
кофемашина
кофемашина xiaomi
xiaomi кофемашина недорого
```

Это может означать:

- естественное уточнение;
- плохую первую выдачу;
- непонимание запроса.

Число запросов нельзя интерпретировать без контекста.

---

# 45. Time to success

Новый поиск должен помогать быстрее прийти к цели.

Важные метрики:

```text
время от первого search до product click
время от search до add_to_cart
время от session_start до purchase
события до успешного действия
```

Если поиск улучшился, duration может уменьшиться. Это может быть положительным результатом.

---

# 46. Simpson's paradox

Общая duration выросла.

Но:

```text
desktop duration снизилась
mobile duration снизилась
```

Рост общего среднего мог произойти потому, что стало больше desktop-трафика, где сессии изначально длиннее.

Поэтому нужны сегменты.

---

# 47. Среднее и тяжёлый хвост

Session duration часто имеет правый хвост.

Причины:

- вкладка оставлена открытой;
- фоновые события;
- бот;
- редкий длинный сценарий.

Кроме среднего полезны:

- median;
- p75;
- p90;
- p95;
- p99;
- max;
- histogram.

Но финальное учебное решение должно соблюдать заданную формулу, а не самовольно менять её.

# 48. Проверки правильности session table

## Инвариант 1

Каждое исходное событие принадлежит одной сессии.

## Инвариант 2

```text
session_start <= session_end
```

## Инвариант 3

```text
duration >= 0
```

## Инвариант 4

```text
event_count >= 1
```

## Инвариант 5

Внутри сессии нет gap > 30 минут.

## Инвариант 6

Между соседними сессиями одного cookie gap > 30 минут.

## Инвариант 7

```text
SUM(event_count) = число исходных событий
```

если фильтрации и дедупликации не было.

# 49. Sanity checks

## Доля одно-событийных сессий

Если почти 100%, возможно:

- cookie меняется на каждом событии;
- неправильный partition;
- timestamp сломан.

## Очень длинные сессии

Возможны:

- bots;
- background events;
- heartbeat;
- неверный timeout.

## Очень большое event_count

Возможны:

- duplicate logs;
- polling;
- frontend loop;
- бот.

## Резкий скачок 17 февраля

Нужно проверить, не изменился ли tracking вместе с поисковым алгоритмом.

# 52. Типичные ошибки начинающих

1. Группировать только по `cookie_id`.
2. Группировать по `cookie_id` и дате до sessionization.
3. Использовать `user_id`.
4. Использовать `>= 30`.
5. Сравнивать с началом сессии, а не с предыдущим событием.
6. Усреднять gaps вместо duration.
7. Удалять одно-событийные сессии из всех метрик.
8. Считать `COUNT(search_keyword)` как число всех событий.
9. Не проверять NULL cookie.
10. Игнорировать timezone.
11. Игнорировать изменение tracking.
12. Запускать сразу на всей таблице без ручного примера.

# 53. Как отлаживать SQL

1. Выбрать один `cookie_id`.
2. Вывести события по времени.
3. Добавить previous timestamp.
4. Проверить gap вручную.
5. Добавить new session flag.
6. Добавить session number.
7. Сгруппировать в сессии.
8. Сверить начало, конец, duration и count.
9. Проверить граничные случаи.
10. Только затем запускать на полном наборе.

# 56. Абстрактная форма будущего SQL

Это не финальное решение, а схема:

```sql
WITH events_with_previous AS (
    -- найти предыдущее событие внутри cookie
),
events_with_flag AS (
    -- отметить начало новой сессии
),
events_with_session_number AS (
    -- накопительно пронумеровать сессии
),
sessions AS (
    -- собрать одну строку на сессию
)
SELECT
    -- рассчитать конкретную метрику
FROM sessions;
```

Финальный запрос пока не даётся, потому что последующие шаги должны уточнить:

- формат результата;
- период;
- группировку;
- имя выходной колонки;
- округление;
- единицу длительности.

# 57. Псевдоалгоритм

```text
для каждого cookie:
    отсортировать события по timestamp

    для каждого события:
        previous_timestamp = timestamp предыдущего события

        если previous_timestamp отсутствует:
            new_session_flag = 1
        иначе если разрыв > 30 минут:
            new_session_flag = 1
        иначе:
            new_session_flag = 0

        session_number =
            накопительная сумма new_session_flag

для каждой пары cookie_id + session_number:
    session_start = минимальный timestamp
    session_end = максимальный timestamp
    duration_sec = session_end - session_start
    event_count = количество строк
```

# 58. Контрольные вопросы

## События 10:00 и 10:30

Одна сессия.

## События 10:00 и 10:30:01

Две сессии.

## События 10:00, 10:25, 10:50

Одна сессия.

Нет.

Первое и последнее наблюдаемое событие совпадают.

Нет, если условие этого не требует.

Сессия может перейти через полночь.

# 59. Что нужно запомнить

1. Исходная таблица имеет уровень события.
2. Сессии восстанавливаются, а не читаются из готового столбца.
3. `cookie_id` определяет независимую последовательность.
4. События сортируются по времени.
5. `LAG` даёт предыдущее событие.
6. Первое событие начинает сессию.
7. Gap строго больше 30 минут начинает новую сессию.
8. Ровно 30 минут остаются одной сессией.
9. Накопительная сумма флага даёт номер.
10. Уникальная сессия — `(cookie_id, session_number)`.
11. Начало — минимум времени.
12. Конец — максимум времени.
13. Duration — разница конца и начала.
14. Event count — количество событий.
15. Одно событие даёт duration 0.
16. Duration 0 исключается из средней длительности.
17. Нулевые сессии не удаляются автоматически из других метрик.
18. Нельзя дробить по календарной дате до sessionization.
19. Нужно проверять данные и граничные случаи.
20. Session metrics требуют бизнесовой интерпретации.

# 60. Финальный технический вывод

Техническое ядро задачи:

```text
LAG
→ временной разрыв
→ флаг новой сессии
→ накопительная SUM
→ session number
→ GROUP BY session
→ метрика
```

Но правильность зависит не только от синтаксиса.

Нужно строго соблюдать:

- `PARTITION BY cookie_id`;
- сортировку по `timestamp`;
- условие `> 30 минут`;
- отдельную обработку первого события;
- session-level агрегацию;
- правило нулевой длительности;
- точный denominator;
- требования следующих шагов.
