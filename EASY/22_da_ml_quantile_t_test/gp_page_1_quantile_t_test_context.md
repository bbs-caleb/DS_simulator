# DA/ML: QUANTILE T-TEST — бизнесовый и технический контекст

## 1. Зачем бизнесу ETA-модель

FlyingFood доставляет еду дронами. До оформления заказа пользователь видит ожидаемое время доставки — ETA, `estimated time of arrival`.

ETA влияет сразу на несколько решений пользователя:

1. Заказывать ли вообще.
2. Какой ресторан выбрать.
3. Готов ли пользователь ждать.
4. Нужно ли отменить заказ.
5. Доверяет ли пользователь сервису после доставки.

Поэтому ETA — не просто число модели. Это публичное обещание продукта.

Если интерфейс показывает 20 минут, пользователь строит планы вокруг этих 20 минут. Он может ждать обед до встречи, заказать еду ребёнку или рассчитывать получить заказ до отъезда. Ошибка модели становится пользовательской и денежной проблемой.

---

# 2. Два направления ошибки

Нужно сначала определить знак ошибки. Удобное определение:

```text
error = actual_delivery_time - predicted_delivery_time
```

## Положительная ошибка

Прогноз: 20 минут. Факт: 60 минут.

```text
error = 60 - 20 = 40
```

Модель **недооценила** время доставки. Это недопрогноз ETA.

Возможные последствия:

- раздражение клиента;
- отмена заказа;
- обращение в поддержку;
- промокод или компенсация;
- плохая оценка ресторана или доставки;
- снижение retention;
- переход к конкуренту;
- недоверие к будущим ETA.

## Отрицательная ошибка

Прогноз: 60 минут. Факт: 20 минут.

```text
error = 20 - 60 = -40
```

Модель дала слишком консервативную оценку.

Последствия:

- пользователь может не оформить заказ;
- ресторан выглядит медленнее конкурентов;
- падает конверсия;
- снижается GMV;
- хороший фактический сервис кажется плохим до покупки.

Обе ошибки неприятны, но их стоимость обычно несимметрична.

---

# 3. Что значит асимметричная цена ошибки

Обычная абсолютная ошибка:

```text
absolute_error = |actual - predicted|
```

считает ошибки `+20` и `-20` одинаковыми.

Но бизнес может оценивать их по-разному:

- `+20`: мы нарушили обещание;
- `-20`: мы дали консервативное обещание и могли потерять конверсию.

Условие говорит, что большой недопрогноз дороже. Значит, особенно важен **правый хвост** signed error.

Сильный DS не начинает с теста. Он начинает с вопросов:

- Как определён знак ошибки?
- Какова денежная стоимость каждого направления?
- Где начинается «тяжёлый» недопрогноз?
- Что является пользовательским SLA?

---

# 4. Что такое распределение ошибок

Каждый заказ создаёт одну ошибку. По тысячам заказов получается распределение.

Пример:

```text
[-5, -3, -2, 0, 1, 2, 3, 4, 6, 35]
```

Большинство заказов имеют небольшие ошибки. Один заказ имеет ошибку 35 минут.

Этот редкий тяжёлый случай находится в правом хвосте.

Хвост важен, когда редкое событие имеет высокую цену:

- крупное опоздание;
- очень большая финансовая потеря;
- extreme latency;
- отказ критического оборудования;
- тяжёлая ошибка безопасности.

---

# 5. Почему среднее может скрыть проблему

Рассмотрим две модели.

## Старая модель

```text
[-1, -1, -1, -1, 9]
```

Среднее:

```text
1
```

## Новая модель

```text
[1, 1, 1, 1, 1]
```

Среднее:

```text
1
```

Средние совпадают, но продуктово модели разные.

У старой модели:

- четыре небольших перепрогноза;
- один тяжёлый недопрогноз;
- опасный правый хвост.

У новой модели:

- одинаковая небольшая ошибка;
- нет катастрофического хвоста.

Проверка только средних не обнаружит эту разницу.

---

# 6. Почему урок называется QUANTILE T-TEST

Основная история урока — сравнение квантилей, то есть точек распределения.

Например:

- p50 — медиана;
- p90 — значение, ниже которого находится 90% наблюдений;
- p95 — значение, ниже которого находится 95%;
- p99 — значение, ниже которого находится 99%.

Для ETA можно сказать:

> p95 signed error равен 18 минутам.

Это означает, что примерно 95% ошибок не превышают 18 минут, а худшие 5% находятся выше.

Но конкретная задача первой страницы пока не просит реализовать тест квантилей. Она просит проверить среднее как дополнительную защитную метрику.

---

# 7. Primary metric и guardrail metric

## Primary metric

Это основная метрика, соответствующая главной гипотезе эксперимента.

В этой истории возможные primary metrics:

- p95 положительной ETA-ошибки;
- p99 положительной ETA-ошибки;
- доля заказов с недопрогнозом больше 15 минут;
- средняя ошибка в худших 5% заказов;
- Expected Shortfall правого хвоста.

## Guardrail metric

Это защитная метрика, которая не должна ухудшиться.

Средняя ошибка здесь является guardrail.

Почему это важно? Команда может «улучшить» хвост очень примитивно:

```text
new_eta = old_eta + 20 minutes
```

Теперь тяжёлых недопрогнозов меньше, но:

- все ETA стали слишком длинными;
- конверсия может упасть;
- сервис выглядит медленным;
- средняя signed error сместится в отрицательную сторону.

Следовательно, хвост можно улучшить неправильным способом. Проверка среднего помогает заметить такой побочный эффект.

# 8. Что дано на вход функции

```python
control: List[float]
experiment: List[float]
```

## Control

Ошибки старого алгоритма в группе A.

```python
control = [2.0, 4.0, -1.0, 7.0]
```

## Experiment

Ошибки нового алгоритма в группе B.

```python
experiment = [1.0, 0.0, 2.0, 3.0]
```

Наблюдения принадлежат двум независимым экспериментальным группам.

---

# 9. Почему недостаточно сравнить два числа

Можно вычислить:

```text
mean_A = 5.2
mean_B = 4.8
```

Разница:

```text
-0.4
```

Но сама разница не говорит, является ли она реальным эффектом модели или случайным шумом.

Нужно учитывать:

- размеры групп;
- дисперсию;
- стандартную ошибку;
- экспериментальный дизайн;
- независимость наблюдений.

Статистический тест отвечает на вопрос:

> Насколько необычна наблюдаемая разница, если истинные средние равны?

---

# 10. Нулевая гипотеза

Обозначим:

```text
mu_A = истинная средняя ошибка старой модели
mu_B = истинная средняя ошибка новой модели
```

Нулевая гипотеза:

```text
H0: mu_A = mu_B
```

Смысл:

> различие выборочных средних можно объяснить случайностью.

---

# 11. Альтернативная гипотеза

```text
H1: mu_A != mu_B
```

Это двухсторонняя альтернатива.

Для guardrail нас интересует любое неожиданное изменение:

- среднее стало выше;
- среднее стало ниже.

Поэтому дефолтный two-sided t-test соответствует постановке.

---

# 12. Что такое p-value

`p-value` — вероятность увидеть наблюдаемое или ещё более экстремальное расхождение при условии, что H0 верна и предпосылки теста выполнены.

Неправильная интерпретация:

> p-value = 0.03, значит вероятность H0 равна 3%.

Правильнее:

> Если истинные средние равны, получить настолько сильное или более сильное расхождение было бы маловероятно — около 3% при используемой модели теста.

---

# 13. Что такое alpha

`alpha` — заранее выбранный порог статистической значимости.

По умолчанию:

```python
alpha = 0.05
```

Правило:

```text
p_value < alpha  -> различие статистически значимо
p_value >= alpha -> недостаточно оснований отвергнуть H0
```

Важно: «не отвергли H0» не означает «доказали равенство».

---

# 14. Почему тест с alpha=0.01 строже

Допустим:

```text
p_value = 0.03
```

При `alpha=0.05`:

```text
0.03 < 0.05 -> True
```

При `alpha=0.01`:

```text
0.03 < 0.01 -> False
```

Именно поэтому функция должна использовать аргумент `alpha`, а не жёстко зашитое значение.

---

# 15. Какой тест нужен

Используется:

```python
scipy.stats.ttest_ind
```

`ind` означает independent — независимые выборки.

Это не:

- `ttest_1samp` — тест одной выборки против фиксированного среднего;
- `ttest_rel` — парный тест;
- Mann–Whitney;
- тест пропорций;
- квантильный bootstrap.

---

# 16. Почему нужно оставить дефолтные параметры SciPy

Условие явно требует использовать дефолтные параметры.

Правильный вызов:

```python
ttest_ind(control, experiment)
```

Не нужно добавлять:

```python
equal_var=False
```

хотя Welch t-test часто полезен в реальной работе.

Дефолт SciPy использует стандартный независимый t-тест с предположением равенства дисперсий и двухсторонней альтернативой.

В учебной задаче важна точность соответствия API и эталонной реализации.

---

# 17. Что возвращает SciPy

`ttest_ind` возвращает статистику теста и p-value.

Концептуально:

```python
statistic, p_value = ttest_ind(control, experiment)
```

В условии нужна только p-value, поэтому статистику можно проигнорировать:

```python
_, p_value = ttest_ind(control, experiment)
```

---

# 18. Что такое t-статистика

Упрощённо:

```text
t = signal / noise
```

Где:

- signal — разница средних;
- noise — стандартная ошибка этой разницы.

Если разница средних велика относительно случайного разброса, абсолютное значение t становится большим, а p-value обычно уменьшается.

---

# 19. Почему группы могут иметь разный размер

Например:

```text
A: 10 000 заказов
B: 3 000 заказов
```

Независимый t-тест умеет работать с разными объёмами выборок.

Но аналитик должен спросить, почему возник дисбаланс.

Возможные причины:

- плановый сплит 80/20;
- staged rollout;
- баг рандомизации;
- потери логов;
- модель B чаще падает;
- eligibility filter применяется по-разному;
- фильтрация после назначения группы.

То есть:

- математически тест вычисляется;
- экспериментально дисбаланс может быть сигналом проблемы.

---

# 20. Sample Ratio Mismatch

Если ожидали 50/50, а получили 63/37, нужно проверить SRM.

SRM может означать:

- сломанный assignment;
- data loss;
- cross-over;
- разную доступность варианта;
- зависимость включения в анализ от результата.

Сильный анализ A/B начинается не с p-value метрики, а с проверки целостности эксперимента.

---

# 21. A/A-тест

A/A-тест — обе группы фактически получают одинаковый алгоритм.

Он нужен, чтобы проверить:

- рандомизацию;
- сбор событий;
- статистический пайплайн;
- ложноположительную частоту;
- SRM.

При `alpha=0.05` даже корректный A/A иногда даст ложноположительный результат. В долгой серии независимых тестов это ожидаемо примерно в 5% случаев при выполнении предпосылок.

Автотест платформы, вероятно, использует данные с явно большим p-value.

---

# 22. Ошибка первого рода

Мы объявили эффект, хотя его нет.

```text
False Positive
```

В ETA-задаче:

- средние на самом деле одинаковы;
- тест показывает различие;
- команда делает неверный вывод.

Alpha контролирует вероятность такого решения в рамках статистической модели.

---

# 23. Ошибка второго рода

Реальное различие есть, но тест его не обнаружил.

```text
False Negative
```

Причины:

- мало данных;
- высокий шум;
- слабый эффект;
- зависимые наблюдения;
- неправильная метрика;
- плохой экспериментальный дизайн.

Связанное понятие — power:

```text
power = 1 - beta
```

---

# 24. Статистическая и практическая значимость

На миллионах заказов разница в `0.01` минуты может стать статистически значимой.

Но бизнесу она может ничего не давать.

Поэтому кроме p-value нужны:

- effect size;
- доверительный интервал;
- MDE;
- денежная оценка;
- влияние на пользователя;
- влияние на операции.

В учебной функции требуется только p-value и bool, но настоящий аналитический вывод должен быть шире.

# 25. Предпосылки t-теста

## 25.1 Независимость

Если один пользователь создаёт много заказов, его наблюдения коррелированы.

Проблема:

- рандомизация по пользователю;
- анализ как будто каждый заказ независим.

Это может занизить стандартные ошибки.

Практические решения:

- агрегировать до уровня пользователя;
- кластерный bootstrap;
- cluster-robust standard errors;
- mixed-effects model.

В учебной задаче это не реализуется, но на работе это критично.

## 25.2 Числовая метрика

ETA-ошибка является количественной величиной.

## 25.3 Поведение среднего

Для больших выборок центральная предельная теорема часто делает распределение среднего близким к нормальному.

Тяжёлые хвосты могут потребовать больше данных и robust-проверок.

## 25.4 Равенство дисперсий

Дефолтный тест предполагает равные дисперсии.

В production часто рассматривают Welch, но здесь по условию оставляем дефолт.

---

# 26. Почему Mann–Whitney не является «тестом на всякий случай»

Распространённая ошибка:

> данные ненормальные, значит использую Mann–Whitney.

Но Mann–Whitney не является прямым тестом равенства средних.

Выбор метода начинается с того, что именно мы хотим оценить:

- среднее;
- медиану;
- квантиль;
- долю;
- всю форму распределения.

Это называется estimand.

---

# 27. Примеры estimand

Разница средних:

```text
E[error_B] - E[error_A]
```

Разница p95:

```text
Q0.95(error_B) - Q0.95(error_A)
```

Разница доли тяжёлых опозданий:

```text
P(error_B > 15) - P(error_A > 15)
```

Средняя ошибка в худшем хвосте:

```text
ES95_B - ES95_A
```

Сначала фиксируется estimand, затем выбирается тест.

# 28. Как сравнивать квантили в реальной задаче

## Bootstrap

1. Сэмплировать A с возвращением.
2. Сэмплировать B с возвращением.
3. Посчитать p95 каждой группы.
4. Посчитать разницу.
5. Повторить много раз.
6. Построить доверительный интервал.

Если рандомизация по пользователю, bootstrap нужен по пользователям, а не по отдельным заказам.

## Пермутационный тест

1. Объединить данные.
2. Перемешать метки A/B.
3. Посчитать разницу квантилей.
4. Сформировать нулевое распределение.
5. Сравнить наблюдаемую разницу.

## Quantile regression

Позволяет оценить treatment effect на условный квантиль с ковариатами.

## Доля превышений

Вместо p99 можно определить бизнес-порог:

```text
error > 15 minutes
```

и сравнивать доли.

## Expected Shortfall

Среднее среди худших 5% может быть устойчивее и информативнее одной крайней точки.

---

# 29. Почему крайние квантили требуют много данных

p99 основан на верхнем 1%.

Если в группе 1 000 наблюдений, хвост около p99 фактически представлен примерно десятью заказами.

Следовательно:

- высокая дисперсия оценки;
- широкий confidence interval;
- низкая мощность;
- сильная чувствительность к сегментам;
- риск нестабильного решения.

Нельзя автоматически выбирать p99 только потому, что он звучит «надёжнее».

---

# 30. Типичные ошибки продуктового анализа

1. Выбрать метрику после просмотра результатов.
2. Проверить p90, p95 и p99 и сообщить только значимый.
3. Смешать недопрогноз и перепрогноз в absolute error.
4. Исключить отменённые заказы и получить selection bias.
5. Рандомизировать пользователя, а анализировать заказы как независимые.
6. Не проверить SRM.
7. Смотреть только p-value.
8. Не учитывать latency новой модели.
9. Не учитывать fallback.
10. Не проверить конверсию и компенсационные расходы.

# 34. Two-sided marketplace

В такси, доставке, аренде и фрилансе есть минимум две стороны.

Изменение для клиента может ухудшить:

- доход исполнителя;
- utilisation;
- acceptance rate;
- supply liquidity.

# 42. Финальное решение страницы

```python
from typing import List, Tuple

from scipy.stats import ttest_ind


def ttest(
    control: List[float],
    experiment: List[float],
    alpha: float = 0.05,
) -> Tuple[float, bool]:
    """Two-sample t-test for the means of two independent samples."""
    _, p_value = ttest_ind(control, experiment)
    result = bool(p_value < alpha)

    return float(p_value), result
```

# 43. Итог

Нужно удерживать две мысли одновременно:

1. Среднее не описывает хвост, поэтому не является главной метрикой для заявленной бизнес-гипотезы.
2. Среднее полезно как guardrail, чтобы новая модель не улучшила хвост ценой общего нежелательного сдвига.

Именно второй пункт реализуется в задаче этой страницы.
