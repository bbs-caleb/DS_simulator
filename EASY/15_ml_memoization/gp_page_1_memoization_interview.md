# Мемоизация на продуктовом и System Design интервью

## 1. Что проверяет интервьюер

Проверяется не знание словаря Python, а способность:

- связать технику с бизнесом;
- уточнить требования;
- оценить нагрузку;
- выбрать компромиссы;
- обеспечить корректность;
- спроектировать метрики;
- учесть сбои.

Слабый ответ:

> Положим результат в Redis, и всё станет быстрее.

Сильный ответ:

> Сначала измерю повторяемость запросов и стоимость вычисления. Затем определю семантически корректный ключ, свежесть, TTL, лимит памяти, fallback и метрики. Для ML включу в ключ версию модели и контекст.

# Часть I. Продуктовое мышление

## 2. Перевод на язык бизнеса

Для рекомендаций:

```text
Высокая latency
→ контент появляется поздно
→ пользователь уходит
→ падает CTR
→ падает конверсия или выручка
```

Для банка:

```text
Долгий скоринг
→ дольше анкета
→ больше незавершённых заявок
→ меньше выдач
```

Для антифрода:

```text
Долгая проверка
→ платёжный timeout
→ легитимный платёж отклонён
→ падает approval rate
```

---

## 3. Вопросы в начале интервью

### Пользователь и сценарий

- Кто вызывает сервис?
- Это online или batch?
- Как задержка влияет на пользователя?
- Возможен ли fallback?

### SLA

- Какой p50, p95, p99?
- Какая доступность?
- Какой error rate?
- Есть ли внешний timeout?

### Нагрузка

- Средний RPS?
- Пиковый RPS?
- Есть ли горячие ключи?
- Какова доля повторов?

### Свежесть

- Как быстро меняется ответ?
- Допустим ли ответ минутной давности?
- Что хуже: старый или медленный ответ?
- Какие события требуют удаления кэша?

### Безопасность

- Есть ли персонализация?
- Есть ли PII?
- Влияют ли права доступа?
- Можно ли разделять кэш между пользователями?

### ML

- Как часто обновляется модель?
- Есть ли A/B-тесты?
- Как версионируются признаки?
- Детерминирован ли inference?
- Какой контекст влияет на ответ?

# Часть II. System Design пример

## 4. Задача

> Спроектируйте сервис рекомендаций на 5000 RPS с p95 меньше 50 ms.

---

## 5. Путь запроса

```text
Client
→ API Gateway
→ Recommendation Service
→ User Feature Store
→ Candidate Generator
→ Item Feature Store
→ Ranking Model
→ Business Rules
→ Response
```

Пример latency budget:

| Компонент | Бюджет |
|---|---:|
| Сеть и gateway | 5 ms |
| User features | 8 ms |
| Candidates | 10 ms |
| Item features | 8 ms |
| Ranking | 12 ms |
| Постобработка | 4 ms |
| Запас | 3 ms |
| Всего | 50 ms |

Кэш можно ставить вокруг разных этапов.

---

## 6. Кэш полного ответа

Ключ:

```text
user_id + context + model_version
```

Плюсы:

- максимальная экономия;
- очень быстрый hit.

Минусы:

- быстро устаревает;
- высокая кардинальность;
- сложная персонализация.

---

## 7. Кэш кандидатов

```text
user_id + candidate_model_version
```

После hit можно заново выполнить свежее ранжирование.

---

## 8. Кэш embeddings

```text
entity_id + embedding_model_version + entity_version
```

Подходит для редко меняющихся сущностей.

---

## 9. Кэш признаков

```text
feature_name + entity_id + feature_snapshot
```

Нужен строгий контроль свежести.

---

# Часть III. Ключ

## 10. Полнота ключа

Ключ должен включать всё, от чего зависит результат.

Плохо:

```text
user_id
```

если ответ зависит ещё от:

- региона;
- устройства;
- времени;
- модели;
- эксперимента;
- прав.

---

## 11. Версия модели

```text
recommendations:{user_id}:{context_hash}:{model_version}
```

Новая модель создаёт другое пространство ключей.

---

## 12. A/B-тест

В ключ входят:

```text
experiment_id + variant
```

Иначе варианты смешиваются и эксперимент становится некорректным.

---

## 13. Права доступа

В корпоративной системе ключ может включать:

- tenant;
- account;
- permission set;
- user;
- область доступа.

Иначе возможна утечка данных.

---

# Часть IV. TTL и инвалидирование

## 14. TTL по бизнес-семантике

| Данные | Подход |
|---|---|
| Статическая категория | часы или дни |
| Embedding неизменного текста | долго |
| Рекомендации | минуты |
| Остаток | секунды |
| Баланс | очень коротко или не кэшировать |
| Историческая транзакция | долго |

---

## 15. Цена устаревания

Для развлечений рекомендация двухминутной давности может быть приемлема.

Для банковского баланса — нет.

Для последней единицы товара старый ответ может привести к отмене заказа.

---

## 16. Стратегии

- TTL-based;
- event-based invalidation;
- versioned key;
- manual purge;
- write-through;
- refresh-ahead.

---

# Часть V. Оценки

## 17. Память

```text
1 000 000 пользователей
× 10 KB на ответ
= примерно 10 GB
```

Реально больше из-за overhead и репликации.

---

## 18. Экономия вычислений

```text
RPS = 5000
hit rate = 80%
hits = 4000 RPS
misses = 1000 RPS
```

Дорогой pipeline запускается только для 1000 RPS.

Но учитываются:

- lookup;
- сеть;
- сериализация;
- запись;
- репликация.

---

# Часть VI. Cache stampede

## 19. Проблема

Популярный ключ истёк.

10 000 запросов одновременно видят miss и запускают одно тяжёлое вычисление.

Последствия:

- пик CPU/GPU;
- падение базы;
- рост latency;
- каскадный отказ.

---

## 20. Защита

- lock per key;
- single-flight;
- stale-while-revalidate;
- TTL jitter;
- pre-warming;
- rate limiting.

---

# Часть VII. Отказоустойчивость

## 21. Кэш не source of truth

Источником истины остаётся:

- база;
- event log;
- feature store;
- объектное хранилище.

При падении кэша возможны:

- обращение к source of truth;
- fallback;
- деградация;
- ограничение нагрузки;
- техническая ошибка.

---

## 22. Graceful degradation

Рекомендации:

```text
Персональные недоступны
→ показать популярные товары региона
```

Поиск:

```text
ML ranking недоступен
→ базовая сортировка
```

Финтех:

```text
Нет безопасно актуального решения
→ ручная проверка или безопасный отказ
```

# Часть IX. Готовый сильный ответ

## 25. Шаблон

> Сначала уточню SLA, пиковый RPS, долю повторных запросов и допустимую свежесть. Затем разложу latency по этапам, чтобы понять, где кэш даст наибольший эффект.
>
> Для ML-сервиса рассмотрю кэш полного ответа, кандидатов, embeddings и признаков. Ключ должен включать идентификатор сущности, контекст, версию модели, версию признаков и экспериментальную группу.
>
> TTL выберу по цене устаревания. Для embeddings он может быть длинным, для остатка или финансового состояния — коротким либо кэширование недопустимо.
>
> Для горячих ключей добавлю single-flight, jitter и stale-while-revalidate, если бизнес допускает старый ответ. Кэш не считаю source of truth; предусмотрю fallback.
>
> Эффект измерю по hit rate, p95/p99, числу вызовов модели, стоимости инфраструктуры и продуктовым метрикам. Отдельно проверю отсутствие смешивания пользователей, A/B-вариантов и версий модели.

# Часть X. Индустриальные акценты

## 26. Финтех

Нужно говорить про:

- актуальность;
- безопасность;
- аудит;
- PII;
- идемпотентность;
- версию модели;
- состояние признаков.

Сильная мысль:

> Решение нельзя кэшировать только по client_id, потому что оно зависит от операции, времени, признаков и версии модели.

---

## 27. E-commerce и marketplace

Нужно говорить про:

- цену;
- остаток;
- регион;
- доставку;
- персонализацию;
- промо;
- продавца.

Сильная мысль:

> Кэшировать embedding безопаснее, чем карточку с текущей ценой и остатком.

---

## 28. Two-sided marketplace

Ключ учитывает состояние обеих сторон, регион, время и доступность.

---

## 29. iGaming

Нельзя мемоизировать функцию, которая обязана генерировать новый независимый случайный исход.

---

## 30. Crypto

Результат анализа состояния нужно привязывать к блоку или хешу состояния.

---

## 31. Dating

Даже при кэше кандидатов перед показом проверяются свежие блокировки, удаление профиля и safety-правила.

---

# Часть XI. Частые ошибки

## 32. «Возьмём Redis»

Нужно объяснить:

- ключ;
- TTL;
- ёмкость;
- отказ;
- метрики;
- пользу.

---

## 33. Нет версии модели

Старые ответы продолжают возвращаться после релиза.

---

## 34. Нет experiment variant

A/B-группы смешиваются.

---

## 35. Нет user/tenant в ключе

Риск утечки данных.

---

## 36. Нет лимита памяти

Кэш может положить сервис.

---

## 37. Нет fallback

Отказ кэша становится отказом всего продукта.
