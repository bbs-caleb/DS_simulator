# NEGATIVE TESTS: продуктовое и ML System Design интервью

## 1. Ответ за одну минуту

> Негативные тесты проверяют поведение системы на некорректных входах, нарушениях бизнес-инвариантов и отказах зависимостей. Цель не просто увидеть падение, а убедиться, что система отказывает предсказуемо: в правильном месте, с правильным типом ошибки и без частичного повреждения данных. В ML-системе я тестирую схемы данных, диапазоны признаков, `NaN`, несовместимую размерность, недоступность feature store, таймауты, дубликаты событий и нарушения постфильтров.

Такой ответ показывает, что кандидат понимает:
- тестирование;
- данные;
- бизнесовые ограничения;
- production ML;
- отказоустойчивость.

---

# 2. Универсальная структура ответа на бизнес-кейс

Когда вас спрашивают, как тестировать функцию или сервис:

1. Определите бизнесовую цель.
2. Сформулируйте допустимые входы.
3. Назовите бизнес-инварианты.
4. Разделите ошибки на типы.
5. Объясните последствия.
6. Назначьте безопасное поведение.
7. Добавьте мониторинг.
8. Разделите unit, integration и end-to-end тесты.

Для CTR:

## Цель

Оценивать привлекательность объявления, рекомендации или элемента интерфейса.

## Инварианты

```text
clicks >= 0
views > 0
clicks <= views
```

## Возможные причины нарушения

- дублирование кликов;
- потеря показов;
- разные временные окна;
- неправильный тип после изменения схемы;
- повторная обработка события;
- ошибка агрегации.

## Последствия

- неправильный бюджет;
- ложный результат A/B-теста;
- ошибочное ранжирование;
- спор с рекламодателем.

## Защита

- валидация;
- негативные unit-тесты;
- data quality checks;
- метрики отклонённых записей;
- алерт на рост ошибок.

---

# 3. Вопрос: какие негативные тесты нужны для CTR

Базовые:
- `clicks` не `int`;
- `views` не `int`;
- отрицательные клики;
- отрицательные просмотры;
- нулевые просмотры;
- кликов больше просмотров.

Расширенные:
- `None`;
- строка;
- bool;
- слишком большие значения;
- `NaN` и `inf`, если функция принимает float;
- несовпадающие окна;
- дубликаты событий.

Важно пояснить уровень:
- типы и диапазоны — unit;
- дубликаты и окна — integration/data quality.

---

# 4. Почему нельзя вернуть 0 при любой ошибке

> Нулевой CTR — валидный бизнесовый результат: показы были, кликов не было. Если техническую ошибку заменить нулём, мы смешаем сбой и настоящий результат. Ошибка станет невидимой, метрики исказятся, а бизнес примет неправильное решение.

Это пример опасности «тихой деградации».

---

# 5. Когда исключение, когда fallback

Исключение уместно, когда:
- нарушен обязательный контракт;
- результат нельзя интерпретировать;
- продолжение повреждает данные;
- операция финансовая или регуляторная;
- вызывающий код обязан решить, что делать.

Fallback уместен, когда:
- есть заранее согласованное безопасное поведение;
- fallback соблюдает все бизнес-фильтры;
- его использование измеряется;
- качество может временно снизиться без критического риска.

Пример:
- рекомендации: можно показать популярные товары;
- платёж: нельзя «примерно» списать сумму;
- кредит: нельзя автоматически одобрить при отсутствии критических данных без утверждённой политики.

---

# 6. Что такое fail fast

> Fail fast — обнаружить нарушение максимально близко к источнику и остановить соответствующую операцию до распространения ошибочных данных.

Важно добавить:

> Это не обязательно означает уронить весь сервис. Можно отклонить один запрос, отправить событие в dead-letter queue, вернуть контролируемую ошибку или включить безопасный fallback.

---

# 7. Что такое бизнес-инвариант

> Бизнес-инвариант — условие, которое должно оставаться истинным во всех допустимых состояниях системы.

Примеры:
- кликов не больше просмотров;
- одна транзакция применяется один раз;
- один заказ не имеет двух активных курьеров;
- вероятность находится в `[0, 1]`;
- удалённый товар не показывается;
- ставка не выше баланса.

# 8. ML System Design: точки проверки

```text
Client
  ↓
API Gateway
  ↓
Schema Validation
  ↓
Feature Service
  ↓
Model Service
  ↓
Post-processing
  ↓
Response
```

## API Gateway
- авторизация;
- rate limit;
- размер запроса;
- формат payload.

## Schema Validation
- обязательные поля;
- типы;
- диапазоны;
- совместимость версии;
- согласованность полей.

## Feature Service
- сущность существует;
- признаки свежие;
- нет критичных `NaN`;
- размерность корректна;
- timestamp не из будущего;
- нет `inf`.

## Model Service
- модель загружена;
- версия совместима;
- score в диапазоне;
- latency в SLA;
- batch size разрешён.

## Post-processing
- нет дубликатов;
- соблюдены ограничения;
- top-k корректен;
- запрещённые объекты удалены;
- результат не пуст без объяснения.

## Response
- схема стабильна;
- есть request ID;
- код ошибки определён;
- персональные данные не попали в лог.

# 9. Как тестировать отказ feature store

Смоделировать:
- timeout;
- 500;
- пустой ответ;
- частичный ответ;
- устаревшие признаки;
- неправильную схему;
- высокую latency.

Проверить:
- timeout ограничен;
- retry не бесконечен;
- нет retry storm;
- fallback срабатывает только по правилам;
- ошибка логируется;
- метрика увеличивается;
- запрос не зависает;
- нет каскадного отказа.

---

# 10. Метрики негативных сценариев

- invalid request rate;
- ошибки по кодам;
- fallback rate;
- timeout rate;
- retry rate;
- dead-letter queue size;
- schema validation failures;
- feature missing rate;
- `NaN` rate;
- model load failures;
- empty response rate;
- post-filter rejection rate;
- p95/p99 latency;
- error budget burn rate.

Логи позволяют расследовать отдельный случай. Метрики показывают системную проблему.

---

# 11. Как избежать хрупких тестов

- Проверять публичный контракт.
- Не зависеть от внутренних переменных.
- Ловить конкретный тип исключения.
- Проверять текст сообщения только если он часть API.
- Один тест — одна причина.
- Использовать минимальные входы.
- Контролировать время и сеть.
- Мокать зависимости.
- Давать ясные имена.

В этом задании проверяется тип исключения, а не точное сообщение. Это разумно, потому что формулировка может измениться без изменения контракта.

# 12. System Design кейс: сервис рекламных метрик

## Запрос

```json
{
  "campaign_id": "abc",
  "clicks": 150,
  "views": 10000
}
```

## Негативные сценарии

- поле отсутствует;
- неправильный тип;
- отрицательное число;
- `clicks > views`;
- `views = 0`;
- неизвестная кампания;
- запрос повторён;
- окно уже закрыто;
- БД недоступна.

## Ошибка API

```json
{
  "error_code": "CLICKS_GREATER_THAN_VIEWS",
  "message": "clicks must be less than or equal to views",
  "request_id": "..."
}
```

Возможные HTTP-коды:
- `400` — неправильный формат;
- `404` — сущность не найдена;
- `409` — конфликт состояния;
- `422` — семантически недопустимые значения;
- `500` — внутренняя ошибка;
- `503` — временная недоступность.

## Защита от частичной записи

- валидация до записи;
- транзакция;
- idempotency key;
- unique constraint;
- audit log;
- dead-letter queue.

# 13. Частые слабые ответы

## «Проверю, что функция работает»

Нет конкретики. Нужно назвать входы, инварианты и реакцию.

## «Оберну всё в try/except»

Неясно:
- какое исключение;
- где ловить;
- что делать;
- как не скрыть проблему.

## «Верну None»

`None` легко забыть проверить. Ошибка появится позже и дальше от причины.

## «Проверю модель»

В production ломаются также:
- схема;
- данные;
- feature store;
- сеть;
- постобработка;
- бизнес-фильтры.

## «Все ошибки — 500»

Клиентские ошибки должны отличаться от внутренних.

---

# 14. Готовый сильный шаблон ответа

> Сначала я определю контракт и бизнес-инварианты. Затем разобью негативные сценарии на неправильный тип, неправильный диапазон, противоречие полей, отсутствие данных, дубликаты и отказ зависимостей. На unit-уровне проверю конкретные исключения. На интеграционном уровне проверю схему API, БД и внешние сервисы. На end-to-end уровне проверю, что ошибка не оставляет частичных побочных эффектов. Для каждого сценария выберу отказ, retry, fallback или ручную обработку. Затем добавлю метрики по типам ошибок и алерты на рост их доли.

Этот шаблон подходит для финтеха, маркетплейсов, ML-сервисов и большинства продуктовых систем.
