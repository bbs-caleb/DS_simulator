# NEGATIVE TESTS: теория, бизнесовый смысл и инженерное мышление

## 1. Что изучаем

Негативное тестирование проверяет не «умеет ли функция работать на красивом примере», а «умеет ли она правильно отказаться от работы, когда входные данные нарушают контракт».

Это критично для промышленного кода. В реальных системах данные бывают:
- неправильного типа;
- отрицательными;
- пустыми;
- противоречивыми;
- устаревшими;
- продублированными;
- частично потерянными;
- повреждёнными при передаче;
- несовместимыми с новой версией схемы.

Хорошая система не обязана успешно обработать любые данные. Но она обязана предсказуемо обработать любой ожидаемый класс ошибки.

---

## 2. Позитивный и негативный тест

Позитивный тест:

```python
assert ctr(20, 100) == 0.2
```

Он проверяет счастливый путь.

Негативный тест:

```python
try:
    ctr(-1, 100)
except ValueError:
    pass
else:
    raise AssertionError
```

Он проверяет, что недопустимое значение отвергается.

Ключевой момент: тест не просто ждёт «любого падения». Он ждёт конкретный тип ошибки. Это часть контракта функции.

---

## 3. Что такое контракт функции

Для функции:

```python
def ctr(clicks: int, views: int) -> float:
```

контракт можно выразить так:

### Допустимо

```text
clicks — целое число
views — целое число
clicks >= 0
views > 0
clicks <= views
```

### Результат

```text
clicks / views
```

### Недопустимо

```text
нецелые clicks
нецелые views
clicks < 0
views < 0
clicks > views
views == 0
```

Тесты превращают этот контракт в исполняемую спецификацию. Если кто-то изменит функцию и сломает правило, тест сообщит об этом автоматически.

---

## 4. Почему тип исключения важен

### `TypeError`

Используется, когда объект имеет неподходящий тип.

```python
ctr(1.5, 10)
ctr("1", 10)
ctr(None, 10)
```

Количество событий моделируется целым числом. Полтора клика в данной модели невозможно.

### `ValueError`

Тип правильный, но значение недопустимо.

```python
ctr(-1, 10)
ctr(20, 10)
```

`-1` и `20` — целые числа, но они нарушают бизнесовые правила.

### `ZeroDivisionError`

Возникает при невозможности деления на ноль.

```python
ctr(0, 0)
```

Важно различать ошибки. Вызывающая система может по-разному реагировать:
- на неправильную схему запроса;
- на нарушение бизнес-инварианта;
- на отсутствие знаменателя.

---

## 5. Fail fast

Fail fast означает: обнаружить нарушение максимально близко к источнику и не пропустить повреждённые данные дальше.

Представим пайплайн:

```text
события → агрегация → признаки → модель → бизнес-решение → отчёт
```

Если отрицательные просмотры не остановить сразу:
1. CTR станет отрицательным;
2. неправильный CTR попадёт в feature store;
3. модель получит значения вне обучающего диапазона;
4. решение станет неверным;
5. бизнес увидит падение метрики;
6. команда будет искать проблему в модели;
7. реальная причина найдётся намного позже.

Ранняя ошибка дешевле поздней.

Плохой код:

```python
def ctr(clicks, views):
    try:
        return clicks / views
    except Exception:
        return 0
```

Почему опасно:
- ноль является настоящим значением CTR;
- технический сбой маскируется под бизнес-результат;
- мониторинг может ничего не заметить;
- неверные данные продолжат движение.

## 6. Разбор функции из задания

```python
def ctr(clicks: int, views: int) -> float:
    if not isinstance(clicks, int):
        raise TypeError(...)

    if not isinstance(views, int):
        raise TypeError(...)

    if clicks < 0:
        raise ValueError(...)

    if views < 0:
        raise ValueError(...)

    if views < clicks:
        raise ValueError(...)

    if views:
        return clicks / views
    else:
        raise ZeroDivisionError(...)
```

Проверки выполняются сверху вниз. Порядок важен.

Вызов:

```python
ctr(1, 0)
```

не даст `ZeroDivisionError`. До него функция увидит:

```python
views < clicks
0 < 1
```

и выбросит `ValueError`.

Чтобы проверить именно нулевые просмотры, используем:

```python
ctr(0, 0)
```

Тогда `0 < 0` ложно, и код доходит до финальной ветки.

Это центральная ловушка задания.

---

## 7. Структура `try / except / else`

```python
try:
    metrics.ctr(...)
except ExpectedError:
    pass
else:
    raise AssertionError(...)
```

### `try`

Выполняется потенциально ошибочный вызов.

### `except ExpectedError`

Если возник именно ожидаемый тип исключения, тест успешен.

### `pass`

Ничего не делать. Исключение уже подтвердило нужное поведение.

### `else`

Выполняется, если исключения вообще не было.

Для негативного теста это провал: функция приняла недопустимые данные.

### `AssertionError`

Явно помечает тест как неуспешный.

Если функция выбросит другой тип ошибки, он не будет перехвачен и тест тоже упадёт. Это правильно: «упало как-нибудь» не равно «контракт выполнен».

---

## 8. Почему нельзя ловить `Exception`

Плохой тест:

```python
try:
    metrics.ctr(1.5, 2)
except Exception:
    pass
```

Он пройдёт при:
- `TypeError`;
- `ValueError`;
- `NameError`;
- `AttributeError`;
- случайной внутренней ошибке.

Такой тест ничего не говорит о точности контракта.

Правильно:

```python
except TypeError:
```

---

## 9. Один тест — одно нарушение

Хороший unit-тест изолирует одну причину.

Для отрицательных просмотров:

```python
metrics.ctr(0, -1)
```

Клики корректны, типы корректны, проблема только в `views`.

Для кликов больше просмотров:

```python
metrics.ctr(3, 2)
```

Оба значения целые и положительные. Нарушено только отношение параметров.

Чем меньше причин одновременно, тем легче понять падение.

## 10. CTR как бизнесовая метрика

CTR — Click-Through Rate:

```text
CTR = clicks / views
```

Пример:

```text
300 кликов / 10 000 просмотров = 0.03 = 3%
```

CTR используется:
- в рекламе;
- поисковой выдаче;
- рекомендациях;
- push-уведомлениях;
- email-рассылках;
- карточках товаров;
- лентах контента;
- промо-баннерах;
- игровых офферах;
- matching-системах.

Неправильный CTR способен привести к:
- неверному распределению рекламного бюджета;
- неправильному победителю A/B-теста;
- деградации ранжирования;
- ошибочной оплате партнёру;
- неверной оценке продукта.

Поэтому даже простая функция метрики имеет бизнесовые инварианты.



# 11. Бизнес-инварианты

Инвариант — условие, которое должно быть истинным во всех допустимых состояниях.

Для CTR:

```text
clicks >= 0
views > 0
clicks <= views
0 <= CTR <= 1
```

Но важно понимать допущения. `clicks <= views` корректно, если:
- события относятся к одной сущности;
- временное окно одинаково;
- клик считается максимум один раз на просмотр;
- нет особой модели множественных кликов.

В реальном проекте сначала уточняют определение метрики, затем кодируют инварианты.

Примеры инвариантов в других системах:
- баланс не изменяется дважды одной транзакцией;
- один заказ не назначен двум курьерам;
- вероятность модели находится в `[0, 1]`;
- удалённый товар не показывается;
- пользователь не видит самого себя в dating-рекомендациях;
- ставка не превышает доступный баланс.

Негативные тесты намеренно пытаются эти условия нарушить.

# 12. Финтех

## Кредитный скоринг

Негативные входы:
- возраст отрицательный;
- дата рождения в будущем;
- доход строкой;
- срок кредита равен нулю;
- сумма кредита отрицательная;
- валюта неизвестна;
- обязательный признак отсутствует;
- модель вернула `NaN`;
- версия признаков не совпала с моделью.

Бизнесовые последствия:
- неверное одобрение;
- кредитный убыток;
- нарушение риск-политики;
- регуляторный риск;
- невозможность объяснить решение.

Безопасное поведение зависит от политики:
- отклонить запрос;
- отправить на ручную проверку;
- запросить данные повторно;
- применить только заранее утверждённый fallback.

## Платежи

Негативные сценарии:
- сумма отрицательная;
- валюта не поддерживается;
- транзакция продублирована;
- подпись неверна;
- idempotency key отсутствует;
- счёт заблокирован;
- баланс недостаточен;
- timestamp слишком старый.

Главный принцип: ошибка не должна приводить к частичному списанию.

# 13. Маркетплейс и e-commerce

## Цены

Проверяем:
- цена отрицательная;
- скидка больше 100%;
- цена после скидки выше исходной;
- валюта неизвестна;
- товар без продавца;
- отрицательный остаток;
- дробное количество штучного товара.

Риски:
- продажа ниже себестоимости;
- массовые отмены;
- компенсации;
- претензии продавцов;
- репутационный ущерб.

## Рекомендательная система

Проверяем:
- пользователь не найден;
- список кандидатов пуст;
- embedding содержит `NaN`;
- размерность embedding изменилась;
- есть дубликаты;
- товар удалён;
- товар недоступен в регионе;
- запрещённый товар прошёл фильтр;
- latency превысила SLA.

Безопасный fallback может быть популярным ассортиментом, но он должен:
- быть бизнесово утверждён;
- соблюдать фильтры;
- логироваться;
- иметь метрику частоты использования.

# 19. Негативные тесты во всём ML-пайплайне

```text
источник → валидация → признаки → модель → постобработка → API
```

## Источник
- файл отсутствует;
- таблица пуста;
- колонка исчезла;
- тип изменился;
- ключи дублируются;
- timestamp неверен.

## Feature engineering
- деление на ноль;
- все значения пропущены;
- неизвестная категория;
- пустое окно;
- `NaN` или `inf`;
- утечка таргета.

## Модель
- файл повреждён;
- версия несовместима;
- размерность неправильна;
- score вне диапазона;
- batch слишком большой;
- модель не загрузилась.

## Постобработка
- дубликаты;
- кандидатов меньше k;
- нарушены бизнес-фильтры;
- запрещённый объект;
- пустой результат.

## API
- JSON невалиден;
- обязательное поле отсутствует;
- тип неправильный;
- запрос слишком большой;
- авторизация отсутствует;
- зависимость недоступна;
- таймаут.

Модель — только один компонент. Большинство production-сбоев возникает вокруг неё.



# 20. Уровни тестирования

## Unit

Проверяют одну функцию без внешних зависимостей.

Это текущее задание.

## Integration

Проверяют взаимодействие:
- API и валидации;
- сервиса и БД;
- модели и feature store;
- очереди и consumer.

## End-to-end

Проверяют полный пользовательский сценарий и побочные эффекты.

Например, при невалидном бюджете:
- кампания не запускается;
- деньги не списываются;
- пользователь видит ошибку;
- событие аудита записано.

## Fault injection / chaos

Искусственно вызываются:
- timeout;
- 500 от зависимости;
- потеря сети;
- дубликат сообщения;
- недоступность БД;
- задержка ответа.

# 21. Как системно придумывать негативные тесты

Для каждого параметра проверяйте:

## Тип
- строка;
- `None`;
- список;
- словарь;
- float вместо int;
- bool.

## Диапазон
- отрицательное;
- ноль;
- максимум;
- слишком большое;
- `NaN`;
- `inf`.

## Пустота
- пустая строка;
- пустой список;
- пустой DataFrame;
- пустой файл.

## Связь полей
- начало позже конца;
- минимум больше максимума;

- дата закрытия раньше создания.

## Повторы
- duplicate event;

## Время
- дата в будущем;
- слишком старое событие;
- неправильная timezone;
- события не по порядку.

## Зависимости
- timeout;
- 500;
- пустой ответ;
- частичный ответ;
- несовместимая схема.

# 22. Особенность `bool` в Python

```python
isinstance(True, int)
```

даёт `True`, потому что `bool` — подкласс `int`.

Поэтому функция из задания примет:

```python
ctr(True, 10)
```

как `ctr(1, 10)`.

В учебном решении это не тестируется, потому что нужно реализовать строго перечисленные сценарии. Но на code review можно предложить:

```python
type(clicks) is int
```

или отдельный запрет `bool`.

Это пример важного различия:
- решение под контракт задания;
- дополнительные улучшения промышленного кода.

# 24. Итоговое инженерное мышление

Зрелый инженер задаёт не только вопрос:

> Как получить правильный ответ?

Он задаёт ещё вопросы:

> Какие входы допустимы?  
> Какие состояния невозможны?  
> Что произойдёт при нарушении?  
> Где должна остановиться обработка?  
> Не останется ли частично записанных данных?  
> Как ошибка будет замечена мониторингом?  
> Есть ли безопасный fallback?  
> Можно ли отличить технический сбой от настоящего нулевого результата?

Негативные тесты — это способ превратить ответы на эти вопросы в код.
