> **Важно для именно этой учебной отправки:** автотест требует, чтобы первой строкой файла была `predict:`. Поэтому `.PHONY` рассматривается ниже как общая теория и промышленная практика, но в отправляемый `Makefile` не добавляется.

# Урок 24. ML: Makefile — бизнесовый и технический контекст

## 1. Что мы изучаем в этой задаче

В этой задаче мы не пишем новую ML-модель и не меняем Python-код. Мы улучшаем **способ управления проектом**.

В исходном проекте есть три отдельных bash-скрипта:

```text
scripts/predict.sh
scripts/preprocess.sh
scripts/docker_build.sh
```

Каждый скрипт содержит одну команду. Задача требует перенести эти команды в единый файл `Makefile`, создать три цели:

```text
predict
preprocess
build-docker
```

После этого каталог `scripts/` больше не нужен.

На первый взгляд это маленькое изменение. Но оно тренирует очень важную инженерную привычку: пользователь проекта не должен помнить внутреннюю структуру репозитория и длинные команды. Ему нужен простой и единообразный интерфейс.

---

## 2. Бизнесовый контекст: почему компании вообще тратят время на Makefile

### 2.1. ML-проект — это не только модель

В учебной задаче ML-проект состоит из нескольких файлов. В реальной компании вокруг модели обычно есть десятки операций:

1. создать или обновить окружение;
2. загрузить данные;
3. проверить качество данных;
4. построить признаки;
5. обучить модель;
6. посчитать метрики;
7. сформировать прогноз;
8. запустить тесты;
9. проверить стиль кода;
10. собрать Docker-образ;
11. отправить образ в registry;
12. развернуть сервис;
13. выполнить миграции;
14. запустить локальный стенд;
15. удалить временные файлы.

Если каждая операция запускается своей длинной командой, люди начинают ошибаться:

- забывают аргумент;
- используют не то окружение;
- запускают скрипт не из того каталога;
- создают Docker-образ с другим тегом;
- выполняют шаги в неправильном порядке;
- передают новичкам устаревшую инструкцию;
- локально получают один результат, а в CI — другой.

Makefile превращает набор технических деталей в короткий интерфейс:

```bash
make data
make train
make test
make build
```

### 2.2. Как Makefile экономит деньги

Makefile напрямую не увеличивает выручку, но снижает инженерные издержки и операционные риски.

Рассмотрим команду из 10 специалистов. Если каждый теряет хотя бы 10 минут в день на поиск правильной команды, исправление путей и восстановление окружения, команда теряет:

```text
10 человек × 10 минут × 20 рабочих дней = 2000 минут
```

Это более 33 часов в месяц. Кроме времени, возникают ошибки, которые могут привести к неверной модели, неправильной сборке или задержке релиза.

Makefile помогает:

- быстрее вводить новых сотрудников в проект;
- стандартизировать локальную разработку;
- уменьшать число ручных ошибок;
- ускорять code review;
- повторно использовать одни команды локально и в CI/CD;
- делать процесс воспроизводимым;
- сокращать время от изменения кода до выпуска в production.

### 2.3. Почему это важно для ML сильнее, чем для обычного скрипта

ML-система чувствительна не только к коду, но и к процедуре запуска.

Например, качество модели может зависеть от:

- версии данных;
- версии Python-библиотек;
- порядка этапов preprocessing;
- выбранного конфигурационного файла;
- seed;
- пути к модели;
- версии Docker-образа;
- переменных окружения.

Если аналитик запускает обучение вручную одной командой, ML-инженер — другой, а CI — третьей, результаты могут отличаться. Makefile не решает всю проблему воспроизводимости, но создаёт одну стандартизированную точку запуска.

## 4. Технический контекст: что такое make

`make` — это программа, которая читает инструкции из файла `Makefile`.

Makefile содержит **цели**, **зависимости** и **рецепты**.

Общая форма:

```make
target: dependencies
	command
```

Где:

- `target` — имя цели;
- `dependencies` — то, от чего зависит цель;
- `command` — команда shell, которую нужно выполнить;
- перед `command` обычно должен стоять символ Tab.

Пример:

```make
test:
	python -m pytest
```

Когда разработчик вводит:

```bash
make test
```

утилита `make` находит цель `test` и выполняет:

```bash
python -m pytest
```

---

## 5. Что означает «точка входа в проект»

Точка входа — это понятный стандартный способ начать работу с системой.

Без Makefile README может содержать множество команд:

```bash
python src/preprocess.py
python src/predict.py
docker build -t mock_ml_image .
```

Пользователь должен знать:

- где лежит каждый файл;
- каким интерпретатором его запускать;
- какой Docker-тег использовать;
- из какого каталога выполнять команду.

С Makefile пользователь запоминает только:

```bash
make preprocess
make predict
make build-docker
```

Внутреннюю реализацию можно менять, сохраняя внешний интерфейс.

Например, сегодня `make predict` вызывает Python-файл, а через месяц может запускать модуль:

```make
predict:
	python -m src.predict
```

Пользователю не нужно менять привычку.

## 7. Что такое bash-скрипт и что мы меняем

Bash-скрипт — это текстовый файл с командами shell.

Например:

```bash
#!/bin/bash
python src/predict.py
```

Строка:

```bash
#!/bin/bash
```

называется shebang. Она указывает, каким интерпретатором запускать файл.

Когда команда переносится внутрь Makefile, отдельный shebang для каждой цели не нужен. `make` сам передаёт рецепт shell.

Мы не «переписываем Python на Makefile». Мы только переносим команды запуска.

---

## 8. Что такое `.PHONY`

В решении используется:

```make
.PHONY: predict preprocess build-docker
```

`make` исторически создан для сборки файлов. Он может интерпретировать имя цели как имя файла.

Предположим, у нас есть цель:

```make
predict:
	python src/predict.py
```

Если в каталоге случайно появится файл с именем `predict`, `make` может решить, что цель уже существует и её не нужно выполнять.

`.PHONY` говорит:

> Эти имена — не файлы. Это команды, которые нужно выполнять при вызове.

Поэтому `.PHONY` — хорошая практика для целей типа:

- `test`;
- `clean`;
- `install`;
- `predict`;
- `preprocess`;
- `build-docker`.

---

## 9. Почему перед командами нужен Tab

Классический синтаксис Makefile требует, чтобы строка рецепта начиналась символом Tab:

```make
predict:
	python src/predict.py
```

Визуально Tab может выглядеть как несколько пробелов, поэтому новички часто ошибаются.

Неправильно:

```make
predict:
    python src/predict.py
```

Здесь могут быть четыре пробела. В таком случае `make` часто выдаёт ошибку:

```text
*** missing separator.  Stop.
```

Правильно — настоящий Tab. В подготовленном файле он уже стоит.

---

## 10. Почему мы используем исходные команды без изменений

Можно было бы написать:

```make
predict:
	python3 src/predict.py
```

Но исходный скрипт использует `python`, а условие требует выполнять те же действия.

Поэтому безопасное решение сохраняет команды буквально:

```make
python src/predict.py
python src/preprocess.py
docker build -t mock_ml_image .
```

Это минимизирует риск несовпадения с автоматическими тестами.

## 11. Что делает каждая команда

### `make predict`

Выполняет:

```bash
python src/predict.py
```

Файл `predict.py` генерирует случайное целое число от 1 до 10 и печатает прогноз срока доставки.

Пример вывода:

```text
Predicted delivery time: 7 days
```

Число может меняться, потому что используется `random.randint(1, 10)`.

### `make preprocess`

Выполняет:

```bash
python src/preprocess.py
```

Скрипт печатает:

```text
Data preprocessing completed!
```

В реальном ML-проекте на этом шаге могли бы выполняться очистка данных, обработка пропусков, агрегации и формирование признаков.

### `make build-docker`

Выполняет:

```bash
docker build -t mock_ml_image .
```

Разбор:

- `docker build` — собрать Docker-образ;
- `-t mock_ml_image` — присвоить образу имя/тег `mock_ml_image`;
- `.` — использовать текущий каталог как build context;
- Docker читает `Dockerfile` из текущего каталога.

---

## 12. Почему каталог `scripts/` нужно удалить

После переноса команд отдельные bash-файлы дублируют Makefile.

Дублирование опасно. Например, кто-то обновит Makefile, но забудет обновить `scripts/predict.sh`. Тогда появятся два разных способа запуска одной операции.

Принцип:

> Один интерфейс — один источник правды.

В учебной задаче таким источником правды становится `Makefile`.

В реальном проекте иногда сохраняют сложные скрипты и вызывают их из Makefile. Например:

```make
train:
	bash scripts/train.sh
```

Это нормально, если скрипт содержит много логики. Но здесь каждый файл содержит только одну простую команду, поэтому отдельные скрипты не дают пользы.

---

## 13. Makefile и соседние инструменты

### Makefile против bash-скрипта

Bash удобен для сложной последовательной shell-логики. Makefile удобен как единый каталог команд и для описания зависимостей.

Они не всегда конкуренты. Makefile может вызывать bash-скрипты.

### Makefile против Docker

Docker упаковывает окружение и приложение. Makefile стандартизирует команды, в том числе команду сборки Docker.

### Makefile против Airflow

Airflow управляет production workflow по расписанию, хранит состояние, делает retries и мониторинг. Makefile — локальный интерфейс запуска команд и простой инструмент автоматизации.

### Makefile против CI/CD

CI/CD выполняет проверки и релизы на сервере. Makefile может предоставить CI стандартные команды:

```yaml
- run: make test
- run: make build-docker
```

Тогда разработчик локально запускает те же операции, что и CI.

### Makefile против Poetry, uv или pip

Poetry, uv и pip управляют Python-зависимостями. Makefile может вызывать их:

```make
install:
	uv sync
```

## 14. Мышление DS/MLE при решении задачи

Правильный ход мыслей:

1. Не писать код до изучения всех входных файлов.
2. Открыть каждый bash-скрипт.
3. Выписать точную команду из каждого скрипта.
4. Сопоставить имя требуемой цели с соответствующим скриптом.
5. Перенести команду без изменения поведения.
6. В обычном проекте добавить `.PHONY`; в этом задании пропустить её из-за строгого автотеста.
7. Проверить, что рецепты начинаются Tab.
8. Запустить цели из корня проекта.
9. Удалить устаревший каталог `scripts/`.
10. Отправить файл с точным именем `Makefile`.

Это и есть инженерное мышление: сначала установить контракт, затем сделать минимальное изменение, затем проверить поведение.

## 15. Частые ошибки

### Ошибка 1. Назвать файл `Makefile.txt`

Windows может скрывать расширения. Проверяющая система не найдёт нужный файл.

### Ошибка 2. Использовать пробелы вместо Tab

Результат — `missing separator`.

### Ошибка 3. Запускать команды из неправильного каталога

Команда:

```bash
python src/predict.py
```

предполагает, что текущий каталог — корень проекта.

### Ошибка 4. Изменить Docker-тег

Нужно сохранить:

```bash
docker build -t mock_ml_image .
```

### Ошибка 5. Создать цель `docker-build` вместо `build-docker`

Автотест может искать точное имя `build-docker`.

### Ошибка 6. Оставить двоеточие без рецепта

Цель должна содержать команду.

### Ошибка 7. Добавить ненужную сложность

Переменные, проверки окружения и дополнительные цели здесь не требуются. Чем больше лишнего кода, тем больше риск отклониться от контракта.
