# Урок 24 — ML: MAKEFILE
## Разбор первой неудачной попытки

## 1. Ошибка

```text
AssertionError: Expected line : predict:, but got .PHONY: predict preprocess build-docker
```

## 2. Что означают две части сообщения

Ожидалось:

```make
predict:
```

Получено:

```make
.PHONY: predict preprocess build-docker
```

Значит, тест читает структуру файла в определённом порядке и не допускает дополнительное объявление перед первой целью.

## 3. Почему первая версия была корректна для make, но некорректна для теста

У решения есть два уровня совместимости:

1. синтаксическая корректность для утилиты `make`;
2. соответствие контракту конкретной проверяющей системы.

Первая версия выполняла первый пункт, но нарушала второй.

В production похожая ситуация возникает, когда API требует точные:

- имена полей;
- типы;
- порядок операций;
- сигнатуры функций;
- имена файлов;
- схемы данных.

## 4. Минимальное исправление

Удалить всё перед `predict:` и оставить только обязательные цели.

```make
predict:
	python src/predict.py

preprocess:
	python src/preprocess.py

build-docker:
	docker build -t mock_ml_image .
```

## 5. Практический вывод

Необязательные улучшения нельзя добавлять вслепую в задачу со строгим автотестом. Сначала нужно выполнить минимальный контракт, затем — при возможности — расширять решение.

## 6. Локальная проверка первой строки

```python
from pathlib import Path

lines = Path("Makefile").read_text().splitlines()
assert lines[0] == "predict:"
```

Для исправленного файла эта проверка проходит.
