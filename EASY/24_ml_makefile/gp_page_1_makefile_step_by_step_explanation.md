# Урок 24 — ML: MAKEFILE
## Пошаговое объяснение с абсолютного нуля

## 1. Что такое Makefile

`Makefile` — обычный текстовый файл без расширения. Утилита `make` читает его и выполняет описанные в нём цели.

Пример запуска:

```bash
make predict
```

Слово `predict` — имя цели, которую нужно найти в `Makefile`.

## 2. Что было в исходном проекте

```text
project/
├── scripts/
│   ├── predict.sh
│   ├── preprocess.sh
│   └── docker_build.sh
├── Dockerfile
├── src/
│   ├── predict.py
│   └── preprocess.py
└── Makefile
```

Каждый `.sh`-файл содержал одну полезную команду. Наша задача — перенести эти команды в одну точку входа.

## 3. Из чего состоит цель

```make
predict:
	python src/predict.py
```

Здесь:

- `predict` — имя цели;
- `:` — разделитель имени и зависимостей;
- зависимостей после двоеточия нет;
- следующая строка — рецепт;
- перед рецептом стоит Tab.

## 4. Что происходит при `make predict`

1. Операционная система запускает программу `make`.
2. `make` ищет файл `Makefile` в текущем каталоге.
3. Находит строку `predict:`.
4. Читает команду под ней.
5. Выполняет `python src/predict.py`.
6. Python запускает код файла `src/predict.py`.

## 5. Цель preprocessing

```make
preprocess:
	python src/preprocess.py
```

Preprocessing — подготовка данных. В реальном ML-проекте это может включать:

- обработку пропусков;
- удаление дублей;
- приведение типов;
- фильтрацию аномалий;
- объединение таблиц;
- построение признаков;
- сохранение датасета.

## 6. Цель Docker build

```make
build-docker:
	docker build -t mock_ml_image .
```

Подробно:

- `build-docker` — выбранное условием имя цели;
- `docker build` — команда сборки образа;
- `-t mock_ml_image` — имя образа;
- `.` — текущая папка, где Docker ищет `Dockerfile`.

## 7. Полный правильный файл

```make
predict:
	python src/predict.py

preprocess:
	python src/preprocess.py

build-docker:
	docker build -t mock_ml_image .
```

## 8. Почему перед командами нужен Tab

Классический синтаксис Makefile различает рецепты по начальному Tab.

Если поставить пробелы, можно получить:

```text
*** missing separator. Stop.
```

В готовом файле Tab уже проставлен правильно.

## 9. Почему мы не добавляем `.PHONY`

Теоретически `.PHONY` сообщает make, что цель является действием, а не именем файла. Например:

```make
.PHONY: predict
```

В обычном проекте это хорошая практика. Но автотест данного задания требует первой строкой `predict:`. Поэтому `.PHONY` нельзя размещать перед ней.

Это пример различия между:

- универсально хорошей практикой;
- строгим контрактом конкретной системы.

## 10. Почему удаляется `scripts/`

После переноса команды уже находятся в Makefile. Если оставить старые скрипты, появятся два места с одинаковой логикой. Позже они могут разойтись.

По условию проекта папка удаляется полностью.

## 11. Как проверить команды без риска

Проверить Python-цели:

```bash
make predict
make preprocess
```

Показать Docker-команду без запуска:

```bash
make -n build-docker
```

## 12. Частые ошибки

### Неправильное имя

```make
docker-build:
```

Условие требует:

```make
build-docker:
```

### Изменение Python-команды

Не нужно заменять `python` на `python3`, если исходный скрипт содержит `python` и тест ожидает точное действие.

### Расширение файла

Неправильно:

```text
Makefile.txt
```

Правильно:

```text
Makefile
```

### Дополнительная строка первой

Комментарии и `.PHONY` перед `predict:` могут сломать строгую проверку.

## 13. Главная памятка

1. Имя файла — `Makefile`.
2. Первая строка — `predict:`.
3. Перед командами — Tab.
4. Три цели имеют точные имена.
5. Команды скопированы без изменений.
6. Каталог `scripts/` удалён.
7. На платформу загружается только `Makefile`.
