# DA/ML: PRICE PARSER

# Полный бизнесовый и технический контекст

## 1. Где мы находимся в бизнесе

Мы стажируемся в условной компании **KarpovExpress**. Команда занимается двумя связанными задачами:

1. динамическим ценообразованием;
2. матчингом товаров с товарами конкурентов.

Компании недостаточно видеть только собственные цены. Для принятия решений нужно понимать рынок:

- сколько стоит аналогичный товар у конкурентов;
- есть ли скидка;
- доступен ли товар;
- кто продавец;
- входит ли доставка;
- действует ли цена для всех;
- относится ли цена к тому же SKU;
- насколько свежи данные.

Источником часто является HTML-страница, а не готовая таблица.

```html
<div class="product-card">
    <h1>Наушники SoundMax Pro</h1>
    <span class="old-price">12 990 ₽</span>
    <span class="current-price">9 990 ₽</span>
    <div class="availability">В наличии</div>
</div>
```

Человек сразу понимает, что текущая цена — 9 990 ₽. Компьютеру нужно формально определить, какой элемент означает актуальную цену и как превратить текст в корректное числовое значение.

# 2. Что такое Price Parser

**Price parser** — программа или функция, которая получает сырые данные о товарном предложении и извлекает структурированную информацию о цене.

Источники:

- HTML;
- JSON API;
- JSON-LD внутри HTML;
- XML-фид;
- мобильный API;
- PDF-прайс;
- текст поставщика;
- OCR со скриншота.

Минимальный результат:

```text
price = 9990.00
currency = RUB
```

Реалистичный результат:

```text
current_price
regular_price
sale_price
loyalty_price
currency
price_type
availability
seller
unit_price
promotion_text
observed_at
parser_version
confidence
raw_price_text
```

## Парсер и скрейпер — разные компоненты

Скрейпер получает страницу:

- HTTP-запрос;
- cookies;
- redirect;
- proxy;
- browser automation;
- rate limiting.

Парсер интерпретирует уже полученное содержимое:

- находит элементы;
- извлекает текст;
- определяет тип цены;
- нормализует число;
- формирует структуру результата.

```text
URL → Fetcher/Scraper → HTML → Parser → Structured Price → Storage/Pricing Engine
```

Это важное разделение для промышленной архитектуры и собеседований.

# 3. Зачем бизнесу мониторинг цен

Цена влияет на:

- конверсию;
- продажи;
- выручку;
- валовую маржу;
- прибыль;
- долю рынка;
- удержание;
- эффективность рекламы;
- оборачиваемость;
- OOS;
- восприятие бренда.

Слишком высокая цена может снизить конверсию и замедлить оборачиваемость. Слишком низкая цена может уничтожить маржу, вызвать дефицит и начать ценовую войну.

Поэтому цель не формулируется как «всегда быть дешевле». Цель — выбрать цену, соответствующую стратегии бизнеса.

## Типовые стратегии

### Быть самым дешёвым

Просто, но рискованно:

- низкая маржа;
- конкуренты автоматически отвечают;
- категория становится убыточной.

### Ценовой паритет

```text
Наша цена должна быть в пределах ±2% от медианы ключевых конкурентов.
```

### Премиальная цена

Цена выше рынка оправдывается:

- брендом;
- доставкой;
- гарантией;
- возвратом;
- сервисом.

### KVI — Key Value Items

Это товары, по которым покупатель запоминает уровень цен магазина:

- молоко;
- яйца;
- хлеб;
- популярный смартфон;
- базовые подгузники.

Их цена может быть особенно конкурентной даже при меньшей марже.

### Максимизация прибыли

```text
profit(price) = expected_demand(price) × (price - cost)
```

Конкурентная цена здесь является одним из признаков, но не единственным.

# 4. Связь с товарным матчингом

Нельзя сравнивать цены без подтверждения, что товары одинаковы.

```text
Apple iPhone 15 128 GB Black
Apple iPhone 15 Pro 128 GB Black
```

Названия похожи, но SKU разные. Ошибка матчинга загрязняет конкурентную аналитику и может привести к неверному repricing.

Типовой pipeline:

```text
Получение страниц
→ извлечение названия, цены и характеристик
→ нормализация
→ генерация кандидатов
→ ML/правила матчинга
→ контроль качества матчей
→ конкурентная цена
→ pricing engine
```

Ошибки возможны на каждом слое:

- страница не загрузилась;
- цена не извлеклась;
- выбрана старая цена;
- потеряна валюта;
- сопоставлен другой товар;
- использована цена стороннего продавца;
- данные устарели.

# 5. Почему цена — не просто число

## 5.1. Простая цена

```html
<span class="price">999 ₽</span>
```

Ожидаемое значение: `999`.

## 5.2. Разделители тысяч и дробной части

```text
1 299 ₽
1,299.50 USD
1.299,50 €
1 299,99 ₽
1'299.50 CHF
```

В американской локали запятая отделяет тысячи, точка — дробную часть. Во многих европейских локалях наоборот. Нельзя применять одно универсальное удаление символов без знания формата.

## 5.3. Старая и новая цена

```html
<span class="old-price">1 499 ₽</span>
<span class="new-price">999 ₽</span>
```

Структура:

```text
regular_price = 1499
sale_price = 999
discount_amount = 500
```

Если взять первое число, результат будет неверным.

## 5.4. Цена по карте

```text
Обычная цена: 1 299 ₽
По карте: 999 ₽
```

Нужно решить, какая цена считается конкурентной. Это продуктовый вопрос, а не только технический.

```text
base_price = 1299
loyalty_price = 999
eligibility = registered_users
```

## 5.5. Цена по подписке

```text
1 190 ₽ без подписки
990 ₽ с подпиской
```

Минимальная цена может быть недоступна обычному пользователю.

## 5.6. Диапазон

```text
от 499 ₽
499–1 299 ₽
```

Причины диапазона:

- размеры;
- цвета;
- комплектации;
- продавцы;
- объёмы упаковки.

```text
min_price = 499
max_price = 1299
is_range = true
```

## 5.7. Цена за единицу

```text
89 ₽ / 100 г
```

Для честного сравнения:

```text
89 × 10 = 890 ₽ / кг
```

В FMCG сравнение только цены упаковки часто бессмысленно.

## 5.8. Рассрочка

```text
от 2 499 ₽ в месяц
```

Это не полная цена товара. Парсер обязан различать:

```text
full_price
monthly_payment
installment_months
```

## 5.9. Процент скидки и отзывы

```text
Скидка 30%. Цена 699 ₽. 120 отзывов.
```

Первая цифра — 30, но цена — 699. Регулярное выражение «найти первое число» недостаточно.

## 5.10. JSON-LD

```html
<script type="application/ld+json">
{
  "@type": "Product",
  "offers": {
    "price": "999.00",
    "priceCurrency": "RUB"
  }
}
</script>
```

JSON-LD часто стабильнее CSS-классов, но тоже требует валидации: цена может быть устаревшей, относиться к другому варианту или быть минимальной среди продавцов.

# 6. Нормализация

Строка:

```text
" 1\u00A0299,90 ₽ "
```

Здесь `\u00A0` — неразрывный пробел.

Типовые этапы:

1. убрать пробелы по краям;
2. заменить неразрывные пробелы;
3. отделить валюту;
4. определить локаль;
5. обработать разделители;
6. преобразовать в число;
7. проверить диапазон;
8. сохранить исходный текст.

Для денег часто лучше использовать:

- целые минимальные единицы, например копейки;
- `Decimal`;
- специализированный money type.

```text
999.90 RUB → 99990 копеек
```

В учебном задании тесты могут требовать `float`, но промышленный выбор типа должен быть осознанным.

# 7. Полный жизненный цикл данных

## 7.1. Выбор источников

Нужно определить:

- ключевых конкурентов;
- категории;
- регионы;
- частоту;
- допустимых продавцов;
- каналы;
- приоритет товаров.

## 7.2. Получение страницы

Варианты:

- HTTP;
- headless browser;
- официальный API;
- партнёрский фид;
- поставщик данных.

Нужно учитывать законодательство, условия использования, robots.txt, договорные ограничения, rate limit и нагрузку на источник.

## 7.3. Извлечение

Источники внутри страницы:

- DOM;
- атрибуты;
- meta tags;
- JSON-LD;
- JavaScript state;
- микроразметка.

## 7.4. Семантическая классификация

Определяем тип:

- current;
- regular;
- sale;
- loyalty;
- subscription;
- installment;
- unit;
- range.

## 7.5. Валидация

Примеры:

```text
price > 0
currency в разрешённом списке
sale_price <= regular_price
данные не старше TTL
изменение не является необъяснимой аномалией
```

## 7.6. Историзация

| product_id | competitor | price | observed_at |
|---|---|---:|---|
| 101 | A | 999 | 2026-07-10 10:00 |
| 101 | A | 949 | 2026-07-11 10:00 |

История позволяет считать:

- изменение;
- минимум;
- медиану;
- частоту промо;
- волатильность;
- реакцию конкурента.

## 7.7. Pricing Engine

Простое правило:

```text
если конкурент дешевле более чем на 5%
и маржа позволяет,
рассмотреть снижение цены
```

Реальная система учитывает:

- себестоимость;
- минимальную маржу;
- спрос;
- эластичность;
- остатки;
- сезонность;
- промо;
- каннибализацию;
- стратегию категории;
- брендовые ограничения.

# 9. Типовые ошибки новичков

## Искать первое число

```text
Скидка 30%, цена 699 ₽, 120 отзывов
```

## Удалять все точки и запятые

```text
1.299,50 → 129950
```

вместо `1299.50`.

## Не сохранять raw text

Без `raw_price_text` сложнее расследовать ошибку.

## Не версионировать парсер

Нужно хранить `parser_version`.

## Считать отсутствие цены нулём

Лучше:

```text
price = null
parse_status = price_not_found
```

## Игнорировать регион

Цена может зависеть от города, склада, адреса и аккаунта.

## Игнорировать продавца

На маркетплейсе минимальная цена неизвестного продавца не всегда репрезентативна.

## Игнорировать доставку

Товар за 900 с доставкой 300 может быть дороже товара за 1100 с бесплатной доставкой.

# 10. Промышленная архитектура

```text
Scheduler
→ URL Queue
→ Fetcher Workers
→ Raw HTML Storage
→ Parser Workers
→ Validation Layer
→ Normalized Price Storage
→ Matching Service
→ Pricing Engine
→ BI / Alerts / API
```

## Scheduler

Определяет частоту и приоритет.

```text
KVI — каждые 15 минут
обычные товары — раз в 6 часов
long tail — раз в сутки
```

## Queue

Даёт:

- масштабирование;
- retry;
- backpressure;
- приоритеты;
- устойчивость.

## Fetcher

Отдельные пулы:

- дешёвые HTTP workers;
- более дорогие browser workers.

## Raw Storage

Нужен для:

- отладки;
- аудита;
- replay;
- regression tests;
- тестирования новой версии парсера.

## Parser Adapters

```text
CompetitorAParser
CompetitorBParser
CompetitorCParser
```

Общий контракт:

```text
parse(html, context) -> ParsedProduct
```

## Validation Layer

Проверяет типы, диапазоны, согласованность, историю, валюту, дубликаты и свежесть.

## Quarantine

Подозрительные данные не должны автоматически влиять на цену.

```text
status = needs_review
reason = 90_percent_price_drop
```

# 12. Тестирование

## Unit tests

```text
"999 ₽" → 999
"1 299,50 ₽" → 1299.50
```

## Negative tests

```text
"Цена по запросу"
"Нет в наличии"

```

Каждый найденный баг превращается в постоянный тест.

## Golden tests

Храним реальный HTML и ожидаемый структурированный результат.

Проверяем свойства:

- цена неотрицательна;
- sale не выше regular;
- currency валидна;
- результат не `NaN`;
- нормализация детерминирована.

# 13. Аномалии

## Аномальный рост

```text
999 → 99 900
```

Причины:

- артикул вместо цены;
- ошибка разделителя;
- цена за короб;
- реальное изменение;
- другой вариант товара.

## Аномальное падение

```text
49 990 → 4 999
```

Причины:

- monthly payment;
- скидка;
- первоначальный взнос;
- аксессуар;
- ошибка.

Историческая проверка:

```text
abs(new_price / median_30d - 1) > threshold
```

Порог должен зависеть от категории: для авиабилетов 50% может быть нормой, для хлеба — нет.

# 16. Математическая база

## Price Index

```text
price_index = our_price / competitor_price
```

Если наша цена 105, а конкурентная 100:

```text
price_index = 1.05
```

Мы на 5% дороже.

## Относительное изменение

```text
(new_price - old_price) / old_price
```

## Unit Margin

```text
selling_price - cost
```

## Margin Rate

```text
(selling_price - cost) / selling_price
```

## Profit

```text
quantity × (price - cost)
```

## Эластичность

```text
% изменения спроса / % изменения цены
```
