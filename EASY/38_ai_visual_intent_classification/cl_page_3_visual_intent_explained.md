### Что вообще такое класс

Класс — это "чертёж" для создания объектов. Представь чертёж дома (класс) и построенные по нему конкретные дома (объекты/экземпляры). Класс описывает: какие данные хранит объект (атрибуты) и что он умеет делать (методы).


- `class ImageProcessor:` — объявляем класс с именем `ImageProcessor`.
- Дальше идёт docstring класса — обязательна для pylint (`C0115`).
- Всё, что написано с отступом внутри — принадлежит классу.

### Метод `__init__` — конструктор

```python
    def __init__(self, image_path: str):
```

#### `def`
Ключевое слово для объявления функции. Функция внутри класса называется **методом**.

#### `__init__`
Специальное "магическое" имя. Python вызывает этот метод **автоматически**, когда ты создаёшь объект:

```python
p = ImageProcessor("payment.png")
#   ↑ здесь Python сам вызовет ImageProcessor.__init__(p, "payment.png")
```

Двойные подчёркивания с двух сторон (`dunder`, от *double underscore*) — признак того, что метод имеет специальное значение для самого языка.

#### `self`
Первый аргумент **любого** метода в классе. Это ссылка на **сам создаваемый объект**. Python подставляет его автоматически — ты его не передаёшь вручную. Через `self` мы сохраняем данные в объект и читаем их обратно.

Аналогия: если класс — чертёж дома, то `self` — это "этот конкретный дом, вот прямо этот".

#### `image_path: str`
Аргумент метода. `: str` — это **аннотация типа**. Она говорит человеку и линтерам "здесь ожидается строка". Python **не проверяет** это на исполнении — аннотация чисто документирующая. Но она резко повышает читаемость.

```python
        """
        Initialize an ImageProcessor instance.

        Parameters:
            image_path (str): The path to the image.

        Raises:
            ValueError: If the image path is empty.
        """
```
Docstring метода. Обязательна (`C0116`). Секция `Raises:` документирует, какие ошибки метод может выбросить.

```python
        if not image_path:
            raise ValueError("Image path must not be empty.")
```

#### Разбираем `if not image_path`
В Python пустая строка `""` считается **ложью** (`False`) в булевом контексте. Также ложью считаются `None`, `0`, пустой список `[]`.

- `not image_path` → `True`, если `image_path` пустая строка или `None`.
- Значит эта строка читается как: "если путь пустой или отсутствует..."

Почему не `if image_path == ""`? Потому что тогда мы не поймаем `None`. `if not image_path` ловит оба случая разом — это идиоматичный Python.

#### `raise ValueError(...)`
`raise` — выбросить исключение (ошибку). Программа немедленно останавливается на этом месте, и ошибка "летит вверх" по стеку вызовов, пока её кто-нибудь не поймает.

`ValueError` — встроенный тип ошибки, означающий "значение правильного типа, но недопустимое". Ровно то, что требует задание: *"Raises: ValueError: If the image path is empty."*

```python
        self.image_path = image_path
```

#### Что тут происходит
Создаём **атрибут** объекта. Читается: "у этого конкретного объекта (`self`) заведи поле `image_path` и положи туда значение из аргумента `image_path`".

Слева `self.image_path` — постоянное поле объекта, живёт пока живёт объект.
Справа `image_path` — временный аргумент метода, исчезнет как только метод закончится.

Они называются одинаково, но это **разные вещи** — это просто общепринятое соглашение (не путать!).

После этой строки:
```python
p = ImageProcessor("payment.png")
print(p.image_path)  # "payment.png"  ← доступно снаружи объекта
```

### Метод `prepare_image`

```python
    def prepare_image(self) -> Image.Image:
```

`-> Image.Image` — аннотация **возвращаемого** типа. Говорит: "этот метод вернёт объект класса `Image.Image`".

Аргументов кроме `self` нет — путь к файлу уже лежит в `self.image_path`, взяли его при создании объекта.

```python
        try:
            image = Image.open(self.image_path)  # keeps format-specific subclass type
        except FileNotFoundError as error:
            raise FileNotFoundError(f"Image not found: {self.image_path}") from error
        except UnidentifiedImageError as error:
            raise ValueError(f"Cannot decode image: {self.image_path}") from error
        except OSError as error:
            raise ValueError(f"Cannot read image: {self.image_path}") from error
        return image
```

#### Конструкция `try / except` — что это

Это механизм обработки ошибок:
- В блок `try:` кладём код, который **может сломаться**.
- В блоки `except SomeError:` кладём код, который выполнится, **если** сломалось именно этой ошибкой.

Если в `try` всё прошло гладко — все `except` пропускаются.

Задание прямо требует: *"Учтите возможные ошибки при чтении... изображения (например, если файл не найден)."* Вот это оно и есть.

#### `Image.open(self.image_path)`

Открывает файл по пути и возвращает объект изображения Pillow.

##### 🔴 САМЫЙ ВАЖНЫЙ МОМЕНТ ВО ВСЕЙ ЗАДАЧЕ

Обрати внимание на комментарий `# keeps format-specific subclass type`.

`Image.open("payment.png")` возвращает объект **не** типа `Image.Image`, а типа `PngImageFile` — это класс-**наследник**. Проверил локально:

```
Image.open('t.png')  ->  PngImageFile   | isinstance(x, Image.Image): True
.convert('RGB')      ->  Image
```

То есть:
- `Image.open(...)` → `PngImageFile` (наследник, помнит про формат PNG). ✅
- `Image.open(...).convert("RGB")` → `Image` (базовый, "обезличенный"). ❌

**Грейдер проверяет тип и ждёт наследника.** Именно поэтому в треде задачи куратор дважды писал "уберите `convert("RGB")`", а Дмитрий Деветьяров в итоге сам докопался до причины:

> *"Ошибка возникала из-за несоответствия типов возвращаемой переменной: у меня возвращал общий тип `Image.Image`, а надо возвращать тип-наследник, связанный с форматом исходного файла."*

Поэтому: **открыли → сразу вернули. Ничего с картинкой не делаем.**

> Наследование объясняю на всякий случай: `PngImageFile` — это `Image.Image` "плюс кое-что про PNG". Любой `PngImageFile` **является** `Image.Image` (поэтому `isinstance` даёт `True`), но не любой `Image.Image` является `PngImageFile`. Аннотация `-> Image.Image` в шаблоне при этом **корректна** — она указывает базовый тип, а вернуть наследника всегда допустимо.

#### Три `except` — почему именно эти три и в таком порядке

| Ошибка | Когда возникает | Что делаем |
|---|---|---|
| `FileNotFoundError` | Файла по пути просто нет | Пробрасываем `FileNotFoundError` с понятным текстом |
| `UnidentifiedImageError` | Файл есть, но Pillow не понимает его как картинку | Превращаем в `ValueError` |
| `OSError` | Всё остальное: нет прав, битый файл, диск отвалился | Превращаем в `ValueError` |

**Порядок критичен!** `FileNotFoundError` и `UnidentifiedImageError` — оба наследники `OSError`. Python проверяет `except` **сверху вниз** и срабатывает на первом подходящем. Если бы `except OSError` стоял первым, он бы перехватил вообще всё, и два других блока никогда бы не выполнились.

Правило: **от частного к общему**.

#### `as error`
Сохраняет объект пойманного исключения в переменную `error`, чтобы можно было к нему обратиться.

#### `f"Image not found: {self.image_path}"`
**f-строка** (formatted string). Буква `f` перед кавычками разрешает вставлять выражения прямо в строку внутри фигурных скобок `{...}`.

```python
self.image_path = "payment.png"
f"Image not found: {self.image_path}"  # → "Image not found: payment.png"
```

Без `f` фигурные скобки остались бы просто текстом.

#### `from error`
Это **цепочка исключений** (exception chaining). Говорит Python: "новая ошибка возникла **из-за** старой, сохрани оригинал в трейсбеке".

Зачем: без `from error` при отладке ты потеряешь информацию о том, что было настоящей причиной. Pylint это тоже требует — без `from` он выдаст `W0707: Consider explicitly re-raising`.

#### `return image`
Возвращаем объект картинки наружу. Стоит **вне** `try/except`, на нулевом уровне отступа внутри метода — выполнится только если ни одно исключение не выстрелило.

## Блок 2. Класс `ClipModel` (25 баллов)

```python
class ClipModel:
    """
    A class for working with a classification model.

    Attributes:
        model (CLIPModel): The CLIP model instance.
        processor (CLIPProcessor): The CLIP processor instance.
    """

    def __init__(self):
        """
        Initialize a ClipModel instance.
        """
        self.model = CLIPModel.from_pretrained(MODEL_ID)  # loaded once, reused
        self.processor = CLIPProcessor.from_pretrained(MODEL_ID)
```

### `from_pretrained(MODEL_ID)` — что это

Это **classmethod** (метод, вызываемый у класса, а не у объекта). Он делает три вещи:
1. Идёт на Hugging Face Hub по идентификатору `MODEL_ID`.
2. Скачивает файлы модели (веса + конфиг) и кэширует их локально (в `~/.cache/huggingface`), чтобы второй раз не качать.
3. Создаёт готовый к работе объект модели с загруженными весами.

Без `from_pretrained` (если написать просто `CLIPModel(config)`) мы получили бы модель со **случайными весами** — она бы ничего не умела.

### Почему грузим в `__init__`, а не в `classify`

Загрузка модели — **медленная и тяжёлая** операция (диск + возможно сеть + распаковка весов в память). Если положить её внутрь `classify`, то она будет выполняться при **каждом** вызове — грейдер уйдёт в таймаут и вернёт `Runtime error`.

В `__init__` она выполнится **один раз** при создании объекта `ClipModel()`, а дальше `self.model` просто переиспользуется.

Куратор в треде прямо на это указал (Дмитрий Войков, 5 июля 2025): *"поправьте загрузку модели в конструкторе класса ClipModel"*.

### Два объекта, а не один — зачем

- `self.model` — **нейросеть**. Умеет только одно: принять тензоры чисел и выдать тензоры чисел. Она не знает, что такое JPEG или буква "а".
- `self.processor` — **переводчик**. Превращает картинку в нормализованный тензор пикселей и текст в тензор токенов (числовых идентификаторов слов).

Это разделение ответственности — стандартный паттерн всей библиотеки `transformers`.

### Метод `classify`

```python
    def classify(self, image: Image.Image, intents: list) -> list:
```

#### 🔴 РОВНО ТРИ АРГУМЕНТА

`self`, `image`, `intents`. Ни больше, ни меньше. Грейдер **проверяет сигнатуру через интроспекцию**. В треде Alexey Yudin сделал два аргумента — получил отказ:

> *"Ваше решение не соответствует шаблону из задания. У вас 2, проверка требует 3."*

Логичный вопрос: "а зачем передавать `intents` снаружи, если можно хранить их внутри `ClipModel`?" Ответ: потому что так задан API. Плюс это архитектурно правильно — `ClipModel` не должна знать про конкретные бизнес-категории, она универсальная. Категории — это уровень `ImageClassifier`.

```python
        inputs = self.processor(
            text=intents,
            images=image,
            return_tensors="pt",
            padding=True,
        )
```

#### Что делает эта строка

Вызываем `self.processor` **как функцию** (у объекта есть метод `__call__`, поэтому объект можно вызывать как функцию). Он готовит данные для модели.

Аргументы:

| Аргумент | Что означает |
|---|---|
| `text=intents` | Список текстовых категорий (3 штуки). Процессор их токенизирует. |
| `images=image` | Наш PIL-объект. Процессор его отресайзит до 336×336, нормализует пиксели и превратит в тензор. |
| `return_tensors="pt"` | `"pt"` = PyTorch. Верни тензоры PyTorch, а не numpy-массивы или списки. Модель — на PyTorch, ей нужны именно они. |
| `padding=True` | Тексты разной длины ("payment-problem" и "profile-problem" дают разное число токенов). Модель требует прямоугольный тензор → короткие тексты добиваются служебными нулями до длины самого длинного. Без этого — ошибка. |

#### Что внутри `inputs`

Это словарь-подобный объект с ключами:
```
input_ids       — (3, seq_len)   токены трёх текстов
attention_mask  — (3, seq_len)   маска: 1 = реальный токен, 0 = padding-заглушка
pixel_values    — (1, 3, 336, 336)  одна картинка, 3 канала (RGB), 336×336 пикселей
```

> Заметь: **процессор сам приводит картинку к RGB и к нужному размеру**. Именно поэтому ручной `.convert("RGB")` в `prepare_image` не просто вреден для грейдера — он ещё и полностью бесполезен функционально.

```python
        outputs = self.model(**inputs)
```

#### `**inputs` — распаковка словаря

Двойная звёздочка `**` перед словарём **распаковывает** его в именованные аргументы. То есть эта строка эквивалентна:

```python
outputs = self.model(
    input_ids=inputs["input_ids"],
    attention_mask=inputs["attention_mask"],
    pixel_values=inputs["pixel_values"],
)
```

`**inputs` — просто короткая и универсальная запись того же самого.

#### Что происходит внутри модели (forward pass)

Ровно та механика, что разбиралась в `cl_page_1_CLIP_architecture.md`:

```
pixel_values (1, 3, 336, 336) ──[Image Encoder]──> image_embeds (1, d)
input_ids    (3, seq_len)     ──[Text Encoder ]──> text_embeds  (3, d)
                                        │
                          L2-нормализация обоих
                                        │
                        image_embeds @ text_embeds.T
                                        │
                              logits_per_image (1, 3)
```

`d` — размерность общего пространства эмбеддингов.

```python
        probs = outputs.logits_per_image.softmax(dim=1)  # (1, n_intents)
```

#### `outputs.logits_per_image`
Тензор формы `(1, 3)`: одна строка (одна картинка) × три столбца (три категории). Каждое число — **сырая** мера сходства картинки с соответствующей категорией (косинусная близость, умноженная на обучаемый температурный коэффициент).

Это **не вероятности** — числа могут быть любыми, например `[2.3, -0.5, 1.1]`. Их сумма не равна 1.

> Есть парный тензор `logits_per_text` формы `(3, 1)` — это та же матрица, но транспонированная. Нам нужен взгляд "со стороны картинки", поэтому берём `logits_per_image`.

#### `.softmax(dim=1)`
Функция softmax превращает произвольные числа в **распределение вероятностей**: все значения становятся положительными и в сумме дают ровно 1.

Формула: `softmax(x_i) = exp(x_i) / sum(exp(x_j))`

Пример:
```
[2.3, -0.5, 1.1]  →  softmax  →  [0.72, 0.04, 0.24]
                                  сумма = 1.0 ✓
```

`dim=1` — по какой оси нормализовать. У нас тензор `(1, 3)`:
- `dim=0` — по строкам (вертикально). Строка одна → каждое значение стало бы 1.0. **Неправильно.**
- `dim=1` — по столбцам (горизонтально), внутри строки. Три категории конкурируют между собой. **Правильно.**

Задание требует вернуть именно *"распределение вероятностей"* — поэтому softmax обязателен.

```python
        return probs.detach().numpy()[0].tolist()
```

Эта строка была **дана в шаблоне задания** — разбираем по частям, слева направо:

| Шаг | Что делает | Что на выходе |
|---|---|---|
| `probs` | Тензор PyTorch | `tensor([[0.72, 0.04, 0.24]])`, форма `(1, 3)` |
| `.detach()` | Отцепляет тензор от графа вычислений (истории операций для автоградиента) | тот же тензор, но без `requires_grad` |
| `.numpy()` | Конвертирует в numpy-массив | `array([[0.72, 0.04, 0.24]])`, форма `(1, 3)` |
| `[0]` | Берёт нулевую (единственную) строку — убирает batch-измерение | `array([0.72, 0.04, 0.24])`, форма `(3,)` |
| `.tolist()` | Превращает numpy-массив в обычный Python-список | `[0.72, 0.04, 0.24]` |

#### Зачем `.detach()`
PyTorch по умолчанию запоминает все операции, чтобы потом уметь считать градиенты для обучения. `.numpy()` **откажется работать** на тензоре, который подключён к этому графу — выдаст ошибку "Can't call numpy() on Tensor that requires grad". Мы не обучаем модель, только предсказываем → отцепляем.

#### Зачем `.tolist()`
Задание требует вернуть **`list`** (`-> list`). numpy-массив — это не `list`, это `numpy.ndarray`. Грейдер может проверять тип строго.

### `self.intents = [...]`

#### 🔴 Именно здесь, и нигде больше

Список категорий **обязан** жить в `ImageClassifier.__init__`. В треде Ильяшенко Роман получил `NameError: intents is not defined`, куратор ответил:

> *"вам необходимо определить атрибут `self.intents` в классе `ImageClassifier`. Для этого задайте список предопределённых интентов в конструкторе `__init__`."*

#### Почему тексты — это ровно сами теги

Задание требует, чтобы `classify_intent` возвращал одну из строк: `"payment-problem"`, `"course-problem"`, `"profile-problem"`. Мы используем **те же самые строки** и как текстовые промпты для CLIP, и как возвращаемые теги.

Что это даёт: связь "индекс в списке ↔ тег" становится **тождественной**. Нам не нужен отдельный словарь-маппинг вроде `{0: "payment-problem", ...}` — можно просто взять `self.intents[индекс]`. Меньше кода = меньше мест, где можно ошибиться.

#### Почему порядок в списке важен

Порядок элементов в `self.intents` определяет порядок столбцов в `logits_per_image`:

```
self.intents = ["payment-problem", "course-problem", "profile-problem"]
                        ↓                 ↓                  ↓
probs        = [       0.72,             0.04,              0.24        ]
                    индекс 0          индекс 1           индекс 2
```

Если поменять порядок в списке — поменяется и порядок вероятностей. Соответствие сохранится автоматически, потому что мы читаем тег из **того же самого списка**, который отдавали в модель.

### Метод `classify_intent` — точка входа

```python
    def classify_intent(self, image_path: str) -> str:
        image = ImageProcessor(image_path).prepare_image()
        probs = self.clip_model.classify(image, self.intents)
        return self.intents[probs.index(max(probs))]  # argmax over probabilities
```

Три строки, и в них весь пайплайн.

#### Строка 1: `image = ImageProcessor(image_path).prepare_image()`

Тут две операции в одной строке:
1. `ImageProcessor(image_path)` — создаём временный объект процессора. Сработает его `__init__`: проверит, что путь не пустой, и сохранит его.
2. `.prepare_image()` — сразу вызываем у него метод. Он откроет файл и вернёт PIL-объект.

Объект `ImageProcessor` больше нигде не нужен → не сохраняем его в переменную, Python сам его удалит после выполнения строки (сборщик мусора). Это идиоматично и читаемо.

Форма данных: путь `str` → объект `PngImageFile`.

#### Строка 2: `probs = self.clip_model.classify(image, self.intents)`

Отдаём картинку и список категорий в модель. Обрати внимание: `self.clip_model` — это тот же самый объект, который создался ещё в `__init__`. Модель **не перезагружается** — она уже в памяти.

Форма данных: `PngImageFile` + `list[str]` → `list[float]` длины 3, например `[0.72, 0.04, 0.24]`.

#### Строка 3: `return self.intents[probs.index(max(probs))]`

Разбираем изнутри наружу — это **argmax**, поиск индекса максимального элемента:

```python
probs = [0.72, 0.04, 0.24]

max(probs)            # → 0.72       ищем максимальное ЗНАЧЕНИЕ
probs.index(0.72)     # → 0          ищем ИНДЕКС этого значения
self.intents[0]       # → "payment-problem"   берём тег по индексу
```

| Функция | Что делает |
|---|---|
| `max(список)` | Возвращает **значение** самого большого элемента |
| `список.index(значение)` | Возвращает **индекс** первого вхождения значения |
| `список[индекс]` | Возвращает элемент по индексу |

Почему не `numpy.argmax`? Потому что `classify` вернул обычный Python-список (мы же вызвали `.tolist()`), а для списка `max` + `.index` — самый прямой и понятный способ, не требующий импорта numpy.

Возвращаемое значение — строка-тег. Именно это ждёт грейдер.
