# cl_page_3_visual_intent_context

# Шаг 1. Контекст задачи: бизнес + техника

## 1. Бизнесовый контекст

### 1.1. Что за компания и что за боль

Karpov.Courses — образовательная платформа. У неё есть **общий чат поддержки студентов**, куда прилетает поток сообщений на самые разные темы: оплата курсов, технические проблемы на платформе, вопросы про личный кабинет и профиль, трудоустройство, и вообще всё что угодно.

Проблема простая и очень денежная: **живой оператор поддержки — самый дорогой ресурс в этой цепочке**. Каждое обращение, которое обрабатывает человек, стоит компании реальных денег (зарплата, время, SLA). При этом большая часть обращений — типовые, и на них можно ответить автоматически.

Решение бизнеса: сделать **Telegram-бота**, который:
1. Сам отвечает на типовые понятные запросы.
2. Эскалирует сложные / непонятные запросы живому оператору.

### 1.2. Почему именно "визуальный" intent

Предыдущая задача курса (Intent Classification) научила бота понимать **текстовые** сообщения. Но реальные пользователи часто **не пишут текст вообще** — они кидают скриншот и всё. "Вот, смотри" + картинка. Или вообще просто картинка.

Типичные скриншоты в поддержке образовательной платформы:
- Скриншот формы оплаты с красной ошибкой → **payment-problem**
- Скриншот страницы курса / задания / плеера с багом → **course-problem**
- Скриншот личного кабинета / настроек профиля с проблемой → **profile-problem**

Если бот не умеет "смотреть" на картинки, то **любое** сообщение с картинкой автоматически падает в "непонятно" → эскалация человеку. То есть весь смысл автоматизации теряется ровно на том сегменте трафика, который проще всего было бы автоматизировать (скриншот ошибки оплаты — визуально очень узнаваемая штука).

### 1.3. Что мы строим в этой задаче

Мы строим **визуальный роутер запросов**: на вход путь к картинке, на выход — один из трёх тегов:

| Тег | Что означает |
|---|---|
| `payment-problem` | Проблема с оплатой |
| `course-problem` | Проблема с курсом / обучением |
| `profile-problem` | Проблема с профилем / личным кабинетом |

Дальше по тегу бот решает, какой сценарий ответа запустить или к какой команде эскалировать.

### 1.4. Как это же мышление применяется в других индустриях

Это не "учебная задача про картинки" — это **паттерн**, который встречается везде:

- **Финтех (SigmaBank)**: клиент кидает скриншот "платёж не прошёл" / "не приходит СМС" / "хочу выписку" → маршрутизация в нужную команду без человека.
- **Маркетплейсы (KarpovExpress, two-sided marketplaces)**: покупатель кидает фото брака / скрин чека / скрин карточки товара → автоматическая категоризация жалобы.
- **iGaming / crypto / casino**: скриншот транзакции из кошелька, скрин ошибки депозита, скрин игрового бага — у каждого провайдера свой UI, размечать вручную нереально дорого, zero-shot спасает.
- **Dating**: модерация — "это фото человека" vs "это скриншот переписки" vs "это неуместное".
- **BigTech (Google / Meta / Anthropic)**: тот же CLIP-подход лежит в основе image search, модерации контента и мультимодальных ассистентов.

Ключевой продуктовый инсайт, который стоит унести: **классификатор картинки здесь — это дешёвый первый фильтр перед дорогими этапами** (человек, большая LLM, ручная проверка). Не финальное решение, а роутер.

## 2. Технический контекст

### 2.1. Что мы используем

**CLIP** (см. `cl_page_1_CLIP_architecture.md`) — модель, которая умеет положить картинку и текст в одно общее векторное пространство и посчитать, насколько они похожи.

Механика ровно та, что разбиралась на page_1:
1. Картинку → через Image Encoder → вектор.
2. Три текстовых описания категорий → через Text Encoder → три вектора.
3. Косинусная близость картинки к каждой категории → softmax → распределение вероятностей.
4. `argmax` → предсказанный тег.

Никакого обучения. Zero-shot. Только forward pass.

### 2.2. Какая конкретно модель

В тексте задания модель названа `CLIP-ViT-tiny-large-patch14`. Реальный идентификатор на Hugging Face, с которым грейдер сдаётся на полный балл (это подтверждено кураторами в треде задачи несколько раз — Дмитрий Войков, 11 марта 2025):

```
yujiepan/clip-vit-tiny-random-patch14-336
```

Это уменьшенная (tiny) версия CLIP. Она маленькая специально — чтобы грейдер не падал по таймауту при загрузке весов.

> **Почему это важно**: если взять "настоящий" `openai/clip-vit-large-patch14`, грейдер, скорее всего, свалится в `Runtime error` по времени/памяти, и все 100 баллов превратятся в 0. Модель берём **ровно ту, что указана**.

### 2.3. Три класса, которые надо реализовать

Задание жёстко задаёт **API** (сигнатуры классов и методов). Это критично:

> *"Если решение не соответствует требованиям API, за все остальные проверки зачисляются 0 баллов."*

```
ImageProcessor   → отвечает за чтение картинки с диска
    __init__(self, image_path: str)      # кидает ValueError если путь пустой
    prepare_image(self) -> Image.Image   # возвращает объект PIL

ClipModel        → отвечает за модель
    __init__(self)                                        # грузит CLIPModel + CLIPProcessor
    classify(self, image, intents) -> list                # РОВНО 3 аргумента!

ImageClassifier  → склеивает всё вместе
    __init__(self)                                        # создаёт ClipModel + self.intents
    classify_intent(self, image_path: str) -> str         # возвращает тег
```

### 2.4. Распределение баллов

| Что проверяют | Сколько |
|---|---|
| `ImageProcessor` работает корректно | 15% |
| `ClipModel` работает корректно | 25% |
| Каждая картинка распознана верно | 60% |

Проходной балл: **90 из 100**.

### 2.5. 🔴 Грабли, на которых спалились люди в треде — читай внимательно

Тред обсуждения этой задачи — это буквально карта мин. Собрал всё:

#### Грабля №1: base64 — ЕГО НЕТ. Задание врёт.

В формулировке задания написано про "ошибки при чтении **и кодировании**" изображения, и раньше там прямо требовалась кодировка в base64. **Это устаревший текст задания.** Кураторы подтвердили несколько раз (Женя Орлова, 25 декабря 2024; Александра Овсянникова, 1 июля 2025):

> *"В эталонном решении кодировки нет."*

`prepare_image` должен просто открыть картинку и вернуть PIL-объект. Никакого base64.

#### Грабля №2: `.convert("RGB")` — НЕ ДЕЛАТЬ

Самая коварная. Куратор (Женя Орлова, 31 октября 2024):

> *"Уберите `convert("RGB")` в `prepare_image` и код должен будет пройти на полный балл."*

И повторно (Дмитрий Войков, 11 марта 2025): *"переводить изображение в RGB не нужно."*

**Почему?** Разгадка — в последнем сообщении треда (Дмитрий Деветьяров, 10 марта 2026):

> *"Ошибка возникала из-за несоответствия типов возвращаемой переменной: у меня возвращал общий тип `Image.Image`, а надо возвращать тип-наследник, связанный с форматом исходного файла."*

Разбираем по-человечески:
- `Image.open("payment.png")` возвращает объект типа **`PngImageFile`** — это класс-наследник от `Image.Image`, привязанный к формату файла.
- `Image.open("payment.png").convert("RGB")` возвращает объект типа **`Image.Image`** — базовый, "обезличенный" тип, потерявший связь с форматом.
- Грейдер проверяет тип возвращаемого объекта и **ждёт наследника** (`PngImageFile`), а не базовый `Image.Image`.

Поэтому: открыли → сразу вернули. Никаких `.convert()`.

#### Грабля №3: `classify` должен принимать РОВНО 3 аргумента

Alexey Yudin (21 ноября 2024) сделал `classify(self, image)` — грейдер упал.

> *"Ваше решение не соответствует шаблону из задания. У вас 2, проверка требует 3."*

Сигнатура строго: `classify(self, image: Image.Image, intents: list) -> list`. Даже если `intents` можно было бы хранить внутри `ClipModel` — нельзя, грейдер проверяет сигнатуру.

#### Грабля №4: `self.intents` живёт в `ImageClassifier`, а не где-то ещё

Ильяшенко Роман (21 октября 2025) получил `NameError: intents is not defined`. Ответ куратора:

> *"вам необходимо определить атрибут `self.intents` в классе `ImageClassifier`. Для этого задайте список предопределённых интентов в конструкторе `__init__`."*

#### Грабля №5: Replicate — НЕ ИСПОЛЬЗОВАТЬ

Встроенный ассистент курса (Ада) несколько раз отправлял людей решать через сервис Replicate. Куратор (Александра Овсянникова, 9 марта 2025):

> *"К сожалению, Ада немного ошибается тут. Решать нужно по описанию из задания: с `transformers`."*

Replicate — платный внешний API, в грейдере его нет. Только `transformers`.

#### Грабля №6: никаких обращений к файлам, которых нет в грейдере

Pavel Kravchuk (25 ноября 2024) получил `Runtime error`. Ответ:

> *"Уберите использование файла, которого нет в грейдере."*

Это значит: **никаких смоук-тестов, которые открывают `payment.png` на уровне модуля**. У грейдера этого файла нет → `FileNotFoundError` → `Runtime error` → 0 баллов. Поэтому в моём решении **намеренно нет** `if __name__ == "__main__"` блока с реальным файлом (хотя обычно я его добавляю) — здесь это прямой риск обнуления.

#### Грабля №7: загрузка модели именно в `__init__` ClipModel

Дмитрий Войков (5 июля 2025):

> *"поправьте загрузку модели в конструкторе класса `ClipModel`: `self.model = CLIPModel.from_pretrained(model_id)`, `self.processor = CLIPProcessor.from_pretrained(model_id)` и используйте `model_id`, указанный в тексте степа."*

То есть модель грузится **один раз** в конструкторе, а не внутри `classify` при каждом вызове.

### 2.6. requirements.txt

```
transformers==4.38.1
```

Версия зафиксирована — в грейдере именно она. Это важно, потому что API `transformers` меняется между версиями.
