# ML: ROTATION BUG — бизнесовый и технический контекст

## 1. Что происходит в задаче

Нам дана функция, которая должна поворачивать изображение:

```python
import cv2
import numpy as np


def rotated_image(image: np.ndarray, angle: int = 45) -> np.ndarray:
    """Rotate image by angle degrees."""
    width, height, channels = image.shape
    transform = cv2.getRotationMatrix2D((height / 2, width / 2), angle, 1)
    result = cv2.warpAffine(image, transform, (width, height))
    return result
```

Код выглядит правдоподобно, но в нём перепутаны соглашения двух библиотек:

- NumPy хранит форму изображения как `(height, width, channels)`;
- OpenCV принимает размер выходного изображения как `(width, height)`.

Из-за этого прямоугольная картинка после обработки получает переставленные местами высоту и ширину.

Однако на этой странице нам не нужно чинить функцию. Нам нужно написать тест, который надёжно докажет существование бага.

Это не формальность. В реальной разработке хороший regression-тест часто важнее устного описания дефекта, потому что он:

1. воспроизводит ошибку автоматически;
2. фиксирует ожидаемое поведение;
3. не даёт багу вернуться после рефакторинга;
4. запускается в CI при каждом Pull Request;
5. помогает безопасно менять preprocessing-пайплайн.

# Часть I. Бизнесовый смысл

## 2. Какой продукт описан в легенде

В условии есть платформа для AI-художников. Команда строит сервис, который анализирует загруженные изображения и определяет, нарушают ли они правила платформы.

Упрощённая схема:

```text
Пользователь загружает изображение
              ↓
Проверка формата и размера
              ↓
Image preprocessing
              ↓
ML-модель
              ↓
Risk score
              ↓
Разрешить / проверить вручную / заблокировать
```

Например, модель возвращает:

```text
P(нарушение | изображение) = 0.92
```

А продуктовая логика применяет пороги:

- меньше `0.30` — разрешить;
- от `0.30` до `0.80` — отправить модератору;
- выше `0.80` — заблокировать или ограничить публикацию.

## 3. Что такое обход модели с помощью поворота

Пользователь понимает, что модель хорошо распознаёт исходную картинку, но хуже работает после небольшого поворота.

Тогда атака выглядит так:

```text
исходное изображение
        ↓
поворот на 5°
        ↓
повторная загрузка
```

Для человека смысл изображения не изменился. Для модели изменилось расположение пикселей, границ и текстур.

Такой сценарий относится к **evasion attack**: нарушитель не взламывает сервер, а подбирает вход, на котором модель ошибается.

Другие похожие преобразования:

- отражение;
- изменение масштаба;
- crop;
- рамка;
- шум;
- blur;
- JPEG-сжатие;
- изменение яркости;
- изменение контраста;
- наложение текста;
- скриншот;
- повторное фотографирование экрана;
- изменение aspect ratio.

## 4. Почему это бизнесовая проблема

### 4.1. Нарушители получают преимущество

Если правило легко обходится, честные пользователи соблюдают ограничения, а злоумышленники — нет. Платформа становится несправедливой.

### 4.2. Растёт ручная модерация

Пусть система обрабатывает:

```text
100 000 изображений в день
```

Если `5%` уходят людям:

```text
5 000 ручных проверок
```

При стоимости одной проверки `$0.10`:

```text
$500 в день
```

Если из-за неустойчивой модели ручная очередь растёт до `12%`:

```text
12 000 × $0.10 = $1 200 в день
```

Дополнительные затраты:

```text
$700 в день
≈ $21 000 в месяц
```

### 4.3. Возникают ложные блокировки

Неустойчивая система может пропускать нарушения и одновременно блокировать нормальные изображения после обычного поворота телефона.

Последствия:

- падение конверсии публикации;
- жалобы;
- рост обращений в поддержку;
- отток авторов;
- репутационный ущерб;
- нагрузка на апелляции.

### 4.4. Нарушается SLA

Если больше объектов отправляется на дорогую модель или ручную проверку, растут:

- latency;
- очередь;
- инфраструктурная стоимость;
- время ожидания пользователя.

Пример SLA:

```text
95% изображений должны получить решение менее чем за 2 секунды
```

Даже маленькая ошибка preprocessing может сделать это недостижимым.

# Часть III. Что такое аугментация

## 12. Определение

Data augmentation — искусственное создание дополнительных обучающих примеров с помощью преобразований.

```text
исходное изображение x
        ↓ T
новое изображение T(x)
```

Если преобразование не меняет смысл, метка сохраняется:

```text
y(T(x)) = y(x)
```

Кот после небольшого поворота остаётся котом.

## 13. Зачем нужны аугментации

### 13.1. Больше разнообразия

Production содержит разные:

- камеры;
- разрешения;
- углы;
- освещение;
- фоны;
- масштабы;
- форматы.

### 13.2. Меньше переобучения

Модель не должна запоминать конкретное положение объекта.

### 13.3. Выше устойчивость

Если модель видит повороты на обучении, ей проще обработать похожие случаи после запуска.

### 13.4. Защита от простых атак

Аугментация не решает всю adversarial security, но уменьшает число дешёвых обходов.

## 14. Почему нельзя применять любые аугментации

Преобразование должно сохранять метку именно в нашей задаче.

Примеры:

- для кошек горизонтальное отражение обычно допустимо;
- для текста зеркальное отражение часто нереалистично;
- для дорожных знаков поворот на `180°` может разрушить смысл;
- для медицинских снимков случайный flip может быть недопустим;
- для документов небольшой deskew полезен, а произвольный сильный rotate — нет.

Главный вопрос:

> Встречается ли такое преобразование в реальном мире и сохраняет ли оно бизнесовый смысл метки?

# Часть IV. Как изображение хранится в NumPy

## 15. Изображение как массив

Цветное изображение:

```python
image.shape == (100, 200, 3)
```

означает:

```text
100 строк пикселей
200 столбцов пикселей
3 канала
```

Следовательно:

```text
axis 0 → height
axis 1 → width
axis 2 → channels
```

Форма:

```text
(height, width, channels)
```

## 16. Почему высота идёт первой

NumPy рассматривает изображение как матрицу:

```text
(rows, columns)
```

Для изображения:

```text
rows = height
columns = width
```

Поэтому сначала высота.

## 17. Индексация и геометрические координаты

В NumPy:

```python
pixel = image[y, x]
```

Но геометрическая точка в OpenCV часто записывается:

```text
(x, y)
```

То есть:

```text
NumPy indexing: image[y, x]
OpenCV point: (x, y)
```

Это один из главных источников ошибок в computer vision.

## 18. Каналы

Для RGB логически есть:

- red;
- green;
- blue.

Но OpenCV часто загружает изображения как BGR.

Это другая распространённая ошибка, но в текущей задаче баг связан не с каналами, а с высотой и шириной.

# Часть V. Построчный разбор исходной функции

## 19. Первая строка

```python
width, height, channels = image.shape
```

Пусть:

```python
image.shape == (100, 200, 3)
```

Python присвоит:

```text
width = 100
height = 200
channels = 3
```

Но реальный смысл:

```text
height = 100
width = 200
channels = 3
```

Названия переменных неверны.

## 20. Центр вращения

```python
transform = cv2.getRotationMatrix2D(
    (height / 2, width / 2),
    angle,
    1,
)
```

OpenCV ожидает центр:

```text
(center_x, center_y)
```

то есть:

```text
(width / 2, height / 2)
```

Из-за ошибочных имён выражение случайно даёт правильные числа.

Для формы `(100, 200, 3)`:

```text
height variable = 200
width variable = 100
```

Следовательно:

```text
(height / 2, width / 2) = (100, 50)
```

А реальный центр:

```text
(real_width / 2, real_height / 2) = (100, 50)
```

Две ошибки здесь временно компенсируют друг друга.

## 21. Размер результата

```python
result = cv2.warpAffine(image, transform, (width, height))
```

`warpAffine` ожидает:

```text
(output_width, output_height)
```

Но получает:

```text
(real_height, real_width)
```

Для нашего массива:

```text
ожидалось: (200, 100)
передано:  (100, 200)
```

Выходной NumPy-массив снова описывается как:

```text
(height, width, channels)
```

Поэтому результат:

```text
(200, 100, 3)
```

вместо:

```text
(100, 200, 3)
```

# Часть VI. Почему баг легко пропустить

## 22. Квадратное изображение скрывает ошибку

Для:

```text
(512, 512, 3)
```

перестановка ничего не меняет:

```text
(512, 512, 3) → (512, 512, 3)
```

Поэтому популярное квадратное изображение может создать ложное ощущение, что функция работает.

## 23. Ручной просмотр недостаточен

Визуализация полезна для исследования, но не заменяет тест:

- человек может не заметить ошибку;
- проверка не автоматизирована;
- её нельзя надёжно встроить в CI;
- она субъективна;
- её легко забыть.

# Часть VII. Как проектировать тест

## 24. Найти инвариант

Инвариант — свойство, которое должно сохраняться.

Для функции с фиксированным canvas:

```text
result.shape == image.shape
```

Особенно очевидно это при `angle=0`.

## 25. Почему угол 0° идеален

При нулевом угле:

- нет реального поворота;
- не должна меняться ориентация;
- не нужны сложные ожидаемые пиксели;
- не влияет обрезание углов;
- не мешает интерполяция.

Мы проверяем только геометрию размера.

## 26. Почему изображение должно быть прямоугольным

Нужно:

```text
height != width
```

Например:

```text
100 != 200
```

Только тогда перестановка осей становится видимой.

## 27. Почему синтетический массив лучше реального файла

Для проверки shape содержание картинки не важно.

```python
np.zeros((100, 200, 3), dtype=np.uint8)
```

Преимущества:

- быстро;
- нет внешнего файла;
- нет пути на диске;
- нет интернета;
- полностью контролируемая форма;
- воспроизводимость;
- понятная причина падения.

# Часть VIII. Связь с TDD

## 28. Цикл Red–Green–Refactor

### Red

Пишем тест, который падает на дефектной функции.

### Green

Исправляем минимум кода, чтобы тест прошёл.

### Refactor

Улучшаем структуру, сохраняя зелёные тесты.

На текущем шаге требуется именно стадия **Red**.

## 29. Regression test

После исправления тест становится защитой от повторного появления ошибки.

Сценарий:

1. баг исправили;
2. через месяц провели рефакторинг;
3. снова перепутали размеры;
4. CI остановил Pull Request.

# Часть XII. Что должен вынести будущий DS/MLE

## 41. Технические уроки

1. NumPy image shape — `(height, width, channels)`.
2. OpenCV point — `(x, y)`.
3. `warpAffine` принимает `dsize=(width, height)`.
4. Квадратные данные маскируют перестановку размеров.
5. `angle=0` — сильный диагностический случай.
6. Синтетический массив идеален для unit test.
7. Тест должен проверять контракт, а не внутреннюю реализацию.
8. Сначала воспроизводим баг, потом исправляем.
9. Preprocessing может испортить даже отличную модель.

## 42. Бизнесовые уроки

1. Простая геометрическая уязвимость может стать реальной атакой.
2. Ошибка в трёх строках влияет на деньги, latency и доверие.
3. Нужны guardrail-метрики, а не только offline accuracy.
4. Нужно оценивать реальные сценарии пользователей и атакующих.
5. Модерация требует апелляций, аудита и ручного fallback.
6. Устойчивость должна измеряться по сегментам.

## 43. Универсальная схема мышления

```text
Что обещает функция?
→ Какой контракт должен сохраняться?
→ Какие библиотеки используют разные соглашения?
→ Какой минимальный вход различает good и bad implementation?
→ Как превратить его в автоматический тест?
→ Как дефект влияет на бизнес и production?
```
