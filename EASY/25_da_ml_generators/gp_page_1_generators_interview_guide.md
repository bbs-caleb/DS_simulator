# Generators — гайд для продуктового и System Design интервью

## 1. Как отвечать не как новичок

Слабый ответ:

> Генераторы экономят память, потому что используют `yield`.

Сильный ответ:

> Генераторы полезны, когда источник данных большой или потоковый, а потребителю не нужны все элементы одновременно. Они снижают peak memory, могут уменьшить time-to-first-result и позволяют построить pipeline с обработкой по одному объекту или батчу. При этом нужно учитывать одноразовость, позднее возникновение ошибок, жизненный цикл файлов и соединений, backpressure, retry и идемпотентность.

Такой ответ показывает, что вы понимаете не только Python, но и эксплуатацию систем.

# 2. Базовые вопросы

## Что такое iterable?

Объект, способный предоставить итератор. По нему можно пройти циклом `for`.

Примеры:

- list;
- tuple;
- str;
- dict;
- set;
- file;
- range;
- generator.

## Что такое iterator?

Объект, который:

- хранит состояние обхода;
- реализует `__iter__()`;
- реализует `__next__()`;
- сообщает о завершении через `StopIteration`.

## Что такое generator?

Разновидность итератора, обычно созданная функцией с `yield` или генераторным выражением.

## В чём разница между `return` и `yield`?

`return` возвращает итог и завершает функцию.

`yield` отдаёт очередное значение, приостанавливает функцию и сохраняет состояние.

## Когда выполняется тело генераторной функции?

Не при простом вызове функции, а при первом запросе элемента через `next()` или `for`.

## Можно ли повторно пройти генератор?

После исчерпания — нет. Нужно создать новый объект.

## Всегда ли генератор быстрее списка?

Нет.

Он может:

- снизить память;
- быстрее отдать первый элемент;
- не вычислять ненужные элементы.

Но готовый список может быть быстрее при многократном использовании.

## `range`, `map`, `filter`, `zip` — генераторы?

Они поддерживают ленивое поведение или возвращают итераторы/итерируемые объекты, но технически не каждый является объектом типа generator. Например, `range` — объект последовательности.

# 3. Типовая задача на интервью

## Условие

Нужно читать файл с миллиардами строк и возвращать только валидные события.

Наивное решение:

```python
def load_valid_events(file_name):
    with open(file_name, "r") as file:
        lines = file.readlines()

    valid_events = []

    for line in lines:
        event = parse(line)

        if is_valid(event):
            valid_events.append(event)

    return valid_events
```

Проблемы:

- весь файл загружается в память;
- все валидные события хранятся в памяти;
- первый результат появляется поздно;
- нельзя естественно остановиться после первых `k` событий.

Генераторное решение:

```python
def iter_valid_events(file_name):
    with open(file_name, "r") as file:
        for line in file:
            event = parse(line)

            if is_valid(event):
                yield event
```

Потребление:

```python
for event in iter_valid_events("events.jsonl"):
    save(event)
```

# 4. Вопросы уровня middle

## Как взять первые десять элементов?

```python
from itertools import islice


first_ten = list(islice(generator, 10))
```

## Как объединить несколько потоков?

```python
from itertools import chain


combined = chain(first_generator, second_generator)
```

Или через `yield from`.

## Как разбивать поток на батчи?

```python
def batched(items, batch_size):
    batch = []

    for item in items:
        batch.append(item)

        if len(batch) == batch_size:
            yield batch
            batch = []

    if batch:
        yield batch
```

## Когда появляется ошибка внутри генератора?

В момент, когда потребитель запрашивает элемент и выполнение доходит до ошибочной строки. Ошибка может возникнуть значительно позже создания объекта.

## Как закрыть генератор?

```python
generator.close()
```

Внутри возникает `GeneratorExit`. Освобождение ресурсов обычно помещают в `finally`.

## Можно ли узнать `len(generator)`?

Обычно нет. Подсчёт через проход исчерпает генератор.

# 5. System Design: выгрузка 300 миллионов транзакций

## Наивная архитектура

```text
DWH -> SELECT ALL -> Python list/DataFrame -> transform all -> write all
```

Проблемы:

- OOM;
- большой time-to-first-write;
- повтор всего процесса при сбое;
- длительная транзакция;
- высокая нагрузка на сеть и БД.

## Более зрелая архитектура

```text
DWH cursor
    -> fetchmany(10 000)
    -> validation
    -> transformation
    -> enrichment
    -> write batch
    -> checkpoint
```

Пример интерфейса:

```python
def fetch_batches(cursor, batch_size):
    while True:
        rows = cursor.fetchmany(batch_size)

        if not rows:
            break

        yield rows
```

## Что обсуждать

- batch size;
- peak memory;
- throughput;
- latency;
- timeout;
- retry;
- idempotency;
- checkpoint;
- ordering;
- partial failures;
- schema evolution;
- data-quality checks;
- monitoring.

# 6. Backpressure

Представим:

- источник создаёт 100 000 событий в секунду;
- потребитель обрабатывает 20 000 событий в секунду.

Если источник работает без ограничений, очередь и память растут.

**Backpressure** — механизм согласования скорости источника со скоростью потребителя.

Синхронный генератор частично создаёт естественный backpressure:

- следующий элемент вычисляется только тогда, когда его просит потребитель.

В распределённой системе дополнительно нужны:

- bounded queue;
- consumer lag;
- rate limiting;
- pause/resume;
- acknowledgements;
- autoscaling;
- контроль batch size.

# 7. Продуктовый кейс: рекомендации маркетплейса

Есть источники кандидатов:

- похожие товары;
- популярные товары;
- повторные покупки;
- embedding retrieval;
- рекламные позиции.

Каждый источник может вернуть тысячи или миллионы кандидатов, но пользователю нужно 100.

Pipeline:

```text
candidate generators
    -> deduplication
    -> policy filters
    -> stock filter
    -> scoring
    -> top-k
```

Важное уточнение:

> Ленивость не решает автоматически глобальный top-k. Если вход не отсортирован по score, нужно пройти весь поток и хранить heap размера `k` либо использовать другой retrieval design.

Это сильная мысль для интервью: генератор уменьшает память на входной поток, но не отменяет алгоритмические требования.

Поток:

Метрики:

- throughput;

- queue lag;
- error rate;
- dropped events;
- fraud capture rate;
- false positive rate;

# 9. Продуктовый кейс: ML dataset generation

Нужно сформировать примеры:

```text
(query, positive_item, negative_item)
```

Генератор может:

1. прочитать positive example;
2. выбрать negative item;
3. построить признаки;
4. отдать обучающий пример;
5. перейти дальше.

Риски:

- sampling bias;
- leakage;
- невоспроизводимость;
- неправильный seed;
- разные результаты у нескольких workers;
- слишком медленный input pipeline;
- starvation GPU.

# 11. Generator vs list vs batches

## Список

Подходит, если:

- набор мал;
- нужен повторный проход;
- требуется индекс;
- нужна полная сортировка;
- элементы переиспользуются.

## Генератор отдельных элементов

Подходит, если:

- данные велики;
- нужен один проход;
- источник потоковый;
- важен ранний результат;
- можно остановиться.

## Генератор батчей

Часто лучший production-компромисс:

- контролируемая память;
- меньше сетевых вызовов;
- эффективные bulk insert;
- удобный retry;
- поддержка векторных операций.

# 12. Ресурс внутри генератора

```python
def read_file(file_name):
    with open(file_name, "r") as file:
        for line in file:
            yield line
```

Файл остаётся открыт, пока генератор:

- не исчерпан;
- не закрыт;
- не уничтожен.

Если потребитель прочитал одну строку и сохранил объект, дескриптор может оставаться открытым.

На интервью стоит говорить о:

- `try/finally`;
- явном закрытии;
- документировании владения ресурсом;
- ограничении времени жизни;
- отмене при разрыве соединения.

# 13. Генератор и транзакция базы данных

Если генератор лениво читает cursor в длинной транзакции, медленный consumer может:

- долго держать snapshot;
- удерживать lock;
- увеличивать MVCC bloat;
- получить timeout;
- потерять соединение;
- мешать vacuum или другим операциям.

Возможные решения:

- server-side cursor;
- keyset pagination;
- короткие транзакции;
- checkpoint по ключу;
- ограниченные батчи;
- идемпотентная запись;
- повторяемые диапазоны.

# 14. Generator vs async generator

Обычный генератор:

```python
def events():
    yield event
```

Асинхронный:

```python
async def events():
    yield event
```

Потребляется через:

```python
async for event in events():
    ...
```

Обычный генератор сам не делает код асинхронным. Async generator полезен, когда между элементами есть неблокирующее ожидание сети или другого I/O.

# 15. Генераторы и параллелизм

Генератор сам по себе:

- не создаёт threads;
- не запускает processes;
- не распределяет вычисления;
- не ускоряет CPU-bound задачу автоматически.

Для этого используют:

- threading;
- multiprocessing;
- asyncio;
- Spark;
- Ray;
- Dask;
- message brokers и consumer groups.

Генератор только задаёт ленивый контракт получения элементов.

# 16. Complexity

Для обработки `n` элементов:

- время обычно остаётся `O(n)`;
- дополнительная память генератора часто близка к `O(1)`;
- готовый список результатов требует `O(n)` памяти.

Но если генератор внутри накапливает большой словарь, cache или heap, память уже не будет `O(1)`.

# 17. System Design вопрос: экспорт истории операций

Хорошая структура ответа:

1. Уточнить максимальный объём данных.
2. Уточнить формат: CSV, JSONL, Parquet.
3. Уточнить SLA и time-to-first-byte.
4. Не загружать всё в память.
5. Читать БД через cursor или keyset pagination.
6. Генерировать строки или батчи.
7. Писать в object storage либо streaming response.
8. Добавить retry/checkpoint.
9. Проверять количество строк и checksum.
10. Защитить PII.
11. Добавить аудит и access control.
12. Обрабатывать отключение клиента.
13. Ограничить время и ресурсы.

Генератор здесь — локальный инструмент одного этапа, а не вся архитектура.
