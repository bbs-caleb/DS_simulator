# ML: URL PARSING — подробный бизнесовый и технический контекст

## 1. Почему эта задача вообще существует

SigmaBank наблюдает рост спам-сообщений, содержащих ссылки.
Злоумышленники используют URL, чтобы перенаправить человека на внешний ресурс, где можно:

- украсть логин и пароль;
- получить данные банковской карты;
- заставить пользователя установить вредоносное приложение;
- выманить одноразовый код;
- имитировать страницу банка;
- собирать персональные данные;
- переводить пользователя в мошеннический чат;
- распространять рекламу и массовый спам;
- обходить внутренние правила платформы.

Наивный фильтр может искать только очевидные строки вида:

```text
https://www.example.com
```

Но один и тот же адрес можно записать по-разному:

```text
example.com
www.example.com
http://example.com
https://example.com
https://www.example.com/
```

Для человека это один и тот же ресурс.
Для программы это разные последовательности символов.

Следовательно, системе недостаточно просто найти слово `http`.
Нужен отдельный компонент, который:

1. обнаруживает потенциальные URL;
2. приводит эквивалентные записи к единому виду;
3. сохраняет важную часть адреса — путь;
4. считает количество упоминаний;
5. проверяет доступность;
6. передаёт результат в следующий слой защиты.

---

## 2. Какую именно ответственность имеет `parse_urls`

Очень важно правильно определить границу задачи.

Функция `parse_urls` **не определяет**, является ли ссылка мошеннической.
Она решает более раннюю и фундаментальную задачу:

> Найти URL в тексте, нормализовать их, проверить доступность и вернуть частоты упоминаний.

Это этап подготовки данных.

В реальной системе конвейер мог бы выглядеть так:

```text
сообщение
   ↓
URL extraction
   ↓
URL normalization
   ↓
availability check
   ↓
reputation enrichment
   ↓
rules / ML scoring
   ↓
allow / warn / block / review
```

`parse_urls` покрывает первые несколько шагов, но не принимает окончательное решение о мошенничестве.

Почему это разделение полезно:

- компонент проще тестировать;
- каждая функция имеет одну ответственность;
- можно отдельно менять правила извлечения;
- можно отдельно менять модель риска;
- одна и та же нормализация подходит разным downstream-сервисам;
- ошибки легче локализовать.

## 3. Бизнесовая ценность в финтехе

### 3.1. Защита клиентов от фишинга

Сообщение:

```text
Ваш аккаунт заблокирован. Срочно подтвердите данные на sigmabank-secure-login.com
```

После извлечения домена система может проверить:

- похож ли домен на официальный бренд;
- присутствует ли он в denylist;
- зарегистрирован ли недавно;
- связан ли с известной вредоносной инфраструктурой;
- встречался ли в жалобах;
- куда ведут редиректы;
- какой сертификат использует сайт;
- какие аккаунты распространяют ссылку.

### 3.2. Защита операторов поддержки

Мошенническая ссылка может попасть не только клиенту, но и сотруднику:

- в тикет поддержки;
- во вложенный комментарий;
- в чат с партнёром;
- в заявку поставщика;
- в письмо якобы от руководителя.

Автоматическое извлечение URL позволяет маркировать опасные обращения до того, как оператор откроет ссылку.

### 3.3. Снижение прямого финансового ущерба

Успешный фишинг приводит к:

- account takeover;
- несанкционированным платежам;
- chargeback;
- компенсациям;
- расследованиям;
- затратам службы поддержки;
- регуляторным последствиям;
- репутационному ущербу.

Поэтому даже небольшой parsing-компонент может быть частью системы, предотвращающей значительные потери.

## 5. Что такое URL на интуитивном уровне

Рассмотрим адрес:

```text
https://www.example.com/catalog/product?id=10
```

Упрощённо он состоит из частей:

```text
https://     www.     example.com     /catalog/product     ?id=10
протокол     префикс  хост/домен       путь                 query
```

В рамках задания нужно удалить:

- `http://`;
- `https://`;
- `www.`.

Но нужно сохранить:

- домен;
- путь после домена.

Например:

```text
https://www.example.com/catalog/product
```

нормализуется в:

```text
example.com/catalog/product
```

---

## 6. Почему путь нельзя отбрасывать

Разные пути на одном домене могут указывать на разные ресурсы:

```text
example.com/login
example.com/help
example.com/payment
example.com/malicious-page
```

Если оставить только домен, вся эта информация потеряется.

Скрытый тест платформы явно проверил это поведение:

```text
https://stackoverflow.com/questions/tagged/python
```

Ожидаемый ключ:

```text
stackoverflow.com/questions/tagged/python
```

а не:

```text
stackoverflow.com
```

Это важный общий урок:

> Нормализация должна удалять только то, что объявлено несущественным, и сохранять всё, что может влиять на идентичность объекта.

## 7. Почему домен и путь по-разному относятся к регистру

### Домен

Имена хостов регистронезависимы:

```text
EXAMPLE.COM
Example.Com
example.com
```

их разумно привести к:

```text
example.com
```

### Путь

Путь на сервере потенциально регистрозависим:

```text
/wiki/Special:Random
/wiki/special:random
```

могут означать разные ресурсы.

Поэтому нельзя делать:

```python
url.lower()
```

для всей строки.

Нужно разделить URL на:

- `host`;
- `path`.

И применить `.lower()` только к `host`.

Пример:

```text
EN.WIKIPEDIA.ORG/wiki/Special:Random
```

должен превратиться в:

```text
en.wikipedia.org/wiki/Special:Random
```

## 8. Что такое регулярное выражение

Регулярное выражение — это формальное описание текстового шаблона.

Нам неизвестны конкретные домены заранее, поэтому нельзя написать:

```python
if "example.com" in message:
```

Нужно описать общую форму URL:

- необязательный протокол;
- необязательный `www.`;
- одна или несколько частей домена;
- точка;
- доменная зона;
- необязательный путь.

Финальный шаблон:

```python
r"(?:https?://)?(?:www\.)?"
r"((?:[a-zA-Z0-9-]+\.)+[a-zA-Z]{2,}(?:/[^\s]*)?)"
```

Подробная расшифровка находится в файле с пошаговым объяснением.

## 9. Зачем нужна нормализация

Без нормализации список может выглядеть так:

```python
[
    "example.com",
    "www.example.com",
    "https://example.com",
    "https://www.example.com/",
]
```

Хотя логически это четыре упоминания одного URL.

После нормализации:

```python
[
    "example.com",
    "example.com",
    "example.com",
    "example.com",
]
```

Теперь `Counter` корректно вернёт:

```python
{"example.com": 4}
```

Нормализация нужна почти в любой аналитической или ML-системе, потому что данные из реального мира записываются неодинаково.

Аналогии:

- телефоны: `+998901234567`, `90 123 45 67`;
- email: пробелы, регистр домена;
- названия компаний: `ООО Ромашка`, `Ромашка ООО`;
- SKU: ведущие нули, разные регистры;
- адреса: сокращения улиц;
- даты: разные форматы.

## 10. Почему используется `Counter`

`Counter` — специализированный словарь для подсчёта частот.

Пример:

```python
Counter(["a", "b", "a"])
```

даёт:

```python
Counter({"a": 2, "b": 1})
```

В задаче это позволяет сразу получить:

```python
{
    "example.com": 2,
    "google.com": 1,
}
```

Важное преимущество для скорости:

- сначала считаются все упоминания;
- затем сеть проверяется только для уникальных ключей.

Если URL встречается сто раз, выполняется один запрос, а не сто.

## 11. Зачем проверять доступность после дедупликации

Пусть в сообщении 100 раз встречается `example.com`.

Плохой подход:

```text
100 совпадений → 100 HTTP-запросов
```

Хороший подход:

```text
100 совпадений
      ↓
Counter({"example.com": 100})
      ↓
1 HTTP-запрос
```

Сетевой запрос значительно дороже локальной операции над строкой.
Он требует:

- DNS resolution;
- открытия соединения;
- ожидания ответа;
- обработки timeout;
- возможного TLS-handshake;
- внешнего ресурса.

Поэтому правильный порядок операций существенно влияет на производительность.

---

## 12. Зачем нужен `timeout=5`

Без timeout запрос может зависнуть надолго.

В production это приводит к:

- росту latency;
- накоплению очереди;
- исчерпанию worker threads;
- каскадному отказу;
- плохому пользовательскому опыту.

`timeout=5` означает, что функция не должна бесконечно ждать сетевой ресурс.
В задании это явное требование API.

---

## 13. Зачем перехватывать `requests.RequestException`

Сеть ненадёжна.

Возможны:

- connection error;
- timeout;
- DNS error;
- redirect problem;
- HTTP error после `raise_for_status()`.

Если исключение не обработать, одна плохая ссылка остановит всю функцию.

Правильное поведение:

```text
один URL недоступен
        ↓
печатаем ошибку
        ↓
продолжаем проверять остальные URL
```

## 14. Что означает требование скорости

Платформа выделяет 10% баллов за speed.

Основная оптимизация здесь алгоритмическая:

1. regex компилируется один раз;
2. нормализация выполняется линейно по найденным URL;
3. `Counter` считает частоты за линейное время;
4. HTTP-запрос выполняется только на уникальный нормализованный URL.

Если:

- `n` — длина сообщения;
- `m` — количество найденных упоминаний;
- `u` — количество уникальных URL;

то локальная часть примерно линейна:

```text
O(n + m)
```

а число сетевых запросов:

```text
O(u)
```

В большинстве сообщений `u <= m`, иногда значительно меньше.

## 15. Ограничения учебного решения

Решение специально минимальное и соответствует задаче, но не является полным промышленным URL parser.

Оно не покрывает полностью:

- IP-адреса;
- localhost;
- IPv6;
- нестандартные порты;
- все Unicode-домены;
- email-адреса с доменной частью;
- сложные скобки в URL;
- query/fragment во всех возможных форматах;
- схемы кроме HTTP/HTTPS;
- скрытые Unicode-символы;
- редиректы и сокращатели;
- SSRF-защиту;
- внутренние адреса.

Это нормально: учебная задача имеет ограниченный контракт.

Ключевой инженерный принцип:

> Не нужно строить весь интернет-браузер, если тестируется небольшой и чёткий API. Нужно точно выполнить контракт и осознавать границы решения.

## 18. Продуктовые ошибки первого и второго рода

### False negative

Вредоносная ссылка пропущена.

Последствия:

- потеря денег;
- кража аккаунта;
- ущерб бренду;
- регуляторные риски.

### False positive

Легитимная ссылка заблокирована.

Последствия:

- ломается коммуникация;
- падает конверсия;
- растут обращения;
- страдают честные пользователи.

Поэтому в реальной системе часто используют несколько действий:

```text
низкий риск → пропустить
средний риск → предупредить
высокий риск → задержать
очень высокий риск → заблокировать
неуверенность → ручная проверка
```

## 19. Главное мышление специалиста

Сильный специалист видит в задаче не просто regex.
Он формулирует цепочку:

1. Какой бизнесовый ущерб предотвращаем?
2. Какова ответственность компонента?
3. Что считается одним и тем же URL?
4. Какие данные нельзя потерять при нормализации?
5. Где дорогие операции?
6. Какие сбои внешней системы возможны?
7. Как компонент тестировать без интернета?
8. Какие ограничения у учебного решения?
9. Как расширить его до production-сервиса?

Именно это превращает маленькую функцию в осмысленный инженерный модуль.
