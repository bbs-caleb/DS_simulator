# Урок 30 — ML: URL PARSING

## Назначение комплекта

Это полный комплект материалов для **первой страницы** урока `ML: URL PARSING`.
Он построен вокруг подтверждённого решения, которое учитывает реальные скрытые проверки платформы.

Здесь есть не только файл для отправки, но и объяснение того:

- какую бизнесовую задачу решает URL-парсер;
- почему обычного поиска строки недостаточно;
- как работает регулярное выражение;
- зачем нужна нормализация URL;
- почему домен можно привести к нижнему регистру, а путь нельзя;
- почему сетевые запросы нужно выполнять только для уникальных URL;
- как рассуждать об этой задаче на продуктовом и system design интервью;
- какие ошибки привели к нулю баллов в первых попытках и как их не повторить.

## Какой файл загружать в форму задания

В проверяющую систему нужно отправлять только:

```text
gp_page_1_url_parsing_solution.py
```

Остальные файлы являются учебными материалами и локальными тестами.

Перед загрузкой убедитесь, что выбран именно новый файл без старых копий в названии:

```text
gp_page_1_url_parsing_solution.py
```

Не выбирайте файлы вида:

```text
gp_page_1_url_parsing_solution(1).py
gp_page_1_url_parsing_solution(1) (2).py
gp_page_1_url_parsing_solution_old.py
```

Такие имена обычно означают, что браузер сохранил несколько разных версий, и легко случайно отправить устаревший код.

## Что делает финальное решение

Функция `parse_urls` выполняет следующий конвейер:

```text
входящее сообщение
        ↓
поиск URL регулярным выражением
        ↓
удаление http://, https:// и www.
        ↓
удаление завершающей пунктуации
        ↓
приведение только домена к нижнему регистру
        ↓
сохранение пути с исходным регистром
        ↓
подсчёт повторений Counter
        ↓
одна HTTP-проверка на каждый уникальный URL
        ↓
исключение недоступных URL
        ↓
словарь {нормализованный URL: количество упоминаний}
```

## Два главных скрытых требования

### 1. Путь является частью URL

Вход:

```text
https://stackoverflow.com/questions/tagged/python
```

Правильный результат:

```text
stackoverflow.com/questions/tagged/python
```

Неправильный результат:

```text
stackoverflow.com
```

Нельзя обрезать всё после домена.

### 2. Нельзя переводить весь URL в нижний регистр

Вход:

```text
https://en.wikipedia.org/wiki/Special:Random
```

Правильный результат:

```text
en.wikipedia.org/wiki/Special:Random
```

Неправильный результат:

```text
en.wikipedia.org/wiki/special:random
```

Домен регистронезависим, поэтому `EN.WIKIPEDIA.ORG` можно нормализовать до `en.wikipedia.org`.
Путь потенциально регистрозависим, поэтому `/wiki/Special:Random` нужно сохранить как есть.

## Как запустить решение локально

Установить `requests`:

```bash
pip install requests
```

Запустить:

```bash
python gp_page_1_url_parsing_solution.py
```

Следует помнить, что результат примера зависит от реальной доступности сайтов в момент запуска.

## Как запустить локальные тесты

Тесты используют стандартный модуль `unittest` и подменяют реальные HTTP-запросы.
Интернет для них не нужен.

Запуск:

```bash
python gp_page_1_url_parsing_local_tests.py
```

Или:

```bash
python -m unittest gp_page_1_url_parsing_local_tests.py -v
```

## Главная инженерная идея урока

Задача выглядит как упражнение по regex, но на самом деле демонстрирует основу реального production-конвейера:

```text
extract → normalize → aggregate → enrich → filter → return
```

Эта схема постоянно встречается в:

- antifraud;
- antispam;
- модерации;
- обработке логов;
- построении признаков для ML;
- data quality;
- threat intelligence;
- системах безопасного просмотра;
- защите LLM-агентов, которые открывают внешние ссылки.
