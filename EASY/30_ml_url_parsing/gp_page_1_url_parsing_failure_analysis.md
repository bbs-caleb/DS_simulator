# URL PARSING — разбор реальных ошибок и скрытых требований

## 1. Зачем нужен отдельный разбор ошибок

Первые варианты выглядели логично и проходили простой пример, но получили 0 баллов.
Это типичная ситуация в реальной разработке:

- happy path работает;
- контракт понят не полностью;
- hidden edge case ломает решение;
- из-за падения correctness не начисляются speed и pylint.

Ниже разобраны две реальные ошибки.

---

## 2. Ошибка №1: сохранён только домен

Платформа проверила:

```text
https://stackoverflow.com/questions/tagged/python
```

Неправильный результат:

```python
{"stackoverflow.com": 1}
```

Ожидаемый результат:

```python
{"stackoverflow.com/questions/tagged/python": 1}
```

### Почему решение казалось разумным

В тексте задания сказано о нормализации URL без `www`, `http://` и `https://`.
Можно ошибочно решить, что нужен только домен.

Но в условии не сказано удалять путь.
Следовательно, путь является частью нормализованного URL.

### Общий урок

При нормализации нельзя удалять данные без явного требования.

Нужно спросить:

> Какие части считаются вариацией записи, а какие определяют идентичность объекта?

Здесь вариации:

- протокол;
- `www.`;
- завершающий одиночный `/`.

Идентичность сохраняют:

- домен;
- путь.

## 3. Ошибка №2: весь URL переведён в нижний регистр

Платформа проверила:

```text
https://en.wikipedia.org/wiki/Special:Random
```

Неправильный результат:

```python
{
    "en.wikipedia.org/wiki/special:random": 1,
}
```

Ожидаемый результат:

```python
{
    "en.wikipedia.org/wiki/Special:Random": 1,
}
```

### Причина

В коде была логика вида:

```python
url.lower()
```

Она нормализовала и домен, и путь.

### Почему это опасно

Домен регистронезависим:

```text
Example.COM == example.com
```

Но путь может быть регистрозависимым:

```text
/Path
/path
```

могут быть разными ресурсами.

### Правильный подход

Разделить строку по первому `/`:

```python
host, separator, path = url.partition("/")
```

Нормализовать только хост:

```python
host.lower()
```

Путь добавить без изменения:

```python
f"{host.lower()}/{path}"
```

---

## 4. Почему после correctness было 0 за speed и pylint

В отчёте:

```text
correctness: 0 / 70
speed:       0 / 10
pylint:      0 / 20
```

Это не обязательно означает, что скорость и стиль были плохими.
Проверяющая система могла остановить последующие проверки после провала основного теста.

В условии прямо сказано:

> Если решение не соответствует требованиям API, за все остальные проверки зачисляются 0 баллов.

Поэтому сначала нужно добиться точной функциональной корректности.

---

## 5. Ошибка управления версиями файла

После исправления существовал риск повторно загрузить старый файл:

```text
gp_page_1_url_parsing_solution(1) (2).py
```

Браузер добавляет такие суффиксы, когда файл с тем же именем уже существует.
Из-за этого легко перепутать версии.

Практика:

1. удалить старые копии из Downloads;
2. скачать финальный файл заново;
3. открыть его и найти ключевую строку;
4. проверить, что отсутствует `.lower()` для всего URL;
5. загрузить именно этот файл.

В рабочей разработке эту проблему решает Git:

- каждая версия имеет commit;
- видно diff;
- нет неясных `(1) (2)`;
- можно воспроизвести отправленный код.

## 6. Набор обязательных regression-тестов

После каждой найденной ошибки нужно добавить тест, чтобы она не вернулась.

### Путь сохраняется

```python
message = "https://stackoverflow.com/questions/tagged/python"
expected = {
    "stackoverflow.com/questions/tagged/python": 1,
}
```

### Регистр пути сохраняется

```python
message = "https://en.wikipedia.org/wiki/Special:Random"
expected = {
    "en.wikipedia.org/wiki/Special:Random": 1,
}
```

### Домен нормализуется

```python
message = "https://WWW.Example.COM/Path"
expected = {
    "example.com/Path": 1,
}
```

### Дубликаты считаются, но запрос один

```python
message = "example.com example.com www.example.com"
expected = {
    "example.com": 3,
}
```

При этом `requests.get` должен быть вызван один раз.

## 7. Как читать hidden-test feedback

Текст вида:

```text
returned X instead of Y
```

нужно анализировать как diff спецификации.

В первом падении:

```text
X: stackoverflow.com
Y: stackoverflow.com/questions/tagged/python
```

Разница говорит:

> Потерян путь.

Во втором:

```text
X: .../wiki/special:random
Y: .../wiki/Special:Random
```

Разница говорит:

> Неверно изменён регистр пути.

Важно не переписывать всё решение, а локализовать минимальную причину.

---

## 8. Инженерный цикл исправления

Правильный цикл:

```text
получить failing example
        ↓
сравнить expected и actual
        ↓
сформулировать нарушенное правило
        ↓
написать regression-тест
        ↓
внести минимальное исправление
        ↓
прогнать старые и новые тесты
        ↓
проверить отправляемый файл
```

Это основа debugging и test-driven development.

---

## 9. Финальные правила контракта

Подтверждённое поведение:

1. Находить URL с протоколом и без него.
2. Удалять `http://` и `https://`.
3. Удалять `www.`.
4. Сохранять путь.
5. Приводить домен к нижнему регистру.
6. Сохранять регистр пути.
7. Удалять завершающую пунктуацию.
8. Удалять одиночный завершающий `/`.
9. Считать количество нормализованных совпадений.
10. Проверять каждый уникальный нормализованный URL один раз.
11. Использовать `timeout=5`.
12. Исключать URL при `requests.RequestException`.
13. Возвращать обычный словарь `Dict[str, int]`.

Эти правила и реализует финальный файл.
