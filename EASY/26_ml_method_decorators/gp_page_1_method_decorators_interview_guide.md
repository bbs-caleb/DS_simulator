# ML: METHOD DECORATORS — продуктовое и System Design интервью

## 1. Как начать обсуждение задачи

На интервью не стоит сразу писать код.

Сначала нужно определить:

1. Что представляет `BankMetrics`?
2. Ставка общая или зависит от продукта?
3. Баланс текущий или среднедневной?
4. За какой период считаются проценты?
5. Нужно ли хранить историю ставок?
6. Что делать при отсутствии счетов?
7. Кто имеет право менять ставку?
8. Нужна ли точность до минимальных денежных единиц?
9. Данные находятся в одном процессе или распределены?
10. Какие требования к аудиту?

В учебной задаче API уже задан, поэтому решение должно соответствовать шаблону. Но профессионал должен видеть вопросы, которые возникнут в production.

## 4. Частый вопрос: `staticmethod` против `classmethod`

Хороший ответ:

> Статический метод не получает автоматически ни экземпляр, ни класс. Метод класса получает `cls`, поэтому может работать с общим состоянием и учитывать наследование. Выбор зависит от требуемого контекста и контракта API.

Для текущей задачи:

```text
adjust_global_bank_rate → staticmethod
calculate_avg_balance   → classmethod
calculate_interest      → classmethod
```

---

## 5. Почему изменение ставки — статический метод

По контракту метод получает только новую ставку:

```python
@staticmethod
def adjust_global_bank_rate(new_rate: float) -> None:
```

Он меняет атрибут класса явным обращением:

```python
BankMetrics.global_bank_rate = new_rate
```

На интервью можно отметить:

> В другой архитектуре эту операцию можно было бы сделать методом класса, но здесь нужно соблюдать заданный API.

---

## 6. Почему средний баланс — метод класса

Метод работает с:

```python
cls.accounts
```

Это общее состояние.

Он не зависит от одного конкретного счёта.

---

## 7. Почему расчёт процентов — метод класса

Метод использует:

```python
cls.global_bank_rate
```

и переданный:

```python
account
```

То есть ему нужен контекст класса и объект счёта.

## 13. Почему простой список не масштабируется

```python

```

существует только внутри одного Python-процесса.

Если есть три сервера:

```text
Server A → свой список
Server B → свой список
Server C → свой список
```

данные будут различаться.

Проблемы:

- потеря после перезапуска;
- отсутствие транзакций;
- отсутствие истории;
- несогласованность;
- отсутствие резервного копирования;
- отсутствие аудита.

## 21. История ставок

Нельзя просто заменить:

```text

```

Нужно сохранять:

- старое значение;
- новое значение;
- момент начала действия;
- пользователя;

- версию;

Каждое начисление должно знать, какую версию ставки оно использовало.

## 23. Идемпотентность

Задача начисления может прийти повторно.

Без защиты проценты начислятся дважды.

Идемпотентный ключ:

```text
account_id + period + calculation_version
```

Повторная обработка должна:

- найти существующий результат;
- не создать второе начисление;
- вернуть тот же бизнес-результат.

## 37. Вопросы интервьюера и сильные ответы

### Почему ставка общая?

> По условию она относится ко всему банковскому продукту, а не к одному счёту.

### Почему баланс индивидуальный?

> У каждого счёта своё значение, поэтому баланс хранится в экземпляре.

### Чем `self` отличается от `cls`?

> `self` указывает на конкретный объект, `cls` — на класс.

### Почему `staticmethod` не получает `cls`?

> Это часть его контракта: Python не передаёт ему контекст автоматически.

### Почему API важен?

> Другие компоненты, пользователи библиотеки и тесты вызывают методы по ожидаемым именам и сигнатурам.

### Что будет при пустом списке?

> Возникнет деление на ноль. В production нужно определить контракт явно.

### Почему не `float` для денег?

> Из-за точности двоичного представления и требований финансового округления.

### Как избежать двойного начисления?

> Использовать идемпотентный ключ и транзакционную запись результата.

### Как поддержать разные продукты?

> Привязать ставки к `product_id`, валюте, сегменту и периоду действия.

### Как пересчитать историю?

> Хранить immutable историю ставок и фиксировать версию в каждом начислении.

## 38. Что важно сказать про автотесты

Сильный ответ:

> Автотест может проверять не только возвращаемое значение, но и структуру API: вид декоратора, имя метода, количество аргументов, типы и наличие атрибутов. Поэтому поведенчески похожий код может не соответствовать контракту.

Это важная production-аналогия:

- интерфейсы должны быть стабильны;
- клиенты зависят от контрактов;
- несовместимое изменение ломает интеграции.

## 40. Сильный итоговый ответ на интервью

> В учебной модели имя и баланс принадлежат конкретному счёту. Ставка и список счетов находятся на уровне класса. Статический метод меняет ставку по заданному API, метод класса считает среднее по общему реестру, а второй метод класса применяет общую ставку к переданному счёту. В production я бы разделил Account Service, ledger, конфигурацию ставок, сервис начислений и аналитику, использовал точные денежные типы, версии ставок, идемпотентность, аудит и централизованное хранилище.
