# AI: Calorie Tracker — контекст FastAPI-этапа

## 1. Что мы строим

Мы разрабатываем прототип сервиса, который получает фотографию блюда и возвращает
примерную пищевую ценность:

- калории;
- белки;
- жиры;
- углеводы.

Полная цепочка работы выглядит так:

```text
Пользователь
    ↓
Streamlit-интерфейс
    ↓ POST /generate_response
FastAPI-сервис
    ↓
LLM-ассистент
    ↓
FastAPI-сервис
    ↓ JSON
Streamlit-интерфейс
```

На текущем этапе настоящий LLM ещё не подключается. Вместо него используется
класс-заглушка `LLMAssistant`, который всегда возвращает заранее заданный результат.

## 2. Бизнесовый смысл этапа

Для бизнеса важно разрабатывать систему по частям. Если ждать полной готовности
LLM, фронтенда и бэкенда одновременно, разработка будет медленной, а ошибки будет
сложно локализовать.

Заглушка позволяет заранее проверить:

1. принимает ли сервер запрос;
2. правильно ли сервер читает поле `image_base64`;
3. корректно ли обрабатывается испорченная строка Base64;
4. возвращаются ли правильные HTTP-коды;
5. сможет ли Streamlit работать с форматом ответа сервера.

Такой подход снижает риск интеграционных ошибок. Когда реальный LLM будет готов,
его можно будет поставить вместо заглушки, не переписывая API.

## 3. Технический смысл этапа

Нужно реализовать POST-эндпоинт:

```text
POST /generate_response
```

Он должен:

1. получить JSON-тело запроса;
2. взять значение поля `image_base64`;
3. проверить, что значение является непустой строкой;
4. проверить, что строка действительно закодирована в Base64;
5. вызвать `assistant.generate_response(...)`;
6. вернуть результат ассистента;
7. преобразовать разные ошибки в правильные HTTP-коды.

## 4. Контракт API

### Корректный запрос

```json
{
  "image_base64": "aGVsbG8="
}
```

### Успешный ответ

HTTP-код:

```text
200
```

Тело ответа:

```json
{
  "status": "success",
  "result": {
    "calories": 100,
    "proteins": 10,
    "fats": 20,
    "carbohydrates": 30
  },
  "error": ""
}
```

### Ошибки

| Ситуация | HTTP-код |
|---|---:|
| Некорректная Base64-строка | 400 |
| Таймаут при обращении к ассистенту | 504 |
| Любая другая непредвиденная ошибка | 500 |

## 5. Почему Base64 нужно валидировать

Обычный JSON предназначен для передачи текста. Изображение состоит из бинарных
байтов, поэтому фронтенд кодирует его в строку Base64.

Однако наличие строки ещё не означает, что она является правильным Base64.
Поэтому используется:

```python
base64.b64decode(image_base64, validate=True)
```

Параметр `validate=True` важен: он заставляет Python отклонить посторонние
символы, а не пытаться молча их проигнорировать.
